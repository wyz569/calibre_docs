DFM DV Analyze

Layer operation

License: Calibre nmDRC-H

Writes layer data to an ASCII format results database (RDB). The results database is organized to assist in analyzing results from a DFM DV operation.

Usage

DFM DV Analyze layer [layer …]  RDB rdb_file
[OVERRIDE] [LOCALIZE] [NOPSEUDO] [MAXIMUM {ALL | count}]

Arguments

layer [layer …]

A required argument specifying one or more layers that are written to the RDB. The layers can be of the following types:

  • An error layer produced by a DFM DV operation with ANNOTATE NETS, called a “DV layer.”

  • A polygon or edge layer with connectivity information, including a net-indexed layer produced by DFM Property Net.

  • An input dvprop_layer to a DFM DV Override operation or the SELECTED output layer from a DFM DV Override operation (see “SELECTED Output” with DFM DV Override). Such layers must include the DFM properties NET1, NET2, and DELTA_V. When such are layers are specified, they must be the only type of layer that is specified and the OVERRIDE keyword must be included.

Within a set of DFM DV Analyze operations writing to the same rdb_file, at least one layer must be a DV layer.

RDB rdb_file

A required keyword set that specifies the name of the RDB file that is created.

OVERRIDE

An optional keyword that specifies that all input layers are either dvprop_layer input layers for the DFM DV Override operation or SELECTED output layers from DFM DV Override. The layers must have attached DFM properties NET1, NET2, and DELTA_V. A warning is issued if this is not the case and the layer is ignored.

For an example using OVERRIDE, see the example included with the DFM DV Override operation. For a description of the output from layers specified with OVERRIDE, see “RDB Content”.

LOCALIZE

An optional keyword that causes the operation to attempt to promote results from the input layer(s) so that the layer objects are reported in the vicinity of the corresponding DFM DV result. Promotion to the same level as the DFM DV result is not guaranteed. See “Promotion With the LOCALIZE Keyword” for details.

NOPSEUDO

An optional keyword that causes layer data in pseudocells to be expanded up to the next level of hierarchy.

Calibre creates pseudocells on an as-needed basis as part of its internal optimization of hierarchy management. By default, layer data in a pseudocell is output in the pseudocell.

MAXIMUM {ALL | count}

An optional keyword set that specifies the maximum number of results per (cell, node number) pair per layer. This is not the total maximum result count in the RDB, instead it limits the number of each type of result, so that in general at least one of each result type is output per layer, but not more than count for each layer/type combination. See the section “RDB Content” for details.

By default there is no limit on the number of results.

Description

Creates a results database (RDB) that can assist in tracing the high and low voltage information on nets that participate in a DFM DV operation.

The output layer produced by DFM DV Analyze is empty. The data generated by the operation is written to the RDB file rdb_file. The operation must be specified inside a rule check, or the layer created by the operation must be output in a rule check.

In typical usage the input layers to DFM DV Analyze are the following:

  • One or more output layers from a DFM DV operation with the ANNOTATE NETS keyword (DV layers).

  • The net-indexed layer ni_layer used in the DV layer derivation.

  • One or more of the layers used to derive the ni_layer.

    When many layers (and therefore geometries) contribute to the derivation of ni_layer, it is useful to filter each layer so that it contains only the geometries that contribute to the voltage definitions, and use the filtered layer in DFM DV Analyze.

Input Layer Accumulation and Concurrent Execution

DFM DV Analyze accumulates input layers across multiple operations when the operations write to the same rdb_file. For example, the following operations:

DFM DV ANALYZE ni_prop Layer_2 Layer_3 RDB result.rdb
DFM DV ANALYZE dv1_out RDB result.rdb
DFM DV ANALYZE dv2_out RDB result.rdb

are executed concurrently as 

DFM DV ANALYZE ni_prop Layer_2 Layer_3 dv1_out dv2_out RDB result.rdb

If there are several rule checks containing DFM DV operations, you can write one DFM DV Analyze operation for the net-indexed layer and the layers used to generate it, and place a DFM DV Analyze operation for the DFM DV output layer in each rule check.

If any concurrent operation specifies NOPSEUDO, all results in the RDB are promoted out of pseudocells. If any concurrent operation specifies “MAXIMUM count”, the minimum count value from all the concurrent operations is used for all operations.

For example, these operations are considered concurrent:

DFM DV ANALYZE ni_prop Layer_2 Layer_3 RDB result.rdb MAXIMUM 3000 
DFM DV ANALYZE dv1_out RDB result.rdb NOPSEUDO MAXIMUM 5000 
DFM DV ANALYZE dv2_out RDB result.rdb

They are executed concurrently as 

DFM DV ANALYZE ni_prop Layer_2 Layer_3 dv1_out dv2_out RDB result.rdb
                  NOPSEUDO MAXIMUM 3000

If a layer appears as an input in multiple concurrent DFM DV ANALYZE operations, some with LOCALIZE and some without, LOCALIZE is always applied to the layer. The same is true for the OVERRIDE keyword.

RDB Content

Output is written to the RDB in the form of rule check output. All DFM properties on the input geometries are written to the check output.

Each FIRST_NET and SECOND_NET property on a DV layer identifies a node number in a particular hierarchical cell. The operation computes the set of all (cell, node number) pairs over all DV layer inputs. Each unique (cell, node number) pair is assigned a unique integer ID (UID).

  • DV layers

    For each DV layer, write out one rule check.

    Check name: <layer>

    where <layer> is the name of the DV layer.

    The properties DELTA_V, FIRST_UID, FIRST_VMAX, FIRST_VMIN, SECOND_UID, SECOND_VMAX, and SECOND_VMIN are included. The SYNC property is attached if it is present on the input DV layer.

    However, if the DV layer was generated using WITH DEFAULT VOLTAGES or WITH VOLTAGE PROPERTIES for one or more input layers, the corresponding FIRST_UID or SECOND_UID property is not included. This is because the DV layer does not have the corresponding FIRST_NET or SECOND_NET property. In addition, geometries from non-DV layers associated with the result are not written out because there is no UID associated with the result.

    If “MAXIMUM count” is specified, DV layer results are limited based on how many times their UIDs have been used in a DV result, distinguishing between usage for a UID's VMIN value and usage for its VMAX value. A result on the DV layer is reported at least once for each UID and each use of that UID for VMIN or VMAX. After count results are reported for any UID in its VMIN or VMAX usage, no more results are reported for that DV layer. Geometries without a FIRST_UID or SECOND_UID property are considered to all have the same UID for counting purposes. The upper bound on the number of results for each DV layer is (2*<number_of_UIDs>*(count+1)).

  • Non-DV layers

    For each non-DV layer, for each UID, write all geometries connected to the net corresponding to that UID to a rule check.

    Check name: <output_layer> uid=<UID> <layer>

    where output_layer is the output layer generated by the DFM DV Analyze operation and layer is the input layer being processed.

    Two nets are connected if they have a common ancestor.

    The net-indexed layer ni_layer contains only attached DFM properties, but no real geometries. The DFM properties associated with UID are written to the check result. The output for non-DV layers also includes the property DN, which is the node number for the geometry.

    If “MAXIMUM count” is specified, layer data for each UID is reported at most count times for each layer that is written to the RDB. The number of geometries reported for each non-DV layer is at most (count + <number_of_UIDs>).

    If a layer has no geometries associated with a particular UID, no check is written for that layer-UID pair.

  • dvprop_layer input to a DFM DV Override operation or output from DFM DV Override with the SELECTED keyword

    For any DFM DV result for which the override DELTA_V value was used, write out the geometries with a NET1/NET2 pair that correspond to the FIRST_NET and SECOND_NET properties of the result. The check name has this form:

    Check name: <output_layer> OVERRIDE uid1=<UID 1> uid2=<UID 2> <layer>

    where output_layer is the output layer generated by the DFM DV Analyze operation and layer is the input layer being processed (the input layer to DFM DV Override).

    The check result includes all DFM properties associated with the geometry.

    If “MAXIMUM count” is specified, layer data for each uid1/uid2 pair is reported at most count times for each layer that is written to the RDB. The upper bound on the number of results for each layer is (<number_of_UID_pairs>*count).

    Note:

    DFM DV results that use DELTA_V values from an override layer are indicated with a SYNC property of 2.0.

Check output is written with the ALL CELLS option from the DFM RDB statement, causing each check to include the geometries for all cells. The DRC Magnify Results statement affects the results.

For local layers, those derived within a rule check, the layer name is preceded by the rule check name: <check>::<layer>.

Promotion With the LOCALIZE Keyword

When LOCALIZE is specified for one or more layers, those layer geometries are reported in the vicinity of the corresponding DFM DV result, if possible. Promotion to the same level as the DFM DV result is not guaranteed. In particular, for a net in a cell, if the geometries for a particular UID are selected in all placements of the cell, the geometries are reported in the cell, rather than being promoted. (The promotion method is similar to that used for Net Area Ratio.)

Examples

This example has two DFM DV checks, between layer1 and layer2, and between layer1 and layer4. For illustration purposes, a simple voltage assignment is used to create the net-indexed layer for the DFM DV operation. DFM DV Analyze is used to output the layers involved in the DFM DV checks, plus layerX, which has connectivity to layer1 and layer4. 

LAYOUT PATH "layout.oas"            
LAYOUT SYSTEM OASIS            
LAYOUT PRIMARY TOPCELL      

DRC RESULTS DATABASE drc.db
DFM DATABASE dfmdb OVERWRITE
DRC CELL NAME YES CELL SPACE XFORM
LAYER layer1 1
LAYER layer2 2
LAYER layerX 3
LAYER layer4 4
CONNECT layer1 layerX
CONNECT layer2
CONNECT layer4 layerX

VARIABLE VH_net1 5
VARIABLE VL_net1 1
VARIABLE VH_other 3
VARIABLE VL_other -3
// net-indexed layer for the DFM DV checks
net_prop_layer = DFM PROPERTY NET layer1 layer2 layer4
   [ vhigh = ( (COUNT(layer1) > 0) ? VH_net1 : VH_other )]
   [ vlow = ( (COUNT(layer1) > 0) ? VL_net1 : VL_other )]
   [ nosync = NONETID()]
// Create debug layers with high and low voltage values
// these are output by DFM DV ANALYZE
layer1_dbg = DFM PROPERTY layer1 net_prop_layer NODAL MULTI
         [vhigh = PROPERTY(net_prop_layer, vhigh)]
         [vlow = PROPERTY(net_prop_layer, vlow)]
layer2_dbg = DFM PROPERTY layer2 net_prop_layer NODAL MULTI
         [vhigh = PROPERTY(net_prop_layer, vhigh)]
         [vlow = PROPERTY(net_prop_layer, vlow)]

layer4_dbg = DFM PROPERTY layer4 net_prop_layer NODAL MULTI
         [vhigh = PROPERTY(net_prop_layer, vhigh)]
         [vlow = PROPERTY(net_prop_layer, vlow)]
// write net-indexed layers and debug layers out with 
// DFM DV ANALYZE. Include layerX, which has connectivity to 
// layer1 and layer4. 
dv_dbg = DFM DV ANALYZE layer1_dbg layer2_dbg layer4_dbg layerX
                  net_prop_layer RDB dv_analyze.rdb
dv_dbg_check { 
   copy dv_dbg
}
dv_12 {
   err_12  = DFM DV layer1 layer2 < 1.5 DVPARAMS net_prop_layer
      "vlow" 0 "vhigh" 5 "nosync" >= 5.0
      ANNOTATE NETS
   DFM COPY err_12
   DFM DV ANALYZE err_12 RDB dv_analyze.rdb
}
dv_14 {
   err_14  = DFM DV layer1 layer4 < 1.5 DVPARAMS net_prop_layer
      "vlow" 0 "vhigh" 5 "nosync" >= 5.0
      ANNOTATE NETS
   DFM COPY err_14
   DFM DV ANALYZE err_14 RDB dv_analyze.rdb
}

The following list of check results is written to the RDB file dv_analyze.rdb for a small layout that produces one result for each of the two DFM DV checks. 

dv_12::err_12
dv_14::err_14
dv_dbg uid=1 layer1_dbg
dv_dbg uid=1 layerX
dv_dbg uid=1 net_prop_layer
dv_dbg uid=2 layer4_dbg
dv_dbg uid=2 layerX
dv_dbg uid=2 net_prop_layer
dv_dbg uid=3 layer2_dbg
dv_dbg uid=3 net_prop_layer

The DV layer output includes the properties DELTA_V, FIRST_UID, FIRST_VMIN, FIRST_VMAX, SECOND_UID, SECOND_VMIN, SECOND_VMAX, and SYNC.

The non-DV layer output includes all DFM properties attached to the layer plus the property DN (the node number). For this example, the *_dbg layers include the vhigh and vlow properties. The net_prop_layer output includes the vhigh, vlow, and nosync properties.
