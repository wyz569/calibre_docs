Extent Cell

Layer operation

Generates a derived polygon layer consisting of rectangles that represent the extents of cells in the given list. Used only in Calibre.

Usage

EXTENT CELL {{cell_name [cell_name …]} | {LIST cell_list_name}} [{ORIGINAL | MAPPED} [OCCUPIED]]
[{[{OVER | UNDER} distance] [RING distance]} | {WITH MATCH [GOLDEN]}]

Arguments

{cell_name [cell_name …]} | {LIST cell_list_name}

A required argument specifying the cells to process.

  • cell_name — Specifies the name of a cell. You can specify cell_name any number of times in one statement.

    Each cell_name may contain one or more asterisk (*) wildcard characters, where the asterisk (*) character matches zero or more characters. When using wildcards, enclose cell_name in quotation marks ( “ ” ). The cell_name can be a string variable. Cell names are case-sensitive.

    Each cell_name may use a regular expression (or regex). A leading slash (/) or exclamation point with a slash (!/) identifies a regex pattern. A regex pattern must be enclosed in quotes since slash is a reserved symbol. The expression should not include a trailing slash and regex flags after a trailing slash are not supported. Matching is case-sensitive. Regular expressions for cell names are not supported if WITH MATCH is specified. For more information about regular expressions, refer to “Regular expression matching” in the Layout Cell List reference page.

  • LIST cell_list_name — Specifies the name of a cell list that is defined by the Layout Cell List statement. The cell_list_name is case‑sensitive and may not be a variable.

{ORIGINAL | MAPPED} [OCCUPIED]

An optional keyword choice that specifies the objects in the input layout database that are used to compute the specified cell extent(s). (By default, the cell extents returned by this operation are the extents of only those objects actually required for the Calibre run.)

  • ORIGINAL [OCCUPIED] — Specifies all objects (geometry and text) in the input layout database are used to compute the specified cell extent(s). The OCCUPIED keyword specifies that only cells containing geometry required in the Calibre run (either in the cell or its subhierarchy) have their extents returned. A cell is ignored if it does not contain geometry required by the run.

  • MAPPED [OCCUPIED] — Specifies all objects in the input layout database that are on simple layers specified by the totality of rule file Layer specification statements are used to compute the specified cell extent(s). The OCCUPIED keyword specifies that only cells containing geometry required in the Calibre run (either in the cell or its subhierarchy) have their extents returned. A cell is ignored if it does not contain geometry required by the run.

{OVER | UNDER} distance

An optional keyword choice that specifies to expand or shrink the results of the Extent Cell operation by the specified distance on a cell‑by‑cell basis. The distance value is specified as a positive numeric in user units. The OVER and UNDER keywords cannot be specified with the WITH MATCH keyword. Flat Calibre applications ignore the OVER and UNDER keywords and generate a warning.

  • OVER distance — Specifies to expand the results by the specified distance.

  • UNDER distance — Specifies to shrink the results by the specified distance.

RING distance

An optional keyword and positive floating-point number in user units that define a distance by which the extent of a specified cell is shrunk (similar to Size with a negative value). This forms an annular boundary region, or “ring,” around the interior of each specified cell, and the ring has a thickness of the specified distance. This ring becomes the output of the operation for a given cell.

If OVER or UNDER is also specified, then that region is computed first and is used instead of the extent of a specified cell. The ring region is then computed and output. May not be specified with WITH MATCH.

WITH MATCH [GOLDEN]

An optional keyword that causes a placed cell in the input layout database hierarchy to be treated as a cell_name argument to the operation if and only if the cell geometrically matches an unplaced cell that is specified as a cell_name argument or is included in the cells listed in the cell_name_list. The WITH MATCH keyword is particularly useful for matching cells to so-called “golden” cells, when the exact name correspondence may not be known.

When WITH MATCH is used, specify only the unplaced cells that you want to match. It is not necessary to specify placed cells; a placed cell is only considered if it is a geometric match to an unplaced cell that is specified. See “WITH MATCH Keyword” for details.

The following conditions apply when using this keyword:

  • May only be used in hierarchical Calibre applications.

  • May not be used in dual‑database mode.

  • May not be specified with RING, UNDER, or OVER.

  • GOLDEN — Specifies that the operation should match only cells that appear in designs specified with Layout Path or Layout Path2 statements that also use the GOLDEN keyword.

    Note:

    The GOLDEN keyword is not recommended. Siemens EDA recommends using the Calibre Pattern Matching tool or the Calibre Auto-Waivers solution.

Description

Generates a derived polygon layer consisting of rectangles that represent the extents of cells in the given list.

By default, the cell extents returned by this operation (as well as the database extent returned by the Extent operation) are the extents of only those objects actually required for the Calibre run. For instance, if a layout database contains objects on layer 1 and layer 1 is not required in the Calibre run (that is, it is not required for producing output from the run), then layer 1 is ignored in any Extent Cell calculations of cell extents. (See Figure 1 for a related example.)

If ORIGINAL is specified, then all geometry in the input layout database is used to compute cell extents, even if the geometry is not required in the Calibre run. Objects not required for other operations in the Calibre run are subsequently ignored as before. If MAPPED is specified, only mapped layers (those appearing in Layer statements in the rule file) are used to compute cell extents.

If OCCUPIED is specified in addition to ORIGINAL or MAPPED, then only cells having geometry required in the flow (including anywhere in their sub-hierarchy) have their extents computed based upon the ORIGINAL or MAPPED keyword. All other cells are considered empty, as in the default behavior.

If you use the * wildcard in a cell_name parameter, the merged extent of all the cells having names that match the search pattern are returned, not the extents of the individual cells.

Note:

Specifying Extent Cell “*” is not recommended as it can cause severe performance degradation in hierarchical Calibre applications. A warning is issued during the hierarchical database construction phase if this is encountered.

By default, a warning is issued if a cell_name parameter cannot be found in the layout, but the run proceeds. You can control the severity of this case through the Layout Input Exception Severity EXTENT_CELL setting.

The OVER, RING, and UNDER keywords are useful in certain results waiver flows. Otherwise, they are of limited utility. OVER and UNDER behave similarly to the Size operation when expanding or shrinking polygons on a layer. The RING keyword causes the output to be a rectangular annulus rather than a convex shape.

WITH MATCH Keyword

The WITH MATCH keyword enables placed cells to be included as part of the Extent Cell calculation based upon geometric matching to an unplaced cell that is specified as a cell_name argument or is in the list of cells specified by cell_list_name. This feature should only be used with validated cells in a hierarchical run. Note that specifying “non-golden” cells (cells that have not been previously validated) can result in warnings or errors during the Calibre run.

Before discussing how WITH MATCH works, the term geometrically equal is defined. Two cells A and B are said to be geometrically equal under the following conditions. For each original layer required in the flow:

  1. Flatten and merge the layer in A and its sub-hierarchy into the coordinate space of A.

  2. Flatten and merge the layer in B and its sub-hierarchy into the coordinate space of B.

  3. Perform a Boolean XOR of the results of Steps 1 and 2.

If there are no results for any XOR, then A and B are geometrically equal.

This definition does not include text objects.

The output of WITH MATCH is as follows.

EXTENT CELL layer1 A1 A2 ... Am WITH MATCH ...

where m >= 1, is made equivalent to

EXTENT CELL layer1 B1 B2 ... Bn

where n >= 0. Each Bj is the name of a placed cell (this name may or may not be known) that is geometrically equal to some unplaced cell A whose name matches some Ai with possible wildcards. The geometry on layer1 that appears in all such cells Bj is output.

If the GOLDEN keyword is specified, then the unplaced cells Ai that the operation attempts to match come from designs specified in Layout Path[2] GOLDEN statements.

See also Extents and Extent Drawn.

Examples

Example 1

x = EXTENT CELL cella cellb
// returns the extents of cella and cellb according to layers
// actually needed in the run

y = EXTENT CELL cella cellb ORIGINAL
// returns the extents of cella and cellb according to all 
// layers in the database

Example 2

Suppose you have a rule for which the poly spacing is different for the memory and non-memory portions of your design. You want to apply two spacing checks, depending on whether the cell is a memory cell or not. In the file golden.gds, the memory cells appear in the cell bitcel.golden. In layout.gds, the names of the memory cells are not known, so you want to match the memory cells geometrically to bitcell.golden. 

LAYOUT PATH layout.gds golden.gds // concatenate the layouts.
// layout cell names are not expected to match.
LAYOUT ALLOW DUPLICATE CELL NO // default
// Find the poly in memory cells in layout.gds by geometric matching
// with golden.gds. The cell name bitcel.golden isn't expected to match
// a cell name in layout.gds; however, it is the known memory cell name
// in golden.gds.

poly_in_bitcel = poly AND (EXTENT CELL bitcel.golden WITH MATCH)  
40X { @ poly spacing is 0.3, except in bit cells where it is 0.2.

        X1 = EXT poly < 0.3 OPPOSITE REGION
        // apply 0.3 spacing measurement for the entire design

        X2 = EXT poly_in_bitcel < 0.3 OPPOSITE REGION
        // apply 0.3 spacing in memory cells; 
        // note X1 has already calculated it so there is no 
        // runtime penalty.

        X3 = X1 NOT X2
        // spacing errors that are not in memory cells

        X3 OR ( EXT poly_in_bitcel < 0.2 OPPOSITE REGION )
        // apply a different measurement for memory cells and OR with
        // other cells' results
}
