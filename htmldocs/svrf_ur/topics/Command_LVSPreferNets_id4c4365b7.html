LVS Prefer Nets

Specification statement

Specifies names of nets that are preferred when assigning names to nets that are shorted during LVS netlist transformations due to LVS Filter SHORT processing.

Usage

LVS PREFER NETS {FILTER name [name …]}
[LAYOUT | SOURCE | LAYOUT SOURCE]

Arguments

FILTER

A required keyword specifying LVS Filter SHORT netlist transformations are processed.

name

A required argument specifying a net name. More than one name may be specified. A name argument may be a string or string variable reference. The string may contain the question mark (?) wildcard, which matches zero or more characters. String variables are dereferenced into name strings for each variable instance, then those dereferenced strings are treated as name strings. All patterns in a string variable are considered to occur at the point of the variable reference, in the order in which they are given in the Variable statement.

LAYOUT

An optional keyword that specifies the statement applies to the layout design.

SOURCE

An optional keyword that specifies the statement applies to the source design.

Note:

The combination LAYOUT SOURCE is used by default if neither keyword is specified.

Description

Specifies names of nets to prefer when assigning names to shorted nets introduced by device filtering. This statement may appear any number of times in a rule file, and name specifications are accumulated across all statements. A name, possibly with wildcard matching, is referred to as a pattern in the remainder of this section.

Pattern order is significant, as each name is applied in the order it is specified for each design type (LAYOUT or SOURCE). When shorting together two or more nets as a result of device filtering, Calibre must choose one of the shorted nets to be the survivor. LVS Prefer Nets provides some control over which net will be selected. Among any two nets in the set, if one of those nets has a user-given name that matches an applicable pattern, and the other does not, the net with the matching name is preferred over the other as the survivor. If both nets have user-given names that match applicable patterns, the net whose name matches the pattern that appears earliest in the ordering is preferred.

If both nets match the same pattern, that pattern does not contribute to determining which net is preferred, but the remaining patterns are checked for further matches. For example, consider this statement:

LVS PREFER NETS FILTER "Q?" "QBAR"

Suppose two nets are to be shorted whose names are QBAR and Q. Both names match the first pattern, so neither is preferred on that basis. Hence, the second pattern is checked, and it matches QBAR. This makes QBAR the survivor. If neither net in a pair is preferred on the basis of LVS Prefer Nets patterns, Calibre uses its default heuristic to choose a preferred survivor.

LVS Prefer Nets takes precedence over other internal rules of precedence for nets shorted due to device filtering. For example, normally a power or ground net is preferred over one that is not. But if a net is preferred over another on the basis of LVS Prefer Nets, the supply status of the two nets does not affect this preference.

Case sensitivity of name matching is determined by the LVS Compare Case statement. Matching is case-insensitive by default, but if LVS Compare Case YES or LVS Compare Case NAMES is specified, matching is case-sensitive.

Also see LVS Prefer Port Nets.

Examples

If you desire to prefer power or ground nets over those whose names match the names specified in LVS Prefer Nets, a reasonable approximation of this may be accomplished conveniently with variables.

VARIABLE PNETS "VDD1" "VDD2" . . .
VARIABLE GNETS "VSS1" "VSS2" . . .

LVS POWER NAME PNETS
LVS GROUND NAME GNETS

LVS PREFER NETS FILTER PNETS GNETS
  . . .
LVS PREFER NETS FILTER "Q?" "R?" SOURCE
LVS PREFER NETS FILTER "IN?" "OUT?" LAYOUT

This rule file excerpt directs Calibre to prefer nets with power and ground names above all others, and then to prefer in the source design nets whose user-given names begin with Q, followed by nets whose user-given names begin with R, while preferring in the layout design nets whose user-given names begin with IN followed by those whose user-given names begin with OUT.

In a hierarchical run, propagation of power and ground nets through the hierarchy occurs after LVS Filter SHORT processing. Hence the power/ground statuses of nets without supply names would not be a factor in any case. The preceding code excerpt directs Calibre to prefer any net with a supply name in any cell, which may yield a somewhat different behavior from preferring only nets with a supply status at the time of LVS Filter SHORT processing. Nevertheless, this behavior is likely desirable, or at least acceptable.
