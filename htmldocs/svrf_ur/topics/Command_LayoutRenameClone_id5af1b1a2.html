Layout Rename Clone

Specification statement

Specifies to use unique identification of clone cells across multiple input files. Used only in Calibre.

Usage

LAYOUT RENAME CLONE {NO | YES}

Arguments

NO

Keyword that instructs Calibre not to use unique identification of clone cells across multiple input files. This is the default behavior when you do not include this statement in the rule file.

YES

Keyword that instructs Calibre to use unique identification of clone cells across multiple input files.

Description

This statement is used to enable unique identification of clone cells across multiple files in the Layout Path or Layout Path2 specification statement (but not across the two). A “clone” cell is a cell created by one of the following:

  • The process of cloning by the Calibre hierarchical database constructor to remove floating-point magnification from placement transforms. In this case, the newly-created cells have a name of the form “<cell‑name>$M<number>$”, where <cell‑name> is the name of the original cell and <number> is an internally-generated sequence of decimal digits.

  • The process of cloning by the Calibre hierarchical database constructor to support the Layout Clone Transformed Placements YES or Layout Clone Rotated Placements YES specification statements. In this case, newly-created cells have a name of the form <cell‑name>$C<number>$, where <cell‑name> is the name of the original cell and <number> is an internally-generated sequence of decimal digits.

  • The process of cloning by the Calibre hierarchical database constructor to support the Layout Clone By Layer specification statement. In this case, newly-created cells have a name of the form <cell‑name>$L<number>$, where <cell-name> is the name of the original cell and <number> is an internally-generated sequence of decimal digits.

When clone cells are created by Calibre, there is no control over their names. Hence, when running Calibre output back through Calibre (for example, in post-tapeout flows), multiple files in a Layout Path can lead to cell name conflicts. You can resolve this by combining duplicate cell names (Layout Allow Duplicate Cell YES) but this is not the intent, since the two clone cells will only be the “same” by accident.

Assume that Layout Path (or Layout Path2) has multiple file parameters, as in a case like this:

LAYOUT PATH file1 file2 ... fileN

Assume that filei, where i > 1, contains a cell that was created by one of the cloning processes described above. In addition, assume this cell name matches a cell name in filej, where j < i (after any renaming has been applied to filej).

When YES is specified, the clone cell is renamed by appending $X<number>$, where X is the ‘M’, ‘C’, or ‘L’ of the original name and <number> is an internally-generated positive-length sequence of decimal digits. This insures the new name of the clone cell will not match a cell name in any filej with j< i.

Because of the complexity in recognizing a clone cell, there is one exception to the renaming rules for clone cells. For example, a clone created by Calibre due to floating-point magnification in one Calibre run may be cloned again by, for example, Layout Clone By Layer, in a subsequent Calibre iteration. Calibre determines that a cell is a cloned cell if the name is of the form:

c…c<clause>[…<clause>]

where

  • c…c is a string of at least one character, and

  • <clause> is a string of the form $Md…d$, $Cd…d$, or $Ld…d$, where d…d is a string of a least one decimal digit.

When renaming, the new name is formed by appending $Xd…d, where X is the M/L/C of the first <clause> and d…d is an internally-generated positive-length string of decimal digits so that the cell name is unique. However, if all clauses are of the form $Cd$, where d < 8, then the cell is not renamed. This is because clones created by Layout Clone Transformed Placements YES or Layout Clone Rotated Placements YES have consistent numerics appended since they are derived from the possible transforms which have been removed by cloning. If d >= 8, or there is more than one decimal digit following the ‘C’ (in any clause), then consistent naming was not possible for some reason, and the cell is renamed as described above.

Note:

Be aware that certain operations that work on cell names may not work on their cloned counterparts.

See also Layout Rename Cell, Layout Rename All, and Layout Rename ICV.

Examples

In the following example, assume there are clone cells with identical names in the layouts due to prior processing by Calibre. With Layout Rename Clone NO, that situation is not allowed:

LAYOUT PATH layout1.gds layout2.gds
LAYOUT RENAME CLONE NO

Using YES causes the duplicate clone cells to be processed as unique.
