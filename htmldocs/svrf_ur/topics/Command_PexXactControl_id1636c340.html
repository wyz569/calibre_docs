PEX XACT Control

Parasitic extraction

Specifies how certain nets are handled during parasitic extraction and netlisting with Calibre xACT. Used only in Calibre xACT.

Usage

Syntax 1:

PEX XACT CONTROL FILE filename

Syntax 2:

PEX XACT CONTROL ‘[’
control_syntax
‘]’

Arguments

FILE filename

A required keyword set that specifies a filename that contains a set of directives used by Calibre xACT for parasitic extraction.

‘[’ control_syntax ‘]’

A set of directives placed between required ([ ]) brackets that specify the parasitic extraction and netlist generation behavior for Calibre xACT.

Description

An optional statement that controls how certain nets are handled during parasitic extraction and netlisting using the Calibre xACT or Calibre xACT 3D direct netlisting flow. This statement can only be specified once, along with a PEX Netlist statement. Directives specified with this statement override global command line and SVRF statements for specific nets. Directives must be specified between required brackets ([ ]) or in a file.

Extraction and netlisting are controlled using the following directives:

  • EXTRACT_CONTROL { … } — The directive used to specify how certain nets are handled during parasitic extraction. This keyword must be followed by a whitespace delimited opening brace ( { ) and closed with a whitespace delimited closing brace ( } ). In this section you can specify the parasitic netmodel and what layers of the net should be extracted. This directive may only be specified once.

  • NETLIST_CONTROL{ … } — The directive used to specify how certain nets are handled during netlisting. This keyword must be followed by a whitespace delimited opening brace ( { ) and closed with a whitespace delimited closing brace ( } ). Specify this section to override any global net model specifications with a local net specification. This directive may be specified more than once.

These sections are independent, but must be consistent from extraction to netlisting. For example, if you want an RCC netlist you must extract RCC as opposed to just R or RC. Consistency checks are done during extraction to ensure valid results.

EXTRACT_CONTROL Contents

EXTRACT_CONTROL ‘{’
	NET: netname [netname …] [LAYOUTNAMES|SOURCENAMES] [TOPLEVEL|RECURSIVE] 
	MODEL: {RCCLK | RCCL | RCC | RCLK | RCL | RC | RLK | RL | R | C | Cg } 
	[{R_LAYERS_SELECT: | R_LAYERS_EXCLUDE:} [layername …] 
		[VIA layer1 layer2 [VIANAME vialayer_name  [vialayer_name …]]… ] 
	[{L_LAYERS_SELECT: | L_LAYERS_EXCLUDE:} [layername …]]
 ‘}’

The EXTRACT_CONTROL section allows you to:

  • Identify nets for special handling.

  • Specify the extraction model to be used for these nets.

  • Optionally limit the layers involved in resistance and/or inductance extraction.

To specify a net for processing in EXTRACT_CONTROL, the NET: and MODEL: keyword syntax must be specified in successive order. If the MODEL: section does not immediately follow the NET: section, an error is generated. Nets not referenced directly by this section are extracted using the parasitic model specified in the command line, such as -rcc or -rc.

Net Specification — This section specifies the nets for special handling and has this syntax:

NET: netname [netname …] [LAYOUTNAMES|SOURCENAMES] [TOPLEVEL|RECURSIVE]

  • NET: — Required keyword followed immediately by a colon.

  • netname [netname...] — Required parameter that specifies one or more net names. The string netname can contain one or more question mark (?) characters. This wildcard character matches zero or more characters. Net names that contain special characters, such as square brackets [ ], must be enclosed in quotation marks.

  • LAYOUTNAMES — Specifies the names are derived from the layout. May not be specified with SOURCENAMES. This is the default.

  • SOURCENAMES — Specifies the names are derived from the schematic. When specifying the SOURCENAMES option, make sure that a valid Source Primary statement identifying a valid primary cell is in the rule file. May not be specified with LAYOUTNAMES option.

  • TOPLEVEL — Specifies the names exist in the top level subcircuit. May not be specified with RECURSIVE. This is the default.

  • RECURSIVE — Specifies the names do not originate in the top level of the design, but in a lower level of the hierarchy. May not be specified with TOPLEVEL.

Model Selection — This section is required in the EXTRACT_CONTROL section to specify the parasitic model that is applied to the nets specified with NET: and has this syntax:

MODEL: RCCLK | RCCL | RCC | RCLK | RCL | RC | RLK | RL | R | C | Cg

  • MODEL: — Required keyword that specifies the parasitic net model for the net(s) during extraction followed immediately by a colon.

  • RCCLK | RCCL | RCC | RCLK | RCL | RC | RLK | RL | R | C| Cg — Specifies the parasitic model. Choose one of the following:

    RCCLK — Specifies the extraction of a distributed network of resistors, inductors, and intrinsic and coupled capacitors along with mutual inductors. Coupled capacitors are not grounded.

    RCCL — Specifies the extraction of a distributed network of resistors, inductors, and intrinsic and coupled capacitors. Coupled capacitors are not grounded.

    RCC — Specifies R-coupled-C extraction.

    RCLK — Specifies the extraction of a distributed network of resistors, inductors, and capacitors along with mutual inductors.

    RCL — Specifies the extraction of a distributed network of resistors, inductors, and capacitors.

    RC — Specifies distributed RC parasitic extraction.

    RLK — Specifies distributed resistance and inductance extraction including mutual inductance.

    RL — Specifies distributed resistance and inductance extraction.

    R — Specifies resistance-only extraction.

    C — Specifies intrinsic and coupled capacitive parasitic extraction.

    Cg — Specifies lumped capacitive parasitic extraction.

Resistance Control — This section specifies the R layers to be extracted for the given set of nets, or the R layers to be excluded for these nets. These keywords can only be specified if the parasitic net model includes resistance (R), otherwise extraction stops with an error. Resistance control has this syntax:

{R_LAYERS_SELECT: | R_LAYERS_EXCLUDE:} [layername …] 
	[VIA layer1 layer2 [VIANAME vialayer_name [vialayer_name …]]…

  • R_LAYERS_SELECT: — A required keyword followed immediately by a colon that controls parasitic resistance extraction on one or more original or derived layers. May not be specified with R_LAYERS_EXCLUDE:.

  • R_LAYERS_EXCLUDE: — A required keyword followed immediately by a colon that specifies which layers of the net are excluded from resistance extraction. May not be specified with R_LAYERS_SELECT:.

  • layername — A optional parameter that specifies one or more original or derived layers. All layers not specified are extracted with the global parasitic net model. If no layer is specified following the R_LAYERS_SELECT: keyword, resistance is not extracted for any layers of the net. If no layer is specified following the R_LAYERS_EXCLUDE: keyword, resistance is extracted for all layers of the net.

  • VIA layer1 layer2 — A keyword set that specifies to extract via resistance on vias connecting layer1 and layer2. The upper layer must be specified first followed by the lower layer. More than one via connection may be specified. May only be specified with R_LAYERS_EXCLUDE: and R_LAYERS_SELECT: keywords.

  • VIANAME vialayer_name — A keyword set that specifies to extract resistance for one or more vias, specified by LVS via layer name, vialayer_name. Specify this keyword set with the VIA layer1 layer2 keyword set to filter vias by LVS via layer name.

Inductance Control — This optional section specifies layer-wise select or exclude filters used to limit inductance extraction on certain layers for a specified set of nets. These keywords can only be specified if the parasitic net model includes inductance (L), otherwise extraction stops with an error. Inductance control has this syntax:

{L_LAYERS_SELECT: | L_LAYERS_EXCLUDE:} [layername …]

  • L_LAYERS_SELECT: — A required keyword followed immediately by a colon. Original or derived layers specified with this keyword are included for inductance extraction. May not be specified with L_LAYERS_EXCLUDE:.

  • L_LAYERS_EXCLUDE: — A required keyword followed immediately by a colon that specifies which layers of the net are excluded from inductance extraction. May not be specified with L_LAYERS_SELECT:.

  • layername — An optional parameter that specifies one or more original or derived layers. All layers not specified are extracted with the global parasitic net model. If no layer is specified following the L_LAYERS_SELECT: keyword, inductance is not extracted for any layers of the net. If no layer is specified following the L_LAYERS_EXCLUDE: keyword, inductance is extracted for all layers of the net.

Inductance extraction that is requested for a given net is limited to the layers specified in L_LAYERS_SELECT: or to the layers not assigned in L_LAYERS_EXCLUDE:. If these inductance controls are not specified, then all the layers involved in resistance extraction are also considered for inductance extraction.

If you want a different set of layers to be selected or excluded for the resistance extraction with respect to inductance extraction, then both resistance control and inductance control statements may be used for the same net. Note that a layer specified for inductance extraction must also be specified for resistance extraction, otherwise extraction stops with an error.

NETLIST_CONTROL Contents

NETLIST_CONTROL ‘{’
	MODEL: {RCCLK | RCCL | RCC | RCLK | RCL | RC | RLK | RL | R 
				| C | Cg | NONE}
	{NETLIST: netlist_name 
	 NETLIST_FORMAT: {DSPF|HSPICE|SPEF|SPECTRE|ELDO|CALIBREVIEW} 
		[LAYOUTNAMES|SOURCENAMES|SOURCEBASED] [TOPLEVEL|RECURSIVE] 
	 [NET: netname [netname …] [LAYOUTNAMES|SOURCENAMES] 
		[TOPLEVEL|RECURSIVE] [MODEL: {RCCLK | RCCL | RCC | RCLK | RCL | RC 
											| RLK | RL |  R | C | Cg | NONE} ]]} ...
‘}’

The NETLIST_CONTROL section allows you to:

  • Specify the default extraction model used during netlist generation.

  • Specify the netlist name and corresponding netlist format. More than one netlist name and format combination can be specified.

  • Optionally identify specific nets of special interest and model selection per net for the specified netlist-netlist format combination.

The first entry in the NETLIST_CONTROL must be the MODEL: which specifies the default model used for netlisting. The next entries must be NETLIST: and NETLIST_FORMAT:. At least one netlist and corresponding format must be specified. The settings specified in NETLIST_CONTROL override the settings specified in your PEX Netlist statement. If the same netlist format is specified in both the PEX Netlist statement and the PEX XACT Control statement, then the netlist format specific keywords specified in your PEX Netlist statement are applied.

Model Selection — This section specifies the parasitic model to be applied during netlisting and has this syntax:

MODEL: {RCCLK | RCCL | RCC | RCLK | RCL | RC | RLK | RL | R 
			| C | Cg | NONE}

  • MODEL: — Required keyword that specifies the default extraction model used during netlisting followed immediately by a colon.

  • RCCLK | RCCL | RCC | RCLK | RCL | RC | RLK | RL | R | C | Cg | NONE — Specifies the parasitic net model. Choose one of the following:

    RCCLK — Includes a distributed network of resistors, inductors, and intrinsic and coupled capacitors along with mutual inductors in the netlist. Coupled capacitors are not grounded.

    RCCL — Includes a distributed network of resistors, inductors, and intrinsic and coupled capacitors in the netlist.Coupled capacitors are not grounded.

    RCC — Includes a distributed network of resistors and capacitors in the netlist.

    RCLK — Includes a distributed network of resistors, inductors, and capacitors along with mutual inductors in the netlist.

    RCL — Includes a distributed network of resistors, inductors, and capacitors in the netlist.

    RC — Includes a distributed network of resistors and capacitors in the netlist. Intrinsic and coupled capacitors are grounded.

    RLK — Includes a distributed network of resistors and inductors along with mutual inductance in the netlist.

    RL — Includes a distributed network of resistors and inductors in the netlist.

    R — Includes resistors in the netlist.

    C — Includes intrinsic and coupling capacitors in the netlist.

    Cg — Includes grounded intrinsic and coupling capacitors in the netlist.

    NONE — Includes only the intentional devices on the net in the netlist. This provides the least detailed net representation.

Netlist Specification — This section specifies the netlist name and format, and has this syntax:

NETLIST: netlist_name 
	 NETLIST_FORMAT: {DSPF|HSPICE|SPEF|SPECTRE|ELDO|CALIBREVIEW} 
		[LAYOUTNAMES|SOURCENAMES|SOURCEBASED] [TOPLEVEL | RECURSIVE]

  • NETLIST: — Required keyword followed immediately by a colon.

  • netlist_name — Required parameter that specifies the netlist name.

  • NETLIST_FORMAT: — Required keyword followed immediately by a colon that specifies the format of the netlist. One of the following must be specified:

    HSPICE — The netlist format is HSPICE.

    DSPF — The netlist format is DSPF.

    SPEF — The netlist format is SPEF.

    SPECTRE — The netlist format is SPECTRE.

    ELDO — The netlist format is ELDO.

    CALIBREVIEW — The netlist format is CalibreView.

  • LAYOUTNAMES — Specifies the names are derived from the layout. May not be specified with SOURCENAMES or SOURCEBASED. This is the default.

  • SOURCENAMES — Specifies the names are derived from the schematic. When specifying the SOURCENAMES option, make sure that a valid Source Primary statement identifying a valid primary cell is in the rule file. May not be specified with LAYOUTNAMES or SOURCEBASED.

  • SOURCEBASED — Specifies to use the circuit pin order and cell hierarchy in the source netlist instead of the layout netlist. If you are creating the source netlist with Calibre nmLVS, you must use Calibre nmLVS-H and the design must pass LVS. May not be specified with LAYOUTNAMES or SOURCENAMES.

    Note:

    Using SOURCEBASED with LVS Push Devices Separate Properties YES generates a warning and SOURCENAMES is used.

  • TOPLEVEL — Specifies the names exist in the top level subcircuit. May not be specified with RECURSIVE. This is the default.

  • RECURSIVE — Specifies the names do not originate in the top level of the design, but in a lower level of the hierarchy. May not be specified with TOPLEVEL.

Net Specification — This optional section specifies special handling of one or more nets for the chosen netlist and has this syntax:

NET: netname [netname …] 
	[LAYOUTNAMES|SOURCENAMES] [TOPLEVEL|RECURSIVE]
	[MODEL: {RCCLK | RCCL | RCC | RCLK | RCL | RC | RLK | RL | R 
				| C | Cg | NONE}]

  • NET: — A keyword that indicates the start of the net specification section. Must be followed immediately by a colon.

  • netname [netname …] — A parameter that specifies one or more net names. The string netname can contain one or more question mark (?) characters. This wildcard character matches zero or more characters. Net names that contain special characters, such as square brackets [ ], must be enclosed in quotation marks.

  • LAYOUTNAMES — Specifies the names are derived from the layout. May not be specified with SOURCENAMES. This is the default.

  • SOURCENAMES — Specifies the names are derived from the schematic. When specifying the SOURCENAMES option, make sure that a valid Source Primary statement identifying a valid primary cell is in the rule file. May not be specified with LAYOUTNAMES.

  • TOPLEVEL — Specifies the names exist in the top level subcircuit. May not be specified with RECURSIVE. This is the default.

  • RECURSIVE — Specifies the names do not originate in the top level of the design, but in a lower level of the hierarchy. May not be specified with TOPLEVEL.

  • MODEL: — Optional keyword that specifies the parasitic net model to apply to one or more nets specified by the NET: section. Choose one: RCCLK, RCCL, RCC, RCLK, RCL, RC, RLK, RL, R, C, Cg, or NONE. When NONE is specified the parasitics for the specified net are not netlisted. The default is RCC.

Control Syntax Requirements and Restrictions

  • Keywords followed by a colon must appear in a certain order.

    For EXTRACT_CONTROL the order is:

    • NET:

    • MODEL:

    • R_LAYERS_SELECT: | R_LAYERS_EXCLUDE:

    • L_LAYERS_SELECT: | L_LAYERS_EXCLUDE:

    For NETLIST_CONTROL the order is:

    • MODEL:

    • NETLIST:

    • NETLIST_FORMAT:

    • NET:

  • All keywords and strings must be space delimited. For example:

    MODEL:RCC is not valid

    MODEL: RCC is valid

  • All keywords must be capitalized. For example:

    netlist_format: CalibreView is not valid

    NETLIST_FORMAT: CALIBREVIEW is valid

  • Names that contain special characters, such as square brackets [ ], must be enclosed in quotation marks. It is good practice to enclose all names in quotation marks to avoid SVRF compilation errors. For example:

    NET: A[12] is not valid

    NET: “A[12]” is valid

  • C-style comments are supported using //, but are limited to the beginning of the line. For comments, the first non-whitespace character on a line must be //. For example:

    NET: VDD // The comment syntax is not valid for this line

    // NET: VDD This comment syntax is valid

For information on performing extraction using PEX XACT Control, see “Calibre xACT Processing Control” in the Calibre xACT User’s Manual.

Examples

Example 1

To extract capacitance and resistance for all nets and layers, and only extract resistance for vias between metal layers on nets VDD? and VSS?, include the following statement in the rule file:

PEX XACT CONTROL [
	EXTRACT_CONTROL {
		NET: "VDD?" "VSS?"
		MODEL: RCC
		R_LAYERS_SELECT: VIA M2 M1 VIA M3 M2 VIA M4 M3 VIA M5 M4
	}
]

Example 2

To produce two RCC netlists in both SPEF and DSPF format, include the following statement in the rule file:

PEX XACT CONTROL [
	NETLIST_CONTROL {
		MODEL: RCC
		NETLIST: netlist1.spef NETLIST_FORMAT: SPEF
		NETLIST: netlist2.dspf NETLIST_FORMAT: DSPF
	}
]

Note that without a EXTRACT_CONTROL section the extraction mode specified in the command line is used.

Example 3

Directives can be defined in a text file that can then be specified using PEX XACT Control.

In this example a text file called my_xact_control_file is created and contains the following: 

// Extraction specification
EXTRACT_CONTROL {
	NET: VDD LAYOUTNAMES
	MODEL: RCC
	R_LAYERS_EXCLUDE: M2
}
// Netlist specifications for RC Models
NETLIST_CONTROL {
	MODEL: RC
	NETLIST: pex1.dspf NETLIST_FORMAT: DSPF NET: "?VDDCOL?" MODEL: Cg
}
// Netlist specifications for RCC Models
NETLIST_CONTROL {
	MODEL: RCC
	NETLIST: pex2.spef NETLIST_FORMAT: SPEF NET: "?VDDCOL?" MODEL: Cg
	NETLIST: pex3.hspice NETLIST_FORMAT: HSPICE
}

Include the following statement in your rule file: 

PEX XACT CONTROL FILE my_xact_control_file

to create three netlists: pex1.dspf, pex2.spef, and pex3.hspice.

The first netlist, pex1.dspf, has all nets netlisted with the RC net models except for the ?VDDCOL? nets that are netlisted with Cg net models.

The second netlist, pex2.spef, has all nets netlisted with RCC net models, except for the ?VDDCOL? nets that are netlisted with Cg net models.

The third netlist, pex3.hspice, has all nets netlisted with RCC net models.

Note that for all three netlists, resistance will not be extracted for nets on layer M2.

Example 4

Suppose ground net VSS is set as a return path for inductance extraction using the following statement:

PEX INDUCTANCE RETURNPATH GROUND

Inductance effects are more pertinent for the part of the interconnect network that is laid out on higher metal layers where layer resistance is lower. Also in large databases where power and ground nets can be massive, it is highly recommended that the L_LAYERS_SELECT: keyword be specified with only the upper metal layer stack.

In this example, the process has eight metal layers M1-M8. Resistance extraction is limited for net VSS? to the via layers between metal layers and the planar M5-M8 layers. Inductance extraction is further limited to the four planar layers M5-M8; if one of these planar layers is not also specified in R_LAYERS_SELECT:, then extraction stops with an error. All signal nets containing the “CLK” string in their layout name are selected to be included in inductance extraction. No layer is excluded from resistance and inductance extraction for ?CLK? nets, for example:

PEX XACT CONTROL [
	EXTRACT_CONTROL {
		NET: "VSS?"
		MODEL: RL
		R_LAYERS_SELECT: M5 M6 M7 M8 VIA M2 M1 VIA M3 M2 VIA M4 M3 
								VIA M5 M4 VIA M6 M5 VIA M7 M6 VIA M8 M7 
		L_LAYERS_SELECT: M5 M6 M7 M8

		NET: "?CLK?"
		MODEL: RCCL
		R_LAYERS_EXCLUDE:
	}
	NETLIST_CONTROL {
		MODEL: RCCL
		NET: "VSS?"
		MODEL: NONE
	}
]

Layer filtering for inductance on signal nets is not recommended to avoid missing driver/receiver ports at lower layers that are used during path length calculations in the inductance engine.

The netlist control section specifies that the netlist contain RCCL parasitics. Net VSS? is not netlisted and any coupling capacitors between VSS? and nearby nets are included in the parasitic model of the nearby nets. Similarly, the RL contribution of VSS? is included in the loop RL values netlisted for signal nets ?CLK?.
