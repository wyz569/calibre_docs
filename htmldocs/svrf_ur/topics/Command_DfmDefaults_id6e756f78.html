DFM Defaults

Specification statement

Specifies default options for subsequent DFM RDB or DFM Fill operations.

Usage

DFM DEFAULTS operation options

Arguments

operation

A required argument that specifies the DFM operation to which the default options apply:

  • RDB [GDS | OASIS] [TOP] — Specifies that DFM RDB operations are affected. The optional keywords GDS, OASIS, and TOP from the DFM RDB operation further restrict which DFM RDB operations are affected.

  • FILL — Specifies that DFM Fill operations are affected.

options

A required argument that specifies default options for the operation. The following values are supported:

  • For operation = RDB:

    [FILE {filename | “PIPE command”} | NOFILE] [NODAL [ALL | OTHER | NULL]] [NOPSEUDO [NULL]] [NOEMPTY [NULL]] [OUTPUT [NULL]] [ALL CELLS [NULL]] [CELL SPACE [NULL]] [JOINED [NULL]] [RESULT CELLS {cell_list | NULL}] [CHECKNAME {check_name | NULL}] [PREFIX string [FILLONLY]] [MAXIMUM {ALL | NULL | value}] [APPEND string [FILLONLY]] [QUOTENAMES [NULL]]

    The NULL keyword cancels the associated default setting so that it no longer applies to subsequent instances of the operation in the rule file. For the FILE keyword, NOFILE is used rather than NULL because NULL is a valid filename keyword.

  • For operation = FILL:

    { [COMPRESS PREFIX cell_name_prefix]  | [COMPRESS VERSION version_number] }

Description

Specifies default options for the associated operation. When this statement is used, the options apply to any subsequent operation in the rule file unless overridden by a NULL or NOFILE keyword or by an incompatible keyword specification for some instance of the operation. The DFM Defaults statement may be specified more than once for a given operation. (Doing so introduces a degree of order dependency to the rule file, which ordinarily is not present.) When this occurs, the options accumulate across DFM Defaults statements for a given operation as they appear in the rules.

For DFM Defaults RDB you can specify multiple options in one statement. For DFM Defaults FILL you must specify separate options in separate statements—they cannot be combined. See “Example 4” for DFM Defaults FILL.

An option literally specified in a rule file instance of the operation has precedence over any corresponding default option currently in effect. If a default option is incompatible with an explicitly specified option, the default option is ignored. Therefore, a DFM Defaults statement cannot cause a subsequent operation not to compile.

Examples

Example 1

In the following code, layers met1 and met2 are written to out.gds.

DFM DEFAULTS RDB GDS FILE out.gds
CHECK1 { DFM RDB GDS DEFAULT met1 met2 }

Example 2

This example shows accumulation and cancellation of default options.

// no defaults in effect.
CHECK1 { DFM RDB met1 "rdb1" }
 
// NODAL keyword is the default for subsequent DFM RDB statements.
DFM DEFAULTS RDB NODAL
CHECK2 { DFM RDB met2 "rdb2" }
// NODAL ALL is incompatible with NODAL. NODAL ALL is used.
CHECK3 { DFM RDB met3 "rdb3" NODAL ALL }
 
// NODAL option is canceled. CHECKNAME now applies.
DFM DEFAULTS RDB NODAL NULL CHECKNAME "%_l_"
CHECK4 { DFM RDB met4 "rdb4" }
 
// ALL CELLS now applies in addition to CHECKNAME.
DFM DEFAULTS RDB ALL CELLS
CHECK5 { DFM RDB met5 "rdb5" }
// NOPSEUDO applies in addition to current defaults.
CHECK6 { DFM RDB met6 "rdb6" NOPSEUDO }

The preceding code is applied by the tool as follows:

CHECK1 { DFM RDB met1 "rdb1" }
CHECK2 { DFM RDB met2 "rdb2" NODAL }
CHECK3 { DFM RDB met3 "rdb3" NODAL ALL }
CHECK4 { DFM RDB met4 "rdb4" CHECKNAME "%_l_"}
CHECK5 { DFM RDB met5 "rdb5" CHECKNAME "%_l_" ALL CELLS}
CHECK6 { DFM RDB met6 "rdb6" CHECKNAME "%_l_" ALL CELLS NOPSEUDO}

Example 3

This example shows how to use the FILE and NOFILE options.

// default output file is rdb2 for subsequent DFM RDB statements.
DFM DEFAULTS RDB FILE "rdb2"
CHECK2 { DFM RDB met2 }
// local specification of rdb3 overrides default.
CHECK3 { DFM RDB met3 "rdb3" }
// default FILE specification canceled.
DFM DEFAULTS RDB NOFILE
// no default filename in effect--generates a compiler error.
CHECK4 { DFM RDB met4 }
// use NULL output for DFM RDB. store layer in DFM Database instead.
DFM DEFAULTS RDB FILE NULL
CHECK5 { DFM RDB met5 }

The preceding code is applied by the tool as follows:

CHECK2 { DFM RDB met2 "rdb2" }
CHECK3 { DFM RDB met3 "rdb3" }
CHECK4 { DFM RDB met4 } // compiler error.
CHECK5 { DFM RDB met5 NULL }

Example 4

This example shows how to use DFM Defaults to specify the cell_name_prefix and VERSION options for a DFM Fill operation with COMPRESS specified. See the DFM Fill operation for details on using the COMPRESS keyword.

// default definition
DFM DEFAULTS FILL COMPRESS PREFIX "test_"
DFM DEFAULTS FILL COMPRESS VERSION 6

// fill specifications
DFM SPEC FILL SHAPE "R1" RECTFILL  0 0 .2 .4 

DFM SPEC FILL Yspec INSIDE OF EXTENT 
   FILLSHAPE "FSbase" OUTPUT "OUT1" "R1"

// create compressed output to write out with DFM RDB
// the options from DFM Defaults are used
R1_fill_Y = DFM FILL Yspec COMPRESS  
R1_fill_Yhier = DFM FILL Yspec HLAYER COMPRESS

// write out the GDS file
writegds {
   DFM RDB GDS "R1_fill.gds" R1_fill_Y 33 R1_fill_Yhier 0
}

The GDS file R1_fill.gds has one compression cell named “test_6FSbase”, where “test” is the prefix, 6 is the version, and FSbase is the base cell name, taken from the basename of the DFM Spec Fill statement.
