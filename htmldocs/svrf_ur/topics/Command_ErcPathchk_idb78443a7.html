ERC Pathchk

Specification statement

Reports polygons from nets that do or do not have a path to power, ground, or labeled nets, or that satisfy a combination of these conditions. Used during circuit extraction in Calibre nmLVS, Calibre nmLVS-H, and ICtrace Mask mode.

Note:

This statement is provided primarily for backward compatibility. The Pathchk layer statement produces geometric output and is generally more useful for conducting ERC checks.

Usage

ERC PATHCHK
{labeled_flag | power_ground_flag [operand power_ground_flag]}
[POLYGONS NETS | POLYGONS | NETS]
[BY LAYER] [BY CELL | FLAT]
[PORTS ALSO] [NOFLOAT] [BREAK FILTERED]
[EXCLUDE UNUSED] [EXCLUDE SUPPLY]

Arguments

labeled_flag

A string that specifies how the tool outputs polygons or nets that do or do not have a path to a top-level labeled net. This option requires top-level connectivity text objects (such as from Text Layer) to be present that are neither power nor ground net labels. Possible choices are these:

  • LABELED — Reports polygons or nets with a path to a labeled net.

  • ! LABELED — Reports polygons or nets with no path to any labeled net.

The exclamation mark (!) is synonymous with logical inverse. A labeled_flag may not be specified with a power_ground_flag.

power_ground_flag

A string that specifies how the tool reports polygons or nets that do or do not have a path to a top-level supply net. This option requires top-level connectivity text objects that are power or ground net labels. Power and ground nets are top-level nets specified in the LVS Power Name and LVS Ground Name statements. Possible choices are these:

  • POWER — Reports polygons or nets with a path to a power net.

  • ! POWER — Reports polygons or nets with no path to a power net.

  • GROUND — Reports polygons or nets with a path to a ground net.

  • ! GROUND — Reports polygons or nets with no path to a ground net.

The exclamation mark (!) is synonymous with logical inverse.

The labeled_flag and the power_ground_flag keywords when not connected by an operand are called primitive conditions.

operand

An optional string of logical symbols that combine two primitive conditions. Possible choices are these:

  • && — Selects polygons or nets that satisfy both primitive conditions: Boolean AND.

  • || — Selects polygons or nets that satisfy any primitive condition. Boolean OR.

POLYGONS NETS | POLYGONS | NETS

Specifies the type of results file to output. Possible choices are these:

  • POLYGONS — Generates a DRC-style results database in ASCII format that contains the polygon output of all nets that satisfy the specified conditions.

  • NETS — Generates a text file that contains a list of all nets that satisfy the specified conditions.

Both are specified by default.

BY LAYER [BY CELL | FLAT]

An optional keyword set that specifies the type of output to include in a generated ASCII results file. These pertain to the POLYGONS option only:

  • BY LAYER — Generates a separate check for each layer. If both BY CELL and BY LAYER are specified, a check is generated for each layer in each cell. Empty checks are not generated.

  • BY CELL — Causes each cell that has cell-specific results to be checked separately. The check contains polygons from nets reported in that cell. The results are reported in top-level coordinate space. BY CELL is ignored in flat execution.

  • FLAT — Causes cell-specific results to be replicated in every placement. FLAT has no additional effect in flat execution.

PORTS ALSO

An optional keyword that specifies to output top-level polygons or nets that are connected to port objects. By default, top-level polygons or nets connected to port objects are not output. Port objects can be specified with the Port Layer Text or Port Layer Polygon in Calibre and the $make_port() command in Pyxis Layout.

NOFLOAT

An optional keyword that specifies to ignore nets with no device pins.

BREAK FILTERED

An optional keyword that causes the statement to stop path tracing at the pins of devices specified by LVS Filter … OPEN [LAYOUT] [MASK] statements without properties. This keyword can be especially effective used in conjunction with the EXCLUDE UNUSED option.

EXCLUDE SUPPLY

An optional keyword that specifies to exclude power and ground nets from the results of an ERC Pathchk statement containing at least one of the keywords POWER or GROUND (not preceded by the ! operator). Supply nets are included if EXCLUDE SUPPLY is not specified.

EXCLUDE UNUSED

An optional keyword that specifies to exclude polygons or nets connected only to unused devices from the output. Unused devices are specified with LVS Filter Unused Option or equivalent LVS Filter Unused statements. Filter options P and Q are not supported with EXCLUDE UNUSED. Filter options O, H, I, and IH are only supported when LVS Filter Unused Option is not specified with the DEVICES keyword.

Description

Reports layout polygons or nets that do, or do not, have a path to top-level power, ground, or labeled nets.

By default, a path is any route that leads through source/drain pins of built-in MOS devices or pos/neg pins of built-in resistor devices. To qualify, a MOS device (types M, ME, MD, MN, MP, or any of the corresponding LDD-type devices) must have at least the following pins: G (or gate), S (or source), and D (or drain). Paths through LDD-type devices are formed in both directions. Capacitor, diode, or bipolar devices can be added to the path using the ERC Path Also statement.

If you want user-defined devices to participate in ERC Pathchk, then you must map these devices and their pins to built-in devices and pins. This is done using the LVS Device Type specification statement. See “User-Defined Devices in Path Check Operations” in the Calibre Verification User’s Manual.

The hierarchical level from which text objects are read is controlled by the Text Depth statement.

Power and ground nets are specified with LVS Power Name and LVS Ground Name specification statements, respectively. This statement utilizes wildcard characters that appear in LVS Power Name and LVS Ground Name specification statements. Power and ground nets break paths.

Power and ground nets are included in the result for the keywords POWER and GROUND. For example, supply nets are included if you specify:

ERC PATHCHK POWER

You can exclude supply nets by specifying EXCLUDE SUPPLY. When specifying EXCLUDE SUPPLY, at least one of the POWER or GROUND keywords must also be specified in the statement. (The EXCLUDE SUPPLY keyword does not apply to the ! POWER or ! GROUND keywords.) For example:

ERC PATHCHK POWER && ! GROUND EXCLUDE SUPPLY

is allowed. However, this form causes a compiler error:

ERC PATHCHK ! POWER && ! GROUND EXCLUDE SUPPLY // bad syntax

Power and ground nets are not included in the result for ! POWER and ! GROUND keywords. For example:

ERC PATHCHK ! GROUND

does not output the power supply net even if there is no path from power to ground.

If a net in the design matches more than one of LVS Power Name or LVS Ground Name name patterns, a warning is issued and conflicts are resolved with power names being given priority over ground names.

The LABELED nets have connectivity extraction text but are not power or ground nets.

By default, shapes (or nets) from top-level nets connected to port objects are not output. The PORTS ALSO option changes this behavior so that such shapes are output.

ERC Pathchk statements respect the LVS Compare Case statement. If either the LVS Compare Case YES or LVS Compare Case NAMES statement is specified, then net names are traced in a case-sensitive manner; otherwise they are case insensitive. These rules on case sensitivity also apply to LVS Power Name and LVS Ground Name matching of nets.

ERC Pathchk observes the Layout Preserve Case specification statement for case-sensitivity of net names. See “Case Sensitivity of Supply Net Names” in the Calibre Verification User’s Manual for more details.

ERC Pathchk applies to Calibre nmLVS/Calibre nmLVS-H and ICtrace Mask mode. It is executed during circuit extraction and only when the layout is geometric. The following Calibre commands execute this statement:

calibre -lvs -flatten rules    // executed flat
calibre -lvs -tl ... rules     // executed flat
calibre -lvs -hier ... rules   // executed hierarchically with 
                               // geometric Layout System
calibre -spice ... rules       

The following Pyxis Layout commands execute this statement:

$lvs_mask()           // executed flat
$write_mask_cnet()    // executed flat

Note the following important considerations:

  • The POWER, GROUND, and LABELED options (without the ! operator) are not recommended because they can generate very large files, consume large amounts of memory, and lead to long run times. In hierarchical ERC, such requests essentially flatten the design hierarchy.

  • In the default mode, with neither FLAT nor BY CELL specified, the polygon output may contain polygons belonging to a cell in two ways:

    • If a net in the cell violates ERC Pathchk requirements in some placements of a cell, the polygons are written out for each placement where the violation occurs in top-level coordinates.

    • If, for a given net, a violation appears in every placement of a cell, the polygons are written out once, in the lowest, leftmost placement.

      This can lead to confusion because, in the same cell, some polygons may be displayed in many placements and others in only one placement. If the results are difficult to interpret, the BY CELL option can be useful.

The FLAT option can also be used to force Calibre to output cell-specific results in all placements. This increases both memory consumption and the size of the polygon file, but can eliminate potential confusion caused by hierarchical output. However, output using the NETS option is unambiguous in all cases.

ERC Pathchk statements are not executed if the specified net is not present. For example, if there is no power net in the layout, then ERC PATHCHK ! POWER generates an error message and generates no output.

The default behaviors of ERC Pathchk and Calibre PERC with regard to paths and break nets are different. ERC Pathchk has a default definition of “path”. In Calibre PERC, you provide the definition of path explicitly, otherwise there are no paths. ERC Pathchk breaks paths at LVS Ground Name and LVS Power Name nets. In Calibre PERC, you define net types and then explicitly specify which net types break paths.

Results Output Options

The POLYGONS option specifies a file named layout_primary.pathchk.erc. It is placed in the Mask SVDB Directory if that statement is specified, otherwise in the current working directory. If layout_primary is not specified, the word “lvs” is used instead. The file consists of lines in the format:

ERC PATHCHK condition [ in cell cell_name ] [ layer_name ]

where condition is the specified condition, such as ! POWER or ! POWER && GROUND. By default, results from each ERC Pathchk statement are written to one check in the result file. If the BY CELL option is specified, then the results are classified by cell and the cell_name field indicates the cell name. If the BY LAYER option is specified, then the results are classified by layer and the layer_name field indicates the layer. Empty checks are not written. If all checks are empty, no file is created.

By default, each ERC Pathchk statement’s result output is limited to 12.5 million polygons. This can be changed through the LVS Pathchk Maximum Results statement.

All polygon results are shown in the top-level coordinate space, and they are written to a DRC-style results database in ASCII format readable by Calibre RVE. By default, hierarchical Calibre shows cell-specific results only once in a representative placement of the cell. This is similar to Calibre nmDRC-H. Results are cell-specific if they can be generated locally in the cell and occur in all placements of the cell in the design.

Note that polygon results generated by ERC Pathchk are not fully merged. This means that distinct polygons can abut or overlap.

The NETS option specifies a file named layout_primary.pathchk.rep. It is placed in the Mask SVDB Directory if specified, otherwise in the current working directory. If layout_primary is not specified, the word “lvs” is used instead.

In hierarchical execution, the report file format is this:

ERC PATHCHK REPORT for layout_primary
  ERC PATHCHK condition: 
    cell cell_name (placement): net1, net2, ...
    cell cell_name (placement): net1, net2, ...

where condition is the specified condition, such as GROUND AND ! POWER, cell_name is a cell name where results were found, placement is a representative placement of that cell, and neti is a net name or number. Cells are listed in bottom-up order. Net names and numbers are in the context of the indicated cells. Cells for which no nets are found by ERC Pathchk are not written to the report file.

In flat execution, the report file format is this:

ERC PATHCHK REPORT for layout_primary
  ERC PATHCHK condition: 
    net1, net2, ...

Nothing is reported for requests in which no nets are found in any cell. If all checks are empty, no file is created.

Requirements and Restrictions

Hierarchical Calibre operations impose a size limit on intermediate data structures generated by ERC Pathchk. The limit is approximately 400 MB. This limit is applied independently to each individual ERC Pathchk statement and applies only to POLYGONS output. Calibre generates incomplete results (some polygons may be omitted from the output) and issues the following warning if Calibre reaches this limit:

WARNING: ERC operation exhausted allowed memory allocation.

Calibre issues the warning in both the transcript and the circuit extraction report. You should not reach this limit in normal operations. You see it only in circumstances when a very large net is selected by the ERC Pathchk statement. Calibre always completes NETS option reporting.

See also ERC Summary Report. The search features in Calibre RVE for LVS may also be of interest—see “Find Path” and “Isolate Path on Layout Net” in the Calibre RVE User’s Manual.

Examples

Example 1

The following statement outputs a text file listing all the nets without a path to a top-level net that is not a supply net. The port net is also part of the output.

ERC PATHCHK ! LABELED NETS PORTS ALSO

Example 2

When not using the [ ! ] LABELED option, the following combinations are possible:

! POWER && GROUND

Nets with no path to power but with a path to ground

! GROUND && POWER

Nets with no path to ground but with a path to power

! GROUND && ! POWER

Nets with no path to ground and no path to power

! GROUND || ! POWER

Nets with no path to ground or no path to power (or both)

POWER && GROUND

Nets with a path to both power and ground

POWER || GROUND

Nets with path to power or a path to ground
