SVRF Rule File Format

Input for: Calibre and ICverify tools

A SVRF rule file includes elements such as layer assignments and definitions, rule checks, comments, specification statements, device recognition statements, conditionals, and macros.

Format

The following broad categories of elements are found in an SVRF rule file:

Layer Assignments

Global Layer Definitions

Comments

Include Statements

Rule Check Statements {

     Local Layer Definitions

     Layer Operations

     Rule Check Comments

}

Specification Statements

Connectivity Extraction Operations

Parasitic Extraction Statements

Device Recognition Operations

Conditionals

Macros

Runtime TVF Functions

See “General Syntactic Conventions” and “SVRF Symbols and Command Parsing” for complete details on the rule file syntax.

Description

Layer assignments and definitions

Layer assignments are made with the Layer specification statement. For example:

LAYER POLY 1
LAYER OXIDE 2

In these statements, layer numbers are assigned to layer names (usually called original or drawn layers). Layer name reassignments are not allowed.

Layer sets are layers with multiple layer numbers or layer names assigned to them. For example:


LAYER MET1 1 2
LAYER ALL_MET MET1 MET2 MET3

In general, layer sets behave like simple layers with few exceptions. Exceptions are discussed in the relevant statements.

Layer definitions are a means for deriving layers from original layers. Once named, the derived layer can then be used in another operation in the rule file by referencing its name. A layer definition has the form:

name = layer operation

Layer definitions within a rule file can exist either inside or outside of a rule check statement. Layer definitions that are not inside of a rule check statement are said to be global. That is, the names of the layers that they define can be used in operations anywhere within the rule file. A layer definition that is inside of a rule check statement is said to be local to that rule check statement, that is, the layer name that it defines is not recognized outside of the rule check statement.

In local scope, the layer names can be reused in different rule check statements. No two global layer definitions can share a name, nor can local layer definitions within the same rule check statement share a name. Any local layer definition supersedes a global definition of the same name within the rule check statement where it appears.

Implicit layer definitions

An implicit layer definition consists of a matching set of parentheses enclosing one layer operation. The layer operation may, in turn, have inputs that are implicitly derived. An implicit layer operation may be used as input to any operation. Example:

taps = (pdiff AND (bulk NOT nwell)) OR 
       (ndiff AND nwell)

Here, there are two implicit layer definitions used as inputs to the OR operation. The first AND operation also has an implicit layer definition as input.

Implicit layer definitions can be used in rule checks also. Example:

rule {EXT < 0.1 (poly NOT oxide) oxide}

Implicit layer definitions cannot be used in the Enclosure, External, and Internal operations when parentheses ( ( ) ) and brackets ( [ ] ) are used as derived edge layer operators.

Comments

Double slash (//) characters start a rule file comment that terminates at the end of the line on which they occur. Multi-line C-style comments (/* … */) are also permitted.

For comments that occur within a rule check (within the braces), the at ( @  )character can be used. These comments can be displayed by Calibre RVE.

The caret (^) character may be used in a rule check comment to dereference the value of a variable (first value for an array) in the comment. For example, if my_var is a numeric variable, then the following example causes the value of my_var to appear in the rule check output.

@ spacing is within ^my_var

The caret (^) can be escaped (treated as literal) with the following methods:

  • In a rule check check (@) comment by using \^

  • In a DFM RDB COMMENT by using ^^

  • In an SVRF Message statement by using \\^

Include statements

Inserts another rule file within a rule file at the location of the Include statement. This statement behaves exactly as if all of the text of the rule file with the given filename were directly included in the rule file where the statement appears. The format is this:


INCLUDE filename

In addition, multiple include statements can appear and the include depth is arbitrary. Recursive calling of included files is not allowed.

Layer operations

These commands manipulate layer data and are commonly used for layer derivation and rule checking. They are often referred to simply as operations. This is in distinction to specification statements, which is the other major command category. See “Layer Operations”.

Rule check statements

Particular to Calibre nmDRC-style applications, rule check statements specify layer operations whose resulting derived layers are placed into the results database when the rule check statement is executed by the application. Rule checks have the following basic form:

    name {

        @ rule check comment

        layer operation or layer definition

        …

        layer operation or layer definition

        layer operation

    }

No two rule check statements in the rule file can share a name. Rule check statements do not need to span multiple lines nor contain comments. A rule check must have at least one standalone layer operation (that is, one operation that is not a layer definition of the form x = layer operation) that has output to the results database, otherwise the rule check will not compile. Rule check names may not use special symbols unless the name is enclosed in quotation marks.

Specification statements

Specification statements describe the overall environment for applications using the rule file. They establish input and output flow and control how various algorithms perform their tasks. See “Specification Statements”.

Connectivity extraction operations, Device operation, and Parasitic extraction operations

These statements control connectivity extraction, device definitions, and parasitic extraction. These are discussed in “Summary of Rule File Elements”.

Conditionals

Conditionals, or pre-processor directives, control how a rule file is compiled. They provide control over which parts of a rule file are used. See “Pre-Processor Directives”.

Macros

Macros are constructs that allow repeated use of SVRF operations in a type of template. Macros can be called multiple times in a rule file. See “Macros”.

Runtime TVF functions

Runtime TVF requires the presence of an SVRF element called TVF FUNCTION to define operations to perform during Calibre runtime. See “Runtime TVF” for more information.
