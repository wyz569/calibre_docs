Pathchk

Layer operation

Constructs layers consisting of polygons that do or do not have a path to power, ground, or labeled nets, or that satisfy a combination of these conditions. Also can output information about nets that satisfy these conditions. Used during circuit extraction in Calibre nmLVS, Calibre nmLVS-H, and ICtrace Mask mode.

Usage

PATHCHK {labeled_flag | {power_ground_flag [operand power_ground_flag]}} [layer]
[PRINT POLYGONS “filename” [BY LAYER] [BY CELL | FLAT]]
[PRINT NETS “filename”]
[PORTS ALSO] [NOFLOAT] [BREAK FILTERED]
[EXCLUDE SUPPLY] [EXCLUDE UNUSED]
[POWERNAME “name” [“name” …]]
[GROUNDNAME “name” [“name” …]]
[LABELNAME “name” [“name” …]]
[BREAKNAME {“break_net” | “cell_name ‘(’ break_net [break_net …] ‘)’ ”} …]

Arguments

labeled_flag

A string that specifies how the tool outputs polygons or nets that do or do not have a path to a top-level labeled net. This option requires top-level connectivity text objects (such as from Text Layer) to be present that are neither power nor ground net labels. Possible choices are these:

  • LABELED — Reports polygons or nets with a path to a labeled net.

  • ! LABELED — Reports polygons or nets with no path to any labeled net.

The exclamation mark (!) is synonymous with logical inverse.

power_ground_flag

A string that specifies how the tool outputs polygons or nets that do or do not have a path to a top-level power or ground net. Power and ground nets are top-level nets specified in the LVS Power Name and LVS Ground Name statement. Possible choices are these:

  • POWER — Reports polygons or nets with a path to a power net.

  • ! POWER — Reports polygons or nets with no path to a power net.

  • GROUND — Reports polygons or nets with a path to a ground net.

  • ! GROUND — Reports polygons or nets with no path to a ground net.

The exclamation mark (!) is synonymous with logical inverse.

The power_ground_flag keyword when not combined by an operand is called a primitive condition.

operand

An optional logical symbol that combines two power_ground_flag conditions. Possible choices are these:

  • && — Selects polygons or nets that satisfy both primitive conditions: Boolean AND.

  • || — Selects polygons or nets that satisfy any primitive condition: Boolean OR.

layer

An optional layer name that directs the Pathchk operation to output polygons only from the specified layer. The layer must have connectivity established. When you do not specify the layer parameter, the Pathchk operation outputs merged polygons from all layers specified in Connect and Sconnect statements.

PRINT parameters are not affected by the layer parameter.

PRINT POLYGONS “filename” [BY LAYER] [BY CELL | FLAT]

An optional keyword followed by a file pathname generates polygon output of all nets that satisfy the Pathchk conditions. The filename is generated in DRC results ASCII format. By default, all results from each Pathchk operation are written to one check in the result file. BREAKNAME nets are not included in PRINT POLYGONS output.

The filename parameter can contain environment variables. For information regarding the use of environment variables in the filename parameter, refer to “Environment Variables in Pathname Parameters” in the chapter “Key Concepts”.

The following keywords may be specified with PRINT POLYGONS; they have no effect on PRINT NETS output:

  • BY LAYER — An optional keyword that classifies PRINT POLYGONS results by layer. The tool creates a separate check for each layer. The check contains net polygons reported for the layer. All results are reported in top-level coordinate space.

    The tool generates a check for each layer in each cell when you specify BY LAYER and BY CELL in the same statement.

  • BY CELL — An optional keyword that classifies PRINT POLYGONS results by cell. The tool creates a separate check for each cell that has cell-specific results. The check contains net polygons reported in that cell. All results are reported in top-level coordinate space. Flat applications ignore this keyword.

  • FLAT — An optional keyword that specifies PRINT POLYGON results to be flat with cell-specific results replicated in every placement. This keyword has no effect in flat applications and cannot be specified with BY CELL.

Creation of empty rule checks by PRINT POLYGONS is controlled by the ERC Keep Empty statement. The output filename is not created if all rule checks are empty and ERC Keep Empty NO is specified.

PRINT NETS “filename”

An optional keyword followed by a file pathname that generates output of all nets that satisfy the Pathchk conditions to the file.

In hierarchical execution, the report file format is this:

PATHCHK REPORT for <layout primary>
PATHCHK <condition>:
  cell <name> (<placement>): net1, net2, ...
        cell <name> (<placement>): net1, net2, ...

where condition is the Pathchk condition (for example, GROUND && ! POWER), name is a cell name where results were found, placement is a representative placement of that cell, and netN is a net name or number if the net has no name. Cells are listed in bottom-up order. Net names and numbers are in the context of the indicated cells. Cells for which no nets are found by Pathchk are not written to the report file.

In flat execution, the report file format is this:

PATHCHK REPORT for <layout primary>
PATHCHK <condition>:
net1, net2, ...

If for a particular request no nets are found (in any cell), nothing is written out for that request. If all requests generate empty results, the report file is not written.

PORTS ALSO

An optional keyword that specifies to output polygons or nets that are connected to top level port objects and that meet the other conditions of the operation. By default, polygons or nets that are connected to top level port objects are not included in the output. This option affects both PRINT POLYGONS and PRINT NETS output. Port objects can be specified with Port Layer Text or Port Layer Polygon in Calibre and with $make_port() in Pyxis layout.

NOFLOAT

An optional keyword that instructs the statement to ignore nets not connected to any devices.

BREAK FILTERED

An optional keyword that causes the operation to stop path tracing at the pins of devices specified by LVS Filter … OPEN [LAYOUT] [MASK] statements without properties. This keyword can be especially effective used in conjunction with the EXCLUDE UNUSED option.

EXCLUDE SUPPLY

An optional keyword that excludes power and ground nets from the results of a Pathchk operation containing at least one of the keywords POWER or GROUND (and not preceded by the ! operator). Supply nets are included if EXCLUDE SUPPLY is not specified.

EXCLUDE UNUSED

An optional keyword that excludes polygons or nets connected only to unused devices from Pathchk output. Unused devices are specified with LVS Filter Unused Option or equivalent LVS Filter Unused statements. Filter options P and Q are not supported with EXCLUDE UNUSED. Filter options O, H, I, and IH are only supported when LVS Filter Unused Option is not specified with the DEVICES keyword.

POWERNAME “name”

An optional keyword and parameter that specifies which nets should serve as power nets in the current Pathchk operation. It overrides the LVS Power Name list in the current operation. This option can only be specified when the [!] POWER keyword is used. More than one name may be specified.

GROUNDNAME “name”

An optional keyword and parameter that specifies which nets should serve as ground nets in the current Pathchk operation. It overrides the LVS Ground Name list in the current operation. This option can only be specified when the [!] GROUND keyword is used. More than one name may be specified.

LABELNAME “name”

An optional keyword and parameter that specifies which nets should be treated as labeled in the current Pathchk operation. This option can only be specified when the [!] LABELED keyword is used. More than one name may be specified.

The string %_p_ can be used as the name to specify all ports in the top cell. Top-level port names are substituted automatically.

BREAKNAME {“break_net” | {“cell_name ‘(’ break_net [break_net …] ‘)’ ”}} …

An optional keyword and parameter group that specifies the names of nets that break paths. The net names can be specified for individual cells. The parameters are as follows:

  • break_net — A name of a net that breaks paths to supply or labeled nets. If the only path to some net N goes through at least one break_net, then N is considered as not having a path to the supply nets or labeled pads. If specified without a cell_name parameter, the break_net is considered to be a top-level net and breaks paths in all instantiated hierarchy. A list of break_net names may be specified. The “?” wildcard character matches zero or more characters in a net name.

  • cell_name — A name of a cell in which specified break_net names are found. When cell_name is specified, the break_net names associated with it must appear in a space-delimited list in parentheses. In this case, a break_net has the effect of breaking paths in the cell in which it is specified and in that cell’s sub-hierarchy. However, no breaking occurs on nets connected to a break_net at a higher level of hierarchy. More than one cell_name group may be specified. No wildcards may be used for cell names in this case. Cell names are always case-insensitive.

See the example with Figure 1 for one use of the BREAKNAME keyword.

Description

This operation constructs layers consisting of layout net polygons that do or do not have a path to top-level power nets, ground nets, labeled nets, or nets that satisfy a combination of these conditions. This layer operation is not node-preserving.

Note:

You cannot use a derived layer from a Pathchk operation in Connect, Sconnect, or Device operations, nor use such a layer to derive other layers that appear in these operations.

By default, a path is any route that leads through source/drain pins of built-in MOS devices or pos/neg pins of built-in resistor devices. To qualify, a MOS device (types M, ME, MD, MN, MP, or any of the LDD-type devices) must have at least the following pins: G (or gate), S (or source), and D (or drain). Paths through LDD-type device pins are formed in both directions. Capacitor, diode, or bipolar devices can also be added to the path as specified in ERC Path Also statements.

If you want user-defined devices to participate in Pathchk (user-defined devices do not participate by default), then you must map these devices and their pins to built-in devices and pins. This is done using the LVS Device Type specification statement. See “User-Defined Devices in Path Check Operations” in the Calibre Verification User’s Manual.

The hierarchical level from which text objects are read is controlled by the Text Depth statement.

Power and ground nets are specified with LVS Power Name and LVS Ground Name, respectively. Power and ground nets break paths in Pathchk (but not by default in Calibre PERC).

LABELED nets are those with connectivity extraction text, but they are not power or ground nets. LABELED nets typically belong to top-level I/O ports.

By default, top-level shapes or nets that are connected to port objects are not output. The PORTS ALSO option changes this behavior so that ports are included.

Pathchk respects the LVS Compare Case statement setting when tracing net names. If either the LVS Compare Case YES or LVS Compare Case NAMES statement is specified, then net names are traced in a case-sensitive manner; otherwise they are case insensitive. These rules on case sensitivity apply to LVS Power Name and LVS Ground Name nets, and to names specified in POWERNAME, GROUNDNAME, LABELNAME, and BREAKNAME options.

By default, each Pathchk operation’s result output is limited to 12.5 million polygons. This can be changed through the LVS Pathchk Maximum Results statement.

The path search features in Calibre RVE for LVS may also be of interest—see “Find Path” and “Isolate Path on Layout Net” in the Calibre RVE User’s Manual.

Supply Name Behaviors

Power and ground nets are included in the result for the keywords POWER and GROUND. For example, supply nets are included if you specify the following in a rule check:

PATHCHK POWER

You can exclude supply nets by specifying EXCLUDE SUPPLY. When specifying EXCLUDE SUPPLY, at least one of the POWER or GROUND keywords must also be specified in the operation. The EXCLUDE SUPPLY keyword does not apply to the ! POWER or ! GROUND keywords. For example, this operation is allowed:

PATHCHK POWER && !GROUND EXCLUDE SUPPLY

However, this syntax causes a compiler error:

PATHCHK ! POWER && ! GROUND EXCLUDE SUPPLY // bad syntax

Power and ground nets are not included in the result for ! POWER and ! GROUND keywords. For example, the supply net itself is not selected by this:

PATHCHK ! GROUND

even if there is no path from power to ground.

If a net in the design matches more than one of LVS Power Name, LVS Ground Name, POWERNAME, GROUNDNAME, or LABELNAME patterns, a warning is issued and the conflicts are resolved according to the following priority rules:

  • Pathchk keyword overrides are given priority over LVS Power Name or LVS Ground Name settings.

  • Power names are given priority over ground names.

Pathchk observes the Layout Preserve Case specification statement for case-sensitivity of net names. See “Case Sensitivity of Supply Net Names” in the Calibre Verification User’s Manual for more details about power and ground names.

BREAKNAME Behavior

BREAKNAME nets are traced to GROUND, POWER, and LABELED nets. BREAKNAME nets are included in the normal output of the operation when they are traced to nets satisfying the criteria of the operation. BREAKNAME nets are also reported in PRINT NETS results, but they are not reported in PRINT POLYGONS results.

BREAKNAME nets interrupt paths at the primary level of a BREAKNAME cell (top level by default) and throughout the hierarchy of such a cell. BREAKNAME nets do not interrupt paths in the hierarchy above a BREAKNAME cell.

Pathchk operation criteria can be mimicked in Calibre PERC initialization procedures; however, the results from the two tools can differ. For example, this Pathchk operation:

LVS POWER NAME VCC

PATHCHK POWER POWERNAME VCC BREAKNAME C1(netA)

performs a similar function as this Calibre PERC code:

LVS POWER NAME VCC

TVF FUNCTION erc_pathchk [/*
  package require CalibreLVS_PERC

  proc init {} {
    perc::define_net_type "Power" {VCC}
    perc::define_net_type "stop" {netA} -cellName {C1}
    perc::create_lvs_path -break {Power || stop}
}

proc test {} {
  perc::check_net -pathType {Power} 
}

(The handling of ground nets has been omitted here, but it can be handled similarly to Power.) Pathchk breaks paths at supply nets but Calibre PERC does not by default, so the perc::create_lvs_path -break option is used. You exclude the VCC net in the “test” proc results by adding the “‑exclude {Power}” option to perc::create_lvs_path. Pathchk includes supply nets in the output by default, but they can be omitted with the EXCLUDE SUPPLY keyword.

Execution of Pathchk Operations

Pathchk is used by Calibre nmLVS/Calibre nmLVS-H and Mask-mode ICverify. It is executed during circuit extraction and only when the layout is geometric. You must use ERC Select Check to enable ERC rule checks in an LVS run. LVS Execute ERC NO disables ERC checks that would otherwise be performed. The following Calibre commands execute this statement:

calibre -lvs -flatten       // executed flat
calibre -lvs -tl            // executed flat
calibre -lvs -hier          // executed hierarchically and 
                            // Layout System is geometric
calibre -spice              

The following Pyxis Layout commands execute this statement:

$lvs_mask()        // executed flat
$write_mask_cnet() // executed flat

Note these important considerations:

  • The POWER, GROUND, and LABELED options (without the ! operator) are not recommended because they can generate very large files, consume large amounts of memory, and cause long runtimes. In hierarchical execution, such requests flatten the design hierarchy.

  • Pathchk operations are not executed if the specified net is not present. For example, if there is no power net in the layout, then Pathchk ! POWER generates a warning message and produces no output. ERC rule checks involving Pathchk operations that abort are annotated in the ERC Summary Report as follows:

    NOT EXECUTED DUE TO PATHCHK ERRORS

    Rule checks involving layers derived from Pathchk operations that abort are annotated in the ERC Summary Report as follows:

    FAILED PATHCHK IN LAYER DERIVATION

    Rule checks that fail with either of the preceding annotations receive the following comment in the ERC Results Database:

    ERROR IN PATHCHK

  • LVS Power Name and LVS Ground Name nets always break paths regardless of POWERNAME or GROUNDNAME overrides. In addition, nets specified in POWERNAME and GROUNDNAME break paths as well.

  • Determination of power and ground names for the purpose of the EXCLUDE UNUSED option is controlled exclusively by the LVS Power Name and LVS Ground Name specification statements; the POWERNAME and GROUNDNAME overrides do not apply.

  • The POWERNAME, GROUNDNAME, and LABELNAME options can be specified in any order, but must be the last options of the Pathchk statement. They work both flat and hierarchically, and they do not apply to ERC Pathchk statements.

  • In Calibre nmDRC/Calibre nmDRC-H and ICrules, the Pathchk operation produces an empty result layer and generates a warning, and PRINT options are not executed.

  • Pathchk includes floating nets when they fit the specified request and you do not specify the NOFLOAT parameter. For example, the following statement’s output includes any floating nets that exist:

    PATHCHK ! POWER

    while the following statement’s output does not include any floating nets:

    PATHCHK ! POWER && GROUND NOFLOAT

  • In the default mode (neither FLAT nor BY CELL specified), the PRINT POLYGONS output may contain polygons belonging to a cell in two ways.

    • If a net in the cell violates the specified requirements in some but not all placements of this cell, the polygons are written out for each placement where the violation occurs in top-level coordinate space.

    • If, for a given net, the violation occurs in all placements of the cell, the polygons of this net are written out only once in the lowest leftmost placement of the cell.

    This could be difficult to interpret because, in the same cell, some polygons may be displayed in many placements while other polygons only appear in one placement.

    If the results are difficult to interpret, you can use the BY CELL option to separate cell-specific results in different checks. The FLAT option can also be used to force Calibre to output cell-specific results in all placements. This eliminates any confusion, but increases both memory consumption and the size of the polygon file. Output using the PRINT NETS option is unambiguous.

  • Results of the operation are optimized for the different outputs produced. Specifically, the output layer produced from a given input layer preserves hierarchy as much as possible, resulting in more compact data with less promotion. Such operations are most commonly used as part of a more complex rule file script where the result layer serves as an input layer for other layer operations involved in ERC checks.

  • The output layer of the operation without an input layer parameter specified, which is commonly used for results presentation, is optimized for readability and clarity. For each net selected by Pathchk which is a pin of a cell, all geometries are promoted to the containing cell and reported in its context as a part of the parent net. Output generated by ERC Pathchk and the PRINT POLYGONS option of Pathchk is optimized in the same way, since it is also intended for results viewing. In no case are the same polygons reported in both child and parent cells.

Performance Considerations

Pathchk operations that specify the same set of net name overrides (POWERNAME, GROUNDNAME, LABELNAME) are executed concurrently as a single operation with multiple output layers. The concurrency of Pathchk operations saves some computations that are the same for all concurrent operations.

For example, if the rule file contains the following Pathchk operations:

PATHCHK POWER &&  GROUND POWERNAME VDD       // 1
PATHCHK POWER && !GROUND POWERNAME VDD       // 1
PATHCHK POWER pgate POWERNAME VDD            // 1
PATHCHK POWER ngate POWERNAME VDD            // 1
PATHCHK POWER &&  GROUND GROUNDNAME VSS      // 2
PATHCHK POWER && !GROUND GROUNDNAME VSS      // 2
PATHCHK POWER pgate GROUNDNAME VSS           // 2
PATHCHK POWER ngate GROUNDNAME VSS           // 2
PATHCHK POWER &&  GROUND                     // 3
PATHCHK POWER && !GROUND                     // 3
PATHCHK POWER pgate                          // 3
PATHCHK POWER ngate                          // 3

then three concurrent sets are executed as shown by the comments above. Notice that Pathchk operations with no overrides are executed in a separate concurrency set.

Hyperscaling does not employ concurrency by default in all HDBs. To enable this, use the ‑hyper pathchk command line option. Be aware that enabling concurrency in all HDBs requires additional memory.

Hierarchical Calibre operations impose a size limit on intermediate data structures generated by Pathchk. The limit is approximately 400 MB and should not be reached under normal conditions. It is applied independently to each Pathchk operation. If the limit is reached, Pathchk generates incomplete results (some polygons may be omitted from the output) and issues the following warning:

WARNING: ERC operation exhausted allowed memory allocation.

Calibre issues the warning in both the transcript and the circuit extraction report. This limit applies only to polygon output (that is, construction of a result layer) and the PRINT POLYGONS option. Net reporting from the PRINT NETS option is always complete.

This limit exists to protect against excessive memory usage in rare cases when a very large net is selected by Pathchk (for example, a power or ground net), and cases where a very large part of the design is selected.

Positive requests like “Pathchk POWER” can generate very large files, consume much memory, and take a long time to complete. In particular, in hierarchical execution such positive requests essentially flatten the design hierarchy. Hence, these types of operations are typically best avoided.

In certain situations, the Pathchk operation consumes large amounts of memory and time if the operation generates large results, and those results are output directly to the results database. For example, consider a design where the power pad is not properly connected to the actual power net, so that a check like this: 

ERR { PATHCHK ! POWER }

attempts to return essentially the entire (disconnected) power routing. This can be very time-consuming, especially when the power routing has the form of a grid. The result can be a huge, flat polygon consisting of all the power routing layers merged together. In such cases, the Pathchk single-layer mode (using the layer argument and writing individual per-layer checks) is likely to reduce resource consumption. Another solution is to use the PRINT NETS option instead of writing the full shapes to the results database. Also, the ERC Pathchk specification statement is better optimized to handle such cases.

Note that if the Pathchk result is operated upon before being output to the results database, then resource consumption may not be an issue. For example, the following is likely to work because the final amount of data written to the results database is reduced:

A = PATHCHK ! POWER 
ERR { A AND CONTACT }

Examples

Example 1

The following example creates a derived layer named erc_poly_layer containing a representation of all shapes on the ipoly layer with no path to POWER.

erc_poly_layer = PATHCHK ! POWER ipoly

Example 2

The following example creates a file named pathchk_no_pwr_nets containing output of all nets with no path to POWER.

erc_rule {PATHCHK ! POWER PRINT NETS "pathchk_no_pwr_nets"}

Example 3

See Example 2 under ERC Pathchk for a discussion of supply-related option combinations.

Example 4

This shows a check for polygons on the ipoly layer that do not have paths to top-level labeled nets in1a and in1b. Shapes from top-level nets connected to port objects are included.

pathchk_not_labelname{
    PATHCHK ! LABELED ipoly PORTS ALSO LABELNAME in1b in1a
}

Example 5

This example shows a check for gates that are not connected either to power or ground.

CONNECT gate poly
gate_not_pwr_or_gnd { PATHCHK !POWER && !GROUND gate }
LVS EXECUTE ERC YES
ERC SELECT CHECK gate_not_pwr_or_gnd

Example 6

This example illustrates the use of the BREAKNAME keyword. In Figure 1, net B1 has a path to the ground signal VSS in the top cell through two resistors. Net B2 has a path to VSS through two resistors in cell C2, as well as through a top-level resistor.

Figure 1. Pathchk BREAKNAME

You can use Pathchk to find all nets with a path to ground (VSS) and designate net A as a break net in cell C1: 

C1_break {PATHCHK GROUND BREAKNAME C1(A)}

In this case, net B1 has a path to ground because the break net A in cell C1 has no effect outside of cell C1. However, net B2 does not have a path to ground because net A breaks paths in cell C1 and C2, which is contained in C1.
