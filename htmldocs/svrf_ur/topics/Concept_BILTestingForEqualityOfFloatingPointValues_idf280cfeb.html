Testing for Equality of Floating-Point Values

Device property computation built-in language.

Testing for equality of floating-point numeric values using the == operator requires care because the floating-point representation of a number is not necessarily equal to the arithmetic value. Additionally, floating-point representations of numbers can vary depending on the operating system and host hardware. The solution for equality testing is to check if values are within a certain tolerance range of each other that guarantees the necessary precision for the evaluation of equivalence.

Suppose a layout measurement value is 0.5 um and you want to test if a parameter has that value in a property computation. You might try this:

if (W == 0.5 * 1e-06) { ...

A potential problem here is W may be 0.5e-06 exactly, but “0.5 * 1e-06” might be stored as a floating-point value of 0.49999999999999919. In that case, the equality test fails.

A consideration in device property computation is the precision at which properties are traced and compared in LVS. The Trace Property statement checks for property equivalence using a value of base × 1e-05, where base is the source design’s value. So equivalence should be checked to at least that precision.

With this in mind, the aforementioned equality test can be performed as follows:

if |a - b| < minimum(a, b) × 1e-05 then ...

In a device property computation program, this might be implemented as in this example:

DEVICE ... [
  PROPERTY W ...

  W = 0.5 * PERIM_CO(dev_layer,pin_layer)
  W_exp = 0.5 * 1e-06

  if (W <= W_exp) {
    min = W
  } else {
    min = W_exp
  }

  if (ABS(W - W_exp) < min * 1e-05) { ...
]

In the TVF functional interface for device property computation, Tcl is used. In that interface, this code performs the same role as the code discussed previously:

if {[expr abs($x - $y) <  min($x,$y) * 1e-05]} { ...
