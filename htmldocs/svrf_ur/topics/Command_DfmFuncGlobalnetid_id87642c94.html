GLOBALNETID

Net based function for DFM expressions

Returns the net ID associated with a geometry on the input layer as type netID. Supported only for DFM Property and DFM Property Merge operations. The netID returned is based upon the cell placement and provides a netID that can be compared to other netIDs in the context of a cell. Unlike the NETID function, GLOBALNETID takes high shorts into account.

Usage

GLOBALNETID(layer[,ordinal])

Arguments

layer

A required layer with connectivity information; may not be an error layer.

ordinal

An optional argument that refers to the index of the secondary layer geometry. The ordinal is required if layer is a secondary layer unless GLOBALNETID(layer) is an argument to VNETID.

The numbering sequence of all secondary layer geometries clustered with a given primary layer is arbitrary, but is consistent for all expressions within a DFM Property operation. The ordinal numbers start with 1. If the ordinal number exceeds the number of geometries in the cluster, the expression evaluation fails; evaluation failure also occurs if the primary layer object has no secondary layer objects clustered with it.

If ordinal is specified and layer is the primary layer, the value of ordinal must be 1.

Return Values

Net IDs of type netID.

Description

GLOBALNETID returns the netID associated with a geometry on the layer argument. This function is supported only in DFM Property and DFM Property Merge operations. The INSIDE OF keyword set for DFM Property cannot be used with the GLOBALNETID function.

The netID returned is based upon the cell placement and provides a netID that can be compared to other netIDs in the context of the cell in which the property expression is evaluated; however, GLOBALNETID does not produce unique netIDs across the complete design hierarchy. Unlike the NETID function, GLOBALNETID considers connectivity at all levels in the hierarchy, and therefore can be used to identify high shorts. In particular, GLOBALNETID values that are not equal in the context of a cell are not connected for any placement of that cell; this is not true for the NETID function.

The function behaves as follows, depending on the layer argument and whether ordinal is used:

  • GLOBALNETID(primary_layer) — Returns the netID of the primary layer object.

  • GLOBALNETID(secondary_layer) — If a secondary layer is referenced without an ordinal, the function must be used as an argument to VNETID; for example, VNETID(GLOBALNETID(secondary_layer)). VNETID creates a vector of netID values with the number of tuples equal to the number of secondary layer objects clustered with the primary layer object, where the tuple value is the netID for the secondary layer object.

  • GLOBALNETID(secondary_layer, ordinal) — Returns the netID associated with a particular secondary layer geometry clustered with the primary layer, where ordinal gives the index of the secondary layer geometry.

Values returned by GLOBALNETID can only be compared within the context of the cell that the property evaluation takes place in; this is always the case when the comparison is made within the property expression. When using Calibre RVE to view netID property values, care must be taken to only compare GLOBALNETID values from results reported in the same cell.

The following cases illustrate some the of concepts regarding promotion and comparing netIDs.

Case 1

The layout has two placements of cellA, where cellA has three layer1 polygons. One placement of cellA has a layer2 polygon in the topcell shorting two layer1 polygons in cellA. The following operation is used:

CONNECT layer1 layer2
gnet_out = DFM PROPERTY layer1 [g_net = GLOBALNETID(layer1 )]

The results for the long rectangular polygons are reported in the top cell because the layer2 polygon in the topcell causes promotion. The result for the two small square polygons is reported in cellA; the netIDs for the two polygons cannot be compared because polygons are in different cell placements. When saved to a hierarchical results database (RDB), the result for the small square layer1 polygon is reported as one hierarchical result in cellA.

Case 2

This case is the same as Case 1, except that the two small layer1 squares are shorted by layer2 shape in the topcell. The results are the same as in Case 1.



The purpose of GLOBALNETID is to give accurate connectivity information in the context of the current cell. The small square polygon in cellA is not connected to anything else in cellA in any placement of cellA, therefore the results are reported in cellA. This is in contrast to the two long polygons in the rightmost placement of cellA, which are connected within that placement of the cell by the shape at the top level.

Case 3

As in case 1, the layout has two placements of cellA, where one placement has a layer2 polygon in the topcell shorting two layer1 polygons in cellA. A layer3 polygon in cellA overlaps all three layer1 polygons. The primary layer is layer3 and is used to cluster layer1 polygons. The following operation is used:

CONNECT layer1 layer2
gnet_out = DFM PROPERTY layer3 layer1 OVERLAP MULTI 
				[Vg_net = VNETID(GLOBALNETID(layer1))]

Clustering of layer1 with layer3 forces all layer1 results to be reported in the same cell, which is the topcell in this case due to promotion. In this example the GLOBALNETID values for the two small layer1 polygons can be compared because the results are reported in the same cell.

Examples

The GLOBALNETID(layer,ordinal) form of the function can be used to store a netID property on the source/drain pins of a MOSFET device. 

gate_p = DFM PROPERTY gate sd OVERLAP ABUT ALSO MULTI
          [N1 = GLOBALNETID(sd, 1)]
          [N2 = GLOBALNETID(sd, 2)]

The following operation determines if the source/drain pins are shorted. The use of GLOBALNETID ensures that shorts higher in the hierarchy are found. If the source/drain pins are shorted the gate shape is written to the output layer. 

gate_sd_short = DFM PROPERTY gate sd OVERLAP ABUT ALSO MULTI
    [SD_NETS_EQ = (GLOBALNETID(sd, 1) == GLOBALNETID(sd, 2)) ? 1 : 0] == 1
