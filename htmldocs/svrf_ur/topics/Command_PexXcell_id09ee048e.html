PEX Xcell

Parasitic extraction

Specifies a hierarchical extraction cell referred to as an xcell. Syntax 1 specifies a hierarchical extraction cell, also referred to as an xcell. Syntax 2 specifies to import GDS data into the Calibre xACT LEF/DEF flow. Syntax 3 specifies to import GDS dummy fill data into the Calibre xACT LEF/DEF flow.

Usage

Syntax 1, use only with Calibre xRC, Calibre xACT, and Calibre xACT 3D:

PEX XCELL layout_cell_name [CONTEXT layout_path
| PCDEF [INTRINSIC] [IDEAL] [CONLY]
| PRIMITIVE [GDS file.gds] [MAP gds_mapfile]
| IDEAL
| NOBLOCK]

Syntax 2, use only within the Calibre xACT LEF/DEF flow:

PEX XCELL layout_cell_name PRIMITIVE GDS file.gds MAP gds_mapfile
[BUSDELIM bus_delim_str]

Syntax 3, use only within the Calibre xACT LEF/DEF flow:

PEX XCELL design_name FILL GDS fillfile.gds MAP gds_mapfile
[DX xtrans DY ytrans]

Arguments

layout_cell_name

In Syntax 1 and Syntax 2, layout_cell_name is a required parameter that specifies the layout name of the xcell. Only one layout_cell_name can be specified per statement. It may contain one or more asterisk (*) characters; the * character is a wildcard that matches zero or more characters. A string that contains a * character must be enclosed in quotation marks because the * is a reserved symbol.

design_name

If Syntax 3 is used, design_name is a required parameter that specifies the name of the DEF design or the GDS top cell. The GDS top cell and design_name must have the same name.

CONTEXT layout_path

An optional keyword and parameter set used for in-context extraction. For more information on in-context extraction, see “In-Context Extraction” in the Calibre xRC User’s Manual.

The layout_path is any valid layout path or “parent_id/cell_id”. See “Discovering Layout Paths for In-Context Cells” in the Calibre xRC User’s Manual to determine a valid layout_path.

If doing gate-level extraction the cell is treated as a primitive and the context ignored. Use only with Calibre xRC, Calibre xACT, or Calibre xACT 3D as shown in Syntax 1.

PCDEF [INTRINSIC] [IDEAL] [CONLY]

An optional keyword that indicates the specified cell is a pre-characterized cell (pcell). The pcell is treated as an xcell and its contents are not extracted. Extraction for the specified pcell behaves as follows:

  • Coupling capacitance from nets outside the pcell to nets inside the pcell are coupled to the associated pin on the pcell.

  • Internal geometries that connect to or can be traced to pcell pins that are within the coupling distance of inside geometries of neighboring pcells, should have coupling capacitance extracted and netlisted.

If the IDEAL keyword is specified with PCDEF, then the contents inside of the pcell will be netlisted.

  • INTRINSIC — An optional keyword specified with PCDEF that causes the pin intrinsic capacitance to be ignored for the device layers within the cell.

  • IDEAL — An optional keyword specified with PCDEF that causes the intentional device of the pcell to be netlisted.

  • CONLY — An optional keyword specified with PCDEF that indicates to ignore capacitance and only extract parasitic resistance for the cell. These resistors are identified in the extracted netlist with the property called $pcell_res=1. Use this keyword when the SPICE model for the cell includes parasitic capacitance but not parasitic resistance.

Use only with Calibre xRC, Calibre xACT, or Calibre xACT 3D as shown in Syntax 1.

PRIMITIVE [GDS file.gds] [MAP gds_mapfile]

PRIMITIVE is a keyword that indicates the cell is a primitive. The contents are not extracted. In gate-level extraction, all xcells are treated as primitives.

  • GDS file.gds — A keyword and parameter set specified with PRIMITIVE that indicates where to find the GDS information. May only be used with LEF/DEF designs. This keyword parameter set is optional for Syntax 1 and required for Syntax 2.

  • MAP gds_mapfile — A keyword and parameter set specified with PRIMITIVE that indicates where to find the LEF/DEF to GDS layer mapping information. May only be used with LEF/DEF designs. This keyword parameter set is optional for Syntax 1 and required for Syntax 2.

This keyword parameter set is optional for Syntax 1 and required for Syntax 2.

FILL GDS fillfile.gds MAP gds_mapfile

FILL is a keyword that specifies to import GDS dummy fill data into the Calibre xACT LEF/DEF flow.

  • GDS fillfile.gds — A keyword and parameter set specified with FILL that indicates where to find the fill GDS information. May only be used with LEF/DEF designs.

  • MAP gds_mapfile — A keyword and parameter set specified with FILL that indicates where to find the LEF/DEF to GDS layer mapping information. May only be used with LEF/DEF designs.

This keyword parameter set is required for Syntax 3. Use only within the Calibre xACT LEF/DEF flow.

BUSDELIM “bus_delim_str”

An optional keyword and parameter set that allows substitution of the bus delimiters used in the DEF design for the delimiters used in the GDS file. The bus_delim_str parameter must be contained within quotes (“”) and be exactly two characters long; for example, “<>”.

Use only in Syntax 2 within the Calibre xACT LEF/DEF flow.

IDEAL

An optional keyword that indicates the cell is an ideal xcell. The contents of the cell are not extracted, but are written to the netlist.

Use only with Calibre xRC, Calibre xACT, or Calibre xACT 3D as shown in Syntax 1.

NOBLOCK

An optional keyword that indicates the cell is not a pcell. The contents of the cell are still extracted and written to the netlist. Use this keyword in when you want to explicitly specify cells to be extracted.

Use only with Calibre xRC, Calibre xACT, or Calibre xACT 3D as shown in Syntax 1.

DX xtrans DY ytrans

An optional keyword and parameter set used to translate the fill cell by the specified number of database units xtrans and ytrans.

Use only within the Calibre xACT LEF/DEF flow as shown in Syntax 3.

Description

Syntax 1 specifies a hierarchical extraction cell also referred to as an xcell in Calibre xRC Calibre xACT, and Calibre xACT 3D.

Syntax 2 specifies to import GDS data into the Calibre xACT LEF/DEF flow.

Syntax 3 specifies to import GDS dummy fill data into the Calibre xACT LEF/DEF flow.

Using Syntax 1

This statement specifies a hierarchical extraction cell also referred to as an xcell. It allows xcells to be specified in the SVRF file and may be used in combination with an xcell list as long as the specified cell does not conflict with an xcell already defined in the xcell list. For example, if an xcell is defined for in-context extraction with two different layout paths, this causes a conflict. If a conflict is detected, then extraction stops with an error message. If an xcell is specified using PEX Xcell and also appears in the xcell file, the resulting conflict may be resolved by using the PEX Xcell Precedence to indicate which definition should take precedence. When performing LEF/DEF extraction, the default xcell list file generated by Calibre will have the lowest precedence and will not trigger warnings or errors. See Hierarchy Control with Xcells in the Calibre xACT User’s Manual for more information on xcell lists.

Syntax 1 Example 1

The following is an example of how to specify xcells in your SVRF rule file:

PEX XCELL NOR IDEAL
PEX XCELL NAND PRIMITIVE
PEX XCELL NMOS PCDEF
PEX XCELL INV PCDEF INTRINSIC

Syntax 1 Example 2

Consider the following cell names:

cfmom
cfmom12
cfmom_no_block1
cfmom_no_block2

Specify the following statement to treat all cells prefixed with “cfmom” as pcells:

PEX XCELL CFMOM* PCDEF

To not treat the cfmom_no_block1 and cfmom_no_block2 cells as pcells and still extract their cell contents, add the following statement to your rule file:

PEX XCELL CFMOM_NO_BLOCK* NOBLOCK

Using Syntax 2

Use the PEX Xcell statement in the following form to import GDS data into the Calibre xACT LEF/DEF flow:

PEX XCELL {layout_cell_name | "*"} PRIMITIVE GDS file.gds MAP gds_mapfile 
	[BUSDELIM bus_delim_str]

The wildcard “*” indicates that all cells contained in the specified GDS file (file.gds) will be imported.

Use the BUSDELIM option when the GDS cellview file and LEF macro files use different characters to delimit bus ports. For example, if the bus ports in the LEF file use brackets ( [ ] ) and the corresponding bus ports in the GDS file use chevrons ( < > ), then specify BUSDELIM “<>” to ensure that Calibre xACT reconciles the two sets of ports when it imports the GDS file.

This statement may be specified more than once in order to import cells from multiple GDS files. All occurrences of this statement that specify the MAP keyword must specify the same GDS mapfile filename (gds_mapfile). Note that file.gds and gds_mapfile file names that contain special characters such as ‘/’ must be enclosed in quotes.

Syntax 2 Example 1

Specify the following statements to import GDS cells from three different GDS files:

PEX XCELL "*" PRIMITIVE GDS "./design/gds/gscl45nm.gds" MAP "mapfile1.txt"
PEX XCELL "*" PRIMITIVE GDS "./design/gds/PADS.gds"
PEX XCELL "*" PRIMITIVE GDS "./design/gds/myram.gds"

It is only necessary to specify the MAP option once if your rule file contains multiple instances of this statement.

Syntax 2 Example 2

Given the LEF file uses brackets ( [ ] ) to delimit bus ports:

A[1]

and the GDS file uses chevrons:

A<1>

Specify the following statement to make sure that Calibre xACT reconciles the two sets of ports when it imports the GDS file:

PEX XCELL "*" PRIMITIVE GDS "./design/gds/gdsmacrocells.gds" MAP "xact_gdscellslayermap" BUSDELIM "<>"

Using Syntax 3

Use the PEX Xcell statement in the following form to import GDS dummy fill data into the Calibre xACT LEF/DEF flow:

PEX XCELL design_name FILL GDS fillfile.gds MAP gds_mapfile

This statement may be specified more than once in order to import cells from multiple GDS files. All occurrences of this statement that specify the MAP keyword must specify the same GDS mapfile filename (gds_mapfile). Note that fillfile.gds and gds_mapfile file names that contain special characters such as ‘/’ must be enclosed in quotes.

Syntax 3 Example

Specify the following statement to import GDS fill from fillfile.gds: 

PEX XCELL design1 FILL GDS "./design/fill/fillfile.gds" MAP "gds_map.txt"

Use the PEX XACT Fill statement to extract dummy fill either as floating or grounded.

GDS Mapfile Format

The gds_mapfile format consists of seven columns in the following order:

  • column 1: LEFDEFlayername

  • column 2: GDSdefaultlayernumber

  • column 3: GDSdefaultlayerdatatype

  • column 4: GDSfilllayernumber

  • column 5: GDSfilllayerdatatype

  • column 6: GDStextlayernumber

  • column 7: GDStextlayerdatatype

For example

metal1 31 0 31 1 31 99
metal2 32 0 32 1 32 99
metal3 33 0 33 1 33 99
....

If you do not know the metal fill layer numbers for your fill data, you can find them by visually inspecting the data using Calibre DESIGNrev or another layout viewer.

It is only necessary to specify the MAPFILE once if the SVRF rule file includes multiple instances of the PEX Xcell statement. If the MAPFILE is specified multiple times, then it must be the same file in each statement.

Application in LVS-H Circuit Extraction

This statement enables LVS-H circuit extraction to support a greater level of hierarchy preservation for specified cells and certain layers. Cells specified with PEX Xcell behave exactly like PEX Preserve Cell List cells in this context.
