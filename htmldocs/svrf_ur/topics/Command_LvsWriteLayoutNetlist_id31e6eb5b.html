LVS Write Layout Netlist

Specification statement

Specifies to output a transformed netlist report from the layout.

Usage

LVS WRITE LAYOUT NETLIST “pathname”

Arguments

“pathname”

A required pathname of the output netlist report.

The pathname parameter can contain environment variables. See “Environment Variables in Pathname Parameters”.

If the pathname ends in either the .Z or .gz suffix, then the file is automatically compressed using the compress or gzip commands, respectively. This assumes these commands are available in your $PATH.

Description

Specifies to output a transformed netlist report from the layout. This statement is typically used only for debugging purposes. It may be specified only once.

The netlist report represents the connectivity of the design as seen by the circuit comparison module after non-corresponding cell expansion, series and parallel device reduction, unused device filtering, and other transformations, but before logic gate recognition. The LVS Write Injected Layout Netlist statement writes a report that takes logic injection into account as well.

Note:

The output of this statement is not canonical SPICE syntax. Use dfm::read_netlist -layout in the Calibre Query Server Tcl shell or in Calibre YieldServer to get a transformed SPICE netlist for downstream tool use.

Hierarchical LVS writes a hierarchical netlist report, subject to the existence of hcells. Flat LVS writes a flat netlist report.

To the extent possible, the output netlists preserve net, device, and instance names from the original design, except that a letter designating the SPICE element type (like M, R, C, X, and so forth) is added in front of device and instance names in the output.

When the original design hierarchy is expanded, then the output may contain hierarchical pathnames with slash ( / ) characters serving as delimiters. For example:

MM1 1 2 3 4 p         $$ Original M1
MX2/X1/M1 1 2 3 4 p   $$ Original X2/X1/M1 (down the hierarchy)
XX3 1 2 XYZ           $$ Original X3
XX4/X1/X1 1 2 XYZ     $$ Original X4/X1/X1 (down the hierarchy)

Each netlist created has a header attached similar to the following example:

** CALIBRE LAYOUT NETLIST **
** CALIBRE VERSION: v8.7_17.1  Fri Jun  4 14:48:36 PDT 1999
** LAYOUT NAME: layout.spi(SR2F_rdc_buffer_L)
** NETLIST FILE: layout.spi.out
** GENERATED: Wed Jun  9 14:52:49 1999

See also LVS Write Source Netlist.

Examples

// Write the layout netlist report after most LVS processing 
// but before gate recognition.
LVS WRITE LAYOUT NETLIST "/usr/mystuff/netlists/layout_xform"
