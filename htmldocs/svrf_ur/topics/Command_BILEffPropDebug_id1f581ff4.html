DEBUG

Effective property computation language function.

Enables debug monitoring during effective property computation.

Usage

DEBUG range1[, range2 …]

Arguments

range1

A device instance ID number or a pair of device instance numbers separated by a hyphen (‑). If the hyphenated form is used, the first number must be less than the second number. In that case, all numbers between and including the numbers specified in the range correspond to instances. At least one range must be specified, although it can contain only a single instance. Multiple ranges are separated by a comma (,) followed by whitespace.

Return Values

None.

Description

Debug monitoring of the effective property computation can be useful in finding errors during the development of new property computation code. When present, DEBUG must follow the EFFECTIVE and EFFECTIVE STRING statements, and precede the optional WARN statement. The debugging trace is written to the transcript. The trace can produce much output, so the use of small ranges is recommended.

This statement can be used in conjunction with the WARN statement, which reports all mathematical computation errors.

An instance ID represents a single flat device to which the input group is reduced. Flat instance IDs are unique across all devices in the design. Instance IDs are assigned by the system and are not guaranteed to be sequential, merely unique.

Examples

If an effective property computation performs division by 0, you will see a warning like this in a flat run transcript:

*****************************************************
EFFECTIVE property user-program WARNING:
Division by zero (result set to NaN) on line 15
while transforming layout with:
LVS REDUCE ... (within EFFECTIVE program beginning on rule file line 13)
Use instance ID# 11 in DEBUG trace range.
*****************************************************
 ADDITIONAL WARNINGS SUPPRESSED - use WARN statement to obtain all LVS
 REDUCE warning messages for this user program.

At this point, you could add the WARN statement to get a report of all impossible calculations in the run. You can also use instance ID# 11 to trace the problem for that instance, like this:

LVS REDUCE SERIES RESISTORS yes
[
  EFFECTIVE r
  DEBUG 11
  WARN
  ...
]

You will then find a debugging trace for instance ID# 11 in the transcript, along with any additional warnings with corresponding instance ID numbers. You can then add the instances to the DEBUG statement, as necessary.
