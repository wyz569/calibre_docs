LVS Property Initialize

Specification statement

Instructs the tool to add numeric properties simultaneously to instances in the layout and source databases in memory and initialize those properties to specified values.

Usage

LVS PROPERTY INITIALIZE component_type [‘(’component_subtype‘)’]
‘[’lvs_property_initialize_program‘]’

Arguments

component_type

A required LVS component type.

‘(’component_subtype‘)’

An optional component subtype of the component_type parameter. If specified the component_subtype must appear within parentheses. If component_subtype is not present, then the LVS Property Initialize statement applies to all instances of the specified component_type regardless of their subtype, except for subtypes that have their own LVS Property Initialize statement (that is, a statement with the same component type and with the component subtype of the instance).

‘[’lvs_property_initialize_program‘]’

A required device property initialization program, which appears between literal brackets. The complete description of this language appears under “LVS Property Initialize Built-In Language”.

Description

Instructs the tool to add numeric properties to instances in the Calibre representation of the layout and source databases and to initialize those properties to specified values. This statement is not designed as a general property insertion mechanism, and it does not apply to netlist extraction.

Note:

You should avoid using this statement except in special circumstances where your source netlist is incomplete or where consistency checking of reduced device properties needs to be disabled.

This statement is useful when you have effective property computations that expect certain properties to be available in the input databases but they are not. Initializing missing property values with this statement allows device reduction to occur without computational errors.

The LVS Property Initialize statement includes a program written in a language similar to device reduction effective property computation programs. The LVS Property Initialize programming language is more restricted, however.

A property that is both added and initialized by LVS Property Initialize is called a generated property. Generated properties do not exist in the input databases. Instead, the properties are created by their appearance in the LVS Property Initialize statement and are given the value computed in the lvs_property_initialize_program for every instance read from input databases. See Example 1.

When instances in the input databases contain property values, those values are made accessible to the lvs_property_initialize_program. Instances can be checked for the existence of property values, as well as what the property values are. The lvs_property_initialize_program can then manipulate the existing property values. See Example 2.

Generated properties can be filtered by the LVS Filter statement as with calculated device properties.

Specifying LVS Property Initialize in the rule file disables all consistency checks related to checking effective device properties (from any of the LVS Reduce family of statements) against Device definitions, regardless of device type, subtype, or property name. For example, the statement:

LVS PROPERTY INITIALIZE MN [ PROPERTY per per = 1 ]

disables all consistency checks regarding any property in any device. This is normally undesirable, so this statement should be used with care. See “Reduction Program Semantic Checking” for a discussion of consistency checks.

When LVS Spice Replicate Devices LPI is specified, any LVS Property Initialize program that calculates M parameters is executed before M parameter replications occur.

Examples

Example 1

This example shows the initialization of three generated properties in layout and source. It also disables all consistency checking for reduction of all device properties specified in the rule file.

LVS PROPERTY INITIALIZE type(subtype) [
// these properties do not exist in the input databases, 
// so they are generated.
   PROPERTY eqpar, eqser, sq2
   eqpar=1
   eqser=1
   sq2 = SQRT(2)/2
]

Example 2

The following example checks MN instances for the existence of property w. If the w property is not found, then w is initialized to 0; otherwise, the w value is maintained. Because w is declared as a property, w must have a value after the user program executes; therefore, the IF statement must guarantee w gets a value from either branch of the conditional block.

LVS PROPERTY INITIALIZE MN [
   PROPerty w
   input_w = INPut_NUMeric_VALue( w )  // get w if present for instances
   IF ( IS_MISSING( input_w ) == 1 ) { // check to see if w is present
     w = 0       // w is missing, initialize it to 0
   } ELSE {
     w = input_w // w is present, so use the input value
   }
]

Example 3

This example shows the assignment of generated properties for a resistor device with a user-given name. The program gets the R, L, and W property values from the device, if they are present. If R is not present on the device, it is either calculated from L and W (if they are present) or it is assigned a fall-through value. If L and W are not present, they are assigned the value 0.

// RMODEL is an R device
LVS DEVICE TYPE RESISTOR RMODEL [ POS=PLUS NEG=MINUS ] SOURCE LAYOUT
LVS PROPERTY INITIALIZE RMODEL [
  PROPERTY R, L, W
  input_R = INPUT_NUMERIC_VALUE( R )  // get R (if present)
  input_L = INPUT_NUMERIC_VALUE( L )  // get L (if present)
  input_W = INPUT_NUMERIC_VALUE( W )  // get W (if present)
  if ( IS_MISSING( input_R ) == 0 ) { // check to see if R is present
    R = input_R   // R is present, so use the input value
  } else if ( ( IS_MISSING( input_L ) == 0 ) && \
              ( IS_MISSING( input_W ) == 0 ) ) {
      R = input_L / input_W           // W and L present
    } else {
        R = 1.0e-5                    // fall-through value
  }
// Must assign values to all referenced properties ...
  if ( IS_MISSING( input_L  ) == 0 ) { L  = input_L  } else { L  = 0 }
  if ( IS_MISSING( input_W  ) == 0 ) { W  = input_W  } else { W  = 0 }
]
