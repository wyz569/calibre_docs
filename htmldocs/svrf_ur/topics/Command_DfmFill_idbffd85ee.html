DFM Fill

Layer operation

License: Calibre YieldEnhancer

Creates a derived polygon layer containing fill polygons generated according to a fill specification.

Usage

DFM FILL spec_name [output_name … ] [COLOR mask_name]
[ {[HLAYER] [VERSION version_number]
          COMPRESS [cell_name_prefix] [TEXTTAG text_string] }
      | {ENVELOPE value} ]
[BLOCKOUT blockout_layer]

Arguments

spec_name

A required argument specifying the name of the fill specification to use when generating the fill layer. The fill specification is defined by a DFM Spec Fill statement.

output_name

An optional argument specifying the fill geometries to output. The outputs must be defined in the DFM Spec Fill statement for spec_name. If more than one output_name is specified, all data is merged into a single output layer.

If no output_name is specified, all output generated by spec_name is merged into a single output layer, regardless of whether an output_name was assigned to individual shapes in the DFM Spec Fill statement.

COLOR mask_name

An optional keyword and argument that enables splitting of fill output into more than one layer, also known as multi-patterning. The mask_name is an arbitrary user-defined string.

The multi-patterning is done such that shapes from the input layer are written to each output layer according to the COLOR SCHEME specified in the corresponding DFM Spec Fill or DFM Spec Fill Shape statement, as shown in the following figure. All shapes on a common mask_name are spatially checked to each other.

Figure 1. DFM Fill COLOR Output

For double patterning, as in the preceding figure, you must specify two DFM Fill operations with the same spec_name and output_name values, and different mask_name and output layer values. See “Example 1: Two Color Fill”. Multi-patterning with three output layers would use three different mask_name values.

Note:

In some special circumstances, you may want to output fill layers with the same spec_name, output_name, and mask_name, creating duplicate layers. This is possible only in a Calibre nmDRC run. It is not allowed in a Calibre DFM run or with Calibre YieldServer execution.

[HLAYER] [VERSION version_number] COMPRESS [cell_name_prefix] [TEXTTAG text_string]

An optional keyword set designed to reduce file sizes when the output layer is written using the DFM RDB operation with the GDS or OASIS keyword. When COMPRESS is specified, the operation replaces repeating fill patterns with an equivalent cell placement. AREF compression is used. The output layer is intended to be used only with the DFM RDB operation with GDS or OASIS specified. The arguments must be specified in the order shown in the usage line.

  • HLAYER — Specifies that the output layer is the hierarchy layer for a set of multi-output or single-output DFM Fill operations. The hierarchy layer contains the hierarchy placement information for all layers in the set. Exactly one hierarchy layer must be specified for each set of multi-output or single-output DFM Fill operations. The output_name argument is not valid with the HLAYER keyword.

    The hierarchy layer must be included when writing one or more output layers using DFM RDB GDS or DFM RDB OASIS.

  • VERSION version_number — Specifies a version number that is included in the name of the compression cell. The version_number argument is a positive integer or a numerical expression that evaluates to a positive integer. The version_number is appended to the cell_name_prefix, if one is specified. If a cell_name_prefix is not specified, version_number is a prefix to the name of the compression cell.

  • COMPRESS — Specifies compression.

  • cell_name_prefix — Specifies a prefix to the name of the compression cell. The cell_name_prefix can be either a quoted string or the name of a variable containing a single string value.

  • TEXTTAG text_string — Specifies a TEXT object, with content text_string, that is placed in each generated compressed cell. The text is placed at the center of the extent of the cell. At least one compressed fill layer must also be output with the DFM RDB operation that outputs the TEXTTAG layer.

    See “Example 4: COMPRESS Keyword With TEXTTAG”.

    TEXTTAG cannot be used with output_name. TEXTTAG is not supported with FILLSTACK.

The cell_name_prefix and version_number arguments are useful for avoiding naming conflicts that can occur when generating compression cell names. The same cell_name_prefix and version_number must be used in a set of multi-output DFM Fill operations that reference the same spec_name. Both arguments must be unique for each spec_name. Both arguments can be set with the DFM Defaults statement (see “Example 4” with DFM Defaults).

See “Compression Cell Naming”, “Example 2: COMPRESS Keyword, Multilayer Output”, and “Example 3: COMPRESS Keyword, Single Layer Output”.

When compressed fill layers are output using DFM RDB OASIS, CBLOCK compression and STRICT mode are used by default. To disable either mode for all OASIS results databases, include the DRC Results Database statement with the NOCBLOCK and/or NOSTRICT keywords.

ENVELOPE env_value

An optional keyword set that generates a fill pattern enclosure layer by expanding the extent of spec_name shapes by the specified env_value, which is a distance in user units. ENVELOPE cannot be specified with the COMPRESS keyword.

The intended application is to create an envelope around groups of nearby fill shapes. When a second fill operation depends on the output of a prior fill operation, the envelop layer can be used as input to the second fill operation, rather than the actual fill shapes. This usually improves the performance of the second fill operation because fewer geometries are used as input to the second fill operation. To create the envelope, the env_value should be greater than half the step distance between fill shapes. When OFFSET is used, the env_value should be large enough to cover the increased corner to corner distances, otherwise holes are created in the envelope layer.

The following figure shows a simple example.

Figure 2. DFM Fill ENVELOPE

When creating an envelope layer for a multilayer FILLSTACK, it is recommended to specify only the spec_name, and not each output_name. The size of the enclosure layer shapes produced with ENVELOPE can vary when multilayer FILLSTACK shapes are specified with output_name. See “Example 5: ENVELOPE Keyword With Multilayer FILLSTACK”.

BLOCKOUT blockout_layer

An optional argument set that specifies keep-out regions for the generated fill layer. Shapes on the output layer that interact with shapes on the blockout_layer are removed. The blockout_layer is a polygon layer. Interaction is determined between fill shapes and blockout_layer shapes; this is different than most spacing measurements in fill operations, which measure distance to the fill shape extent. BLOCKOUT filtering is done after fill is generated, in contrast to the SPACE set of keywords in DFM Spec Fill, which affect the calculated fill region.

If separate DFM Fill operations with the same spec_name specify BLOCKOUT with the same blockout_layer but different output_name arguments, the generated fill layers are processed together by BLOCKOUT. Shapes in the same fill cell are processed as a unit, and if a shape on any one of the fill layers interacts with the blockout_layer, all layers in the fill cell with the same blockout_layer are removed.

These conditions must be met when using BLOCKOUT:

  • One output_name must be specified, and only one.

  • DFM Fill operations with the same spec_name and output_name must have the same BLOCKOUT specification.

  • FILLSHAPE, RECTFILL, and POLYFILL are the only supported fill pattern keywords in the corresponding DFM Spec Fill statement. If FILLSHAPE is specified, the fill shape must be RECTFILL or POLYFILL. If RECTFILL or POLYFILL is specified in DFM Spec Fill, the OUTPUT keyword must be specified.

  • COLOR is not supported.

See “Example 6: BLOCKOUT”.

Description

Creates an output layer consisting of fill polygons.

The output of this operation is a derived polygon layer containing the generated fill. When adding fill shapes, the DFM Fill operation treats the design as if it were flat and creates the initial fill shapes at the top level of the hierarchy. The operation then cleans up the fill shapes and, wherever applicable, pushes repeated shapes lower in the hierarchy, thereby reducing the final design file size and improving performance of future operations.

The sizes, shapes, and positions of these fill polygons are controlled through a fill specification, which you must define with the DFM Spec Fill statement. The following figure shows the basic use model for this operation. For a complete discussion, refer to “Fill Layers in Calibre YieldEnhancer” in the Calibre YieldAnalyzer and YieldEnhancer User’s and Reference Manual.

Figure 3. Typical Use Model for DFM Fill

All DFM Fill operations referencing the same DFM Spec Fill statement are processed concurrently.

Compression Cell Naming

When the COMPRESS keyword is specified, the compression cell name has this general format:

cell_name_prefixversion_numberbasecell_suffix

  • cell_name_prefix — The cell_name_prefix in the DFM Fill operation, if used.

  • version_number — The version_number in the DFM Fill operation, if used.

  • basecell — The basename specified for the corresponding fill shape in the DFM Spec Fill statement. The basename string cannot contain spaces or underscores (_).

    If basename is not specified, spec_name is used for basecell. If spec_name includes underscores or characters not allowed in a cell name, those characters are replaced with a dollar sign ($). If more than one fill shape is defined, “$index” is appended to spec_name, where index is an integer starting at 1.

  • suffix — A suffix that is added by the tool if needed to generate unique cell names. This occurs for FILLSTACK and STRETCHFILL shapes, and when the COLOR keyword is used. The suffix has the following form:

    Snum — “S” indicates STRETCHFILL. num is the stretch increment.

    Cmask_name — “C” indicates color.

    fsrange — “fs” indicates FILLSTACK. “fs” is followed by a single layer number or a range of FILLSTACK layer numbers.

Note:

Cell name prefixes and suffixes defined with the PREFIX and APPEND keywords in DFM RDB are applied after the compression cell name is generated by DFM Fill.

Examples

Example 1: Two Color Fill

Generates RECTFILL fill in two colors. 

LAYOUT PATH "layout.oas" 
LAYOUT SYSTEM OASIS 
LAYOUT PRIMARY TOPCELL 

DRC RESULTS DATABASE drc.db
DFM DATABASE dfmdb OVERWRITE
DRC CELL NAME YES CELL SPACE XFORM

LAYER layer1 1
LAYER layer4 4
DFM SPEC FILL SHAPE "R1" 
   RECTFILL  0 0  1 .2 
   SPACE .01 layer4
   STEP .2
   COLOR  SCHEME CHECKER
DFM SPEC FILL fillR1
   INSIDE OF LAYER layer1
   INITIAL FILLREGION 
   FILLSHAPE OUTPUT "outR1" "R1" 
// generate fill 
fill_A = DFM FILL fillR1 "outR1" COLOR A
fill_B = DFM FILL fillR1 "outR1" COLOR B
oas_out {
DFM RDB OASIS fillout.oas fill_A 10 0
DFM RDB OASIS fillout.oas fill_B 11 0
}
Figure 4. DFM Fill with Two Color Fill

Example 2: COMPRESS Keyword, Multilayer Output

This example shows how the COMPRESS keyword is used with a set of multilayer DFM Fill operations.

The fill specification M1_M2_M3_FILL (not included) defines three outputs: M1, M2, and M3. In the following code, there is a hierarchy layer (hier_layer) and three output layers: m1_fill, m2_fill, m3_fill. All four layers are written to a GDS file using a DFM RDB GDS operation. Layer m1_fill is written to layer 81, m2_fill is written to layer 82, and so on.

m1_fill = DFM FILL M1_M2_M3_FILL M1 COMPRESS
m2_fill = DFM FILL M1_M2_M3_FILL M2 COMPRESS
m3_fill = DFM FILL M1_M2_M3_FILL M3 COMPRESS
hier_layer = DFM FILL M1_M2_M3_FILL HLAYER COMPRESS
write_gds {
   DFM RDB GDS "compressed_fill.gds" 
            m1_fill 81 
            m2_fill 82 
            m3_fill 83 
            hier_layer 0
}
Tip

You can use the METACELL keyword in DFM RDB to create one or more subcells below the top cell and place layer data in those subcells. When METACELL is used, the hierarchy layer is specified only once in the set of DFM RDB operations writing to the same output file.

Example 3: COMPRESS Keyword, Single Layer Output

This example generates single layer fill output and specifies a cell name prefix “_M1_”.

m1fill_compress = DFM FILL Xspec COMPRESS "_M1_"
hier_layer = DFM FILL Xspec HLAYER COMPRESS "_M1_"
write_gds {
   DFM RDB GDS "compressed_fill.gds" 
               m1fill_compress 81 
               hier_layer 0
}

Example 4: COMPRESS Keyword With TEXTTAG

This example creates two text objects. Three cases are included, showing how to output text objects on the same layer, different layers, and on the same layer as an output polygon layer.

All three cases use the same set of DFM Fill operations:

m1_fill = DFM FILL FillSpec M1OUT COMPRESS
m2_fill = DFM FILL FillSpec M2OUT COMPRESS
m3_fill = DFM FILL FillSpec M3OUT COMPRESS
text1 = DFM FILL FillSpec COMPRESS TEXTTAG "text1"
text2 = DFM FILL FillSpec COMPRESS TEXTTAG "text2"
hier_layer = DFM FILL FillSpec HLAYER COMPRESS

Case A: Output all text objects to the same GDS layer 

write_gds {
   DFM RDB GDS "m_fill_A.gds" 
            m1_fill 81 
            m2_fill 82 
            m3_fill 83 
            text1   100 2
            text2   100 2
            hier_layer 0
}

Case B: Output text objects to different GDS layers 

write_gds {
   DFM RDB GDS "m_fill_B.gds" 
            m1_fill 81 
            m2_fill 82 
            m3_fill 83 
            text1   100 2
            text2   101 2
            hier_layer 0
}

Case C: Output text objects to a polygon layer 

write_gds {
   DFM RDB GDS "m_fill_C.gds" 
            m1_fill 81 
            m2_fill 82 
            m3_fill 83 
            text1   83
            text2   83
            hier_layer 0
}

Example 5: ENVELOPE Keyword With Multilayer FILLSTACK

This example uses the ENVELOPE keyword with a multilayer FILLSTACK. It shows how the size of the enclosure layer produced with ENVELOPE can vary when multilayer FILLSTACK shapes are specified with output_name rather than only with the spec_name, as recommended.

Here is the FILLSTACK specification: 

DFM SPEC FILL SHAPE "RH" 
    RECTFILL  0 0.2  0.5 0.3 
    SPACE .05 layer3

DFM SPEC FILL SHAPE "RV" 
    RECTFILL 0.2 0 0.3 0.5
    SPACE .05 layer2
DFM SPEC FILL fstack 
     INSIDE OF LAYER layer0
     INITIAL INSIDE OF LAYER LL
     FILLSTACK "basestack" 
        OUTPUT 1 RH "RH"
        OUTPUT 2 RV "RV"
     STEP 0.2

fsH = DFM FILL fstack RH
fsV = DFM FILL fstack RV

Example 6: BLOCKOUT

This example uses the BLOCKOUT keyword. 

LAYER layer1 1
LAYER block_v 4
LAYER block_h 5

DFM SPEC FILL SHAPE "RH" 
   RECTFILL  0 0.2  .5 .3 
   STEP .2
DFM SPEC FILL SHAPE "RV" 
   RECTFILL 0.2 0 0.3 0.5
   STEP .2
DFM SPEC FILL fillBlock
   INSIDE OF LAYER layer1
   INITIAL INSIDE OF LAYER
   FILLSHAPE "baseB" OUTPUT RH "RH"
                     OUTPUT RV "RV" 

fill_BV = DFM FILL fillBlock "RV" BLOCKOUT block_v
fill_BH = DFM FILL fillBlock "RH" BLOCKOUT block_h

The block_v shape along the top of the fill region causes vertical fill shapes that overlap or touch the block_v shape to be removed, but does not affect the horizontal fill shapes. Similar behavior occurs for the block_h shape and the horizontal fill shapes.
