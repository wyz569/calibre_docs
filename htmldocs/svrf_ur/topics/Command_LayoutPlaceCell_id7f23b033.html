Layout Place Cell

Specification statement

Specifies to construct a GDSII or OASIS database from parameters given in this statement. Used only in Calibre.

Usage

LAYOUT PLACE CELL filename [ GDSII | GDS | GDS2 | OASIS ] top_cell
{placed_cell_info [placed_cell_info …] | EMPTY}

Arguments

filename

Required parameter giving the filename for the constructed database.

GDSII | GDS | GDS2 | OASIS

Optional parameter giving the database type. GDSII is the default.

top_cell

A required argument that gives the name of the top cell in the created database. Only one cell record is created.

placed_cell_info

A required parameter that gives the information necessary to instantiate a cell placement. The parameter placed_cell_info has the following syntax:

  • cell_name magnification angle reflection x y

with these parameter definitions:

  • cell_name — The name of the instantiated cell. Cells referenced by this parameter must be defined in an input database.

  • magnification — An undimensioned positive floating-point number, which gives the magnification. It may be a variable.

  • angle — An undimensioned floating-point number, which gives the angular rotation in degrees of the cell placement. It may be a variable.

  • reflection — An integer constant that must be either 0 or 1, with 1 meaning x-axis reflection and 0 meaning no reflection.

  • x y — Dimensioned floating-point numbers that give the coordinates of the cell placement origin in user units. They may be variables. Unlike other SVRF operations, negative numbers may not appear in parentheses ( ).

EMPTY

If specified instead of placed_cell_info, this parameter creates a top-level cell in the database with no placements.

Description

Specifies construction of a GDSII or OASIS database within the Calibre application.

Note:

The filename specified for the GDS or OASIS database should be unique, in order to avoid overwriting an existing file.

Calibre applications generally begin by compiling the rule file. After compilation, the application determines if a Layout Place Cell specification statement is present. If present, Calibre immediately constructs a GDSII or OASIS database with the given filename. Only one cell record (BGNSTR in GDSII) is created, and the name of the cell is top_cell. The database precision is that given in the rule file Precision. For GDSII databases, the LIBNAME is “drc.db”.

The process for instantiating the cell placement involves performing the magnification, placing the magnified cell at the location specified by placed_cell_info, performing any reflection of the x-axis, and then rotating the cell placement based on the specified angle. The transformations occur in that sequence.

Each placed_cell_info argument instantiates a placement (SREF in GDSII) of a cell having the cell_name within top_cell. The cell given by a cell_name parameter appears in some other database specified in the Layout Path. The placement transform is given by the magnification, the angle, the reflection and the (x, y) location of the cell origin.

The placed_cell_info and EMPTY arguments are mutually exclusive. The EMPTY argument creates a top-level cell with no placements for use with MDP Embed and Fracture direct data entry.

The usage model is normally that the Layout Path and Layout Primary specification statements reference this new database. The Layout System is specified accordingly.

Note:

If Layout Place Cell is used in conjunction with Layout Use Database Precision, then the filename may not be the first file referenced in a Layout Path statement.

Examples

Example 1

This example demonstrates how to place primary cells in separate layouts into the same design.

LAYOUT PATH new.gds A.gds B.gds    // A.gds contains cell A 
                                   // B.gds contains cell B
LAYOUT PRIMARY new_top
LAYOUT SYSTEM GDSII
...
// The new input layout database.
LAYOUT PLACE CELL new.gds GDS new_top 
	A 1.5 90  0 65.3  89.7 // places cell A
	B 1.0 270 1 987.3 0    // places cell B

Example 2

Layout Primary “*” does not recognize multiple primary cells. It recognizes only the first one in the data stream.

This example demonstrates how to place two primary cells from the same layout into a common design. Placing both primary cells under a new top cell allows both primary cells to be read.

// layout.gds contains primary cells A_top and B_top
LAYOUT PATH new.gds layout.gds 
LAYOUT PRIMARY top // cannot use "*" to recognize multiple primary cells
LAYOUT SYSTEM GDSII
...
// The new input layout database.
LAYOUT PLACE CELL new.gds GDS top 
	A_top 1 0 0 0.0    0.0 // places primary cell A_top
	B_top 1 0 0 1000.0 0.0 // places primary cell B_top
