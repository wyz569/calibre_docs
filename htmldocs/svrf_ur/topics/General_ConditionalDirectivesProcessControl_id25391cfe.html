About Conditional Directives for Compilation Control

The portions of the rule file that Calibre or ICverify compiles can be controlled with the #DEFINE, #UNDEFINE, #IFDEF, #IFNDEF directives within your rule file. These directives define and undefine variables and provide conditional execution based on the variable definition. Variables for compilation control can also be defined as environment variables in the shell. Usage information and examples are provided.

Refer to #IFDEF CALIBRE_VERSION for a pre-processor directive that provides conditional compilation based on the Calibre version.

The following sections are included:

  • Defining Variables for Process Control

  • Undefining Variables

  • Conditional Directives (#IFDEF and #IFNDEF)

  • Examples

Defining Variables for Process Control

You can define variables for technology process control with the #DEFINE directive or in the shell environment.

#DEFINE Usage

     #DEFINE var_name [var_value]

The variable name is given by var_name and an optional value is given by var_value. You should enclose var_name within quotes if var_name includes spaces or characters other than letters and numbers.

You can also implicitly define a variable in the shell environment, in which case the #DEFINE statement is not needed.

A variable var_name is considered defined if either of the following are true:

Explicit definition

The variable name (var_name) appears in a #DEFINE directive before it is referenced in a conditional block, and the var_name does not appear in an #UNDEFINE statement prior to its reference in a conditional block.

Implicit definition

The variable name (var_name) appears with a dollar character ($) before it, and var_name is defined as a non-null shell environment variable.

The following is an example of an implicit definition of a variable in the C shell environment:

setenv process 7lm

This is an equivalent example for the Bourne shell:

export process=7lm

With the preceding implicit variable definitions, the string “$process” represents the variable name in an conditional statement (#IFDEF or #IFNDEF, which are discussed later in this topic). In these examples, the “$process” variable resolves to a value of “7lm”.

In an explicit variable definition, the following construct establishes “process” as defined, with no assigned value:

#DEFINE process

The following construct defines the variable name “process” and assigns it the value “7lm”: 

#DEFINE process 7lm

This is analogous to the previous environment variable assignment examples, although “process” and “$process” are not the same variable name.

The value of the variable (var_value) may not be specified as an environment variable in a #DEFINE directive. For instance, $value is not interpreted as a variable in the following rule file excerpt:

//in the shell, $value = 7lm
#DEFINE process $value // value is not a variable here

In this example, $value is treated as a literal string, not a variable.

You can override an implicit variable definition in the rule file. Suppose you define an environment variable (for example, setenv myvar xyz), such that the variable is specified with an implicit definition ($myvar resolves to xyz in conditional statements). The following statement redefines $myvar:

#DEFINE $myvar newvalue

This statement causes $myvar to resolve to newvalue (for pre-processor commands only), regardless of the shell environment. In other words, #DEFINE directives can override implicit variable definitions from the shell environment, but they have no effect on the shell itself.

Undefining Variables

The #UNDEFINE directive is used to undefine a variable. It can be used to undefine an explicitly defined variable (one defined with #DEFINE), or an implicitly defined variable (one defined in the shell), or a variable that has not been previously defined. In the latter case, the purpose would be to initialize the variable in the undefined state.

#UNDEFINE Usage

     #UNDEFINE var_name

The following example undefines a variable defined in the shell:

#UNDEFINE $process //undefines the implicit name definition

This example undefines a variable explicitly defined with a #DEFINE statement in the rule file:

#DEFINE process
...
#UNDEFINE process  //undefines the explicit name definition
...

The two preceding examples do not undefine the same names. #UNDEFINE statements do not affect the shell environment, only the rule file pre-processor.

The way to change the state of a conditional compilation pre-processor directive is to change the directive to its opposite, not to comment it out. For example, suppose you have the following conditional directive:

#DEFINE name

You should not precede this with a comment character because the pound sign (#) must be the first character on the line for any pre-processor directive. C-style comments (/* … */) do not hide the directive because they are processed after pre-processor directives when there is a dependency. The following statement is the correct way to negate the definition:

#UNDEFINE name

Conditional Directives (#IFDEF and #IFNDEF)

Conditional directives provide conditional compilation for sections of the rule file. #IFDEF (if defined) and #IFNDEF (if not defined) test whether a variable is defined or undefined, respectively, and optionally compare to a specified value. An #IFDEF or #IFNDEF statement can include an optional #ELSE statement, which is executed if the preceding #IFDEF or #IFNDEF conditional evaluates to false.

#IFDEF and #IFNDEF Usage

{#IFDEF var_name [var_value] | #IFNDEF var_name [var_value]}

          rule_file_text

[ #ELSE

          rule_file_text ] ...

#ENDIF

The variable name is var_name; it is a string that is defined with the #DEFINE directive or implicitly defined in the shell environment. If var_name is defined in the shell environment, it should appear with an initial dollar character ($) when used in a rule file conditional directive.

The var_value is an optional string that appears in a #DEFINE directive or it is the value assigned to a variable in the shell environment. If var_value is present, the value of the variable var_name is compared by a case-insensitive string comparison to var_value.

The string rule_file_text includes rule file statements and operations to be performed, depending on the conditional directive. The rule_file_text string can also contain conditionals, which creates nested pre-processor directives. (The SVRF Error statement is useful for inserting compiler error messages within conditional constructs.)

For the following discussion of conditionals, “associated value” refers to one of these cases:

  • The value assigned to the variable var_name in the most recent #DEFINE directive, prior to the #IFDEF or #IFNDEF conditional.

  • The non-empty string, if any, assigned to the var_name variable in the shell environment.

“#IFDEF var_name [var_value]” enables or disables conditional compilation as follows:

  1. If var_name is not defined, then compilation is disabled.

  2. Else if var_name is defined with a value, but var_value is not specified in the #IFDEF, then compilation is enabled.

  3. Else if var_name is defined with no associated value and var_value is specified, then compilation is disabled.

  4. Else if var_value equals the assigned value of var_name (by case-insensitive string comparison), then compilation is enabled.

  5. Else compilation is disabled.

“#IFNDEF var_name [var_value]” enables or disables conditional compilation as follows:

  1. If var_name is not defined, then compilation is enabled.

  2. Else if var_name is defined with a value, but var_value is not specified in the #IFNDEF, then compilation is disabled.

  3. Else if var_name is defined with no associated value andvar_value is specified, then compilation is enabled.

  4. Else if var_value equals the assigned value of var_name (by case-insensitive string comparison), then compilation is disabled.

  5. Else compilation is enabled.

Examples

Example 1

Process P1 states that metal6 is the top metal layer, process P2 states that metal5 is the top metal layer, and in all other processes metal4 is the top metal layer. The arguments P1 and P2 would appear in #DEFINE or #UNDEFINE statements in a control file, which would include the following code:

LAYER metal4 23
LAYER metal5 26
LAYER metal6 29

#ifdef P1 
  LAYER top_metal metal6
  #else
    #ifdef P2
      LAYER top_metal metal5
      #else
        LAYER top_metal metal4
    #endif
#endif

Example 2

In this example, the name parameter P1 is initially defined for the conditionals, then it is undefined (order is important):

#DEFINE P1
…			//rule file statements
#IFDEF P1
…			//conditional statements are compiled
#ENDIF
#IFNDEF P1
…			//conditional statements are not compiled
#ENDIF
#UNDEFINE P1
#IFDEF P1
…			//conditional statements are not compiled
#ENDIF
#IFNDEF P1
…			 //conditional statements are compiled
#ENDIF

Example 3

This shows use of name value pairs in an explicit definition. P1 is explicitly defined:

#DEFINE P1 7lm //the associated value of P1 is “7lm”
#IFDEF P1
…        //conditional statements are compiled
#ENDIF
#IFDEF P1 7lm
...      //conditional statements are compiled 
         //P1 = 71m from the #DEFINE directive
#ENDIF
#IFDEF P1 P1
...      //conditional statements are not compiled 
         //P1 = 7lm from the #DEFINE directive
#ENDIF
#IFNDEF P1
...      //conditional statements are not compiled
#ENDIF
#IFNDEF P1 7lm
...      //conditional statements are not compiled 
         //P1 = 7lm from the #DEFINE directive
#ENDIF
#IFNDEF P1 P1
...     //conditional statements are compiled 
#ENDIF

Example 4

Here the name is implicitly defined as a shell variable. Note the #UNDEFINE directive does not affect the environment variable declared in this example. “$P1” is a non-null shell environment variable:

…         //rule file statements
#IFDEF $P1
...       //conditional statements are compiled
          //$P1 is defined in the shell
#ENDIF
#IFNDEF $P1
...       //conditional statements are not compiled
#ENDIF
#UNDEFINE P1
#IFDEF $P1
...       //conditional statements ARE compiled because 
          //the previous #UNDEFINE does not affect $P1
#ENDIF
#IFNDEF P1
...       //conditional statements are compiled
          //P1 is currently undefined
#ENDIF

Example 5

This example uses an implicit definition to assign a value to a variable. “$P1” is a non-null shell environment variable assigned the value 71m:

#IFDEF $P1 7lm
...       //conditional statements are compiled
          //$P1 = 7lm in the shell environment
#ENDIF
#IFNDEF $P1 7lm
...       //conditional statements are not compiled
#ENDIF
#UNDEFINE P1
#IFDEF $P1
          //conditional statements are compiled
          //previous #UNDEFINE P1 does not affect $P1
#ENDIF
#DEFINE $P1 P1 //$P1 reassigned to value P1
#IFDEF $P1 P1
...       //conditional statements are compiled
#ENDIF
