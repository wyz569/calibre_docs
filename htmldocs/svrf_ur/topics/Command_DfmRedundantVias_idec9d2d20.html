DFM Redundant Vias

Layer operation

License: Calibre YieldAnalyzer

Identifies redundant or non-redundant polygons on a layout and produces a derived polygon layer containing the identified polygons.

Usage

DFM [NON] REDUNDANT VIAS layer vra_spec
[HIERARCHICAL | FLAT] [QUANTIFY [RGID] [RGFS] [RGLS] [PRGID] [PRGFS]]

Arguments

NON

An optional keyword that instructs the operation to identify non-redundant polygons instead of redundant polygons. If this argument is not specified, the operation identifies redundant polygons.

layer

A required argument that specifies the layer on which redundant or non-redundant polygons are identified. This layer is typically a via layer, and must be a Connect layer.

vra_spec

A required argument that specifies the name of a via redundancy analysis specification created with the DFM Spec Via Redundancy statement.

HIERARCHICAL | FLAT

An optional keyword pair that specifies whether the operation runs in hierarchical or flat mode. The default is hierarchical.

In hierarchical mode, every redundant or non-redundant polygon is saved in the lowest-level cell that fully contains the hierarchical net to which the polygon belongs. The layer connectivity defined in the vra_spec must be compatible with the global layer connectivity defined by the Connect statements so that correct connectivity information can be found on all layers specified in the vra_spec. Hierarchical mode supports MT and MTflex processing where the nets are processed in parallel.

In flat mode, the operation processes an internally-flattened layout and produces flat output — all redundant or non-redundant polygons are saved in the top-level cell. In this mode, connectivity is not required to be present: there are no restrictions are placed on the connectivity defined in the vra_spec.

QUANTIFY [RGID] [RGFS] [RGLS] [PRGID] [PRGFS]

An optional keyword set that enables the output of additional information for redundant groups of polygons in terms of vias; a redundant via group is the largest set of vias that are connected together when all single vias are removed. The keywords RGID, RGFS, RGLS, PRGID, and PRGFS enable the creation of properties on the output layer. Specifically, the RGID, RGFS, and RGLS keywords pertain to vias that are electrically redundant, whereas PRGID and PRGFS pertain to vias that are both electrically and polygonally redundant (as for the case of vias in the same net).

  • RGID — When specified, a property named rgid (redundant group ID) is created on the output layer. This property contains a numeric value that is assigned to all polygons associated with a particular group of redundant polygons. The property value may change between runs, especially between multithreaded runs. Each group is assigned a unique rgid value.

  • RGFS — When specified, a property named rgfs (redundant group full size) is created on the output layer. This property contains a numeric value that indicates the number of shapes of the redundant group to which a particular polygon is associated with, for all layers in the group.

  • RGLS — When specified, a property named rgls (redundant group layer size) is created on the output layer. This property contains a numeric value that indicates the number of shapes of the redundant group to which a particular polygon is associated with, for the input layer only.

  • PRGID — When specified, a property named prgid (polygonal redundant group ID) is created on the output layer. This property contains a unique numeric value that is assigned to each group of polygonally redundant vias. Within one group, each via has the same PRGID value. The property value may change between runs, especially between multithreaded runs. Each group is assigned a unique prgid value.

  • PRGFS — When specified, a property named prgfs (polygonal redundant group full size) is created on the output layer. This property contains a numeric value that indicates the number of redundant polygons in each redundant group. Depending on the definitions in the DFM Spec Via statement, this value may contain vias from multiple layers.

By default, all properties are output when none of the keywords RGID, RGFS, RGLS, PRGID, and PRGFS are specified. The QUANTIFY keyword set applies only to the DFM Redundant Vias operation. The QUANTIFY keyword may not be used with the DFM NON REDUNDANT VIAS operation.

Description

Identifies redundant or non-redundant polygons on a layout and produces a derived polygon layer containing the identified polygons. A redundant polygon is defined as a polygon that may be removed without affecting the connectivity of the input layers.

Note:

This operation outputs only functional polygons. DFM Redundant Vias and DFM NON Redundant Vias may not be complementary operations if a design contains non‑functional polygons. Refer to “Definition of Functional and Non‑Functional Vias” for via redundancy.

Definition of Functional and Non‑Functional Vias

Polygons are defined as functional vias for via redundancy analysis when all the following conditions are met:

  • The design meets DRC constraints.

  • The interacting metal layer shapes overlap.

  • The representation point of the via shape is enclosed by both the metal layers.

    The representation point of a via shape is defined in the following way:

    • If the ratio of the via shape area and its bounding box area is greater than 0.9, then the via representation point is defined as the center of the via shape bounding box area. See Figure 3 case (a).

    • If the ratio of the via shape area and its bounding box area is less than 0.9, then the via representation point is defined as the top right vertex of the via shape. See Figure 3 case (b).

If the functional via constraints are not met, a via is considered to be non‑functional and is not included in the via redundancy analysis. Non‑functional vias can affect the count of functional vias in the same net and create the potential for a non‑single via to be counted as a single via.

Figure 3 demonstrates the effect on via redundancy analysis when functional and non‑functional vias lie in the same net.

case (a)

One non‑functional via due to a via center representation point that is not enclosed by the layer 1 metal shape. The resulting via count is reported as three single vias.

case (b)

One non‑functional via due to a non‑rectangular via shape with a top right vertex representation point that is not enclosed by the layer 1 and layer 2 metal shapes. The resulting via count is reported as three single vias.

case (c)

All functional vias result in a reported via count of four redundant vias.

case (d)

All functional vias result in a reported via count of four redundant vias.

Hierarchy Treatment

The DFM Redundant Vias operation offers similar functionality to that of DFM Copy; however, it offers advantages in how hierarchy is treated. Particularly, any connection that causes a redundant via but is outside the cell hierarchy where the original via exists is included in the output. This operation must be used in a hierarchical Calibre nmDRC or DFM run; flat runs produce empty output.

Run Scalability

  • The CALIBRE_DFM_VRA_MAX_EDGES_PER_TILE environment variable can be used to improve the scalability of this operation when running the job in multithreaded mode or with Calibre MTflex. This variable specifies the upper bound for the tile size as the average flat edge count per tile, and should be set to a reasonable positive value. The default value is 10,000,000.

  • In order to run DFM REDUNDANT VIAS or DFM NON REDUNDANT VIAS on remote hosts efficiently with Calibre MTflex in hyperscaling mode, a MERGE operation should be performed on the via layers being analyzed. The merged via layers must appear in CONNECT statements and DFM SPEC VIA REDUNDANCY statements.

    See the Calibre Administrator’s Guide for complete run mode information.

  • The recommended command line arguments for the run mode are as follows:

    calibre {‑drc|‑dfm} ‑hier ‑turbo [#CPUs] ‑remotefile filename \
		‑hyper remote ‑remotedata ‑recoveroff rules

Examples

Specify connectivity layers and a via redundancy analysis specification with DFM Spec Via Redundancy. Then use DFM Redundant Vias to identify redundant and non‑redundant vias and produce a derived layer containing the identified vias:

VIA1 = MERGE (VIA1square OR VIA1bar)
CONNECT M1 M2 BY VIA1
DFM SPEC VIA REDUNDANCY m1m2_connectivity
CONNECTIVITY
	M1 VIA1 M2 ;
END CONNECTIVITY
// find redundant vias on layer VIA1
m1m2_redundant_via1 = DFM REDUNDANT VIAS VIA1 m1m2_connectivity
// find non-redundant vias on layer VIA1
m1m2_nonredundant_via1 = DFM NON REDUNDANT VIAS VIA1 m1m2_connectivity
