device_numeric_value

Device annotation language function.

Returns the numeric value associated with a property.

Usage

DEVice_NUMeric_VALue(“property_name”)

Arguments

property_name

A required DEVice property name.

Return Values

Floating-point numeric value.

Description

Returns the numeric value associated with the property_name. The property_name is expected to be numeric and is calculated by the Device operation.

This function can access properties that are removed by LVS Push Devices SEPARATE PROPERTIES. Doing so causes promotion of seed shapes for which the properties are calculated to the appropriate hierarchical level in order to compute the properties. For access to succeed, two conditions must be met:

  • All device shapes selected by the Device Layer operation must originate from a single seed layer. Failure to ensure this results in a DVL7 compiler error.

  • A property accessed by this function must have the same LVS-actionable status for all devices selected by the Device Layer operation (more than one device model can be selected if no model name is used). Failure to ensure this results in a DVL8 compiler error. LVS-actionable status is discussed under LVS Netlist Property.

To overcome these errors, you can add a model name to the Device Layer operation or use the SELECT statement in the device annotation program. Alternatively, in the case of the DVL8 error, you can change the LVS-actionable status of the devices by declaring them in LVS Netlist Property statements (although this is not the only method).

Examples

Example 1

Consider the following Device statement:

DEVICE mp(p) pgate pgate(g) sd(s) sd(d) bulk(b)

By default, properties L and W are calculated for built-in MOS devices. You could use this code to access the L property in a device annotation program:

X = DEVICE LAYER mp(p) ANNOTATE dfm_layer [
    PROPERTY p
    p1 = DFM_NUM_VAL( dfm_layer, "P1" ) // DFM property value
     L = DEV_NUM_VAL( "L" ) // DEVice property value
     p = p1 * L
]

LVS ANNOTATE DEVICES X

Any property that is calculated for a device can be accessed by DEVice_NUMeric_VALue using a similar method.

Example 2

Consider the following rule file excerpt:

LVS PUSH DEVICES YES
LVS PUSH DEVICES SEPARATE PROPERTIES YES
DEVICE MN(N1) L1 L2(g) L2(s) L2(d)
[
    PROPERTY AS, AD
    AS = AREA(s)
    AD = AREA(d)
]
DEVICE MN(N2) L1 L2(g) L3(s) L3(d)
[
    PROPERTY AS, AD
    AS = AREA(s)
    AD = AREA(d)
]
TRACE PROPERTY MN(n1) AS AS 1
TRACE PROPERTY MN(n1) AD AD 1
//TRACE PROPERTY MN(n2) AS AS 1
//TRACE PROPERTY MN(n2) AD AD 1
L99 = DEVICE LAYER MN ANNOTATE [
    PROPERTY A1, A2
    A1 = DEVice_NUMeric_VALue("AS")
    A2 = DEVice_NUMeric_VALue("AD")
]

This configuration generates a DVL8 compiler error because the properties AS and AD do not have the same LVS-actionable status for both MN devices. This is due to the differing Trace Property configurations. You can mitigate the error by declaring the device properties in an LVS Netlist Property statement.
