LVS Expand On Error

Specification statement

Specifies whether to examine LVS comparison discrepancies and, in certain cases, to automatically perform additional comparison runs with fewer hcells.

Usage

LVS EXPAND ON ERROR {NO | YES}

Arguments

NO

A keyword that specifies not to remove hcells from the current hcell list during comparison. This is the default behavior.

YES

A keyword that specifies to examine the LVS comparison results. If there are potentially resolvable discrepancies that are likely due to the hcell list, then certain cells are removed from the list and an additional LVS comparison run occurs with the reduced list. This can occur iteratively.

Description

Controls the expansion of hcells that cause false LVS comparison discrepancies. This statement may be specified at most once. Mask SVDB Directory must be specified when YES is used.

Some hcells lead to LVS errors that can be mitigated by expanding them. For example, if an hcell in the source design contains a device that is in its parent hcell in the layout design, there will be LVS errors unless that hcell is expanded.

When YES is specified, hcells are examined to determine if they can cause false discrepancies. If there are potentially resolvable discrepancies that are likely due to the hcell list, then hcells are iteratively removed from an internal hcell list maintained by the tool. Additional comparison runs may occur with problematic hcells automatically removed from the internal list, thus expanding them. This can mitigate false discrepancies.

As part of the hcell expansion process, a bad-hcell prediction algorithm is used. The prediction algorithm assesses whether an hcell will likely contribute to LVS discrepancies. If no hcells are predicted to cause LVS discrepancies, LVS comparison follows immediately.

Bad-hcell prediction occurs after all flattening and transforming has been done, just prior to comparison. If bad hcells are identified, they are removed from the internal hcell list. LVS is immediately restarted before comparison occurs, so no LVS report is generated for the partial run. After restarting, Calibre reads and transforms the netlists again. Then it reruns bad-hcell prediction, which may find additional bad hcells, leading to their removal from the internal list followed by another restart. When no more bad hcells are found, LVS moves on to comparison with the revised hcell list.

The comparison results are evaluated for bad hcells when the results are INCORRECT. Bad hcells are marked for removal from the internal list if found (more details are discussed later). In cases where post-comparison evaluation removes hcells and restarts LVS, bad-hcell prediction runs again in each new pass.

When bad-hcell prediction finds bad hcells, this is reflected in the transcript as follows:

Performing PRECOMPARISON PREDICTION work ...

Predicting BAD HCELLS ...

  Predicted bad hcells:
    cell1   cell1
    cell2   cell2

BAD-HCELL prediction completed.  CPU TIME = 0  REAL TIME = 0  LVHEAP = 1/
3/3  MALLOC = 40/40/40  ELAPSED TIME = 0

PRECOMPARISON PREDICTION work completed.  CPU TIME = 0  REAL TIME = 0  
LVHEAP = 1/3/3  MALLOC = 40/40/40  ELAPSED TIME = 0

Hcells were predicted bad. Restarting LVS.

Initializing LVS ...

If no bad hcells are found, none are listed, and LVS is not restarted. This appears in the transcript:

Performing PRECOMPARISON PREDICTION work ...
...
PRECOMPARISON PREDICTION work completed.  CPU TIME = 0  REAL TIME = 0  
LVHEAP = 1/3/3  MALLOC = 40/40/40  ELAPSED TIME = 0

Sometimes bad-hcell prediction finds hcells that cannot be expanded because they are listed in cell lists, or because instances of them are contained in other hcells listed in cell lists. The transcript output for such cases looks like this:

Predicted bad hcells:
    cell1   cell1   (not expandable due to cell list)
    cell2   cell2

If none of the bad hcells can be expanded, LVS does not restart, and they are only listed in the transcript. There is no other side effect.

When bad hcells are identified after comparison is performed and automatic expansion occurs, a supplemental LVS report including only INCORRECT cells is stored in the Mask SVDB Directory. Each report has the suffix “.pass_n” where n is an index number corresponding to the comparison run, starting from 1. No design cross-reference information is recorded for any but the final comparison run.

The LVS reports contain a section called EXPAND ON ERROR DETAILS. This section contains a running list of supplemental report paths and a list of the hcells that were expanded during the run.

Bad hcells detected by the prediction algorithm that were expanded are listed in the EXPAND ON ERROR DETAILS portion of an LVS report. Here is an example:

*************************************************************************  
                             EXPAND ON ERROR DETAILS                         
*************************************************************************

 The INCORRECT LVS report prior to EXPAND ON ERROR is available in:     
                                
         qaout.svdb/qaout.lvs.rep.pass_1    
                                                               
 CELLS EXPANDED DUE TO BAD-HCELL PREDICTION:      
                                                      
    Result         Layout                        Source
    -----------    -----------                   --------------    
    EXPANDED       cell1                         cell1            
    EXPANDED       cell2                         cell2 

CELLS EXPANDED DUE TO ERRORS PRIOR TO EXPANSION:    

    Result         Layout                        Source
    -----------    -----------                   --------------   
    EXPANDED       cell3                         cell3

There is also an overall status message when bad-hcell prediction expands hcells:

Warning:  Some hcells were predicted bad prior to comparison and expanded (LVS EXPAND ON ERROR).

Hcells to be expanded in a subsequent LVS comparison run appear in an LVS EXPAND ON ERROR WILL REMOVE THE FOLLOWING HCELL(S) IN THE NEXT RUN subsection at the end of the report. In any supplemental report after the first one, a warning is issued in the overall status summary for that run:

Warning:  Some hcells were expanded with recoverable errors (LVS EXPAND ON ERROR).

Conditions Affecting Hcell Expansion After Comparison

These conditions apply after LVS comparison has run and the prospect of reducing the tool’s internal hcell list is analyzed.

If either a containing hcell or a contained hcell is specified in an LVS Cell List statement, then the contained cell will not be removed from the hcell list.

The conditions that cause hcell expansions when YES is specified include unbalanced numbers of instances, parallel or series reduction of devices across the hierarchy, and many-to-one hcell correspondences for which an expansion occurs on either the “many” or the “one” side of the relation.

Taking the simplest case first, for many-to-one hcells, all the cells in the many-to-one relation are in a group of equivalent cells. If any member of the group is removed from the hcell list, then all other hcells in the group are removed.

Unbalanced numbers of instances occur when a layout hcell contains a different number of instances than a corresponding source hcell. In this situation, the LVS report shows the discrepancy in the summary section for the cell. For example:

NUMBERS OF OBJECTS AFTER TRANSFORMATION
---------------------------------------

               Layout    Source         Component Type
               ------    ------         --------------

Ports:           3433      3433
Nets:            7823      7823
Instances:         41        40    *    MN (4 pins)
                   48        48         MP (4 pins)

...

    * = Number of objects in layout different from number in source.

Automatic expansion occurs as follows for hcells in unbalanced pairs:

  • For a member of an unbalanced hcell pair, if expanding a certain type of contained hcell instance can reduce the difference in instance counts, then that contained hcell is removed from the hcell list.

    For example, consider cell1 and cell2 designated as hcells with cell2 placed inside cell1. A resistor is misplaced by moving it from cell2 to cell1 in one of the designs. After the first comparison, both cell1 and cell2 are INCORRECT.

    * layout                  * source
.subckt cell2 1 2         .subckt cell2 1 2
R0 1 2 r=1                .ends
.ends

.subckt cell1             .subckt cell1
X0 1 2 cell2              R0 1 2 r=1
.ends                     X0 3 4 cell2
                          .ends

    LVS Expand On Error YES will cause cell2 to be removed from the hcell list, and the subsequent LVS comparison will be CORRECT.

  • For an unbalanced hcell pair, if some hcell instances within one of those cells are also members of unbalanced pairs, and expanding all of those hcell instances resolves the imbalance with the containing hcell, then all of the cells referenced by those instances are removed from the hcell list.

    For example, consider cell1, cell2, and cell3 designated as hcells with cell2 and cell3 placed inside cell1. All three pairs are unbalanced.

    * layout                  * source
.subckt cell2 1 3         .subckt cell2 1 2
R0 1 2 r=1                .ends
R1 2 3 r=1
.ends

.subckt cell3 1 2         .subckt cell3 1 3 
R0 1 2 r=1                R0 1 2 r=1
.ends                     R1 2 3 r=1
                          .ends

.subckt cell1             .subckt cell1
X0 1 2 cell2              R0 1 2 r=1
X1 2 3 cell3              R1 2 3 r=1
R0 3 4 r=1                X0 3 4 cell3
.ends                     .ends

    Resistors in cell1, cell2, and cell3 would produce an INCORRECT result, but the aggregate resistor count for all the cells together is correct. LVS Expand On Error YES will cause cell2 and cell3 to be removed from the hcell list, and the subsequent LVS comparison will be CORRECT.

  • For a balanced hcell pair, if one of those cells contains instances of hcells that are members of unbalanced pairs, and the original pair is still balanced after expanding the instances whose hcells are unbalanced, then all of the cells referenced by those instances are removed from the hcell list.

    For example, consider cell1, cell2, and cell3 designated as hcells with cell2 and cell3 placed inside cell1. cell2 and cell3 are unbalanced.

    * layout                  * source
.subckt cell2 1 2         .subckt cell2 1 2
R0 1 2 r=1                .ends
.ends

.subckt cell3 1 2         .subckt cell3 1 2
.ends                     R0 1 2 r=1
                          .ends

.subckt cell1             .subckt cell1
X0 1 2 cell2              R0 1 2 r=1
X1 2 3 cell3              X0 2 3 cell2
R0 3 4 r=1                X1 3 4 cell3
.ends                     .ends

    Resistors in cell2 and cell3 would produce an INCORRECT result, but the aggregate resistor count for all the cells together is correct. LVS Expand On Error YES will cause cell2 and cell3 to be removed from the hcell list, and the subsequent LVS comparison will be CORRECT.

The expansion of an hcell cannot undermine any containing hcell’s balance. If a balanced hcell pair becomes unbalanced after expanding all contained instances in a member of the pair, then the contained instances' cells will not be expanded, even if they help to improve other containing hcells’ balances.

Reducing parallel and series devices across the hierarchy occurs when the device instances in a cell “C” can be reduced with other device instances in cells contained in C, or several instances in different cells within C can be reduced together. For this to occur, either the containing cell or the contained cell has an unequal instance count for that reducible device.

For the purposes of LVS Expand On Error, the reduction can be either parallel or series as specified in applicable LVS Reduce statements for built-in devices. Device reduction using the generic LVS Reduce statement is unsupported.

When cells with unequal numbers of devices can have their devices reduced together with devices in other cells, which themselves may have equal or unequal device counts, then the relevant cells are removed from the hcell list which permits the reduction.

Consider cell1 and cell2 as hcells with cell2 placed inside cell1. In one netlist, two parallel resistors are placed in cell1 and cell2, with one resistor in each cell. In the other netlist, a single reduced resistor is placed in cell1. 

*layout                 *source
.subckt cell2 1 2       .subckt cell2 1 2
R0 1 2 r=1              .ends
.ends

.subckt cell1           .subckt cell1
R0 1 2 r=1              R0 1 2 r=0.5
X0 1 2 cell2            X0 1 2 cell2
.ends                   .ends

After the first comparison, both cell1 and cell2 are INCORRECT due to mismatched resistors. LVS Expand On Error YES will cause cell2 to be removed from the hcell list, and the subsequent LVS comparison will be CORRECT.

In the preceding example, layout instance cell1:R0 could instead be in an instance X1 of cell2, such that the resistor placements are split as X0/R0 and X1/R0. The source could have an extra instance cell1:X1 of the empty cell2. The outcome would be the same.

See also LVS Auto Expand Hcells, LVS Expand Unbalanced Cells, LVS Exclude Hcell, LVS Hcell Report.

Examples

Example 1

// potentially reduce the number of inappropriate hcells during
// circuit extraction and automatically perform additional LVS comparison
// runs with reduced hcell lists if false discrepancies occur.
// provide a detailed hcell report.

LVS REPORT lvs.report
MASK SVDB DIRECTORY svdb QUERY
LVS HCELL REPORT hcells.details

LVS AUTO EXPAND HCELLS PRESET
LVS EXPAND ON ERROR YES
