DFM Analyze

Layer operation

License: Calibre YieldAnalyzer

Identifies areas on a layout where manufacturing yield can potentially be improved.

Usage

DFM ANALYZE input_layer [input_layerN …]
[‘[’expression‘]’ [constraint]]
[ANNOTATE ‘[’ name = “value” ‘]’] …
[COMMENT comment_string] …
[INSIDE OF {
   EXTENT |
   {x1 y1 x2 y2} |
   {LAYER layerB [CENTERED value]}
   }
]
[RDB [COORD] [DV] [DX] [ONLY] filename]
[BY CELL
   [ALLDEPTH]
   [NOHIER]
   [NOPSEUDO]
   [RESULT CELLS cell_list]
| WINDOW {w | wx wy}
   [BACKUP]
   [CENTERS [c | cx cy]]
   [{NUMWIN {n | nx ny} [OVERLAP {o | ox oy}]} | STEP {s | sx sy}]
]

Arguments

input_layer [input_layerN …]

A required original layer, derived polygon layer, derived edge layer, or derived error layer. DFM Analyze accepts derived error layers as input for use in certain measurement functions. For more information see “Measurement Functions”. Any number of input_layers can be specified.

‘[’expression‘]’ [constraint]

An optional expression that performs a calculation involving layer data. The expression may be accompanied by an optional constraint.

The expression may involve any of the following:

  • Numbers, including numeric variables

  • Variables

  • Math Operators (+, -, !, ~, ^, *, /, +, -, &&, ||)

  • Parentheses ( )

  • Supported Math Functions in DFM Expressions

  • Conditional Expressions

  • User-defined functions created by DFM Function

  • DFM Properties

  • Most Measurement Functions

The standard rules of operator precedence apply, and parentheses may be used to establish precedence within the expression.

  • The expression must be enclosed in brackets ([ ]).

  • The default expression is:

    [f1(layer1) + f2(layer2) + … + fN(layerN)]

    where:

    • fi is AREA(layeri) if layeri is an original or derived polygon type

    • fi is LENGTH(layeri) if layeri is a derived edge layer

    • fi is COUNT(layeri) if layeri is a derived error layer

  • If an expression is not specified, the constraint defaults to greater than or equal to zero (>= 0).

  • Division by 0 never satisfies a constraint.

For more information on the expression, refer to “DFM Expressions”.

  • constraint — Specifies operators and numbers in user units that can only be specified with an expression. The value specified by the constraint can be positive or negative. If no expression or constraint is specified, the constraint defaults to greater than or equal to zero (> = 0). For more information see “Constraints”.

ANNOTATE ‘[’name = “value” ‘]’

An optional keyword that associates meta data with the _detail layer that is written to a DFM database. The _detail layer is produced in addition to the output layer normally produced by the operation. Unlike the output layer normally produced, which contains merged geometries, the _detail layer contains unmerged geometries. This keyword is enabled only when running Calibre with the ‑dfm command line option. It is ignored when running Calibre with the ‑drc command line option. You can specify the ANNOTATE keyword any number of times.

  • ‘[’name = “value” ‘]’

    The following constraints apply to the name and value pair:

    • Brackets ([ ]) must enclose the name and value pair.

    • The name may be quoted or unquoted, and does not need to be unique.

    • The value must be enclosed in quotes.

Each keyword occurrence adds additional annotations (specified by the name and value pair) to the layer.

COMMENT comment_string

An optional keyword that adds comments to the RDB. Each COMMENT keyword writes a comment_string as text to the RDB. See “Scores RDB File Format” in the Calibre YieldAnalyzer and YieldEnhancer User’s and Reference Manual for more information. You may specify multiple COMMENT keywords.

  • comment_string

    Specifies the comment string. A comment_string that contains spaces or special characters must be enclosed in double quotes (“ ”). There is no default comment_string.

INSIDE OF {EXTENT | {x1 y1 x2 y2} | LAYER layerB [CENTERED value]}

An optional keyword set that defines a rectangular boundary within which a data capture window moves. If you do not specify INSIDE OF, the default boundary is the extent of the layout data read in for the run. If you specify WINDOW with a size that is greater than any INSIDE OF argument, the window size is truncated at the data capture extent defined by the value of the argument.

  • EXTENT

    Specifies that the boundary is the extent of the input layer. This is the default argument.

  • x1 y1 x2 y2

    Specifies the corners of a rectangle in user units. Negative values must be enclosed in parentheses.

  • LAYER layerB [CENTERED value]

    An optional keyword that specifies the boundary as the area of the polygonal extents of layerB, and not the polygons themselves. The layerB argument must be an original or derived polygon layer and may not be a layerN parameter previously referenced within an expression.

    • CENTERED value

      An optional keyword used by INSIDE OF LAYER that specifies the data capture extents are squares with a side length of the value in user units. The center of each square data capture extent is the center of the extent of each layerB polygon. The data capture extent may extend beyond the layerB extent depending on the value parameter, as shown in Figure 1.

Note:

When CENTERED is specified, the size of the data capture extent is determined by value, not the polygons on layerB. Depending on value, the size of the data capture extent may be larger or smaller than the extent of the shapes on layerB.

RDB [COORD] [DV] [DX] [ONLY] filename

An optional keyword that generates a Scores RDB with the given filename. The filename must be specified after the keyword options. Within the filename, the pattern substitution facility replaces all occurrences of the pattern “%_t_” with the name of the top cell for the design. See “Scores RDB File Format” in the Calibre YieldAnalyzer and YieldEnhancer User’s and Reference Manual for details on the RDB format. The keyword options COORD, DX, DY, and ONLY control what is written to the RDB or standard DRC database.

  • COORD

    An optional keyword specifying the window coordinates for the capture window.

  • DV

    An optional keyword specifying the value of the DFM Analyze expression per window (DV property).

  • DX

    An optional keyword specifying the values of a sub-expression, such as the DA properties for AREA measurement.

  • ONLY

    An optional keyword specifying that data is sent only to the Scores RDB. No geometric data is sent to the standard Calibre nmDRC results database.

By default, the operation writes all these types of data to the RDB. There is no keyword provided to not write data to the RDB, because this would result in meaningless RDB output.

Note:

If you create an RDB with partial data, you may not be able to use that RDB as input to some downstream tools. For example, Calibre RVE requires that COORD (window coordinates) be present in the RDB.

BY CELL [ALLDEPTH] [NOHIER] [NOPSEUDO] [RESULT CELLS cell_list]

An optional keyword, mutually exclusive with WINDOW, that specifies that data is to be partitioned on a per-cell basis. BY CELL may be specified separately from RDB in this operation, which differs from other DFM operations. BY CELL is mutually exclusive with WINDOW or NUMWIN. If neither this keyword, nor the WINDOW keyword is explicitly specified, then the default WINDOW behavior is used.

In flat Calibre applications, BY CELL causes the DFM operation to evaluate a single capture defined by the total rectangular extent of the input layers for the operation. No further adjustments are made for flat applications.

  • ALLDEPTH

    An optional keyword that affects calculation of measurement functions. When ALLDEPTH is specified, the measurement functions for all placements in a given cell are added to the measurement functions for that cell. ALLDEPTH may result in the same geometries being counted numerous times. However, the results for the top cell and any lower cell match the results of the design when flattened. When ALLDEPTH is not specified, measurement functions in a cell do not include results from lower cells.

    When ALLDEPTH and NOHIER are specified together, each geometry is measured within the cell it is placed, and the results are summed up through the hierarchy. If geometries overlap across the hierarchy, they are counted multiple times.

  • NOHIER

    An optional keyword that affects the processing of geometries that have hierarchical overlap. When NOHIER is specified, these geometries are processed in each cell independently, ignoring cells above and below. When NOHIER is not specified, geometries in a cell that is part of a larger flat geometry contribute to measurements only partially. For COUNT(), the geometries are counted only in the cell in which the entire geometry is assembled and not in any lower cells. For other measurement functions, only the part of the geometry that is in the cell, and not overlapped by above geometries, is counted. If there is no hierarchical overlap between geometries in different cells, NOHIER has no effect.

  • NOPSEUDO

    An optional keyword that specifies that pseudocells created by Calibre during the run are not part of the output. This can only be specified with the BY CELL keyword. Calibre creates pseudocells on an as-needed basis as part of its internal optimization of hierarchy management. These pseudocells appear in the output by default. This keyword causes any results in a pseudocell to be expanded up to the next level of hierarchy.

  • RESULT CELLS cell_list

    An optional keyword that controls expansion of results into a set of result cells specified by cell_list, where the cell_list is defined by a Layout Cell List statement. When this keyword is used, results are partially flattened in such a way that only the specified result cells contain results. Results from any cell not listed as a result cell are expanded into the nearest containing result cell. The top cell is always considered to be a result cell even if it is not explicitly specified as such, and so all results that are not contained within one of the specified result cells are reported in the top cell. This keyword enables NOPSEUDO, since pseudo cells cannot be specified as result cells. For example:

    LAYOUT CELL LIST rcells "A*" "B*"
DFM ANALYZE metal1 BY CELL RESULT CELLS rcells [ AREA(metal1) ] > 0
    Note:

    Calibre performs extensive hierarchy transformations, and there is no guarantee that any specified result cell is still present after these transformations. The Layout Preserve Cell List specification statement, which also uses the Layout Cell List facility, can be used to suppress expansion of the result cells. However, enforcing preserved cells can have a significant impact on Calibre performance.

WINDOW {w | wx wy} [BACKUP] [CENTERS [c | cx cy]] [{NUMWIN {n | nx ny} [OVERLAP {o | ox oy}]} | STEP {s | sx sy}]

An optional keyword, mutually exclusive with BY CELL, that specifies a data capture window within which DFM checking is performed. The default window is the database extent. WINDOW may be specified separately from the RDB keyword, which differs from other DFM operations.

The w parameter specifies a square window with a width in user units, and must be a positive real number. The wx and wy parameters specify a rectangular window with a separately specified width and height in user units, and must be positive real numbers.

WINDOW keyword usage complies with these parameters:

  • WINDOW and BY CELL keywords are mutually exclusive.

  • NUMWIN and STEP arguments are mutually exclusive.

  • NUMWIN ensures that the selected window is the minimum acceptable size. If the number of data-capture windows specified by NUMWIN covers an area less than that specified by WINDOW, the selected window is the smallest one that is greater than or equal to the specified WINDOW size, and also evenly divides the data extent.

  • If WINDOW is specified with a size greater than INSIDE OF, the window size is truncated to the extent defined by the INSIDE OF value.

For details of how data capture windows work refer to “Partitioning by Window” in the Calibre YieldAnalyzer and YieldEnhancer User’s and Reference Manual.

  • BACKUP

    Optional keyword that specifies that if a window overlaps the right or top edges of the design extent, the window is shifted left or down until it no longer overlaps the design extent. This keyword is similar to the BACKUP keyword for the Density operation.

    In WINDOW mode, by default, the data capture windows that extend outside of the design extent are truncated so that their sizes are less than the value specified by the WINDOW option. When BACKUP is specified, the data capture windows that cross the right edge of the design extent are instead shifted to the left until their right edges line up with the right edge of the design extent. Similarly, the windows that cross the top edge of the design extent are shifted down until their top edges line up with the top edge of the design extent. The window in the right top corner can be shifted in two directions at once. As a result, all windows have the same size; however, the shift of the backed-up windows is smaller than the specified STEP value.

  • CENTERS [c | cx cy]

    An optional keyword that generates an output layer containing window marker geometries annotated with DFM properties containing the values of the DFM Analyze command, in addition to coordinates of the data capture window represented by each output polygon. This layer is generated instead of the regular output layer of the DFM Analyze command.

    CENTERS directs the output layer of the DFM Analyze operation to contain one rectangle for each window that satisfies the DFM Analyze constraint, or for every window if the constraint is not specified. The size of the rectangle can be specified by the optional parameters c, or cx and cy, which determine the size of the output rectangle in horizontal and vertical directions. If c is specified, then cx and cy are assumed to be equal and squares are generated. If no dimensions are specified, square markers of 4*4 database units are generated by default.

    The rectangles on the output layer generated by a DFM Analyze operation with CENTERS specified cannot abut or overlap. The specified size of the marker rectangle is reduced, if necessary, to prevent adjacent rectangles from overlapping. Therefore, the size of the rectangles cannot exceed the size of the window or the step. Because the windows that touch the top and right boundaries of the chip may be reduced if they do not completely fit inside the boundary, the corresponding marker rectangles may be smaller. For example, if WINDOW and STEP are specified, and the CENTERS value is larger than the WINDOW size, the marker size is reduced from the specified value and the output layer would contain separated non‑overlapping rectangles separated by gaps.

    All geometries generated by the DFM Analyze operation with the CENTERS keyword are in the top cell.

    The output layer is annotated with DFM properties for all values computed by the DFM Analyze operation. The DV property is always present on the output layer. In addition, other properties normally reported in the RDB generated by the DFM Analyze operation are present. For example, the DA property is present if the AREA(layer) measurement function is computed.

    The output layer is also annotated with DFM properties containing original coordinates of the data capture window represented by each output polygon. These properties have integer values and cannot be referenced with the PROPERTY() function in DFM expressions, but they can be accessed using Calibre YieldServer.

  • NUMWIN {n | nx ny} [OVERLAP {o | ox oy}]

    An optional keyword, mutually exclusive with STEP, that specifies the number of data-capture windows. The parameters n, or nx and ny, must be integers that are greater than or equal to one (>= 1).

    • OVERLAP {o | ox oy}

      An optional keyword that specifies the fractional amount by which a data-capture window overlaps the previous data-capture window. The parameters o, or ox and oy, must be greater than or equal to zero and less than one (>= 0 < 1). The default value for OVERLAP is 0.

  • STEP {s | sx sy}

    An optional keyword, mutually exclusive with NUMWIN, that specifies the fixed distance a window moves within a given boundary.

Description

Constructs a derived polygon layer consisting of data capture regions such that the expression meets the given constraint. The output layer may be empty.

The DFM Analyze command is a layer operation that is used to identify areas on a layout where manufacturing yield can potentially be improved. Specifically, it provides output of layout regions that meet specified constraints for various layout properties. These properties include area, perimeter, length, and count.

DFM Analyze is one of the few SVRF operations that accepts derived error layers as input (see “Derived Error Layers” in the Calibre YieldAnalyzer and YieldEnhancer User’s and Reference Manual). DFM Analyze accepts this type of data as input for use in measurement functions such as COUNT, EC, and EW; see “Measurement Functions”.

Concurrency

DFM Analyze operations are performed concurrently if they share the following:

  • The same input layers in the same order.

  • The same WINDOW options:

    • Same window coordinates and step or same number of windows and overlap.

    • Same INSIDE OF settings.

RDB Output

RDB output is optional, but helpful in most cases where a detailed analysis of the results is desirable. The arguments COORD, DX, and DY control what is written to the RDB. ONLY controls whether data is sent to the standard DRC database. See “Scores RDB File Format” in the Calibre YieldAnalyzer and YieldEnhancer User’s and Reference Manual for details on the RDB format.

Limitations

For input layers containing skew edges, AREA values can differ between flat and hierarchical runs. When using derived error layers as input, the COUNT values can differ between flat and hierarchical runs.

Examples

Example 1

This example shows a rule check that finds the average area of metal polygons in a layout database of output 100 x 100 um regions where the average area per polygon is below a certain value z.

avg_polygon_area {
	DFM ANALYZE metal WINDOW 100
	[AREA(metal) / COUNT(metal)] < z
}

This is how the rule check works:

  • The operation uses a 100 x 100 um data collection window that is triggered by WINDOW 100 and scans the metal layer. The exact details of how this scan occurs are covered in the “Partitioning by Window” section of the Calibre YieldAnalyzer and YieldEnhancer User’s and Reference Manual.

  • The AREA(metal) function calculates the area of the metal layer in the capture window.

  • The COUNT(metal) function counts the number of metal polygons enclosed by the window.

  • The entire expression in the brackets [ ] gives an idea of the average area of polygons in the window.

  • Windows with an average metal polygon area less than z are output as check results.

If an expression denominator is 0, the entire expression is undefined because division by 0 is undefined. In this case, the expression does not satisfy the constraint. For example:

[AREA(metal) / (COUNT(metal)] < z

If there are no metal polygons in the data capture window, the denominator of this expression is 0. This does not satisfy the constraint, by definition.

Example 2

The previous example can be altered to compute the ratio of metal to the area of the data capture window as follows:

metal_to_window { 
DFM ANALYZE metal WINDOW 100
	[AREA(metal) / AREA()] < z
}

If you do not specify a WINDOW parameter, the default data capture window is the extent of the layers read in for the run. Remember that Calibre only reads in the layers it actually needs for the run, which is not necessarily the entire layout database. DFM Analyze allows you to specify the WINDOW parameters without additionally specifying an RDB. This is different from other DFM operations.

Example 3

You can organize the output by cell instead of by window. The following example shows this syntax:

metal_to_window { 
DFM ANALYZE metal
	[ AREA(metal) ] < z BY CELL 
}

Example 4

You can use the DFM Analyze operation to compute the ratio of the areas of the metal layer to the data capture window. The CENTERED keyword specifies that the data capture extents are 10 x 10 um (user unit) squares, where the center of each square data capture extent is the center of the extent of each marker layer polygon. The operation constructs an output layer that consists of the capture windows that meet the expression constraint.

metal_to_window { 
DFM ANALYZE metal WINDOW 1.0
	[AREA(metal) / AREA()] > 0.0
	INSIDE OF LAYER marker CENTERED 10
}
