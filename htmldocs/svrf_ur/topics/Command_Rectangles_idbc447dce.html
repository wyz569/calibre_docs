Rectangles

Layer operation

Generates an output layer consisting of an array of rectangles with the specified dimensions and spacing.

Usage

RECTANGLES x_length y_length {spacing | {x_spacing y_spacing}}
[OFFSET {offset | {x_offset y_offset}}]
[{INSIDE OF x1 y1 x2 y2} | {INSIDE OF LAYER layer}]
[MAINTAIN SPACING]

Arguments

x_length

A required positive, floating-point number that specifies the x dimension of the rectangle in user units.

Note:

In hierarchical applications, the x and y directions are determined in the local cell frame of reference, after possible promotion. See “Hierarchical Applications”.

y_length

A required positive, floating-point number that specifies the y dimension of the rectangle in user units.

spacing | x_spacing y_spacing

A required argument set that specifies the spacing between rectangles. The values are positive floating-point numbers in user units.

  • spacing — The spacing in both the x and y directions.

  • x_spacing y_spacing — The spacings in the x and y directions, respectively.

OFFSET {offset | {x_offset y_offset}}

An optional argument that specifies the horizontal and vertical offsets between adjacent rectangles. The values are non-negative, floating-point numbers in user units.

  • offset — The offset in both the x and y directions.

  • x_offset y_offset — The offsets in the x and y directions, respectively..

See Figure 1. If an offset value is greater than the corresponding spacing value, the minimum spacing between rectangles is controlled by the corresponding spacing value. See the discussion in the “Description” section.

{INSIDE OF x1 y1 x2 y2} | {INSIDE OF LAYER layer}

An optional argument that specifies the region to be filled with rectangles. By default, the rectangles fill the database extent that is read in at runtime.

  • INSIDE OF x1 y1 x2 y2 — Specifies a region to be filled. A set of four floating-point numbers in user units specify the lower-left (x1, y1) and upper-right (x2, y2) corners of the region to be filled. If a coordinate is negative, it must be enclosed in parentheses ( ).

  • INSIDE OF LAYER layer — Specifies a layer having polygons to be filled with rectangles, where layer is the name of an original or derived polygon layer.

MAINTAIN SPACING

An optional keyword that controls the spacing of rectangles, such that a halo area is constructed around each rectangle, whereby no other rectangle may fall within the spacing, or x_spacing and y_spacing arguments.

When MAINTAIN SPACING is not specified and OFFSET values are less than or equal to the corresponding spacing values, it is possible for corner-to-corner spacing to be less than the specified spacing values.

Description

Generates an output layer consisting of an array of rectangles with the specified dimensions and spacing. You can control the size, spacing, offset, and extent of the generated layer using the appropriate arguments. The Rectangles operation does not generate zero-area rectangles.

By default, the generated rectangles fill the portion of the database extent actually read in at run time. See the Extent operation for a discussion of database read-in. You can specify the INSIDE OF or INSIDE OF LAYER keyword to fill a specified region instead of the database extent.

The DFM Rectangles operation is similar to Rectangles, but supports x_length, y_length, spacing, and offset values that are provided by DFM properties on an input layer.

Flat Applications

The following describes the semantics of the Rectangles operation in flat Calibre and ICverify applications.

Rectangles are aligned with the x_length dimension along the x‑axis and the y_length dimension along the y-axis. Spacing of the rectangles is given by the spacing and offset arguments.

The placement of the first rectangle always begins at the lower left corner of the database extent. If the rule file contains a Resolution value, the lower-left corner of the first rectangle is snapped (up and right) by that value. This ensures that the rectangles are placed on grid if the x_length, y_length, spacing, and offset arguments are appropriately specified.

The default algorithm is used if each optional offset value is less than or equal to the corresponding spacing value and MAINTAIN SPACING is not specified. The algorithm constructs columns of rectangles beginning in the lower-left corner of the area to be filled. In each column, a rectangle always begins at the point (X + AX, Y + AY), where:

  • X is the lower-left x-coordinate of the previous rectangle.

  • AX is the x_offset (or offset) value.

  • Y is the lower-left y-coordinate of the previous rectangle.

  • AY is the y_length value plus the y_spacing (or spacing) value.

This process continues until no further rectangles can be placed within the extent.

When moving left-to-right, a new column begins at the point (X + BX, Y + BY) where:

  • X is the lower-left x-coordinate of the first rectangle in the previous column.

  • BX is the x_length value plus the x_spacing (or spacing) value.

  • Y is the lower-left y-coordinate of the first rectangle in the previous column.

  • BY is the y_offset (or offset) value.

This process continues until no further rectangles can be placed within the extent (see the following figure).

Figure 1. Rectangles Arguments

When using the offset options, a starting point can occur where a full rectangle (including any spacing requirements) would fit below it. In this case, the column is started at (X + CX, Y - CY) where:

  • X is the lower-left x-coordinate of the first rectangle in the previous column.

  • CX is the x_length value plus the x_spacing (or spacing) value minus the x_offset (or offset) value.

  • Y is the lower-left y-coordinate of the first rectangle in the previous column.

  • CY is the y_length value plus the y_offset (or offset) value.

These left-to-right algorithms ensure a constant corner-to-corner separation of (DX, DY) between columns where:

  • DX is the x_spacing (or spacing) value minus the x_offset (or offset) value.

  • DY is the y_spacing (or spacing) value minus the y_offset (or offset) value.

This corner-to-corner separation is determined by the distance between the second rectangle in the first column and the first rectangle in the second column, when moving from left-to-right.

When moving from bottom-to-top (filling the upper-left region of the database extent), the process is similar to that of the left-to-right algorithms described above, with appropriate modifications of the (X, Y) argument order.

Corner-to-corner spacing (DX or DY) is zero when an offset value equals its corresponding spacing value.

If you specify a y_offset or x_offset (or offset) that exceeds the corresponding spacing value, or if you specify the MAINTAIN SPACING optional keyword, then Rectangles spacing is controlled so that, given any constructed rectangle B, no other constructed rectangle is strictly within a halo area around B defined by the x_spacing and y_spacing (or spacing) as shown in the following figure:

Figure 3. Rectangles With MAINTAIN SPACING or Large Offset

This spacing is controlled in the algorithm, along with allowing offset values to exceed their corresponding spacing values, by ensuring that the first rectangle in each new column obeys the x_spacing to all rectangles in the previous column. This occurs within the y-extent of that first rectangle plus the y_spacing in both directions.

All rectangles must fit entirely within the extent in order to be output.

If you specify INSIDE OF, then the specified area substitutes for the database extent. This area must be a valid, orthogonal rectangle.

If you specify INSIDE OF LAYER, then the rectangles are placed inside of the input layer polygons, according to the following:

  1. For each polygon on the layer, compute its rectangular extent.

  2. Fill the rectangular extent of the polygon according to the general steps discussed above.

  3. Output complete rectangles that are inside the polygon boundary.

Hierarchical Applications

In hierarchical applications, the algorithm used is more complicated.

As in the flat algorithm, the placement of the first rectangle always begins at the lower left corner of the database extent. If the rule file contains a Resolution value, the lower-left corner of the first rectangle is snapped (up and right) by that value. This ensures that the rectangles are placed on grid if the x_length, y_length, spacing, and offset arguments are appropriately specified.

If you do not specify INSIDE OF LAYER, then the output rectangles are instantiated hierarchically as much as possible.

Cells in which rectangle instantiation occurs are selected to achieve an optimal hierarchical distribution. In each cell where rectangle instantiation occurs, a fill area is determined. The algorithm selects the fill area to ensure no minimum spacing violation between rectangles in different cell placements at any point in the hierarchy. The spacing value is the larger of the specified spacing values (spacing, x_spacing, or y_spacing). The algorithm then distributes the rectangles in the fill area with the width-axis along the x-axis of the cell extent.

The hierarchical fill algorithm described above attempts to achieve a hierarchical distribution at the expense of these items:

  • Density — In general, the rectangles may not be as densely packed (from the flat point of view) as in the flat applications, especially when positive offsets are specified. This is due to a need to observe minimum spacing of rectangles (as described above) between any two cell placements at any point in the hierarchy. Loss of density is normally less than 5%.

  • Uniform rotation — In a hierarchical application, the rotation of placements can cause some blocks of rectangles to be aligned with the x_length dimension along the y-axis from the flat viewpoint.

  • Uniform alignment — Hierarchical rectangle placement can generate multiple fill areas in each cell, which are filled separately (such as with the use of the INSIDE OF LAYER option in flat applications). This can cause blocks of rectangles in a cell to be positively offset from each other. This would not be seen in flat applications.

If you specify INSIDE OF LAYER, then the polygon layer is converted to fully-merged form, and in each cell where the fully-merged form exists, the operation follows the flat application’s algorithm for INSIDE OF LAYER with the x_length dimension along the x-axis of the geometry extent in cell space. A layer is converted to fully-merged form in hierarchical applications by promoting geometry up the hierarchy until only full polygons (in the flat view) are present in a cell; that is, it is not possible for a polygon in a cell to grow due to any hierarchical interaction of the cell’s placements.

However there is an important modification to the previously-stated placement rule. This modification insures transform-invariance. This means that the specified x_length, y_length, spacing, and so forth of the generated rectangles must also be aligned on the x-axis and y-axis, respectively, from the point of view of the top-level cell coordinate space.

Transform-invariance is accomplished by not placing rectangles (and adjusting upward the hierarchical position of the INSIDE OF LAYER geometry) in cells with multiple placements having inconsistent transforms from the flat point of view. For cells with consistent non-identity flat transforms, the INSIDE OF LAYER geometry is transformed to top-level (flat) space where the rectangles are generated, and then they are transformed back to cell-space and instantiated.

If the polygon layer for INSIDE OF LAYER is flat (for example, output of a Density operation) or is in fully-merged form only at the top-level cell (for example, output of an Extent operation), then the output rectangles are instantiated at the top-level. This overrides the hierarchical distribution described previously. The appearance of the output rectangles in these cases will be more similar to their appearance in flat applications.

Examples

Example 1

Generate a grid of rectangles with x_length 3, y_length 4, and spacing of 1 unit in the x- and y‑directions.

fill1 = RECTANGLES 3 4 1

Example 2

Generate rectangles with x_length 2, y_length 4, x_spacing 2, y_spacing 3, and an offset of 2 units in the x- and y‑directions.

fill2 = RECTANGLES 2 4 2 3 OFFSET 2

Example 3

Generate rectangles with x_length 4, y_length 6, spacing of 3 units in the x- and y‑directions, x_offset 1, y_offset 2. Use the INSIDE OF LAYER keyword to generate the rectangles inside of the shapes on the poly layer.

fill3 = RECTANGLES 4 6 3 OFFSET 1 2 INSIDE OF LAYER poly

Example 4

Generate rectangles with x_length 2, y_length 5, spacing of 2 units in the x- and y-direction, offset 1 in the x- and y-directions. Use INSIDE OF to generate the rectangles inside of a region with corners (20, 20) and (200, 200).

fill4 = RECTANGLES 2 5 2 OFFSET 1 INSIDE OF 20 20 200 200

Example 5

Create 2x4 fill rectangles at 2 user unit spacing in the inverse of the active layer. Observe spacing of 2 user units to active and poly layers and 3 user units to nwell edges.

X1 = SIZE ACTIVE BY 2                      // For spacing of 2 to active.
X2 = SIZE POLY BY 2                        // For spacing of 2 to poly.
X3 = ( SIZE NWELL BY 3 ) NOT ( SIZE NWELL BY -3 ) // For spacing of 3 to 
                                                  // nwell edges.
X4 = ( EXTENT ) NOT ( X1 OR ( X2 OR X3 ) ) // Area to fill.
X5 = RECTANGLES 2 4 2 INSIDE OF LAYER X4   // Fill pattern.
