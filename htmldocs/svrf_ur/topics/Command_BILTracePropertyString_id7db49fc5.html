PROPERTY STRING

Trace Property computation language function.

Defines string properties to be compared in layout and source.

Usage

PROPerty STRing prop_name1[, prop_name2, …]

Arguments

prop_name1

A required string device property name. If more than one is specified, then names are separated by a comma (,) followed by whitespace.

Return Values

String.

Description

This statement declares the names of the properties for which string values are made available from layout and source input pairs. The property values are compared based on the Trace Property computation program.

Either this statement or a PROPERTY statement is required in a Trace Property program. The PROPerty statement may precede or follow the PROPerty STRING statement, but they must both precede the optional DEBUG and WARN statements, if present.

Properties may only be accessed by data retrieval and discrepancy reporting functions, the property values may not be altered.

Property names must be simple names; property-name/SPICE-parameter combinations such as instpar(w) are not accepted (these are typically found in Pyxis Layout designs). To handle such combinations, use the LVS Property Map specification statement.

A property name cannot be declared in both an PROPerty and PROPerty STRING statement within the same property computation procedure. For example, the following causes a compiler error:

TRACE PROPERTY ... [
    PROPERTY x
    PROPERTY STRING x  // ERROR.  'x' already mentioned in PROPERTY
    IF ( lay_num_val(x) == 0 ) {
        report_message( "unexpected zero value for 'x'" )
   }
]

Property names must be declared in order to be used in data retrieval or discrepancy reporting functions. For example, the following causes compiler errors:

TRACE PROPERTY t(st) [
    PROPERTY STRING q
    report_string_discrepancy(q)         
    // ERROR. 'q' is not a declared PROPERTY STRING name
    local = lay_str_val(l)  
    // ERROR. 'l' is not a declared PROPERTY STRING name
]

Properties mentioned in Trace Property computation PROPerty STRing declaration statements are considered LVS-actionable properties, because they are declared for the purposes of trace comparison by a user procedure. As LVS-actionable properties, they trigger missing property errors when such properties are missing from a device in either instance of the input pair. See LVS Netlist Property for a complete discussion of LVS-actionable properties.
