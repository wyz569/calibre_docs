Macros

Macros are functional templates, like subroutines, that can be called multiple times in a rule file. They are similar to the macros of the C and C++ languages. There is a macro definition called DMACRO, which contains rule file code, and a call to the macro, called CMACRO, which can pass arguments to the DMACRO. These structures support a modular organization of the rule file. This can be convenient when layer operations or Device statements are executed multiple times with varying parameters.

DMACRO

A macro definition (DMACRO) defines a macro and has this syntax:

DMACRO macro_name [arguments] {
   SVRF_code
}

where macro_name is the name of the procedure, arguments are input argument names, and SVRF_code is valid SVRF syntax.

A macro_name must be unique, may only contain letters, numbers, underscores (_), and periods (.), and may not conflict with SVRF keywords. Handling DMACRO names as string constants can override these limitations in some cases.

Each argument must be a name (generally a layer or variable name) that conforms to the same syntax requirements as a macro_name and may not be duplicated in the same DMACRO arguments list. Any name from the arguments list used in the body of the DMACRO definition is substituted by the corresponding value from the CMACRO call to the DMACRO.

The SVRF_code may not contain syntax that contains braces “{ }”, such as the complete body of a rule check. If using DFM property computations, derived layer names within the SVRF_code should not be the same as string arguments to DFM functions.

The following example defines a rule check that can be called multiple times, with lay and val being values passed in from the calling environment (a CMACRO): 

DMACRO WIDTH_CHECK lay val {
S1 = INT lay < val ABUT < 90 SINGULAR REGION  
S2 = INT lay < val ANGLED == 2 PARALLEL OPPOSITE REGION
S3 = INT lay < val  CORNER TO EDGE REGION
( S1 OR S2 ) OR S3
}

DMACRO definitions may not be nested or called recursively.

Any of the rule file comment characters may be used in a DMACRO definition; however, user comments (specified by the “at” symbol @) do not appear in the Calibre nmDRC results database or in Calibre RVE. Instead, user comments should be placed in the rule check that calls the DMACRO.

CMACRO

A macro is invoked by the keyword CMACRO (call macro) followed by a DMACRO name and a list of zero or more arguments, as follows: 

CMACRO macro_name [arguments]

The macro_name must match that of some DMACRO definition. Each argument may be either a name (generally a layer or variable name) or a numeric constant. A sufficient number of arguments must be present after the CMACRO macro_name to match what the DMACRO specifies as arguments. A CMACRO may not appear within the scope of brackets ([ ]).

The DMACRO definition is internally placed in the CMACRO call, with argument substitution. For example, these CMACRO calls refer to the previously-shown DMACRO named WIDTH_CHECK:

poly_width { CMACRO WIDTH_CHECK poly 0.5 }
metal_width { CMACRO WIDTH_CHECK metal 0.6 }

The arguments poly and 0.5 are substituted into the DMACRO named WIDTH_CHECK in order to construct the rule check poly_width. A similar substitution occurs for the rule check metal_width.

You can also do this:

width_checks {
   CMACRO WIDTH_CHECK poly 0.5
   CMACRO WIDTH_CHECK metal 0.6
}

Layer Names, Derivations, and Variables in Macros

Layer definition names in DMACRO statements are locally-scoped by default when the DMACRO is instantiated in a CMACRO. This is done by generating a unique (rule-file-wide) name for each layer definition in the DMACRO construct when the DMACRO is instantiated. References to each layer definition name in the DMACRO are also substituted appropriately. This is illustrated by the previously shown width_checks example. The compiler is not confused by multiple intermediate derived layers having the same names (S1, S2, and S3 in the DMACRO named WIDTH_CHECK) in the width_checks rule.

If an argument to a DMACRO statement matches the name of a derived layer within the DMACRO block definition, then the corresponding CMACRO argument becomes the name of a global layer. For example:

DMACRO MAC IN1 IN2 OUTPUT { // OUTPUT occurs as an argument to DMACRO
  X = IN1 AND IN2
  OUTPUT = SIZE X BY 0.01   // OUTPUT is a derived layer name
}
CMACRO MAC L1 L2 RES     // OUTPUT is assigned the global layer name RES

In this example, OUTPUT is used as a DMACRO argument name and is the name of a derived layer in the DMACRO; therefore, OUTPUT is treated as a global name for the derived layer. The global name OUTPUT becomes the global layer name RES because the CMACRO argument called RES corresponds to OUTPUT.

For the DFM Property and DFM Property Select Secondary operations, the :EDGE, :EXTENT, and :EXTENT_EDGE layer derivation notations cannot be used within the DMACRO, but must be passed in as an argument. For an example with DFM Property, see “Example 11”. For an example with DFM Property Select Secondary, see “Example 8”.

Variable statements inside of a DMACRO are locally-scoped and are not accessible outside of the DMACRO. This enables you to use the same variable name with different values inside multiple DMACRO definitions. Variable statements that are specified outside of a DMACRO are globally-scoped and available for any DMACRO in the rule file. If present, a locally-scoped variable will override a globally-scoped variable of the same name.

Nesting in Macros

DMACRO definitions may themselves contain nested CMACROs. For example, the following constructs are allowed:

DMACRO skew_edge inlayer {
   ANGLE inlayer > 0 < 45
   ANGLE inlayer > 45 < 90
}
DMACRO acute_angle inlayer grid_size {
   INT inlayer < 4*grid_size ABUT > 0 < 90 INTERSECTING ONLY
   EXT inlayer < 4*grid_size ABUT > 0 < 90 INTERSECTING ONLY
}
DMACRO illegal_shape met grid_size { // nested CMACROs
   CMACRO skew_edge met
   CMACRO acute_angle met grid_size
}
test {CMACRO illegal_shape metal1 .001}

However, no CMACRO can call a DMACRO that contains a call to that CMACRO (recursion is not allowed).

Macros for Keywords and Arguments

Although the preceding DMACRO examples define complete SVRF operations or statements, this is not required. The DMACRO can define a set of arguments—this is useful in cases where the arguments are lengthy and repeated often.

A set of Density arguments is defined in this example. 

DMACRO DENS_ARGS { 
WINDOW 10 
INSIDE OF LAYER marker
GRADIENT > 0.25 
BACKUP
}
dens2 { DENSITY metal2 < 0.50 CMACRO DENS_ARGS }
dens4 { DENSITY metal4 < 0.35 CMACRO DENS_ARGS }

If the macro is defining a DFM property expression, the brackets ([]) enclosing the property expression must be within the macro.

Macros in User-Defined Device Property Calculations

You can use macros for user-defined property calculation in Device operations. This also applies to other built-in language programs that support device-related computations.

Argument substitution is performed for arguments that appear within brackets ([ ]) in the DMACRO definition. Note the brackets must appear in the DMACRO definition.

DMACRO mosprop seed {
  [
    property W, L
      weffect = 0   // Specify width effect constant here
                    // (built-in default is 0).
      W = 0.5 * ( perim_co( S, seed ) + perim_in( S, seed )
        + perim_co( D, seed ) + perim_in( D, seed ) )
      L = area( seed ) / W
      if ( ( bends( seed ) != 0 ) && ( weffect != 0 ) ) {
        if( W > L ) W = W - weffect * bends( seed ) * L
        else L = L - weffect * bends( seed ) * W
      }
  ]
}
DEVICE MP gate gate psd psd CMACRO mosprop gate
DEVICE MN gate gate nsd nsd CMACRO mosprop gate

The gate layer in the CMACRO call in each Device statement gets passed in as the seed argument in the mosprop macro.

CMACRO calls within brackets are not expanded. For example, the following code is not valid:

DEVICE MP gate gate psd psd [ CMACRO MOSPROP2 gate ] // Bad.
DEVICE MN gate gate nsd nsd [ CMACRO MOSPROP2 gate ] // Bad.

LOCAL() Modifier

The LOCAL() modifier can be used to create local names that are valid only in the scope of each CMACRO call. This modifier is useful for defining specification statements within DMACROs. Consider the following example:

DMACRO COPY_C L1 L2 C {
   LAYOUT CELL LIST LIST1 C
   L2 = DFM COPY L1 CELL LIST LIST1
}

Assume that you want to use this DMACRO as follows:

LAYER A 1
LAYER B 2
CMACRO COPY_C A A1 "A*"
CMACRO COPY_C B B1 "B*"

This use is invalid because the Layout Cell List specification statement is duplicated with the name LIST1. To resolve this problem, add a LOCAL() modifier to the DMACRO:

DMACRO COPY_C L1 L2 C {
   LAYOUT CELL LIST "LOCAL(LIST1)" C
   L2 = DFM COPY L1 CELL LIST "LOCAL(LIST1)"
}
CMACRO COPY_C A A1 "A*"
CMACRO COPY_C B B1 "B*"

The name “LOCAL(LIST1)” is replaced by a name that is unique to each CMACRO call. Within one CMACRO call, multiple instances of LOCAL(name) are always replaced by the same local name (such as “LOCAL(LIST1)” in the preceding example). Different names used with LOCAL() in the same macro remain different after the replacement. Note that the quotes around the string “LOCAL(LIST1)” are required.
