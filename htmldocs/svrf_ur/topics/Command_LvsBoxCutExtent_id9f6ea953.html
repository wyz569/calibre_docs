LVS Box Cut Extent

Specification statement

Specifies how layers derived using Extent-based operations are handled in LVS Box BLACK and GRAY flows.

Usage

LVS BOX CUT EXTENT {YES | NO}

Arguments

YES

A keyword that specifies the areas (extents) of LVS Box BLACK and GRAY cells are excluded from Extent operation computations of top-level layers. This is the default behavior.

NO

A keyword that specifies the areas of LVS Box BLACK and GRAY cells are included in Extent operation computations of top-level layers.

Description

This statement controls how extent-based operations are performed for LVS Box BLACK and GRAY cells. The following figure shows the basic difference between YES (the default) and NO.

Figure 1. LVS Box Cut Extent

The formation of such an Extent layer can adversely affect connectivity whether YES or NO is specified. Careless use of Extent-based layers with the YES keyword being active can contribute to connectivity problems due to the lack of layer material in the region that is cut out around the box cell. For example, assume the following rule file excerpt:

LVS BOX BLACK "BOX" // could also be GRAY
LVS BOX CUT EXTENT YES

LAYER AB 1

nsub = EXTENT
nplug = AB AND nsub

CONNECT nplug ...

Notice that by cutting out the area corresponding to the BOX cell, the nplug layer is split, and this is undesirable. Hence, care must be taken that such things do not occur in the layout when YES is used.

Conversely, using the NO keyword can contribute to unintentional short circuits. This is usually not a problem for BLACK box cells because only port polygons are used within their extents, so that flow is not considered in the ensuing discussion.

Recall that LVS Box GRAY processes layer operations in two distinct contexts. Firstly, the layers are processed inside each gray cell independent of the top level. Secondly, the layers are processed at the top level without any consideration of the layers inside a gray cell. During connectivity extraction, the Connect layers inside and outside the box cell are used to determine connectivity. Hence, if a connectivity layer is derived based on the Extent operation, there is a possibility that such connectivity layers can be handled improperly due to exclusion of box cell geometry.

Assume the following rule file excerpt:

LVS BOX GRAY "BOX"
LVS BOX CUT EXTENT NO

LAYER PWELL 1

sub = EXTENT
nsub = sub NOT PWELL

// psd is P source/drain and nsd is N source/drain
CONNECT psd PWELL
CONNECT nsd nsub

Together, PWELL and nsub constitute the extent of the sub layer in the BOX cell.

As noted previously, the layer operations also take place in the top level context but without consideration of the drawn layers that exist inside the BOX cell. Hence, the nsub layer could appear as follows at the top level:



because the PWELL layer in the BOX cell is not processed at the top level, so nsub has the same extent as sub at the top level. Since connectivity extraction considers both the Connect layers inside and outside the BOX cell, there is the possibility that nsub from the top level may be short-circuited to PWELL inside the BOX cell. Hence, care must be taken this does not occur in the layout when NO is used.

See also LVS Box Peek Layer.

Examples

This is a typical gray box sequence:

LVS BOX GRAY "ramcell"
// exclude extent-based layers from extent of box cells
LVS BOX CUT EXTENT YES
