Constraints

Certain SVRF layer operations are measurement-oriented and therefore carry constraints. Constraints are ranges of non-negative real numbers; input data that falls within the specified constraint of an operation is generally output as result data.

The syntax for constraints uses the six keywords (operators) shown in the first six rows of Table 1. The syntax for the last four constraint forms in the table is a combination of first four operators; in most cases a >= 0 and a < b. The constraint “< 0” is not allowed because strictly negative constraint values are not possible in SVRF operations. The constraint “<= 0” is permitted.

As an example, the constraint “< 4” denotes all non-negative numbers less than 4, and the constraint “>= 5 < 7” denotes all numbers greater than or equal to 5 and less than 7.

Not all operations accommodate all types of constraints or all values of the numbers a and b. For example, a = 0 in the following constraint is not valid for use with the With Width operation because a polygon with zero width is not possible for this operation.

WITH WIDTH layer1 >= a < b // 0 not valid for a

Constraint restrictions are discussed in the description of the operation, where applicable.

Constraints that have two numeric values, such as the final four forms in the following table, are called interval constraints.

Constraints for Angled or Skew Edge Data

Angled or skew edge data can sometimes present a problem for measurement constraints or the testing of equality of measurement values. Such edge data may result in computed coordinate values that do not end up precisely on a layout database coordinate grid, which has an impact on whether a constraint is met or not. More generally, such edge data may adversely impact any computed value derived from it.

If a measurement calculation generates a coordinate value that is between available values in a layout database, the measurement tool must select which database point to “snap” the coordinate value to. For example, Euclidean distance is determined by the expression √(x2 + y2). This distance is converted to a floating-point representation of the Euclidean value. If that value happens to be between coordinate values representable by a layout database at a given precision, the value is “snapped” to the database grid. The selected grid point is not generally predictable due to differences in floating-point arithmetic performed in computational engines of operating systems, in various CPU hardware, and in software compiled using different methods.

For the reasons just discussed, it is unreasonable to expect consistent behavior for snapping coordinate values to grid at an arbitrary precision for angled and skew edge data. This is true for any layer operation that handles such data. Therefore, appropriate measurement tolerances must be used any time angled or skew edge data are present.

In particular, constraints that check for exact equality (==) can give unexpected results any time grid snapping occurs. Therefore, an equality constraint (==) should be avoided for skew or angled data. Instead, using a measurement tolerance is desirable.

In general, the check for equality between two numbers A and B should follow this expression, where epsilon is a suitably small number, and A is a reference value for determining relative tolerance:

( | A - B | )/A < epsilon

If the expression is true, then the numbers are considered equal.

To demonstrate this, the following operation may be undesirable depending on whether any grid snapping occurs:

# possibly bad
area_1 = AREA L1 == 0.5

This would be better, where tol is a suitable tolerance factor:

VARIABLE tol 0.004
area_1 = AREA L1 >= 0.5 < 0.5*(1 + tol)

The same applies in DFM property computations. This type of property check is frequently undesirable:

# possibly bad
[ AreaA = (AREA(A) == AREA(B)) ? 0 : AREA(A) ] > 0

Using a small tolerance is better:

[ AreaA = ((ABS(AREA(A) - AREA(B))/AREA(A)) < 0.002) ? 0 : AREA(A) ] > 0

See “Comparison of DFM Properties” for more details about DFM property constraint evaluation.
