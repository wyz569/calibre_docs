Layout Rename ICV

Specification statement

Specifies to use unique identification in ICV cells across multiple input files. Used only in Calibre.

Usage

LAYOUT RENAME ICV {NO | YES}

Arguments

NO

Keyword that instructs Calibre not to use unique identification of ICV cells across multiple input files. This is the default behavior when you do not include this statement in the rule file.

YES

Keyword that instructs Calibre to use unique identification of ICV cells across multiple input files. This option should be used if there are ICV cells in the input layout files, and there is more than one file specified.

Description

This statement is used to enable unique identification of ICV cells across multiple files in the Layout Path or Layout Path2 specification statement, respectively, as in a case like this:

LAYOUT PATH file1 file2 ... fileN

When pseudo cells are created by Calibre, there is no control over their names, which are “ICV_<number>”. Hence, when running Calibre output back through Calibre (for example, in post-tapeout flows), multiple files in the Layout Path can lead to cell name conflicts. You can resolve this by combining duplicate cell names (Layout Input Exception Severity DUPLICATE_CELL) but this is not the intent, since the two ICV_<number> cells will only be the “same” by accident.

When YES is specified, input ICV cells are renamed so they are unique. More specifically, assume that filei with i > 1 contains a cell named ICV_*, where * is any string of non-negative length. When this cell name matches a cell name in filej, where j < i, then this cell is renamed to ICV_*$ICV<number>$, where <number> is internally generated, and does not match a cell name in any filej where j < i.

See also Layout Rename Cell, Layout Rename All, and Layout Rename Clone.

Examples

In the following case, if cell ICV_n exists in both input files, this causes a duplicate cell error at runtime:

LAYOUT PATH layout1.gds layout2.gds
LAYOUT RENAME ICV NO

Using YES avoids the error.
