DFM RDB

Layer operation

License: Calibre nmDRC-H. The NODAL, GDS, and OASIS keywords require a Calibre YieldEnhancer license.

Writes original or derived layer data to a results database (ASCII by default).

Usage

Syntax 1:

DFM RDB layer1 filename [layer2 …]
[NODAL [ALL | OTHER]] [NOPSEUDO | TOP] [NOEMPTY] [OUTPUT]
[MAXIMUM {ALL | NULL | value}]
[JOINED] [ALL CELLS | TRANSITION] [SAME CELL] [ABUT ALSO]
[RESULT CELLS cell_list]
[CHECKNAME name]
[COMMENT “comment_string”]
[ANNOTATE ‘[’ “name” = “value” ‘]’]…
[CELL SPACE]
[HIGHPREC]
[QUOTENAMES]

Syntax 2:

DFM RDB layer1 NULL [layer2 …]
[NODAL [ALL | OTHER]] [TOP]
[COMMENT “comment_string”]
[ANNOTATE ‘[’ “name” = “value” ‘]’] …
[QUOTENAMES]

Syntax 3:

DFM RDB layer1 filename CSG
[NOPSEUDO | TOP]
[COMMENT “comment_string”]
[ANNOTATE ‘[’ “name” = “value” ‘]’]…
[QUOTENAMES]

Syntax 4:

DFM RDB {GDS | OASIS} {filename | DEFAULT | “PIPE command”}
layer1 [layer1_number [layer1_datatype]] [layer2 [layer2_number [layer2_datatype]] …]
[USERMERGED | PSEUDO | USER | COMBINE | TOP]
[KEEPEMPTY] [MAXIMUM {ALL | value}]
[PREFIX prefix_string [FILLONLY]] [APPEND suffix_string [FILLONLY]]
{[AUTOREF [prefix]] | [PROPERTIES]} [NOEMPTYDB]
[METACELL cell_suffix]

Arguments

layer1

A required argument specifying the name of the layer to be written to the database.

  • If the optional keyword NODAL is specified, this first input layer must contain connectivity data.

  • If the TRANSITION keyword is specified, this first input layer plays the role of a via layer and contains the added vias.

  • If layer2 is specified, then layer1 is the primary layer — refer to “Multilayer DFM RDB Mode”.

This layer can be an original or derived layer. Note that edge and error layers are only supported for single layer RDBs. See the “Description” section for the definition of the modes mentioned in the following table.

filename

A required argument in Syntax 1, Syntax 2, and Syntax 4 that specifies the name of the file to which the layer data is written when calibre -drc is used. This argument is ignored when layer data is written to a DFM Database (calibre -dfm). This argument must placed as indicated in the syntax.

When not generating GDS or OASIS output using Syntax 4, you can write the results of multiple DFM RDB operations to a single RDB as follows:

  • The first time an RDB file is opened, the run clears the file and writes the header line consisting of top-level cell name and precision.

  • Subsequent opens are in append mode, and append additional RDB rule checks to the file. The header line is not repeated.

  • Within the filename, the pattern substitution facility replaces all occurrences of the pattern “%_t_” with the name of the top cell for the design.

To avoid overwriting results, filename should not be the same file that any other results output statement in the rule file writes to. There is no check that a DFM RDB filename does not coincide with any other Calibre results database created in the run, and caution is advised. If an output file cannot be opened, then a warning is issued, and the run continues as if the operation were not specified.

In Syntax 4 you can alternatively specify the output using the DEFAULT or “PIPE command” arguments, which are defined later. The filename argument is not used in Syntax 2.

NULL

A required keyword in Syntax 2. This is useful when writing layer data to a DFM Database (calibre -dfm). If a rule file containing a DFM RDB NULL operation is processed using calibre -drc, the operation is ignored.

When specified, multiple input layers do not need to be of the same layer type. Only the keywords shown in the Syntax 2 usage line may be used with the NULL keyword.

layer2 …

An optional argument that specifies the names of one or more layers to be written to the database in addition to layer1. These layers must be original or derived polygon layers. This argument can only appear after the required arguments.

  • If the NODAL ALL keyword is specified, connectivity is required on all input layers.

  • If the TRANSITION keyword is specified, you must specify exactly two additional layers. These layers represent the added metal enclosure for the vias on layer1.

  • If the TRANSITION keyword is not specified, these layers play the role of the secondary layers—refer to “Multilayer DFM RDB Mode”.

Not used in Syntax 3.

NODAL [ALL | OTHER]

An optional keyword set used to control whether connectivity data is stored in the RDB. When specified, the operation attaches NET and NETNAME properties to geometries in the RDB if that information was available on the input layer. Used only in Syntax 1 and 2. Use of the NODAL keyword requires a Calibre YieldEnhancer license. Refer to the Calibre Administrator’s Guide for licensing information.

  • NODAL — Attach connectivity data to geometries from layer1.

  • NODAL ALL — Attach connectivity data to geometries from all input layers.

  • NODAL OTHER — Attach connectivity data to geometries from all layers except layer1.

Connectivity is required for the layers to which connectivity data will be attached. The optional keywords ALL and OTHER can be used only when operating in the multilayer mode.

NOPSEUDO

An optional keyword that controls whether the RDB has the pseudo-hierarchy created by Calibre when processing hierarchical data. When specified, pseudo-cells (indicated by the prefix ICV_) are expanded. These pseudo-cells appear in the RDB by default. This option may not be used with unmerged input layers; see “Unmerged Polygon Layers” in the Calibre Verification User’s Manual for a list of operations that produce unmerged polygon layers. Used only in Syntax 1 and 3. This keyword is mutually exclusive with TOP.

TOP

An optional keyword that causes Calibre nmDRC-H to transform geometries to the top-level cell without merging prior to results database output. Use this option in cases where you do not want to fill in lower level cells with the input layers, such as when using a routing-based primary cell. When used in a compressed fill flow, the output results in “standard layers,” which are uncompressed layers promoted to the primary cell during output. The USERMERGED, USER, PSEUDO, NOPSEUDO, COMBINE, and TOP keywords are mutually exclusive.

ALL CELLS can be specified with TOP in order to suppress the cell number suffix in the check name (see Table 3 with the CHECKNAME keyword).

NOEMPTY

An optional keyword that specifies to omit cells with no geometries from the output. Used only in Syntax 1.

OUTPUT

An optional keyword that generates a non-empty output layer when the operation produces any results. This layer has one rectangle representing the extent of the results written out by the DFM RDB operation. If the DFM RDB operation does not write any results, the result layer is empty. Note that this keyword has no effect in calibre -dfm mode, since the behavior of the DFM RDB operation in this mode is to save its input layers to the DFM Database. Used only in Syntax 1.

MAXIMUM {ALL | NULL | value}

An optional keyword set used in Syntax 1 and 4, which is used to control the maximum number of results written to the output.

If not specified, the operation writes all results on the input layer to the output.

  • MAXIMUM ALL — Specifies that all results are output, with no limit. This is the default.

  • MAXIMUM NULL — Equivalent to MAXIMUM ALL. This combination is not used in Syntax 4.

  • MAXIMUM value — Limits the number of polygons written to value. The value is a positive integer. Note that the MAXIMUM value applies to the entire set of results, not to each check. When the MAXIMUM value is reached, no more results are written into the RDB and a warning is printed in the transcript like this:

    WARNING: Maximum result count of 1000 exceeded in DFM RDB for layer MET1::<1>

    When specified in Syntax 4, all DFM RDB operations writing to the same file must use the same MAXIMUM value setting.

Calibre YieldServer: In Calibre YieldServer, using the following operation:

dfm::new_layer -rdbs_as_files -svrf

applies the DFM Defaults MAXIMUM setting when the DFM RDB operation is executed and writes out an ASCII RDB.

Note that the MAXIMUM keyword has no effect in calibre -dfm mode where the whole layer is saved into the output.

JOINED

An optional keyword that outputs error clusters with additional lines joining the edges. These added lines convert error clusters into contiguous chains of edges and line segments. This keyword is useful for identifying the edges that comprise a particular error cluster.

ALL CELLS

An optional keyword in Syntax 1 that causes results data from all cells to be written out as a single check. The default is to write rule check results per each layout cell.

When you specify ALL CELLS, the operation adds an additional property to the first result in each cell. This property is named CN and holds three values: cell name, the global-to-cell-coordinates transform matrix, and flat placement count for the current cell, which shows how many instances of the cell are in the entire design.

Geometry coordinates are written in top-level coordinates by default. The ALL CELLS keyword does not affect this, but the CELL SPACE keyword does.

This keyword may not be used with TRANSITION.

TRANSITION

An optional keyword used in Syntax 1 that causes the data to be written using the DFM Transition by cell format. When this keyword is specified, you must supply exactly three input layers for the operation. This keyword may not be used with ALL CELLS.

SAME CELL

An optional keyword used in Syntax 1 to disable hierarchical promotion when generating a multilayer geometric results database. See “Multilayer RDB File Format” in the Calibre YieldAnalyzer and YieldEnhancer User’s and Reference Manual. Specifying this keyword limits geometry clusters to the same level of hierarchy; that is, the cluster is assembled in the same cell as the primary layer polygon, and only the secondary layer polygons in that cell are added to the cluster. Secondary layer polygons that overlap a primary layer polygon but belong to a different cell are not assembled into any cluster. This keyword can be specified only when you supply multiple layers as input to the operation.

By default, the operation performs a hierarchical promotion when assembling clusters for multilayer RDBs.

ABUT ALSO

An optional keyword that specifies secondary layer polygons that abut with a primary layer polygon (that is, share only an edge or a part of an edge) are grouped with the primary layer polygon. By default, abutting secondary layer polygons are not assembled in a cluster. Can only be specified with multiple input layers.

RESULT CELLS cell_list

An optional keyword set that controls expansion of results into a set of result cells specified by cell_list, where the cell_list is defined by a Layout Cell List statement. This keyword is only used in Syntax 1. When this keyword is used, results are expanded such that only the specified result cells contain results. Results from any cell not listed as a result cell are expanded into the nearest containing result cell. The top cell is always considered to be a result cell even if it is not explicitly specified as such, and so all results that are not contained within one of the specified result cells are reported in the top cell. This keyword implicitly activates the NOPSEUDO option, since pseudo-cells cannot be specified as result cells.

Note that Calibre performs hierarchy transformations, and there is no guarantee that any specified result cell is still present after these transformations. The Layout Preserve Cell List specification statement, which also uses the Layout Cell List facility, can be used to suppress expansion of the result cells. However, enforcing preserved cells can have a significant negative effect on Calibre performance.

COMMENT “comment_string”

An optional keyword and argument used to add comments to ASCII output. This keyword set is not used in Syntax 4 or with the TRANSITION keyword. When present, the operation writes comment_string as check text in the results database, which is rendered in Calibre RVE.

An arbitrary number of COMMENT keyword sets is allowed, and each one adds one line of check text to the appropriate rule check.

The comment_string must be a single string, so comments containing spaces or special characters must be enclosed in double quotes (“ ”). Double quotes are recommended for all comments. Pattern substitution is supported—see “Pattern Substitution”.

If this keyword is encrypted in the rule file, the check comment appears as <ENCRYPTED COMMENT>.

Rule file variable values can be referenced in the comment string by preceding the variable name with a caret (^). This is the same behavior as for “@” rule check comments. In the COMMENT specification, a ^ is escaped by another ^.

CHECKNAME name

An optional keyword set used in Syntax 1 to control the name of the rule checks under which results are stored in the RDB. The name is a user-specified string. It must be a single string, so comments containing spaces or special characters must be enclosed in double quotes (“ ”). You can also use pattern substitution as discussed later under “Pattern Substitution”.

By default, the name of the rule check is generated from the name of the derived layer created by the rule check or layer statement in which the operation was executed. The cellNumber is assigned by the tool.

ANNOTATE ‘[’“name” = “value” ‘]’

An optional keyword and argument set used to associate meta-data with the input layer when writing them to a DFM Database. This keyword is processed only when running in calibre ‑dfm mode. It is ignored when running with the -drc switch. Not used in Syntax 4.

Within the arguments for this keyword, the following conditions apply:

  • The brackets [ ] are required.

  • The name can be a quoted or unquoted string and need not be unique.

  • The value must be a quoted string.

You can specify this keyword as many times as needed. Each occurrence adds another annotation (name and value pair) to the layer.

If an input layer specified for this operation already exists in the DFM Database, the ANNOTATE keyword is the only keyword evaluated for that layer. In this situation, the annotations are added the existing layer in the DFM Database. Previously existing annotations are unaffected.

CELL SPACE

An optional keyword used in Syntax 1 that specifies to output results in cell coordinates. This is similar to output from DRC Cell Name YES CELL SPACE. The resulting output shows the coordinates are the cell’s coordinates. The CN header contains the cell transform instead of inverse transform preceded with the letter C, to indicate a cell transform, as follows:

CN cell_1 C 1 0 0 1 24167 1873 1

CELL SPACE is not valid with layer2 specified or with the TRANSITION keyword.

HIGHPREC

An optional keyword in Syntax 1 that causes properties to print with twelve digits of precision. By default, six digits of precision are printed.

QUOTENAMES

An optional keyword the causes quoted DFM property names with embedded whitespace to be written to the output with the enclosing quotation marks. Not used in Syntax 4.

CSG

A required keyword used only in Syntax 3. This keyword is used to write files compatible with the Calibre LFD CSG flow. See “About the CSG Function” in the Calibre Litho-Friendly Design User’s Manual. These files are used to annotate device properties computed by Calibre LFD in a subsequent Calibre LVS run. When CSG is specified, the DFM RDB operation must be given exactly one input layer of any type.

In CSG mode, the input layer must be annotated with one or more DFM properties. Only numeric polygon properties are considered. Other properties, such as vector properties, NETID properties, or BY NET properties are ignored.

See “CSG Output” for information on the format of the output file.

If the same output filename is specified in more than one DFM RDB CSG command, the output generated by all operations is accumulated.

GDS | OASIS

A required keyword choice used only in Syntax 4 that specifies generation of a GDSII or OASIS database. These keywords are intended primarily for use with input layers generated by a DFM Fill operation with the COMPRESS keyword. Syntax 4 requires a Calibre YieldEnhancer license.

All layer arguments must be polygon layers.

If USERMERGED, PSEUDO, USER, TOP, or COMBINE is used in a Syntax 4 operation, then all DFM RDB operations that write to the same file must use the same combination of these keywords.

DFM properties and layout properties are not preserved when GDS or OASIS is specified. By default, text objects from the input database are not included in the output. They can be included by using DRC Map Text YES. However, when the input layers are generated with DFM Fill COMPRESS, text objects can be added with the TEXTTAG keyword in DFM Fill.

filename | DEFAULT | “PIPE command”

A required argument choice in Syntax 4 that specifies where to write the output in a calibre ‑drc run. In calibre ‑dfm mode, the input layers are written to the DFM Database. The argument must be placed as shown in the syntax.

  • filename — Output is written to the file filename. For OASIS output, CBLOCK and STRICT modes are disabled if output is to a named pipe.

  • DEFAULT — Output is written to the filename specified in a DFM Defaults statement with the RDB GDS or RDB OASIS keyword. Used only in Syntax 4.

  • “PIPE command” — A keyword and argument in quotation marks that specifies to open a pipe to the specified command. Output is sent to that command as standard input. The command usually is a shell command. For OASIS output, CBLOCK and STRICT modes are disabled if “PIPE command” is used. Used only in Syntax 4.

layer1_number [layer1_datatype] [layer2 [layer2_number [layer2_datatype]] …

An optional argument set that may be placed after each layer argument when GDS or OASIS is specified (Syntax 4). The arguments provide a GDS or OASIS layer number and optional datatype.

Data on the associated layer is written to the specified layer number and datatype. The defaults are layer number 0 and datatype 0.

USERMERGED | PSEUDO | USER | COMBINE | TOP

An optional keyword choice in Syntax 4 (with GDS or OASIS) that affects what is written to the database.

  • USERMERGED — Suppresses the output of pseudo-cells in the hierarchical database. Geometries in pseudo-cells are transformed up the hierarchy to the first user cell and then merged. This keyword applies to Calibre nmDRC-H only and is the default behavior.

  • PSEUDO —Includes pseudo-cells (ICV_ cells) in the hierarchical database. This keyword only applies to Calibre nmDRC-H.

  • USER — Suppresses the output of pseudo-cells in the hierarchical database. Geometries in pseudo-cells are transformed up the hierarchy to the first user cell and are not merged.

  • COMBINE — Outputs only original cell placements that contain compressed fill layers.

  • TOP — Causes Calibre nmDRC-H to transform geometries to the top-level cell without merging prior to database output. Use this option in cases where you do not want to fill in lower level cells with the input layers, such as when using a routing-based primary cell. When used in a compressed fill flow, the output results in “standard layers,” which are uncompressed layers promoted to the primary cell during output. This keyword is not restricted to Syntax 4.

KEEPEMPTY

An optional keyword used in Syntax 4 to control whether empty cells are written to the database. When not specified, cells with no results in their immediate hierarchy are omitted from the database.

PREFIX prefix_string [FILLONLY]

An optional keyword set used in Syntax 4 that specifies a prefix for all output cell names. The prefix_string argument is a string or a string variable defined by a Variable statement. When this keyword set is used, all DFM RDB operations writing to the same file must specify PREFIX with the same prefix_string.

The FILLONLY option specifies to apply prefix_string to fill generated cell names only and not all output cell names.

APPEND suffix_string [FILLONLY]

An optional keyword set used in Syntax 4 that specifies a suffix for all output cell names. The suffix_string argument is a string or a string variable defined by a Variable statement. When this keyword set is used, all DFM RDB operations writing to the same file must specify APPEND with the same suffix_string.

The FILLONLY option specifies to apply suffix_string to fill generated cell names only and not all output cell names.

AUTOREF [prefix]

An optional keyword set used in Syntax 4 with GDS (not OASIS) that provides automatic rectangle array compaction in the output.

The optional prefix argument is a string or string variable that is used as a prefix for all generated cells. The prefix “A” is used by default.

DFM RDB operations writing to the same file can have some that specify AUTOREF and some that do not. If a prefix is used in any DFM RDB operation, then all DFM RDB operations that specify AUTOREF and write to the same file must use the same prefix.

The AUTOREF keyword applies only on non-fill layers, or fill-generated layers without compression. It is a compiler error for two or more DFM RDB GDS statements with different AUTOREF prefixes to output to the same file, or with one prefix explicitly specified and one taking the default value. Not used with PROPERTIES.

PROPERTIES

An optional keyword used in Syntax 4 that causes DFM property values to be exported as GDS text objects (when GDS is specified) or OASIS properties (when OASIS is specified). This behavior only applies for hierarchical runs. This keyword may not be specified with AUTOREF.

DFM vector properties, DFM Property BY NET ONLY properties, and DFM Property derived layers that have been processed with DFM Copy UNMERGED are not supported with this keyword.

NOEMPTYDB

An optional keyword used in Syntax 4 that specifies an empty database is not output if the input layer is empty. When no database is written, this warning is issued to the transcript:

WARNING: No output is produced. File <filename> will not be generated.

METACELL cell_suffix

An optional keyword set used in Syntax 4 that creates a subcell below the top cell and places all layer data in that subcell. The string cell_suffix is appended to the subcell name. If APPEND is specified, cell_suffix is placed after the APPEND suffix_string.

You can use multiple DFM RDB operations with METACELL to create multiple subcells below the topcell. For example: 

DFM RDB OASIS "output.oas" layer1 1 METACELL _cellA
DFM RDB OASIS "output.oas" layer2 2 METACELL _cellA
DFM RDB oasis "output.oas" layer3 3 layer4 4 METACELL _cellB

The preceding code results in this cell hierarchy:

TOPCELL
--TOPCELL_cellA  // all layer1 and layer2 data
--TOPCELL_cellB  // all layer3 and layer4 data

When METACELL is used, all DFM RDB operations that write to the same file must specify METACELL. Within a set of DFM RDB operations writing to the same file, each layer can be written out only once; that is, the same layer cannot be written to multiple subcells.

Description

Writes original or derived layer data to the specified database. This statement must occur inside a rule check. These databases are written:

  • A DFM Database instead of the filename whenever “calibre -dfm” is used for the run. The results are organized by layer rather than by rule check.

  • An ASCII RDB when the DFM RDB operation contains a filename, the GDS, OASIS, or CSG keywords are not used, and either of the following is true:

    • The rule file is processed using “calibre -drc”.

    • The DFM RDB operation is used inside the Calibre YieldServer dfm::new_layer command, and the keyword ‑rdbs_as_files is also supplied.

    If the rule check in which DFM RDB appears has no other output to the DRC Results Database when the -drc option is used on the command line, then the results count sent to the DRC Results Database from that rule check is 0. This is indicated in the DRC Summary Report as follows: 

    RULECHECK rule ... TOTAL Result Count = 0 (0)
TOTAL DFM RDB Result Count = 3 (3)

    The ASCII data format is consistent with RDB data generated by other operations, and specific to the requirements of this particular operation. The RDB created by this operation uses the precision and magnification statements a DRC run uses. That is, the precision is printed at the top of the RDB and magnification factor is applied to all results, with rounding to the nearest integer as needed. See “Results Databases” in the Calibre YieldAnalyzer and YieldEnhancer User’s and Reference Manual for details.

  • GDS or OASIS output when the keywords of the same names are specified and the rule file is processed using “calibre -drc”. By default, text objects from the input database are not included in the output. This can be changed by using DRC Map Text YES.

    If OASIS output is specified, CBLOCK compression and STRICT mode are used by default. To disable either mode for all OASIS results databases, include the DRC Results Database statement with the NOCBLOCK and/or NOSTRICT keywords.

  • CSG output for the Calibre LFD CSG flow when the CSG keyword is used. See “CSG Output”.

This operation operates in any of three modes:

  • Single-Layer DFM RDB Mode — Writes a single layer as a single-layer RDB, or as a layer in a DFM database.

  • Multilayer DFM RDB Mode — Writes multiple layers as a multilayer RDB or as multiple layers in a DFM Database.

  • Transition Mode — Writes three layers in DFM Transition format.

You control the mode through the number of layers you supply and the keywords you include. Each mode is described in its own section.

While this operation behaves like a layer constructor, the derived layer created in memory by this operation is always empty and does not persist. The recommended way to use this operation is to include it in a rule check, which has empty output itself. For example:

RDB1 { DFM RDB M1 "met1.db" NOPSEUDO NOEMPTY }
RDB2 { DFM RDB (NET M2 "A?") "met2_net.db" NODAL NOPSEUDO NOEMPTY }

In this case, the RDB1 and RDB2 rule checks produce no results themselves. Only the DFM RDB operations produce results to separate results databases.

  • Multilayer DFM RDB Mode

  • Transition Mode

  • Pattern Substitution

  • CSG Output

Single-Layer DFM RDB Mode

When only one layer is specified, the operation creates either a layer in the DFM Database or an RDB file containing one rule check for each cell in the layout. Note that the following keywords generate a compilation error if used in this mode: NODAL ALL or NODAL OTHER, SAME CELL, TRANSITION, ABUT ALSO. See “Single Layer RDB File Format” in the Calibre YieldAnalyzer and YieldEnhancer User’s and Reference Manual.

Multilayer DFM RDB Mode

When multiple layers are specified, the operation either saves the layers in the DFM Database or writes the layer data from all the layers into the same RDB file. By default, the RDB file contains one rule check for every cell in the layout. When ALL CELLS is specified, the RDB is organized into a single rule check with results grouped by cell and cluster. Within a rule check, cell data is organized by clusters representing the results from several layers, as described in “Multilayer RDB File Format” in the Calibre YieldAnalyzer and YieldEnhancer User’s and Reference Manual.

Data is organized into clusters in this mode. Clustering is similar to that used by the DFM Property operation with the OVERLAP keyword, although the keywords used are not the same. See “Clustering Methods in DFM Property”. The keywords ABUT ALSO and SAME CELL affect the clustering in multilayer DFM RDB operation.

If a polygon from a secondary layer (layer2 …) overlaps two or more polygons from layer1, that polygon is grouped arbitrarily in a cluster with one of the layer1 polygons, and a warning is generated in the transcript. If a polygon from secondary layer does not overlap any polygons from layer1, it is not written into the RDB and a warning is generated in the transcript.

Transition Mode

When TRANSITION is specified, the operation writes data in the Transitions RDB File Format format, which is a format designed for storing layout modifications, such as the results from the DFM Transition operation. This mode lets you do the following:

  • Generate layers representing layout modifications using the method of choice, then save them in a familiar format for viewing and manipulation using analysis and post-processing tools such as Calibre RVE.

  • Perform post-processing on layers generated by the DFM Transition operation and then writing the post-processed geometries in the format that would have been used had they been written to the RDB from DFM Transition directly.

There are several restrictions on the TRANSITION mode DFM RDB operation:

  • The operation must have exactly three input layers. The first input layer plays the role of a via layer and contains the added vias. The two additional input layers contain the added metal enclosure for these vias.

  • NODAL ALL cannot be used, but NODAL can be used.

These restrictions are enforced by the rules compiler.

The format of an RDB generated by the DFM RDB TRANSITION operation is identical to the format of an RDB generated by the DFM Transition operation in BY CELL mode with the following exception: the statistical data present in the check text of a DFM Transition RDB are not available in the DFM RDB TRANSITION RDB. Zero values are written out instead.

The optional NODAL keyword causes the RDB to be annotated with net numbers and names in the same format as an RDB produced by a DFM Transition operation. Only the net numbers from the first input layer, which should be the via layer, are used.

Optional SAME CELL and ABUT ALSO keywords are supported in TRANSITION emulation mode, and have the same effect on the clustering algorithm as they do in a regular multilayer RDB operation.

Pattern Substitution

Arguments of the COMMENT and CHECKNAME keywords support pattern substitution for the following patterns:

  • %_t_ — Replaced by the top cell name.

  • %_l_ — Replaced by the input layer name of the DFM RDB operation. If a DFM RDB operation contains multiple input layers, this pattern is replaced by the primary layer.

The patterns are case-insensitive. For example, both %_t_ and %_T_ are equally valid patterns.

CSG Output

Syntax 3 with the CSG keyword is used to write files compatible with the Calibre LFD CSG flow. See “About the CSG Function” in the Calibre Litho-Friendly Design User’s Manual. These files are used to annotate device properties computed by Calibre LFD in a subsequent Calibre LVS run.

When CSG is specified, the DFM RDB operation must be given exactly one input layer of any type. The input layer must be annotated with one or more DFM properties. Only numeric properties are considered. Other properties, such as vector properties, NETID properties, or BY NET properties are ignored.

The output file consist of a header section and Layout Text specification statements.

The following header, consisting of SVRF comment lines, is written at the beginning of the output file:

// count: time
// comment
// IL: layer

where:

  • count — The number of Layout Text lines written by this DFM RDB operation.

  • time — The date and time of the run.

  • comment — The comment string specified in the DFM RDB command. One line is written for each specified COMMENT.

  • layer — The name of the input layer of the DFM RDB operation.

For each geometry on the input layer, one line is written to the output file for each property. The line is a valid Layout Text specification statement with the following format:

LAYOUT TEXT value X Y name cell

where these definitions apply:

  • value — The value of the property converted to meters—the SPICE unit of length. This conversion takes into account the value of Unit Length if it is specified in the rules. The default unit length is 1e-6.

  • X Y — The coordinates of the center of the extent of the geometry in user units, that is polygon, edge, or edge cluster, depending on the input layer type. Conversion to the user units takes into account Precision and DRC Magnify Results magnification, if specified. All coordinates are at the chip level.

  • name — The name of the DFM property.

  • cell — The name of the top-level cell containing the geometry. The cell name is omitted if Calibre is running flat.

The lines for all properties of one geometry are written one after the other, then the lines for the next geometry.

The same output filename can be specified in more than one DFM RDB CSG command; the output generated by all operations is accumulated. The same filename should not be used as the output file for other syntax forms of DFM RDB.

Examples

Creates an RDB file with data from layer M12, the checkname “M12”, and the check comment “Layer M12 in cell ram1”.

LAYOUT PRIMARY ram1

M12 = MET1 AND MET2

DFM_CHECK {
DFM RDB M12 "dfm.rdb" CHECKNAME "%_l_" COMMENT "LAYER %_l_ in cell %_t_" 
}
