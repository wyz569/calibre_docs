NORMALIZE_XY

Vector property function for DFM expressions

Acts as a transformation function that shifts the first elements of input vector tuples by a fixed amount. By default, the shift is to the ordinate axis of the vector data space. The transformation removes the dependence of the input data on the coordinates of particular geometries, and allows placement-independent characterization and comparison of the vector values.

Usage

NORMALIZE_XY(vector[, base_value[, orientation_flag]])

Arguments

vector

A required vector argument containing at least one tuple <c m1 m2 … mn>, where c is an interval endpoint coordinate in user units, and each mi is a measurement value. All values are floating-point numbers.

A vector argument may be a comma-delimited list of tuples of the proper form. Conceptually, if “myvec = {<a b c> <d e f>}”, where “a” through “f” are numeric values, then myvec can be properly formed as a vector argument to the function.

base_value

An optional numeric value that is subtracted from all first elements of tuples in the vector. By default, the value of the first tuple’s initial (0th) element is used.

orientation_flag

An optional numeric value. The value is treated as a boolean that determines the orientation of the vector. A value of 0 means “false”, and any other value means “true”. By default the value is 1 (“true”), and the input vector is unaltered. If 0 (“false”) is specified, the additive inverses of the 0th elements of all tuples are used, and the order of the tuples is reversed appropriately.

Return Values

Vector of the same dimension as the input vector.

Description

This function is typically applied to the output of SORT_MERGE_XXY or related functions. That output represents a sequence of horizontal segments in some vector data space. Intuitively, the NORMALIZE_XY function effectively translates that sequence so that the leftmost segment in the sequence starts at x' = 0 or some other point, where x' maps to some X or Y coordinate in cell space.

Here is a simple example to introduce the function’s behavior. Assume an input vector, where 10, 20, and 30 represent some X or Y coordinate values, and 1, 2, and 0 represent measurement values at the corresponding coordinates:

{<10 1> <20 2> <30 0>}

This vector could correspond to measurements along an edge, such as here:



Recall that the vector is from SORT_MERGE_XXY or a related function, which always assigns a measurement value of 0 to the final tuple. NORMALIZE_XY by default transforms this vector to the following:

{<0 1> <10 2> <20 0>}

Essentially the vector is translated so the leftmost coordinate element is at the ordinate axis of the vector space, such as here:



This code produces the preceding result:

vec_layer = DFM PROPERTY layer 
             [ V = CONCAT( VECTOR(10,1), VECTOR(20,2), VECTOR(30,0) ) ]
vec_norm  = DFM PROPERTY vec_layer 
             [ norm_V = NORMALIZE_XY( VPROPERTY(vec_layer,V) ) ]

If only one argument is provided to NORMALIZE_XY, the behavior is similar to the preceding example.

If two arguments are provided, then the base_value is used to perform the vector translation instead of the first tuple’s initial element.

If three arguments are provided and orientation_flag is nonzero, the behavior is the same as if only the first two arguments had been provided. If orientation_flag is 0, then the base_value is subtracted in the usual way, the additive inverse of each tuple’s 0th element is taken, and the order of the tuples is reversed, except for the measurement value of the final tuple. For example, assume vector V is the same as the one used previously:

V = {<10 1> <20 2> <30 0>}

NORMALIZE_XY(V,10,0) returns this:

<-20 2> <-10 1> <0 0>

The final 0 is unaltered, as this value is immaterial. (Recall that in SORT_MERGE_XXY and related functions, the final measurement value is assigned as 0 simply to guarantee the proper dimension of the tuple.) This figure shows the effect graphically:



To define the concept of vector reversal more precisely, recall that the vector represents a piecewise constant relation, with the break points (call them x' values, corresponding to X or Y coordinates in cell space) given by the 0th elements of the tuples, and the relation values themselves (call them y' values, corresponding to measurements in cell space) given by the other tuple elements. The reversal of the vector takes the additive inverses of the 0th elements of the tuples, and the order of the tuples is reversed appropriately so the resultant vector represents reflecting the piecewise constant relation across the y' axis. As mentioned in the preceding paragraph, the final element of the final tuple is immaterial to the relation (except for preserving dimension), so that element remains unchanged.

The orientation_flag argument is typically obtained by comparing the values of ECBEGIN and ECEND for a primary layer. Recall that ECBEGIN and ECEND are measurement functions closely related to ECMIN and ECMAX. For concreteness, consider the result of applying these functions on an edge layer. Suppose that an edge E on the layer is fully merged in a cell C. If measurements are performed in C, then these equalities are true for E:

ECMIN = ECBEGIN
ECMAX = ECEND

When the edge layer is promoted into higher-level cells for clustering with other geometries, the values of these EC functions are transformed into the local coordinate space of the promoted layer. ECMIN and ECMAX may be swapped to ensure that ECMIN <= ECMAX, but ECBEGIN and ECEND always have values corresponding to the endpoints of E with which they were associated in C. This means that if E has swapped orientation during promotion from C to the higher cell, ECBEGIN is greater than ECEND. Checking for this condition allows the orientation_flag to be set appropriately. The following example demonstrates this.

Examples

Consider this layout, where enclosure measurements of a gate polygon by a well polygon are taken:



Following is code to process the measurements:

VARIABLE wproxd 5 // well proximity background distance

// perpendicular ngate orientation edges
ngate_side_orient = COINCIDENT EDGE ngate sd
// ngate enclosure by pwell < wproxd
ngate_side_enclosure = ENC ngate_side_orient pwell < wproxd OPPOSITE

ngate_side_norm = DFM PROPERTY ngate_side_orient ngate_side_enclosure
                  OVERLAP ABUT ALSO MULTI
                  [ PER_NORM = NORMALIZE_XY(
                                 SORT_MERGE_XXY(
                                   VECTOR(
                                     ECMIN(ngate_side_enclosure),
                                     ECMAX(ngate_side_enclosure),
                                     EW(ngate_side_enclosure)
                                   )
                                 ),  
                                 ((COUNT(ngate_side_enclosure) == 0) ? 0 :
                ((ECBEGIN(ngate_side_orient)< ECEND(ngate_side_orient))
   ? MIN(ECMIN(ngate_side_enclosure)) : MAX(ECMAX(ngate_side_enclosure))
                 )
                                 ),
         (ECBEGIN(ngate_side_orient) < ECEND(ngate_side_orient)) ? 1 : 0
                               ) ]

If ngate_side_norm is output to a DFM RDB, the PER_NORM properties for the measurement vectors could be these for a given device instance:

PER_NORM <0 0.3> <0.3 0>
PER_NORM <0 3.05> <0.15 0.2> <0.3 0>

where 204.35 is used as the base_value, the orientation_flag is 1 (because the vectors are not reversed), and the maximum enclosure of ngate by pwell is 3.05u.
