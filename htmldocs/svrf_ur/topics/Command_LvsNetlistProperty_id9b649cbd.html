LVS Netlist Property

Specification statement

Specifies Device computed properties that behave as LVS-actionable properties during netlist extraction.

Usage

LVS NETLIST PROPERTY component_type[‘(’component_subtype‘)’] property [property …]

Arguments

component_type

A required LVS component type.

‘(’component_subtype‘)’

An optional component subtype of the component_type parameter. If specified, the component_subtype must appear within parentheses. If component_subtype is not present, then the statement applies to all instances of the specified component_type regardless of their subtype.

property

A required name of a computed device property. More than one property may be specified.

Description

Specifies properties computed in a property_specification block of a Device statement to be preserved in the extracted SPICE netlist, even if they are not LVS-actionable. This statement may be specified once for each component_type and optional component_subtype.

This statement is only used in the LVS Push Devices SEPARATE PROPERTIES YES (PDSP) flow, where certain properties can be stripped from the extracted netlist because they are not used for LVS comparison purposes.

LVS-actionable properties are those required by the LVS comparison phase because they are used in one or more of the following rule file statements:

  • Trace Property, including user-defined properties.

  • LVS Filter family of statements.

  • LVS Reduce family of statements using the TOLERANCE keyword. Only the tolerance properties are used.

    Note that properties mentioned in the EFFECTIVE section of LVS Reduce statements are not LVS-actionable. This is in keeping with treatment of properties mentioned for these statements themselves.

  • LVS Split Gate Ratio

  • LVS Recognize Gates Tolerance

Specifying a property in an LVS Netlist Property statement makes that property behave as if it were in any of the preceding statements.

Only properties specified in a Device property_specification block are LVS-actionable. (This does not include a device_annotation_program block.)

In Calibre PERC, the effect of this statement is the same as in LVS. It applies to layout netlist extraction only, not to reading properties from the netlist.

Examples

In an LVS Push Devices SEPARATE PROPERTIES YES flow, LVS Netlist Property can be used to treat certain computed properties as LVS-actionable, even though they are not used for circuit comparison. In this example, the AD, AS, PD, PS, NRD, and NRS computed properties are not traced, so they are not used for circuit comparison. However, because they are specified in LVS Netlist Property statements, they do appear in the extracted layout netlist.

Note:

Typically, you do not want to handle non-actionable properties in this way.

DMACRO mosprop seed sd{
  [
    property L, W, AD, AS, PD, PS, NRD, NRS
    weffect = 0      
    W = 0.5 * ( perim_co( S, seed ) + perim_in( S, seed )
                + perim_co( D, seed ) + perim_in( D, seed ) )
    L = area( seed ) / W
    if ( ( bends( seed ) != 0 ) && ( weffect != 0 ) ) {
      if( W > L ) W = W - weffect * bends( seed ) * L
      else L = L - weffect * bends( seed ) * W
    }
    AD = AREA(D)
    AS = AREA(S)
    PD = PERIM(D)
    PS = PERIM(S)
    NRD = 1/AD
    NRS = 1/AS
  ]
}
DEVICE MP gate pgate psd psd CMACRO mosprop gate psd
DEVICE MN gate ngate nsd nsd CMACRO mosprop gate nsd

TRACE PROPERTY MN L L 0
TRACE PROPERTY MN W W 0
TRACE PROPERTY MP L L 0
TRACE PROPERTY MP L L 0

LVS PUSH DEVICES SEPARATE PROPERTIES YES
// preserve MOS PEX simulation properties in the layout netlist
LVS NETLIST PROPERTY MP AD AS PD PS NRD NRS
LVS NETLIST PROPERTY MN AD AS PD PS NRD NRS
