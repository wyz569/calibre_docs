LVS Report Option

Specification statement

Controls the level of detail and contents of an LVS Report. Not used in Calibre PERC.

Note:

Some of the options cause the LVS Report to be long and difficult to read, especially options A, AV, C, and CV. These options should be avoided in production rule files as they also increase the size of the SVDB (standard verification database), which causes long load times in Calibre RVE.

Usage

LVS REPORT OPTION {NONE | option [option…]}

Arguments

NONE

Keyword that specifies the concise LVS Report format and transcript output is used. This is the default if the statement is not specified.

option

A case-insensitive keyword that specifies the verbosity for a particular section of the LVS Report or transcript. You may specify option any number of times in one statement, separated by spaces. The list of keywords is given in the following table.

Table 1. Keywords for LVS Report Option

Option

Description

A

Enables the reporting of detailed instance connections on matched nets that are classified as INCORRECT (except for Short Circuit and Open Circuit discrepancies). Regardless of whether this option is specified, much of this information is also reported in a more concise form in the DETAILED INSTANCE CONNECTIONS section located at the bottom of the default LVS report. See Example 1.

AV

Enables the same reporting as option A, with the exception that the AV option does not report correct devices on incorrect power and ground nets. For example, specifying this option does not generate the Correct Devices On This Net section in the LVS report when power or ground nets are reported as incorrect. Power and ground nets are defined by LVS Power Name and LVS Ground Name statements in the rule file. Specifying both the A and AV options generates the LVS report based only on the AV option.

B

Enables the reporting of detailed instance connections in Short Circuit and Open Circuit discrepancies. See Example 2.

BV

Enables the same reporting as option B, except for reporting correct devices on power and ground nets. In other words, if the net for which detailed connections are being reported is a power or ground net, correct instances are not reported. Power and ground nets are defined by LVS Power Name and LVS Ground Name statements in the rule file. Specifying both the B and BV options generates the LVS report based only on the BV option.

BX

Causes LVS Box cells to be listed in the setup section of the LVS report. Cells are listed by layout and source, respectively. The cells listed are those specified in the rule file (all LVS Box statements are concatenated), whether they are present in the design or not.

C

Enables the reporting of detailed instance connections in Missing Net and No Similar Net discrepancies. See Example 3.

CV

Enables the same reporting as option C, except for reporting correct devices on power and ground nets. In other words, if the net for which detailed connections are being reported is a power or ground net, correct instances are not reported. Power and ground nets are defined by LVS Power Name and LVS Ground Name statements in the rule file. Specifying both the C and CV options generates the LVS report based only on the CV option.

D

Enables the reporting of detailed instance connections reporting in Missing Instance and Missing Gate discrepancies. See Example 4.

E

Causes LVS to ignore missing properties. Subsequently, missing properties are not reported and do not cause the overall result to be INCORRECT. Only the number of missing properties are reported in the statistics section. By default, missing properties are reported as discrepancies and cause the overall result to be INCORRECT. This option should be used with caution, as it disables a potentially important diagnostic.

E1

Causes LVS to ignore missing properties that are used exclusively in TOLERANCE clauses of LVS Reduce statements, and that are not used for any other purpose. Such missing properties are not reported as missing and do not cause the overall result to be INCORRECT. Only the number of missing properties is reported in the Statistics section.

Missing properties that are used in other contexts (for example, in Trace Property specification statements) continue to be reported as discrepancies and cause the overall result to be INCORRECT, even if they also appear in TOLERANCE clauses. (Unless such reporting is disabled by other options, such as option E.)

As is normally the case, when a property used in a reduction TOLERANCE statement is missing on a device, then, with respect to that property, the device participates in the reduction as if the TOLERANCE statement were not present.

EB

Causes bad device warnings that occur during circuit extraction to produce an INCORRECT status during LVS comparison for the cells in which the devices occur. The discrepancy is reported like this:

BAD DEVICE(s) detected by extraction in this cell. See extraction report for details.

EC

Causes Sconnect conflicts that occur during circuit extraction to produce an INCORRECT status during LVS comparison for the cells in which the conflicts occur. The discrepancy is reported like this:

SCONNECT error(s) detected by extraction in this cell. See extraction report for details.

See also option S.

EO

Causes open circuit warnings that occur during circuit extraction to produce an INCORRECT status during LVS comparison for the cells in which the text opens occur. The discrepancy is reported like this:

Open circuit(s) detected by extraction in this cell. See extraction report for details.

ES

Causes short circuit warnings that occur during circuit extraction to produce an INCORRECT status during LVS comparison for the cells in which the text shorts occur. The discrepancy is reported like this:

Short circuit(s) detected by extraction in this cell. See extraction report for details.

F

Disables missing smashed MOSFET warnings, the unbalanced smashed MOSFET summary warning at the top of the LVS report, and respective warnings in the Information and Warnings section for the LVS report. Also see option O.

FX

Enables DETAILED ERROR ANALYSIS section of the report. This section can improve comprehension of the report and simplify debugging of numerous common LVS error types. For complete details of this section of the report, see “Detailed Error Analysis Reporting” in the Calibre Verification User’s Manual.

G

Enables the reporting of detailed instance connections in Property Error discrepancies.

H

Disables warnings in Calibre nmLVS‑H about cell correspondence names not found in the layout or source. The tool normally prints these warnings in the transcript.

I

Disables the listing of isolated nets in the LVS report. However, named top-level isolated nets are always retained and participate in LVS comparison.

LPE

Causes the overall comparison result to be INCORRECT if there are unmatched passthrough nets present in the layout. By default, passthrough nets in the layout are reported as warnings in the INFORMATION AND WARNINGS section of the LVS report. Passthrough nets are nets that are connected to ports but not connected to anything else inside the subcircuit. Calibre nmLVS-H removes such nets and their ports unless they are named and can be used as initial correspondence points. For a full discussion and examples, see the section “Passthrough Nets” in the Calibre Verification User’s Manual. See also options SP and SPE.

MC

Controls the reporting of device instance counts based upon model name (subtype). By default, initial instance counts and instance counts after transformations are reported per element name (component type). For example, one count is reported for all MN devices, even if there are MN devices with different model names. When MC is specified, instance counts are reported separately for each element name and model name. See Example 5.

N

Causes LVS to report layout names that are missing in the source as errors. LVS also reports the overall comparison result as INCORRECT and generates the secondary comparison status message:

Error: Layout names missing in source.

Additionally, a NAME ERRORS section is created in the report for any instance, net, or port name errors, respectively.

NCA

Controls reporting of NOT COMPARED cells. If a cell with the status of NOT COMPARED is detected, all remaining cells that have not been fully compared are also reported as NOT COMPARED (no attempt is made to finish comparing the remaining cells) and the run terminates with a NOT COMPARED status. Cells that may have been compared as either CORRECT or INCORRECT prior to the detection of the first NOT COMPARED cell retain their compared statuses.

NE

Causes LVS to report a NOT COMPARED status for corresponding cells that have differing instance counts in layout and source. Both instance type (element) and subtype (model) are checked when comparing instance counts. In flat LVS, this option causes the top-level comparison status to be NOT COMPARED if the instance counts do not match between layout and source.

NOK

Causes Calibre nmLVS-H to provide detailed comparison results only for hcells in which discrepancies occur. Hcells that compare correctly have just their object counts reported. By default, in Calibre nmLVS-H, all hcells are reported in detail whether they compare correctly or not. If you specify NOK, only INCORRECT hcells have all details reported. In that case, the CELL COMPARISON RESULTS section header changes to CELL COMPARISON RESULTS (INCORRECT CELLS ONLY). For CORRECT hcells, the section header is CORRECT CELL CONTENTS SUMMARY. The overall results of all cells are still reported in the OVERALL COMPARISON RESULTS section.

NP

Causes the LVS report to omit correctly matched pins. This makes reading the report easier when the number of pins is large but only a few are incorrect.

O

Enables reporting of ambiguity resolution and unbalanced smashed MOSFETs status messages in the OVERALL COMPARISON RESULTS and CELL COMPARISON RESULTS sections. This reporting would be otherwise disabled by LVS Report Option options F, R, or RA. If LVS Report Option O is specified, options F, R, and RA disable reporting of ambiguity resolution points and unbalanced smashed MOSFETs, respectively, but the overall status messages are printed followed by the string “(reporting disabled)”.

P

Disables the reporting of “Direct connections between different ports” warnings in the connectivity extractor.

PG

Causes Calibre nmLVS-H to report all hcells that have mismatched power and ground nets. Calibre nmLVS-H reports the comparison status as NOT COMPARED for hcells that have different supply nets in layout and source (for example, power net is missing in layout). By default, however, this status is reported only if logic gate recognition is enabled, and at least one gate is recognized in the cell in either layout or source. The PG option ignores the dependency on logic gate recognition and reports any hcells with power and ground net mismatches.

R

Disables reporting of ambiguity resolution points in LVS when the overall comparison result is CORRECT. This option controls reporting of ambiguity resolution points in the Information and Warnings section of the LVS report, as well as reporting of ambiguity resolution status messages in the OVERALL COMPARISON RESULTS and CELL COMPARISON RESULTS sections of the LVS report. When the overall comparison result is other than CORRECT, ambiguity resolution points are reported as usual, regardless of this option. Also see options O and RA.

RA

Disables reporting of ambiguity resolution points in LVS independent of the overall comparison result. This option is similar to LVS Report Option R, which also disables reporting of ambiguity resolution points (but only if the overall comparison result is CORRECT). Both options control reporting of ambiguity resolution points in the INFORMATION AND WARNINGS section of the LVS report.

RD

Marks reduced instances in the LVS report.

If LVS Report Option RD is specified, names of reduced instances in the LVS report are followed by the string “(SL)” for layout instances or “(SS)” for source instances. Note that these annotations are the same as the ones used in instance cross-reference files to mark reduced instances.

For example, a property error report involving a reduced layout instance could appear as follows:


1    m2 (SL)  MP(P)             m1  MP(P)
     w: 2 u                     w: 1 u             100%

S

Enables the detailed reporting of Sconnect conflicts in the LVS run transcript and the circuit extraction report. The report format is discussed under “LVS Connectivity Extraction Messages” in the Calibre Verification User’s Manual.

See also option EC.

SP

Enables reporting of passthrough nets in the source. Such nets are reported in the INFORMATION AND WARNINGS section of the LVS report as warnings. Passthrough nets are nets that are connected to ports but not connected to anything else inside the subcircuit. Calibre nmLVS removes such nets and their ports, unless they are named and can be used as initial correspondence points. For a full discussion and examples, see the section “Passthrough Nets” in the Calibre Verification User’s Manual. See also option SPE.

SPE

Causes the overall comparison results to be INCORRECT if there are unmatched passthrough nets in the source. The nets are reported as for option SP. For a full discussion and examples, see the section “Passthrough Nets” in the Calibre Verification User’s Manual. See also option LPE.

UP

Causes unattached ports detected during circuit extraction to be passed through to the LVS comparison module and to cause an INCORRECT result in the LVS report. The section of the report is entitled “Errors passed through from extraction.” The error message is this:

Unattached port(s) detected by extraction in this cell. See extraction report for details.

Additionally, a SPICE comment beginning with “*.CALIBRE WARNING PORTS” and containing the preceding error message is written to the extracted netlist for any affected subcircuits.

V

Causes the connectivity extractor in Calibre and ICverify to report virtual connections (for example, from the Virtual Connect Name or Virtual Connect Colon YES specification statements). Virtual connections are reported as notes in the connectivity extraction report. Specifying this option is equivalent to specifying Virtual Connect Report YES.

W

Causes LVS to report an error rather than a warning when source and layout refer to the same data. When the source and layout refer to the same data, this option causes the overall comparison result to be INCORRECT.

X

Causes Calibre nmLVS-H to provide a list of representative instances for each configuration of pins reported in the COMPONENT TYPES WITH NON-IDENTICAL SIGNAL PINS section of the report. Each list of instances is preceded by its parent cell name. The final pin configuration (after discarding of extra pins) is shown as well. See the section “Component Types with Non-Identical Signal Pins” in the Calibre Verification User’s Manual.

XR

Disables creation of cross-reference database (XDB) when comparison result is CORRECT.

By default, creation of the cross-reference database is controlled by options of the Mask SVDB Directory specification statement, as well as command-line arguments -ixf and -nxf. If option XR is specified, then the database is not created if there are no LVS errors and the overall comparison status is CORRECT. This option can be useful if the cross-reference database is used only to debug LVS errors and generating it takes considerable time. This option has no effect if the cross-reference database is not requested.

Description

Controls the level of detail of an LVS Report and related run transcript entries. The default report format is concise, that is, no report options are set or NONE is specified. Commonly used options include: F, FX, NOK, R, RD, S, V, and W.

Some options such as those in the A, B, C, and D families cause voluminous output that can cause the report to become difficult to read. Such options are meant primarily for debugging purposes when the information they provide can help to isolate an existing discrepancy.

Statements can be repeated and options are accumulated.

You can specify LVS Report options in Calibre Interactive—nmLVS. See “Specifying Calibre nmLVS Report Options” in the Calibre Interactive User’s Manual.

See also LVS Report Pinswapped, LVS Report Trivial Ports, LVS Report Warnings Hcell Only, and LVS Report Warnings Top Only.

    ** unmatched connection **     X1/M5:D

Option A specified:

5   Net 8(800,500)                 DOG
    ----------------               ----------------

    --- Incorrect Devices On This Net ---

      s: 2(200,500)                  s: B

      d: 7(700, 500)                 ** CAT **

Example 5

This example illustrates the default output followed by output of LVS Report Option MC:

Default:

INITIAL NUMBERS OF OBJECTS
--------------------------

                Layout    Source         Component Type
                ------    ------         --------------
 Instances:         20        30    *    mn (4 pins): g (s d) b

NUMBERS OF OBJECTS AFTER TRANSFORMATION
---------------------------------------

                Layout    Source         Component Type
                ------    ------         --------------
 Instances:         10        11    *    mn (4 pins): g (s d) b

It may be difficult to determine why there is an extra device in the source. With LVS Report Option MC, the counts are reported separately for all MN devices with different model names:

Option MC specified:

INITIAL NUMBERS OF OBJECTS
--------------------------

                Layout    Source         Component Type
                ------    ------         --------------
 Instances:         20        15    *    mn(N) (4 pins): g (s d) b
                     0        15    *    mn() (4 pins): g (s d) b

NUMBERS OF OBJECTS AFTER TRANSFORMATION
---------------------------------------

                Layout    Source         Component Type
                ------    ------         --------------
 Instances:         10        10         mn(N) (4 pins): g (s d) b
                     0         1    *    mn() (4 pins): g (s d) b

Now it is clear that the extra device in the source has a different model name. It is likely that this device was not reduced together with other MN devices because device reduction operates only on devices with identical element and model names.
