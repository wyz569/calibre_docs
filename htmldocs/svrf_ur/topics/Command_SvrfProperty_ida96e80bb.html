SVRF Property

Specification statement

Defines a name in the rule file corresponding to an OASIS property in the layout.

Usage

SVRF PROPERTY SVRF_name [EXACT | SUB | SUPER] OASIS_name
[{value_type value}…]

Arguments

SVRF_name

A required name to assign to the SVRF property. Names that differ by text case are considered different names. The name may be a character string or string constant.

EXACT

An optional keyword that indicates the OASIS property values and SVRF value parameters are equal in count and the property values match in the order specified in the rule file.

SUB

An optional keyword that indicates each value parameter matches some value in the OASIS property.

SUPER

An optional keyword that indicates each value in the OASIS property matches some value parameter.

OASIS_name

A required name of the OASIS property to match. The name may be a character string or string constant. This name may contain one or more wildcard characters (*), which match zero or more characters.

value_type

An optional integer that specifies an OASIS value type to match. The following are valid values:

  • 0 — Specifies value is a numeric constant. It matches OASIS value types 0 - 9 if the numeric values (after sign application, division, and so forth) are equal. Note that numeric values may be coded as strings, which falls under value_type 2.

  • 1 — Specifies value is a pair of integer constants N D representing the ratio N / D. It matches OASIS value types 0 - 9 if the numeric values (after sign application, division, and so forth) are equal.

  • 2 — Specifies value is a character string and matches OASIS value types 10, 11, and 12 if the strings match in a case-sensitive manner.

A value_type is specified with a value. More than one value_type value pair may be specified. If no value_type is specified, then a matching OASIS property has no value.

value

An optional value of the OASIS property. The value can be numeric or string. The value is specified after value_type. If a value_type of 2 is specified, the value string may contain one or more wildcard characters (*) which match zero or more characters in OASIS.

If the OASIS property value is an a-string and if the SVRF Property value does not contain leading or trailing space characters, then leading and trailing space characters are removed from the OASIS string prior to comparison.

Description

Defines a name in the rule file corresponding to an OASIS property in the layout. This statement provides a simple way to reference a potentially complex OASIS property by a single user-defined name.

The SVRF_name is referenced in Inside Cell [NOT] WITH PROPERTY operations and is used for property-based polygon selection.

Unlike Layout Polygon or Layout Text, this statement only specifies the property name in the rule file, not in the Calibre representation of the layout database.

The OASIS specification1is maintained by SEMI®2. Refer to their web site to obtain the OASIS document:

http://www.semi.org

From the OASIS specification, a property (after the required resolution of all name and string values and modality processing) has the following syntax:

<propname-string> <prop-value-count> <prop-value>*
	<propname-string> -> n-string
	<prop-value-count> -> unsigned integer
	<prop-value> -> refer to sections 7.3 and 7.8 in the OASIS document

The <prop-value> types 13, 14, and 15 must be resolved during layout read (stream in) and are never visible after reading the file. Calibre replaces byte zero in b-strings with the space character.

A value_type parameter must correspond to the appropriate OASIS property value type in order for a property match to occur. If no value_type value parameter set is specified, then the property match is based upon the property name only.

The SVRF Property matches its OASIS counterpart if the value specified for OASIS_name matches the case-sensitive OASIS <propname-string> and one of the following is true:

  • If none of the EXACT, SUB, or SUPER keywords is specified, each value specified in the SVRF Property statement matches some value in the OASIS property and each value in the OASIS property matches some value in the SVRF Property statement.

  • If EXACT is specified, the number of property values is equal, and the property values match in the order specified in the rule file.

  • If SUB is specified, each value specified in the SVRF Property statement matches some value in the OASIS property.

  • If SUPER is specified, each value in the OASIS property matches some value specified in the SVRF Property statement.

Examples

Example 1

In this example, sprop is the property name used in the rule file that corresponds to the OASIS property oprop having the value POWER. Because SUB is specified, this means POWER must match some value in the OASIS property oprop in order for sprop to match anything. The sprop parameter is then used in the Inside Cell operation to derive a layer consisting of all L1 polygons having property sprop.

SVRF PROPERTY sprop SUB oprop 2 "POWER"
L1_prop = INSIDE CELL L1 "*" WITH PROPERTY sprop

Example 2

Assume there is a layout polygon having two OASIS property records attached to it. The OASIS property name is “prop” and has the values 1 and 2 (assume these are unsigned positive integers of OASIS type 1), respectively:

     First record: prop 1

     Second record: prop 2

This statement in the rule file does not match the aforementioned polygon’s properties:

SVRF PROPERTY sprop prop 0 1 0 2

The reason this does not match is each of the values in the statement must match some value in the OASIS properties. However, each of the OASIS property records has only one value associated with it, whereas two are needed for a default match.

This statement causes a match of both of the OASIS property records:

SVRF PROPERTY sprop SUPER prop 0 1 0 2

The matches occur because SUPER specifies that each value in an OASIS property matches some value in the specification statement.
