DRC Results Database

Specification statement

Specifies the filename and type of the DRC results database. Used only in Calibre nmDRC and Calibre nmDRC-H.

Usage

DRC RESULTS DATABASE {filename | “PIPE command”}
[ASCII | {GDSII | GDS | GDS2} | {OASIS [AUTOMAP] [INDEX [NOVIEW]]}]
[PREFIX string] [APPEND string]
[CBLOCK | NOCBLOCK] [STRICT | NOSTRICT]
[USER | PSEUDO | USER MERGED | TOP | COMBINE]

Note:

Beginning with the 2016.2 release, the CBLOCK and STRICT keywords are the default for the OASIS output. For releases prior to 2016.2, the NOCBLOCK and NOSTRICT arguments are the default.

Arguments

filename | “PIPE command”

A required argument choice that specifies where to write the results database.

  • filename — Write the Calibre nmDRC results database to the file filename.

    The filename argument can contain environment variables. See “Environment Variables in Pathname Parameters”.

  • “PIPE command” — Open a pipe to the specified command. The quotation marks are required. The command usually consists of a shell command. See “PIPE Output”.

    This argument may not be specified with ASCII. If specified with OASIS, then neither CBLOCK nor STRICT may be specified.

ASCII | {GDSII | GDS | GDS2} | {OASIS [AUTOMAP] [INDEX [NOVIEW]]}

An optional keyword choice that specifies the format of the results database. See the “Description” section for additional information on each format.

  • ASCII — ASCII format. This is the default if you do not specify a keyword from this set.

  • GDSII | GDS | GDS2 — GDSII format.

  • OASIS [AUTOMAP] [INDEX [NOVIEW]] — OASIS format. These optional keywords affect the OASIS database generation:

    • AUTOMAP — Causes the layer numbers in the output database to be indexed starting with an integer that is one more than the highest DRC Check Map output layer number (possibly 0, for which the starting index is 1). The output layer numbers are incremented consecutively. If the maximum output layer number of 65535 is exceeded, then the layer number 0 is assigned. A datatype of 0 is always assigned.

      When AUTOMAP is used, warnings associated with outputs having the default layer number of 0 are disabled.

    • INDEX [NOVIEW] — Specifies the OASIS database is indexed. Indexing improves job deck loading and pattern file viewing in Calibre WORKbench.

      The NOVIEW keyword specifies that indexing will omit some computations intended only to improve interactive viewing speed. This keyword, only intended for direct input to Calibre MDP functions, generates the index more quickly but the drawing speed in the viewer is substantially slower.

PREFIX string

An optional keyword specifying a string to prefix to all to cell names. This keyword is only valid for GDSII or OASIS format databases. This behavior applies only to results from Calibre nmDRC-H. This keyword set specifies the global default for all GDSII or OASIS databases generated during the run.

APPEND string

An optional keyword specifying a string to append to all cell names. This keyword is only valid for GDSII or OASIS format databases. This behavior applies only to results from Calibre nmDRC-H. This keyword set specifies the global default for all GDSII or OASIS databases generated during the run.

CBLOCK | NOCBLOCK

An optional keyword choice that specifies whether Calibre embeds compressed data in an OASIS results database. The default depends on the Calibre version.

This keyword choice specifies the global behavior for all OASIS databases generated during the run.

  • CBLOCK — Instructs Calibre to embed compressed data in an OASIS results database. CBLOCK is the default beginning with the Calibre 2016.2 release.

    The number and location of the CBLOCK records are at the discretion of Calibre. In general, using the CBLOCK keyword provides a larger degree of compression in the output database(s). It may also be faster to write OASIS with CBLOCK records, since less CPU-intensive geometric arraying is done if CBLOCK records are written.

  • NOCBLOCK — Instructs Calibre to not embed compressed data in an OASIS results database. NOCBLOCK is the default for releases prior to the Calibre 2016.2 release.

STRICT | NOSTRICT

An optional keyword choice that specifies whether Calibre uses strict-mode output for an OASIS database. The default depends on the Calibre version.

This keyword choice specifies the global behavior for all OASIS databases generated during the run.

  • STRICT — Instructs Calibre to create strict-mode output for all fields in the table-offsets structure of an OASIS database. STRICT is the default beginning with the Calibre 2016.2 release.

  • NOSTRICT — Instructs Calibre to not create strict-mode output for all fields in the table-offsets structure of an OASIS database. NOSTRICT is the default for releases prior to the Calibre 2016.2 release

USER | PSEUDO | USER MERGED | TOP | COMBINE

An optional keyword choice that controls the type of results output from hierarchical Calibre applications. These keywords apply only to Calibre nmDRC-H. See “Hierarchy Output” for additional discussion.

  • USER — Suppresses the output of pseudocells in a hierarchical results database. Geometry in pseudocells is transformed up the hierarchy to the first user cell but is not merged.

    USER is the default except in the following cases when GDSII or OASIS is specified:

    • Fracture, Litho, or DFM Fill operations are required by the flow.

    • Layout Turbo Flex, Layout Ultra Flex, or Layout Mondo Flex is specified.

  • PSEUDO —Includes pseudocells in a hierarchical results database.

    PSEUDO is the default when GDSII or OASIS is specified and either of the following conditions is met:

    • Fracture, Litho, or DFM Fill operations are required by the flow.

    • Layout Turbo Flex, Layout Ultra Flex, or Layout Mondo Flex is specified.

  • USER MERGED — Suppresses the output of pseudocells in a hierarchical results database. Geometry in pseudocells is transformed up the hierarchy to the first user cell and then merged.

  • TOP — Transforms geometry to the top-level cell (without merging) prior to results database output. In other respects, output using this keyword is similar to USER. Use of this keyword is limited and it should be used with caution.

  • COMBINE — Suppresses the output of pseudocells in a hierarchical results database. Cells created by cloning are treated as pseudocells. Placements are not written. Geometry in pseudocells is transformed up the hierarchy to the first user cell but is not merged.

Description

Specifies the pathname and type of the Calibre nmDRC results database. This statement must be specified once. Detailed information about Calibre nmDRC results database formats can be found under “DRC Results Database” in the Calibre Verification User’s Manual.

To specify the DRC Results Database in Calibre Interactive—nmDRC, see “Specifying Outputs for a DRC Run” in the Calibre Interactive User’s Manual or “Outputs Page in Calibre Interactive nmDRC” in the Calibre Interactive (New GUI) User’s Manual.

Layout text is not written to mask format DRC Results Database output by default. Text objects can be when DRC Map Text YES is used.

ASCII Output

ASCII database format is the default. This format can be read by Calibre RVE for debugging your layout. See “Using Calibre RVE for DRC” in the Calibre RVE User’s Manual for more information.

Note:

Calibre nmDRC-H only reports a rule check violation once per cell for an ASCII results database. The hierarchically lowest, leftmost result is reported. This suppresses multiple instances of the same result, which would normally occur in a flat run. Viewing or debugging of generated layers for a hierarchical run may be easier to do in GDSII or OASIS output because multiple results are not suppressed for these formats.

The ASCII DRC Results Database has, as its final entry, a rule check containing the names of all RDB files generated during the run, if any.

The format for this database is described under “ASCII DRC Results Database Format” in the Calibre Verification User’s Manual.

GDSII Output

The GDSII format is a standard GDSII stream representation of the Calibre nmDRC results database where:

  • The GDSII version number in the header record is 6.0.

  • The modification and last access times in the BGNLIB and BGNSTR records are the date/time of database creation.

  • The library name in the LIBNAME structure is drc.db by default. It can be changed using the DRC Results Database Libname specification statement.

  • The UNITS record is drawn from the rule file Precision and Unit Length statements, or their defaults.

Note:

When writing a GDSII-type results database, Calibre nmDRC aborts if a coordinate exceeds the 32-bit space capacity of GDSII.

OASIS Output

The OASIS format is a standard OASIS representation of the Calibre nmDRC results database. It offers advantages over GDSII, particularly in size. OASIS cell placements are stored in arrays automatically. OASIS LAYERNAME records are output to an OASIS results database for all Calibre nmDRC rule checks that map to that database. The record consists of the name of the rule check and the layer/datatype specified in the appropriate DRC Check Map statement (or the default of layer 0). For information about the Calibre implementation of OASIS, see Calibre for the Open Artwork System Interchange Standard.

The CBLOCK and STRICT keywords are recognized only by Calibre nmDRC applications. In addition, when explicitly specified, they are global attributes that apply to all OASIS-type results databases.

Specifying the STRICT keyword instructs Calibre nmDRC to create strict-mode output for all fields in the table-offsets structure. The table-offsets structure contains six pairs of unsigned integers. Each pair consists of a flag field and a corresponding byte-offset field, in the order shown in the following table.

The table-offsets structure is stored in the START record if the offset-flag (an unsigned-integer) is 0, and in the END record if the offset-flag is 1. The offset-flag is located in the START record. Calibre always stores the contents of each section referred to in the table-offsets structure after the START record. Each flag is 0 indicating non-strict-mode for the corresponding records and 1 indicating strict-mode.

If STRICT mode is not specified, then all flags and offsets listed in Table 1 are zero. If STRICT mode is specified, then CELLNAME, TEXTSTRING, PROPNAME, PROPSTRING, and LAYERNAME records are enabled, the corresponding flags are set to 1, and the byte-offsets are set to address the beginning of each table. The actual location of each table in the file is at the discretion of Calibre. In addition, as required by strict-mode, all references to the corresponding classes of objects are made exclusively by reference-number. XNAME records are not written by Calibre and remain in non-strict-mode.

Additionally, in strict-mode, each CELLNAME record in the CELLNAME table has an adjoining S_CELL_OFFSET standard property whose value is the byte-offset of the CELL record that references the associated CELLNAME record. If compressed output is specified using the CBLOCK keyword, the byte-offset value in the S_CELL_OFFSET property is the byte-offset of the CELL record before decompression.

If AUTOMAP is specified, then the output layers are indexed consecutively as follows. For each OASIS DRC results database D:

  1. Let M be the highest layer number specified in all rule check outputs for D. Layer numbers may be specified explicitly in a DRC Check Map statement or default to 0 when not explicitly specified in a DRC Check Map statement, or when there is no DRC Check Map statement specified for a rule check.

  2. Let S be the set of rule check outputs for D that have a default layer number (0).

  3. Assign consecutive layer numbers starting with M+1 to each rule check output in S. If M+1 > 65535, then assign the layer number 0. Assign a datatype of 0.

PIPE Output

The PIPE keyword enables you to send the results database output through a pipe rather than to a file. For example, gzip results output can be created as follows:

DRC RESULTS DATABASE "PIPE gzip > drc_results.gdsii.gz" GDS

If PIPE output is specified, then Calibre nmDRC forks a child process, has it execute “⁄bin⁄sh ‑c command”, and pipes the results database output to the standard input of the child process. The PIPE keyword applies only to GDSII and OASIS output. OASIS output may not have CBLOCK or STRICT records. Exceptions to these restrictions are treated as runtime errors.

Hierarchy Output

Cells in the layout hierarchy are output according to the Calibre hierarchical database structure (which is not necessarily identical to the original input database structure). The USER, PSEUDO, USER MERGED, COMBINE, and TOP keywords control the type of results output from hierarchical Calibre applications.

If you specify USER or USER MERGED, then the pseudocells created by Calibre nmDRC-H applications are not included in the results database. If you specify USER MERGED, then the data residing within pseudocells is transformed up to the next level of user hierarchy and merged (except for clustered edges). If you specify USER, or if it applies as the default, then the data residing within pseudocells is transformed up to the next level of user hierarchy but is not merged. The main advantage of the USER keyword over USER MERGED is that USER is typically faster.

For ASCII databases, geometry transformation to the top level proceeds as usual. For GDSII-type or OASIS-type results databases, neither cell records nor placement records for pseudocells are written.

If you specify PSEUDO, Calibre applications include the pseudocells that are internally generated and the geometry is not transformed up the hierarchy. The cells are named ICV_n, where n is a unique integer. For an ASCII results database, using the PSEUDO keyword can act, in many cases, as an error-suppression mechanism. For layout databases used for mask preparation, using the PSEUDO keyword can dramatically reduce output database size as well as improve performance. PSEUDO is recommended for these situations.

The PSEUDO and USER keywords exhibit much greater scalability in multithreaded (MT) mode and Calibre MTflex runs than does USER MERGED when you specify OASIS results.

If any of the hierarchy output keywords is explicitly specified in a DRC Results Database statement, then that setting is the default for any DRC Check Map statement that does not override the global setting. Otherwise the DRC Check Map local setting takes precedence.

Multiple Output Databases

You can output multiple Calibre nmDRC results databases during one run by using the DRC Check Map specification statement. Results output by DRC Check Map statements are not sent to the results database specified in a DRC Results Database statement if the database names are not identical in both statements. For example:

DRC RESULTS DATABASE "drc.db" GDSII
DRC CHECK MAP rule_x other.db

The results sent to the other.db are not sent to drc.db by default. If you want the results to go to both databases, include the following:

DRC CHECK MAP rule_x drc.db

Calibre nmDRC-H creates a geometric results database with exactly one cell record. The name of the cell is the value of the Layout Primary statement with the specified string appended or prefixed if the APPEND or PREFIX secondary keywords are included. If there is no Layout Primary statement (that is, the input layout system was not GDSII or OASIS), the cell name is drc.

If a DRC Results Database statement uses the APPEND or PREFIX keywords, or both, then these keywords act as global defaults for all GDSII or OASIS results databases generated during the run. A DRC Check Map statement may not have different APPEND or PREFIX keyword settings from the DRC Results Database statement if both statements write to the same file. A DRC Check Map statement may have different APPEND or PREFIX keywords from the global default if that statement writes to a different file than the DRC Results Database statement.

Calibre applications print a warning for each rule check that outputs to a GDSII- or OASIS-type results database with a default layer number of 0. This may occur because a layer number is not specified in the DRC Check Map specification for the rule check, or because no DRC Check Map statement is specified for the rule check.

ICrules Details

Flat ASCII DRC results databases can be converted by ICrules using the RESTORE DRC RESULTS command. You can use the DATABASE EXPORT command in ICrules to create an ASCII results database.

Related statements: DRC Results Database Precision, DRC Maximum Results, DRC Maximum Cell Name Length, DRC Maximum Vertex, DRC Select Check, DRC Unselect Check, DRC Summary Report, and Layout Property Audit.

Examples

Example 1

The following statement outputs an ASCII format of the results database to the specified location. You can use Calibre RVE to analyze the results or bring the (flat) results database into ICrules for viewing.

DRC RESULTS DATABASE "~/design1/drc_database" ASCII

Example 2

The following statements show how use of the APPEND keyword affects the resulting cell names. Calibre generates subcells, provided that the data exists in the subcells and the subcells do not need to be flattened. Assume that the original cell names in the database are: TOPCELL, SUBCELLA, and SUBCELLB.

The first statement does not use the APPEND keyword:

DRC RESULTS DATABASE drc.db GDSII

Calibre does not alter the original cell names in the database in this case. The second statement uses “_new” as the APPEND string:

DRC RESULTS DATABASE drc.db GDSII APPEND _new

Calibre nmDRC-H alters the original cell names, and saves them to the results database as the following: TOPCELL_new, SUBCELLA_new, and SUBCELLB_new. Flat Calibre nmDRC flattens the input database cells and produces the following file: TOPCELL_new.

Example 3

The following rule file excerpt shows how to output multiple results databases using DRC Results Database as the global database, and DRC Check Map to output an additional database:

LAYOUT PATH "in.gds"
LAYOUT PRIMARY "top"
LAYOUT SYSTEM GDSII

// GDSII output is the default
DRC RESULTS DATABASE "out.gds" GDSII APPEND _new

LAYER MEx1 1

copy_MEx1 { copy MEx1 }

DRC CHECK MAP copy_MEx1 10 APPEND _new
// results go to layer 10 in out.gds

DRC CHECK MAP copy_MEx1 ASCII drcdb.ascii
// results go to drcdb.ascii
