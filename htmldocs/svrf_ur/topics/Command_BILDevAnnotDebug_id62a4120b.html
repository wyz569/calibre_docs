DEBUG

Device annotation language function.

Enables debug monitoring during device annotation computation.

Usage

DEBUG range1[, range2 …]

Arguments

range1

A required device instance ID number or a pair of device instance numbers separated by a hyphen (‑). If the hyphenated form is used, the first number must be less than the second number. In that case, all numbers between and including the numbers specified in the range correspond to instances. At least one range must be specified, although it can contain only a single instance. Multiple ranges are separated by a comma (,) followed by whitespace.

Return Values

None.

Description

Debug monitoring of the device annotation computation can be useful in finding errors during the development of new property computation code. When present, DEBUG must precede the PROPERTY statement. The debugging trace is written to the transcript. The trace can produce much output, so the use of small ranges is recommended.

Errors in the computation program are reported in the run transcript, such as here:

************************************************
ERROR: Numeric error during property evaluation.
   Rule file: rules
   Device Layer: MN (line 103 of rule file)
   Cell: TOPCELL
   Argument for sqrt must be zero or positive.
************************************************

The instance function can be used in a Device statement property computation to write instance IDs as properties in the extracted netlist. The property values can then be used to choose which instance IDs to specify in the DEBUG command.

Instance IDs are not unique in a hierarchical run, but the device annotation flow is hierarchical. The debug trace will contain evaluations for all instances in all cells that have the specified instance ID number. For example, you could see two traces for instance 0:

EVALUATION: (promoted from cell: inv #2) 
   Instance: 0  x: 5.38  y: 14
...
EVALUATION: (promoted from cell: nand #0) 
   Instance: 0  x: 5.38  y: 14

For this reason, specifying the parent cell of the devices being debugged as the Layout Primary cell can be useful in reducing the DEBUG trace output.

Examples

Assume there are discrepancies in property annotation that you want to debug. Determine which cell is the parent of the devices you want to debug and make that the Layout Primary cell.

In the corresponding Device property computation, assume the I property is generated as follows:

device mn (n) ngate gate (G) diff (S) diff (D)
[
    property W, L, I
    I = inst()
...

After running circuit extraction without hcells, the layout netlist indicates this device is of interest:

M2 7 IN1 GND GND n L=1.25e-06 W=1.5e-05 i=11 $X=5375 $Y=14000 $D=0

This code shows how to obtain a debugging trace of instance 11:

annot_layer = DEVICE LAYER MN ANNOTATE ngate_per
  [ 
    DEBUG 11
    property SEFFA, SEFFB, W, L
  ...
  ]
