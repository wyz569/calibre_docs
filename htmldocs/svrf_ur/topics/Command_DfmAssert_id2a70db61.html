DFM Assert

Rule file compilation control statement

Specifies requirements for various rule file objects and generates a compiler error if the requirements are not met.

Usage

DFM ASSERT { 
{LAYERS [‘[’ layer_restriction …‘]’] layer_name [layer_name …] }
| {NUMBER [‘[’number_restriction …‘]’] expression [expression …] }
| {SCOPE {GLOBAL | LOCAL}}
}

Arguments

One of the LAYERS, NUMBER, or SCOPE keyword sets must be specified.

LAYERS [‘[’layer_restriction …‘]’] layer_name [layer_name …]

A keyword set that specifies one or more layer names preceded by any number of optional layer_restriction arguments.

  • layer_restriction — The optional layer_restriction arguments limit the input layers to be of the specified types and must be enclosed by brackets ([ ]). Choices for layer_restriction are as follows:

    • EDGE — Derived edge layer.

    • ERROR — Derived error layer.

    • GLOBAL — Global layer derived outside of a rule check.

    • LOCAL — Local layer derived inside a rule check.

    • NODAL — Connectivity layer.

    • ORIGINAL — Original layer.

    • POLYGON — Original or derived polygon layer.

    By default, all layer_restriction conditions are checked, and if any of the conditions is true for the layer, no error is given. This is equivalent to specifying that the layer must be defined. If more than one layer_restriction keyword is explicitly specified, then the layer must meet all specified conditions or an error is given.

  • layer_name — One or more layer names.

NUMBER [‘[’number_restriction …‘]’] expression [expression …]

A keyword set that specifies one or more numeric expressions along with any number of optional number_restriction arguments.

  • number_restriction — The optional number_restriction argument limits the value of the expression to be of the specified type. Possible choices for number_restriction are the following:

    • POSITIVE — Number is positive.

    • NONNEGATIVE — Number is greater than or equal to zero.

    • NONZERO — Number is not be equal to zero.

    • INTEGER — Number is an integer.

    • GRID — Number multiplied by the precision must be an integer.

    If more than one number_restriction type is specified, each expression value can be any of the specified types. If no number restrictions are specified, any real number is allowed.

  • expression — One or more required numeric expressions that must evaluate to a number. See “Numeric Expressions” for a list of operators and math functions that are supported in the numeric expression.

SCOPE {GLOBAL | LOCAL}

A keyword set that specifies the scope from which the statement is accessed.

  • GLOBAL — The scope of the calling environment is outside of a rule check.

  • LOCAL — The scope of the calling environment is inside of a rule check.

Description

Specifies requirements on layers, numeric expressions, or the scope of a DFM Assert statement in a rule file. If the specified requirements are not met, Calibre generates a compilation error.

The DFM Assert statement can be used with regular or encrypted SVRF (including CMACRO and DMACRO statements) and TVF.

DFM Assert supports the creation of intellectual property (IP) rules. One of the problems often encountered when creating IP rule file code involves error checking. If the inputs are incorrect, or the containing file renders the IP incorrect, the errors are reported by whatever SVRF statement detects them. If the IP is encrypted (without any assertion conditions), the errors are suppressed. If the IP is not encrypted, the errors can refer to internal objects such as layers derived from input layers.

To handle such error detection problems, a syntax error is generated by the rule file compiler for any DFM Assert conditions that are unmet. The errors are the same as the errors generated by any operation with the same requirements on the input. For example, the statement DFM Assert LAYER L generates error RES2 if the identifier “L” is not a defined layer. The errors generated by DFM Assert statements are not suppressed by encryption; hence, even if the DFM Assert statement is contained within encrypted SVRF or TVF, the complete error message is reported.

Examples

Example 1

The following statement specifies that ARG1 and ARG2 are defined layers:

DFM ASSERT LAYERS ARG1 ARG2

Consider that DFM Assert statement in the following SVRF context:

DMACRO CHECK ARG1 ARG2 {
  DFM ASSERT LAYERS ARG1 ARG2
  ...
}

LAYER M1 0
LAYER M2 1
m1m2 = M1 OR M2
m2_ext = EXT [M1] < 0.3
CMACRO CHECK m1m2 m2_ext
CMACRO CHECK m1m2 m3route

In the first CMACRO call, the arguments ARG1 and ARG2 resolve to the layer names m1m2 and m2_ext, respectively. Since there are no restrictions on the types of layers they must be, this call compiles. In the second CMACRO call, the argument ARG2 resolves to the name m3route. Assuming that there is no layer m3route defined in the rest of the rule file, the use of this macro triggers a RES2 error on layer name m3route.

Example 2

The following statement asserts that ARG1 and ARG2 are original or derived polygon layers, and have connectivity:

DFM ASSERT LAYERS [POLYGON NODAL] ARG1 ARG2

Consider the above DFM Assert statement in the following SVRF context:

DMACRO CHECK ARG1 ARG2 {
  DFM ASSERT LAYERS [POLYGON NODAL] ARG1 ARG2
  ...
}

LAYER M1 0
LAYER M2 1
CONNECT M1 M2
m1m2 = M1 OR M2
CMACRO CHECK m1m2 M2

The argument ARG1 resolves to the layer name m1m2. This is a polygon layer, but it does not have connectivity information. Consequently, the use of this macro triggers an EXT3 error on layer m1m2.

Now consider the preceding DFM Assert statement in the following SVRF context:

DMACRO CHECK ARG1 ARG2 {
  DFM ASSERT LAYERS [POLYGON NODAL] ARG1 ARG2
  ...
}

LAYER M1 0
LAYER M2 1
CONNECT M1 M2
m1_length = LENGTH M1 > 1
CMACRO CHECK m1_length M2

The argument ARG1 resolves to the layer name m1_length. This is a layer with connectivity, but it is an edge layer. Therefore, the use of this macro will trigger a TYP5 compilation error on layer m1_length.

Example 3

Consider this DFM Assert statement in the following SVRF context:

DMACRO CHECK ARG1 ARG2 {
  DFM ASSERT SCOPE GLOBAL
  ...
}

LAYER M1 0
LAYER M2 1

RES { 
  CMACRO CHECK M1 M2
}

The CHECK macro is called inside the rule check RES, which is not in the global scope. This results in a LOC3 error.

Example 4

The following DFM Assert statements specify that ARG1 is a polygon layer and ARG2 is a positive number:

DMACRO CHECK ARG1 ARG2 {
  DFM ASSERT LAYERS [POLYGON] ARG1
  DFM ASSERT NUMBER [POSITIVE] ARG2
  ...
}

LAYER M1 0
LAYER M2 1

VARIABLE X 2
CMACRO CHECK M1 X-5

The argument ARG1 resolves to the layer name M1, which is a polygon layer. The macro argument ARG2 resolves to the numeric expression “X-5”, which evaluates to -3 after variable substitution, which is non-positive. Therefore, the use of this CMACRO triggers a NUM3 error.

Example 5

Consider the following TVF procedure, which is contained in a file named my_proc.tvf:

proc my_proc {layers} {
	foreach {layer} $layers {
	  tvf::VERBATIM "DFM ASSERT LAYERS \[POLYGON\] ${layer}"
	  tvf::RULECHECK check_${layer} {
		tvf::SETLAYER ${layer}_int = INT ${layer} <= 0.2
		tvf::OUTLAYER "DFM RDB ${layer}_int out.rdb CHECKNAME ${layer}_int"
	  }
	}
}

The DFM Assert statement specifies that its input layer must be a polygon layer. Consider this procedure in the following TVF context:

#!tvf

tvf::VERBATIM {
  LAYER M1 0
  LAYER M2 1
  m1_edge = EXT [M1] < 0.3
}

source my_proc.tvf

my_proc {M1 M2}
# my_proc {m1_edge M2}

The first call to my_proc compiles because M1 and M2 are both polygon layers. However, the second call (if uncommented) to my_proc generates a TYP5 error because m1_edge is not a polygon layer.
