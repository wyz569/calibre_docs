LVS Filter

Specification statement

Filters out source and layout instances having properties with the specified values.

Usage

LVS FILTER {{component_type [‘(’component_subtype‘)’]
[property [‘(’spice_parameter‘)’] [filter_constraint]]} |
{COMPONENTS component_type [‘(’component_subtype‘)’]
    [component_type [‘(’component_subtype‘)’]] …
    [PROPERTY property [‘(’spice_parameter‘)’] [filter_constraint]]}
}
{{{SHORT [{pin_name1 pin_name2} [pin_name3 …]]} …} | OPEN}
[LAYOUT | SOURCE | LAYOUT SOURCE]
[DIRECT | MASK | DIRECT MASK]

Arguments

component_type

A required component type to which the filter applies. The tool filters out all instances of this component type that meet the other criteria specified in the statement. The component_type may be any SPICE device type, primitive subcircuit name, or any subcircuit name that is used in an LVS Box statement. The asterisk (*) wildcard is allowed and matches zero or more characters. When using this wildcard, enclose the component_type in double quotes.

When the COMPONENTS keyword is not used, this parameter must be specified exactly once. When the COMPONENTS keyword is used, this parameter must be specified at least once following the COMPONENTS keyword and preceding the PROPERTY keyword (if used).

‘(’component_subtype‘)’

An optional component subtype enclosed in parentheses and specified with a component_type. The filter disregards subtypes when you do not specify this parameter. The asterisk (*) wildcard is allowed and matches zero or more characters. When using this wildcard, enclose the component_subtype in double quotes.

property

An optional name of a device property to which the filter applies. When present, the filter applies only to instances that own the specified property. Otherwise, the filter applies to instances regardless of their properties.

‘(’spice_parameter‘)’

An optional string in parentheses that is an upper or lowercase letter specified with the property parameter. The letter instructs LVS to treat property values as strings in SPICE-like syntax (see the section “SPICE-Like Property Syntax” in the ICverify User’s and Reference Manual) and to parse the strings to obtain the specified SPICE value. Possible values are as follows:

W

MOS width

C

capacitance

L

MOS length

A

diode area

R

resistance

P

diode perimeter

When this parameter is present, the filter applies only to instances that own a property with the specified property parameter. The property value contains the specified spice_parameter.

filter_constraint

An optional constraint (see the “Constraint Notation” column of Table 1 under “Constraints”), specified with the property parameter. The constraint limits the value of the property. Either numeric or string constants, numeric variables, or, in Pyxis Layout, process variables can be used in the constraint. A numeric value can be positive or non-positive. String constants are case-insensitive. Interval constraint endpoints a and b (such as > a <= b), must be both numbers or both strings, and a must be less than b.

COMPONENTS

A keyword that enables more than one component_type parameter to be specified, along with any associated component_subtype parameters.

PROPERTY

A keyword used when COMPONENTS is specified that precedes a property parameter group. The PROPERTY keyword follows all component_type parameters in that usage.

SHORT [{pin_name1 pin_name2} [pin_name3 …]]

A keyword set that specifies pins whose associated nets are connected together after device filtering. Either this keyword set or OPEN must be specified.

If SHORT is specified without any pin name arguments, all remaining pin nets after filtering of a device are connected together. In this case, SHORT may be specified only once.

If a pin name list is used, then the pin nets that remain after filtering, whose pins match the specified names, are connected together. Each pin_nameN is a device pin name. Pin names must exist in the device definition in order for filtering to occur. Standard pin abbreviations for built-in devices are permitted (like p for pos, n for neg, g for gate, and so on). When specified, only the nets of listed pins are shorted together; other pin nets are left unconnected. If a pin is part of a swappable set of pins, then all of those swappable pins must be specified in the pin name list.

The SHORT keyword set with the pin_nameN arguments may be specified up to four times. Each such keyword set is called a short group, and the groups each conform to the behaviors mentioned in the preceding paragraph. Additionally, a pin name specified in a short group may only be specified once in that group and may not appear in any other short group in the same LVS Filter statement.

If the same device is specified in more than one LVS Filter SHORT statement with a pin name list, then all such statements must specify the same pins to be shorted in all short groups, respectively. The order of short group listings and the orders of pins in each list are unimportant, but the pins in each short group must match across all statements pertaining to the same device.

OPEN

A keyword that specifies all remaining pin nets after filtering are left unconnected. Either this keyword or SHORT must be specified.

LAYOUT

An optional keyword that specifies the statement applies to the layout design.

SOURCE

An optional keyword that specifies the statement applies to the source design.

Note:

The combination LAYOUT SOURCE is used by default if neither keyword is specified.

DIRECT

An optional keyword that places the operation in the Direct verification set. This keyword applies only to ICtrace.

MASK

An optional keyword that places the operation in the Mask verification set. This keyword applies only to Calibre.

Note:

The combination DIRECT MASK is used by default if neither keyword is specified.

Description

Filters out source and layout instances that conform to the criteria specified by the statement’s parameters. The filtering applies only during LVS comparison, not netlist extraction.

Either OPEN or SHORT must be specified. If you specify OPEN, then all nets that connect to filtered components are left open. If you specify SHORT with no pin list, then all nets that connect to filtered components are shorted together.

If SHORT is followed by a pin_name list, LVS shorts together all nets associated with the listed pins. Nets connected to the remaining pins of a filtered instance are left unconnected. Pin names that are also keywords for the LVS Filter statement must be enclosed in quotes, for example: SHORT IN1 IN2 “SOURCE”. A SHORT keyword with a pin_name list may be specified up to four times.

Preferences for names of nets shorted by device filtering can be set using LVS Prefer Nets.

To filter devices by property, the property must appear in a Trace Property or LVS Property Initialize statement.

The following consistency checks are implemented for pin names when the rule file is compiled:

  1. No pin name may appear more than once in each LVS Filter statement (if case-sensitivity is enabled, names that are different only by case are considered different names and are allowed).

  2. If two LVS Filter statements have the same component_type and component_subtype, apply to the same design (LAYOUT or SOURCE), and both specify SHORT, then both statements must have identical pin_name lists.

  3. If several device pins are swappable, either all must appear in any SHORT pin list, or none. If the rule file contains DEVice statements for the device referred to by an LVS Filter statement, this check is performed when the rule file is compiled.

When LVS matches LVS Filter statements to device or cell templates, and the LVS Filter statement has a non-empty list of SHORT pins, LVS applies the filter to a template only if all pins in the list exist on the template (standard abbreviations for built-in device pins, such as p for pos, g for gate, and so forth, are allowed). Pins in the SHORT list that do not exist for the specified device are ignored.

This statement may be specified any number of times.

Each Mask-mode LVS Filter specification statement can have a set of corresponding rule file Device operations; however, Device statements are not required in netlist-to-netlist comparison, for example. Be aware that Layout Case, Source Case, and LVS Compare Case statements can have an effect on device filtering.

In Calibre, if Layout System specifies a geometric layout, then a syntactical check is performed on all LVS Filter statements in the LAYOUT set that have a property parameter. When the syntactical check is performed, it enforces the following:

  • A spice_parameter cannot be used as part of the property being checked because properties for extracted devices do not have SPICE-like values as described previously in the Parameters section.

  • There must be at least one Device operation having a component type corresponding to the LVS Filter specification statement in which a property parameter appears.

  • The property parameter must be computed as follows:

    • It must be computed in all Device operations that correspond to the LVS Filter specification statement.

    • It must have the correct type (numeric or string) in all Device operations that correspond to the LVS Filter specification statement. No string-valued filter constraints are allowed in this situation because extracted devices do not compute string-valued properties.

    • The property must not appear as a vector type in any Device operation that corresponds to the LVS Filter statement. SUM() is an example of a vector function in a Device property calculation.

This check in Pyxis Layout is performed on all LVS Filter statements in the MASK LAYOUT set that have a property parameter.

See also LVS Spice Cull Primitive Subcircuits and LVS Spice Exclude Cell.

Examples

Example 1

The following statement filters out all mp-type layout instances that own a w property with a value greater than 3 but less than 5. The connecting nets are left open.

LVS FILTER mp w > 3 < 5 OPEN LAYOUT

Example 2

The following statement filters out all c-type source and layout instances. The connecting nets are left open.

LVS FILTER c OPEN

Example 3

The following examples filter resistors with a resistance value equal to zero out of a SPICE netlist:

  • For resistor r0 in the SPICE netlist having a zero value:

    .subckt zcell porta portb
r0 porta 4 0 $.MODEL=red
r1 1 portb 0.3333 $.MODEL=blue
r2 1 4 0.3333 $.MODEL=blue
.ends zcell

    To filter devices by property, the property must be traced: 

    TRACE PROPERTY R(red) r r 0
LVS FILTER R R == 0 SHORT SOURCE // a null value does not equal zero

  • For all resistors in the SPICE netlist having a model property of red:

    .subckt zcell porta portb
r0 porta 4 0 $.MODEL=red
r1 1 portb 0.3333 $.MODEL=blue
r2 1 4 0.3333 $.MODEL=blue
.ends zcell

    the corresponding rule file statement is:

    LVS FILTER R(red) SHORT SOURCE

Example 4

This example shows how to filter a cell from a source netlist and short the connections to the cell.

LVS BOX SOURCE "cellA" // make cellA a box cell in the source
LVS FILTER "cellA" SHORT SOURCE // filter cellA and short it in the source

If cellA is primitive, then no LVS Box statement is needed.

Example 5

The following statement filters out all r type and x subtype source instances that own an instpar property with a resistance value equal to zero. The resistance is specified in the instpar property using SPICE-like syntax in ICtrace(D). The connecting nets are shorted together.

LVS FILTER r(x) instpar(r) == 0 SHORT SOURCE

Example 6

Consider the following statements:

LVS FILTER "EMPTY_CELL*" SOURCE OPEN
LVS FILTER COMPONENTS "EMPTY_CELL*" SOURCE OPEN
LVS FILTER COMPONENTS EMPTY_CELL EMPTY_CELL2 SOURCE OPEN

the source netlist:

.SUBCKT EMPTY_CELL PINA
.ENDS
.SUBCKT EMPTY_CELL2 PINB
.ENDS

.SUBCKT TOP
M0 A B C VSS N
X1 VDD EMPTY_CELL
X2 VDD EMPTY_CELL2
.ENDS

and this layout netlist:

.SUBCKT TOP
M0 A B C VSS N
.ENDS

These netlists compare correctly with any of the three rule file statements.
