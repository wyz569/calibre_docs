LVS Annotate Devices

Specification statement

Specifies to annotate device shapes with properties stored on an annotation layer. Annotation properties are available using the Calibre Connectivity Interface. Used only in Calibre nmLVS‑H and Calibre PERC.

Usage

LVS ANNOTATE DEVICES device_layer

Arguments

device_layer

A required layer that must be derived from a Device Layer ANNOTATE operation.

Description

Specifies whether to annotate extracted device shapes with DFM properties taken from the device_layer. The device_layer must touch the appropriate device seed layer to have the properties annotated to the device. This statement is used during circuit extraction. It may be specified any number of times with unique device_layer parameters.

DFM Property names starting with the string ICV_ are reserved for internal use of the tool and are not copied to devices by this statement.

Use of this statement follows this flow:

  1. Generate measurements of interest for a particular device and apply them with the DFM Property operation. These measurements might include external measurements from a device gate polygon to nearby gate polygons. See “Example Device Annotation Programs Using DFM Property”.

  2. Generate a layer using the Device Layer ANNOTATE operation. This creates a layer containing recognized device seed shapes that are annotated with all NUMBER, STRING, and DFM VECTOR properties from the ANNOTATE layer, as well as device-specific information that can be used by the Query Server’s CCI interface to create netlists that have additional properties that were not present in the original LVS netlist.

  3. Pass the output from Device Layer as the device_layer parameter to LVS Annotate Devices. The layer is stored in the PHDB database of the Mask SVDB Directory (specify the CCI option).

  4. Run circuit extraction to have the properties annotated to the appropriate devices.

    The annotated properties are not in the extracted netlist used for LVS comparison.

  5. Run the Calibre Connectivity Interface annotated device commands to produce layout annotated SPICE netlists or annotated geometry files (AGF) that contain the properties. See “Annotated Device Commands” in the Calibre Query Server Manual.

Device annotation can avoid seed promotion issues when LVS Push Devices YES is used. In the LVS Push Devices SEPARATE PROPERTIES YES flow, using a Device annotation program block is the recommended implementation. See “Property Definitions and Netlist Behavior for CCI Flows” in the Calibre Query Server Manual for related information.

Examples

Consider the following Device statement:

// MP Device:
DEVICE MP PGATE PGATE(g) SD(s) SD(d) NW(b) [
  property w,l
  w=perimeter_co(PGATE,SD) * 0.5
  l=perimeter_outside(PGATE,SD) * 0.5
]

To annotate the area of overlapping polygons from layer L1 onto the device, the following sequence of layer operations is required:

// Create a layer that has L1 area annotated onto pgate polygons:    
pgate_with_area = DFM PROPERTY pgate L1 OVERLAP ABUT ALSO MULTI
                  [ A_L1 = AREA( L1 ) ]

// Generate the annotated device layer from MP devices that interact with
// pgate_with_area
dev_layer_mp = DEVICE LAYER MP ANNOTATE pgate_with_area

// Pass device property layer on to LVS ANNOTATE DEVICES.
LVS ANNOTATE DEVICES dev_layer_mp

MASK SVDB DIRECTORY svdb QUERY CCI

MP devices that interact with dev_layer_mp will have the property A_L1 associated with them in the PHDB.

Annotation properties can be accessed using Calibre Connectivity Interface annotated device commands. The properties appear in the AGF and extracted netlist output from CCI. Here is an example script:

qs::set_agf_options -annotated_devices YES
qs::write_agf annotated_MP.agds
dfm::set_layout_netlist_options -annotated_devices YES
dfm::write_spice_netlist annotated_MP.sp
qs::quit

In a pushdown separated properties (PDSP) flow, this statement would be used in the rule file:

LVS PUSH DEVICES SEPARATE PROPERTIES YES

You would omit the dfm::set_layout_netlist_options command from the previous CCI script and use this instead:

qs::write_separated_properties pdsp_MP.data
