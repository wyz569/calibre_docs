DFM Properties

DFM properties are a form of metadata associated with objects on derived layers. They are often numerical values used for calculating scores and filtering data. Some operations support net number, string, and vector properties. Each property has a name and a value.

DFM properties are different from “layout properties” and “database properties,” which are attributes stored on geometries in the original layout database. DFM properties are associated with objects on derived layers; DFM properties can be created with the DFM Property family of operations or with Calibre YieldServer. The DFM Property operation creates a derived layer containing some or all the geometries on the primary layer annotated with the specified properties. DFM Properties are calculated with DFM Expressions.

A derived layer that contains objects with DFM properties can be used as an input layer to any Calibre operation that supports the layer type. However, DFM properties are ignored unless the operation has special handling for properties on input layers. This means that for most operations, when a layer is derived from a layer that has properties, these properties are not carried over to the derived layer.

Standard layout database formats such as GDS and OASIS do not support DFM properties. To write DFM properties to a file, you must save the layout as either a DFM database or DFM RDB (using the DFM RDB operation).

The DFM RDB operation writes out all DFM properties for its input layers as part of the check results, with one property (name-value pair) per line. Also see “DFM RDB Reference” in the Calibre YieldAnalyzer and YieldEnhancer User’s and Reference Manual.

Property names are case-sensitive. A property name can be “-” or an arbitrary string not beginning with the “-” character. White space is not recommended, as Calibre RVE interprets the space character as a delimiter. The property name “-” and property names which begin with the “+” character have special handling; see “Non-Persistent DFM Properties”.

DFM properties can be the following types:

  • Numeric — Contain floating-point numbers.

  • Net number — Contain the net ID for a geometry.

  • String — Contain string values which may be either user-defined or come from a GDS property, an OASIS property, or a text object.

  • Vector — A vector is a two-dimensional array (or matrix) of numbers composed of tuples. Vectors can contain numeric values, strings, or net numbers.
