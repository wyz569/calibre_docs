Calibre nmSRAF Recipe Development

When developing recipes, Siemens EDA recommends you start from isolated layout features and transition to layouts with greater feature density.

Priority Usage

Rules should be prioritized to apply closer SRAFs first and secondary SRAFs later. The secondary SRAFs should be placed at a minimum distance from SRAFs with the highest priority.

Global and Priority Precedence

When using priority, global keywords and their respective options must be specified first, otherwise nmSRAF issues warning messages. The specification of any keyword after the priority keyword forces the specification to be priority-based and not global. For any parameter to be considered global, it must be specified before the priority keyword.

Use Priority for Improved Processing and Treatment

Although formulating recipes is often easier when specifying keywords globally, a number of important considerations may impact processing and treatment efficacy:

Runtime

In general, Siemens EDA recommends using lineendmerge 0 or no, and intrapriorityalign and interpriorityalign, and conditional MRCs such as lineend2long, long2lineend, and square2long per priority (restricted to required applications only), not globally, to avoid runtime impact.

Consistency

The consistency of generated SRAFs may be negatively impacted using priority-related keywords such as interpriorityalign, intrapriorityalign, maxlength, and others.

Predictability

Results of intrapriorityalign, and particularly, interpriorityalign may be unpredictable at tile boundaries.

Hybrid Recipes

Hybrid recipes are defined as combinations of Calibre cnSRAF- and Calibre mbSRAF-generated SRAFs applied to the same design (whether executed together or separately). Calibre mbSRAF provides an SRAF generation safety net for ensuring SRAF coverage that may be potentially missed. If the initial SRAF generation is rule-based, the secondary model-based SRAF generation provides complete SRAF coverage.

This requires that Calibre mbSRAF process the entire design, achieving complete coverage, with an expected increase in associated simulation runtime. Alternatively, if the rule-based SRAF generation can be defined exclusively for a large portion of the design (for example, SRAM) or a complete rules-based recipe can be generated, SVRF data-processing rules can be used to isolate regions which do not require SRAF coverage within a specific distance of the region. These isolated shapes and edges can be passed as targets to Calibre mbSRAF, with the rules-based SRAFs as an sblayer. Because of the limited target areas requiring simulation, this method provides complete SRAF coverage with potentially no additional mbsraf runtime.

minarea, minlength, and minwidth Keyword Interaction

The minwidth, minlength, and minarea keywords define the minimum size of SRAFs. Width and length always refer to the length of the shorter and longer edges of rectangular SRAFs, respectively, regardless of SRAF orientation; width is the distance between the two long edges and length is the distance between the two short edges in a rectangular shape.

If any two of the keyword values are specified but not the third, the value of the third is not derived, and must be explicitly defined. Siemens EDA recommends that all these keyword values be specified. In addition, all values and priorities of these keywords are printed to the session transcript, providing useful information for debugging purposes.

If the specified values of any keyword are incompatible with each other, Calibre nmSRAF issues a warning message and no values are adjusted. Additionally, a minarea value is not assumed from the product of minwidth and minlength if no minarea specification is made. Incompatible keyword values may include:

minlength < minwidth or minlength * minwidth > minarea

A warning is issued if width or length are specified for a template and are incompatible with the specified keywords. For example, where:

width < minwidth or length < minlength or width * length < minarea

No corrective measures are taken in these cases, but the template does not generate any valid shapes.

Additionally, the dynamicsizing, sbwidth, and sraf keywords define minimum constraints for dynamic sizing and conflict resolution operations. They do not define constraints implemented by minarea, minlength and minwidth and they do not override those keyword values. However, during parsing they are checked against the specified min... keywords. If they are found incompatible, a warning is given and the values of dynamicsizing, sbwidth, and sraf are increased to comply with min... keyword constraints.

If min* keyword constraints for square SRAFs are specified, the sraf keyword that defines the size of square SRAFs inserted into certain conflict regions is checked against the square SRAF-related keyword values only. The sraf keyword is not checked against minarea, minlength and minwidth keyword values.

If minarea, minlength, and minwidth are not specified, the dynamicsizing, sbwidth, and sraf keywords and the specified width and length values of the templates are checked to define the appropriate limits for the SRAF sizes. For each priority, the minimum of any of the specified keywords dynamicsizing, sbwidth, and sraf and the nominal SRAF width and length is used as the minwidth constraint. Calibre nmSRAF then issues an informational message stating the actual constraints it uses.

SRAFs do not change orthogonal orientation if the square SRAF-related keywords are specified. If separate square keywords are not specified, minwidth and minarea are used as constraints, and square SRAFs may become rectangular in shape and are placed at either orientation during cleanup operations.

Use the sraf_options Statement

Rules should be formatted within an sraf_options statement. This insures developers free-format whitespace for comments and easier template and keyword manipulation than using the setlayer command alone.

Layer Handling

Most Calibre nmSRAF templates should require only one setlayer statement. Additional setlayer statements do not cause additional runtime, unlike additional templates within a previous setlayer. However, additional setlayers only process the output of previously-generated SRAFs.

In general, no input layers may be used with the same name more than once. If the same layer is required, it must be derived and output under a different name prior to its subsequent use.

You must specify the targetlayer first in the argument list to RET MBSRAF. You must specify only one targetlayer. If the targetlayer is required in other areas of the recipe, before you process it you must derive it again under a different name.

Using the offsetlayer can help manage SRAF placement. Never use the target layer as the offsetlayer. Additionally, if sblayer or offsetlayer is specified, they must appear in either the sraf_options statement or the setlayer command, but never both.

Siemens EDA recommends using the correctionlayer for best SRAF-generating performance. The correctionlayer is a polygon-based layer acting as a corrected target layer and is used to calculate the gradient. Only one correctionlayer may be specified.

Debug Layers

Siemens EDA recommends you enable debug layers so you can see what happens in each step of the SRAF generation process. If SRAFs are not placed where you expect them, it may be because the minoffset or minfeaturespace distance checks are being violated. You can examine the field debug layer for any affected templates.

Note:

Siemens EDA recommends never using the output of debug layers for SVRF-related post-processing. The only purpose for using the cnsraf debug capability is for debugging SRAF-generated intermediate layers within small regions. A high possibility exists that generated debug layers may be missing or not MRC clean near tile boundaries. Therefore, Siemens EDA does not guarantee clean generation of debug-specific SRAF layers for large designs or hierarchical runs.

Multiple Mapping of Debug Layers

Multiple mapping and output of debug layers within Calibre nmSRAF is not permitted as of Calibre release 2019.2. A compile-time error message is generated if multiple mapping is attempted. The following code fails and generates the error message shown:

setlayer sraf = cnsraf target OPTIONS all
setlayer p10_iso = cnsraf debug sraf isolate 10
setlayer p10_iso_2 = cnsraf debug sraf isolate 10
p10_iso = RET SBAR target FILE sraf_gen MAP p10_iso
p10_iso_2 = RET SBAR target FILE sraf_gen MAP p10_iso_2
Trying to use the debug layer "isolate" for priority 10 in more than one setlayer command

Tags, Templates and Conflict Resolution

Upper bounds for the newtag internal keyword are critical:

  • When using the internal keyword to generate tags used in the context of rectedge center strictcenter template, you must provide an upper bound to the internal constraint to enable limits during edge collection.

  • When applying the internal keyword to any pitch-based template, you must provide an upper bound to the constraint to enable limits during edge collection.

  • As a default, the promotion distance is used for the upper bound.

Templates using the conflictonly keyword are generally placed at a higher priority than other templates.

When defining multiple tags in the setup file, Siemens EDA recommends that no gaps are left between the individual tagging bins. This ensures that all the target edges in the design are accounted for.

SRAF interaction with optics are localized effects and effective control requires localized movement. If SRAF fragmentation is not allowed during co-optimization of OPC, conflict resolution shapes should be restricted in size.

Match MRC and OPC-MRC Specifications

The values assigned to options minoffset, minfeaturespace, minwidth, sbwidth, and sraf within the Calibre nmSRAF LITHO FILE should be set in accordance with the final MRC values defined for SRAFs in the Calibre nmOPC setup file. If the SRAFs input to nmOPC are not MRC clean, nmOPC cannot guarantee MRC clean SRAF output after the sraf_print_avoidance operation.

Disable SMT

Simultaneous multi-threading (SMT) treats each physical core as two logical CPU cores (one physical and one virtual). The processor causes the operating system to behave as if there are twice as many CPU cores. This halves the available memory per core.

To turn off SMT, set the environment variable CALIBRE_MT_SMT_FACTOR to 0.

For detailed information regarding disabling SMT, see the Calibre Local Printability Enhancement User’s and Reference Manual.

AVX and SOCS Recommendations

Use SOCS shared memory for better memory management during SRAF generation. As of Calibre release 2016.4, processes running on the same physical host can share the directory that contains SOCS calculations. To enable SOCS memory management, you set LITHO_SOCS_KERNELS_SHARED to 1 in your environment.

Use AVX for performance benefits, enabled at the command line.

CAUTION:

Both recipe development and production execution must be performed using identical AVX and non-AVX environments to eliminate potential SRAF XOR differences resulting from varied gradient values between these respective environments.

For usage information regarding both AVX and SOCS, see the Calibre Post-Tapeout Flow User’s Manual.
