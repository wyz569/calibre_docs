intrapriorityalign

Cleanup Keywords

Aligns SRAFs of same priorities during cleanup. Used by RET SBAR and RET MBSRAF.

Usage

intrapriorityalign value [centermove value] [spacerule 1 | ‑1]

Arguments

value

A required positive floating point number. The value is the maximum distance lower priority SRAFs can be moved to align with higher priority SRAFs.

centermove value

An optional positive floating point number. This is the maximum amount of movement allowed for center SRAFs. The default value is the intrapriorityalign value.

spacerule 1 | ‑1

An optional argument, either 1 or ‑1. A value of ‑1 moves SRAFs away from target shapes during alignment. A value of 1 moves SRAFs toward target shapes during alignment. The default value is ‑1.

Description

An optional keyword that enables valid, generated SRAFs of same priorities to align and merge during SRAF cleanup. This option can be set globally or per priority. If set globally and not specified for any priority, the global setting is adopted. Before alignment, SRAFs are trimmed with respect to target shapes because the projecting distance and shielding are not well defined when SRAFs overlap targets. You may use maxlength in conjunction with intrapriorityalign.

Note:

The intrapriorityalign keyword is incompatible with cleanupversion 0 and results in a compile-time error.

The fundamental alignment location rules include:

  • The initial position of SRAFs are given greater emphasis during alignment processing.

  • Symmetric alignment solutions are given greater emphasis during alignment processing.

  • The widths of SRAFs may be adjusted by intrapriorityalign during alignment and line‑end merging.

For alignment of SRAFs with the different priority, these conditions must be satisfied:

  • The orientation must be the same.

  • Diagonal or square SRAFs are not supported.

  • A minimum distance is established from SRAFs to target shapes through specification of minfeaturespace.

  • SRAFs cannot align if they are shielded by a target shape. SRAFs are not shielded where their long-end projections are overlapping. The shielded area is determined by the bounding box of the two SRAFs shown; two edges of the shielded area touch the lineend of SRAFs and the other two edges touch the boundary of the shielded area. In the example, the red box is the shielded area of the two green SRAFs. The SRAFs are shielded by the blue target shape where it appears within the shielded area, and therefore no alignment is performed.

  • In the example, widths of SRAFs are 0.02. The first three SRAFs ascending from the left can be aligned and the three SRAFs descending from the right can be aligned together, but all four of them cannot be aligned into a single SRAF. No alignment scenarios exceeding the SRAF widths would be handled by intrapriorityalign, but other cleanup steps may be available to resolve this situation.

  • The lineend distance between two SRAFs must be less than or equal to the specified lineendmerge value. The lineendmerge keyword is required for intrapriorityalign to work. A parsing error is issued if lineendmerge is not specified. There is no default value for lineendmerge.

  • Siemens EDA recommends values of intrapriorityalign to be less than maximum width of its associated priority. If the value exceeds this amount, a warning message is issued.

The intrapriorityalign keyword demonstrates these behavioral characteristics:

  • The value specified for intrapriorityalign acts as the search distance for the alignment of eligible SRAF.

  • Only two SRAFs of the same width can be merged.

  • The maximum movement for a center-placed SRAF is equivalent to the value specified for intrapriorityalign.

  • During alignment, SRAFs are moved away from target shapes.

  • The widths of SRAFs may be adjusted by intrapriorityalign during alignment and line‑end merging.

Note:

Siemens EDA recommends always specifying intrapriorityalign by priority. A warning message is issued to the transcript when intrapriorityalign is specified globally.

Examples

Example 1 — intrapriorityalign Code

In Figure 1, all SRAFs generated by the rectedge template specifying an offset of 0.050 would be moved to align with the SRAFs generated by the rectedge template specifying an offset of 0.080. The dashed line represents the feasible range of merged SRAFs. If no SRAFs are found within the range, SRAFs are merged in the middle of the range. The pink area is the bounding box of the SRAFs found in the range. The merged SRAF (shown in yellow) is generated between the SRAFs forming the bounding box.

rectEdge tag tag1 extension 0.020 width 0.030 offset 0.050 prior 1
rectEdge tag tag1 extension 0.020 width 0.040 offset 0.080 prior 1
priority 1 lineendmerge 0.020 intrapriorityalign 0.030
Figure 1. intrapriorityalign

Example 2 — intrapriorityalign Using spacerule

In Figure 2, field SRAFs for priority 1 and priority 2 are identical. In the illustration, intrapriorityalign with a priority -1 moves SRAFs away from the target shape (the default behavior), and intrapriorityalign with a priority 2, using spacerule, moves SRAFs toward the target shape.

rectEdge tag longEdges extension 0.01 width 0.004 offset 0.040 prior 1
rectEdge tag longEdges extension 0.01 width 0.004 offset 0.040 prior 1
priority 1 lineendmerge 0.07 intrapriorityalign 0.030
priority 2 lineendmerge 0.07 intrapriorityalign 0.030 spacerule 1
