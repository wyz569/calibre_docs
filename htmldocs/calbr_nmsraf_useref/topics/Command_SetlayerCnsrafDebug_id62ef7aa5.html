setlayer cnsraf debug

LITHO FILE Statement and Commands

Generates and saves in-process debugging layers. Used by RET SBAR only.

Usage

setlayer output_layer = cnsraf debug layer debug_keyword priority [template index]

Arguments

setlayer output_layer

A required command and layer name to which SRAFs are output.

layer

A required name of the SRAF layer to which all debugging layers apply.

debug_keyword

You must specify one of the following keywords. They are listed in the order they are processed by cnsraf debug:

  1. tags — Outputs edges of target shapes used to generate SRAFs. These edges are selected using a tagging keyword. These edges highlight those target layer edges contributing to a template during recipe development.

  2. candidates — Outputs SRAFs formed from selected seedpixels prior to selection and placement by priority. Applicable to mbSRAF only.

  3. initialfield — Outputs SRAFs generated by pairtemplate prior to further processing.

  4. field — Outputs all SRAFs by priority prior to cleaning.

  5. lineend — Outputs the line-end keepout areas derived by the minfeaturespace lineend keyword. If the offsetlayer keyword is specified, shapes derived for offsetlayer are merged and output with the lineend layer.

  6. preclean — Outputs SRAFs after the first step of cleaning only for cleanupversion 0 (specifically, removing SRAF violations against target shapes), unless cleanmode postsizing is specified. For cleanupversion 1, preclean outputs a copy of field.

  7. align — Outputs all SRAFs generated by interpriorityalign and intrapriorityalign keywords, after lineendmerge (as applicable), but prior to any cleaning.

  8. process — Outputs SRAFs which are processed by dynamicsizing prior to conflict resolution. This debug mode is accumulative and includes the output of previous priorities.

  9. hardconflict — Outputs the conflict regions prior to filtering. These regions include non-rectangular shapes and may be larger than the maxconflict constraint. Non-rectangular shapes may be converted to rectangular regions by conflictmode bbox for potential processing. Conflict regions are either merged or not merged depending on conflictmode settings merge or nomerge, respectively.

  10. conflictbbox — Outputs rectangular bounding boxes of conflict regions as determined by conflictmode bbox, and is used only for the conflictmode bbox keyword. All bounding boxes are processed separately and are not merged, even though they may overlap.

  11. conflict — Outputs regions less than or equal to maxconflict in both x- and y-directions that are filtered by cnsraf.

  12. conflictbar — Outputs conflict resolution bars (formed as rectangles or squares) prior to resolving conflicts between them and or any other shapes. Conflict bar shapes may overlap each other or overlap target or SRAF shapes. Further cleanup or removal is performed to resolve these conflicts.

  13. resolution — Outputs the SRAFs generated by processing conflict regions.

  14. isolate — Outputs the final SRAF result by priority.

These additional debug output keywords are listed by function:

Dynamicsizing Related

  • dynamic — Outputs SRAFs after symmetric dynamic sizing.

  • unsizabledynamic — Outputs SRAFs that are unsizable after symmetric dynamic sizing.

Barsplitting Related

  • barsplit — Outputs all SRAFs after barsplitting.

  • unsizablebarsplit — Outputs SRAFs that are unsizable after barsplitting.

  • barsplit1 — Outputs sizable SRAFs after successful barsplitting.

  • unsizablebarsplit1 — Outputs SRAFs that fail barsplitting.

  • ubarsplitpostcr — Outputs unsizable SRAFs barsplit after conflict resolution.

Symmetry Related

  • asymmetric — Outputs SRAFs after asymmetric dynamicsizing.

  • unsizableasymmetric — Outputs SRAFs that are unsizable after asymmetric dynamicsizing.

Line-end Related

  • lemu — Outputs SRAFs after initial lineendmerge of overlapping SRAFs.

  • lem — Outputs SRAFs after lineendmerge.

Centermerge Related

  • centermerge0 — Outputs SRAFs after initial centermerge with dynamicsizing or trimming.

  • centermerge — Outputs SRAFs after centermerge with dynamicsizing or trimming.

Intersection Related

  • intersection — Outputs SRAFs after adjustment of existing intersections and construction of new intersections.

  • unsizable_intersection — Outputs unsizable SRAFs after intersection construction is complete.

  • post_intersection — Outputs SRAFs after construction of intersections and subsequent cleanup operations.

  • unsizable_post_intersection — Outputs unsizable SRAFs after construction of all intersections and subsequent cleanup operations.

Final-Stage Related

  • oprotected — Outputs keep-out regions around final SRAF shapes.

  • positive — Outputs positive SRAFs (in Boolean terms, SRAFs NOT Target).

  • negative — Outputs negative SRAFs (in Boolean terms, SRAFs AND Target).

  • maxlength — Outputs SRAFs after maxlength splitting is performed, either during per priority cleanup or after all priorities have been processed.

  • sraf_pn — Outputs a debug layer containing SRAFs only of the specified priority. For a priority 1 template, the sraf_pn debug keyword is specified as sraf_p1; for a template of priority 2, sraf_p2, and so on.

priority

A required integer specifying the priority declared by any template. All SRAFs generated by the template are output by the specified priority. Exceptions include unsizable_barsplit, positive, negative, sraf_pn, and others.

template index

An optional keyword and non-negative integer argument pair that specifies a template to be output for any given priority. This argument pair is valid only for the field debug keyword. This argument permits the debug output of SRAFs limited to a single priority and template combination.

Description

The debug command receives output data from the setlayer cnsraf command and, during cnsraf processing, generates layers for analysis that are viewable in Calibre WORKbench. This intermediate data helps you develop rules and debug cnsraf processing.

You must specify setlayer cnsraf with a target layer. Generated SRAF layers must be output to the layout database for any associated debug layers to be saved and subsequently viewed. No error message is reported when generated SRAF layers are not saved.

Note:

Debug layers may be clipped at tile boundaries, giving unexpected results. Siemens EDA recommends using setlayer tilegen for the generation of tile boundaries to observe debug layer interaction.

Examples

Example 1

This example contains a partial list of debug keys and their respective layers.

LITHO FILE cnsraf_su [ 
	layer layer1 
	... 
	sraf_options OPTS {... prior 1 ... prior 2 ...} 
	setlayer cnsb1 = cnsraf target OPTIONS OPTS 
	 
	setlayer field_layer        = cnsraf debug cnsb1 field        1 
	setlayer hardconflict_layer = cnsraf debug cnsb1 hardconflict 1 
	setlayer isolate_layer      = cnsraf debug cnsb1 isolate      1 
	setlayer tags_layer         = cnsraf debug cnsb1 tags         1 
	setlayer lineend_layer      = cnsraf debug cnsb1 lineend      1 
] 
field_layer        = RET SBAR target FILE cnsraf_su MAP field_layer 
hardconflict_layer = RET SBAR target FILE cnsraf_su MAP hardconflict_layer 
isolate_layer      = RET SBAR target FILE cnsraf_su MAP isolate_layer 
tags_layer         = RET SBAR target FILE cnsraf_su MAP tags_layer 
lineend_layer      = RET SBAR target FILE cnsraf_su MAP lineend_layer

Example 3

This example demonstrates usage of the field debug keyword (in red) with priority 60 (in green). The code snippet shown generates the example output in Figure 2.

LITHO FILE cnsraf_su [ 
	layer target 
	sraf_options TEMPLATE {
		rectedge tag sram width 0.03 offset 0.04 prior 60
		rectedge tag sram_2 width 0.05 offset 0.08 prior 60
	}
	setlayer cn_sraf = cnsraf target OPTIONS TEMPLATE
	 
	setlayer 60_field_t1 = cnsraf debug cn_sraf field 60 template 1 
	setlayer 60_field_t2 = cnsraf debug cn_sraf field 60 template 2 
 
] 
60_field_t1 = RET SBAR target FILE cnsraf_su MAP 60_field_t1
60_field_t2 = RET SBAR target FILE cnsraf_su MAP 60_field_t2

Example 4 — Saving Tags for Debugging

These code examples output tags for cnsraf and mbsraf debugging purposes. The tag name is shown in red. The debug keyword required to output tags is green. The rulecheck and database layer names must be identical. They are shown in magenta.

Note:

Siemens EDA recommends never using the output of debug layers for SVRF-related post-processing. The only purpose for using the cnsraf debug capability is for debugging SRAF-generated intermediate layers within small regions. A high possibility exists that generated debug layers may be missing or not MRC clean near tile boundaries. Therefore, Siemens EDA does not guarantee clean generation of debug-specific SRAF layers for large designs or hierarchical runs.

LITHO FILE cnsraf_setup [
	...
	sraf_options cnsraf_opts {
		cleanupversion 1
		newtag tag_name external < 0.20
		...
	}
	setlayer cn_sraf = cnsraf target OPTIONS cnsraf_opts ...
	setlayer tag_name = cnsraf debug cn_sraf tags 1
]
tag_name = RET SBAR target MAP tag_name FILE cnsraf_setup
tag_out {copy tag_name} drc check map tag_out 10
LITHO FILE mbsraf_setup [
	...
	sraf_options mbsraf_opts {
		algorithm 1
		newtag tag_name external < 0.20
		...
	}
	setlayer mb_sraf = mbsraf target OPTIONS mbsraf_opts ...
	setlayer tag_name = mbsraf debug mb_sraf tags 1
]
tag_name = RET MBSRAF target MAP tag_name FILE mbsraf_setup
tag_out {copy tag_name} drc check map tag_out 10
