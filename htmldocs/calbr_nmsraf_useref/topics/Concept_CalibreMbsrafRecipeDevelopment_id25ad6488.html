Calibre mbSRAF Recipe Development

Application of best practices aid in recipe development.

Role of Input Layers During Gradient Calculation

These input layers affect gradient map generation for mbsraf:

  • correctionlayer

  • offsetlayer

  • sblayer

  • targetlayer

Although taglayer and prioritylayer are input layers, their role is limited to SRAF candidate selection, after seed pixel generation is complete, using the computed gradient. Therefore, these input layers are not shown in this list. Various input layers specified to mbSRAF can be classified as one of three types:

Gradient

This layer is used for determining pixel evaluation points for computing the gradient. Currently, only the targetlayer is used for deciding pixel evaluation points for computing the gradient.

Light

This layer is used to inform where the light passes through. The gradient computer uses this layer to compute the gradient at each pixel evaluation point. It is not used during seed pixel generation, candidate generation, or MRC checks. If the correctionlayer and sblayer are defined, they are used to determine where light passes through the mask. If not, the targetlayer is used.

MRC

This layer is used to perform enforcement of MRC checks and define keepout areas. The targetlayer, sblayer, and offsetlayer are all used during the MRC cleanup process. The correctionlayer is not used during MRC checking. Its role is limited only to the gradient calculation.

The target layer is a required layer because the shapes provided on the target layer define the evaluation points for the gradient calculation. If a portion of the original target layer is being processed in mbSRAF (typically using mbSRAF as SRAF fill after cnsraf rules-based insertion), the selected region of the target layer must be large enough to provide an empiric topological environment for any subset of target shapes in order for the gradient to be representationally accurate. If a shape is artificially isolated, the gradient can be erroneous due to inadequate sampling of the empiric target environment.

The sblayer and correctionlayer are layers containing shapes that are simulated to provide the image from which the gradient is calculated. If neither of these layers are available, the target layer is used for the image simulation.

When using mbSRAF as fill or an SRAF safety net after rules-based insertion, all layers processed must contain a large enough region centered on the selected shape needing further lithographic validation through an optical diameter, but the generated SRAFs produced by mbSRAF must only be constrained to a smaller halo around the center target (using a value equal to or smaller than maxoffset).

Using algorithm 1

Added in Calibre release 2016.3, algorithm 1 significantly enhances geometric consistency during SRAF generation. Siemens EDA recommends always enabling algorithm 1 within all Calibre mbSRAF recipes. Enabling algorithm 1 requires these keyword settings:

  • The minfeaturespace keyword must be explicitly set.

  • The minoffset keyword must be explicitly set.

  • Because algorithm 1 provides optimal SRAF generation through internal settings, Siemens EDA recommends that none of these keywords be modified:

    • cornerpref

    • cornerprefnegative

    • mode

    • peaktype

Calibre mbSRAF Optimization Steps

The foundation of Calibre mbSRAF is the gradient calculation for a process condition that defines SRAF placement locations. In order for the gradient to be as robust as the process can support, it is important that the print image at the desired process conditions roughly matches the target shape dimensions. One empirically consistent rule used for many customer designs and processes is the area of image should be 80% to 120% of the target area for any given process condition. A correction layer derived from a SIZE operation or from a few iterations of OPC may be required to achieve images of this range.

The next value to specify for proper gradient-to-SRAF conversion is gradientshift. The optimal amount can best be measured on an isolated target. An application note is available on Support Center, entitled Application note MG585866 which details methods for choosing this setting. The gradientshift keyword may be required in resist processes using negative tone development (NTD) where dark areas in a bright mask are the desired target locations.

When the gradient is largely shifted to a positive range, the SRAFs generated from seed pixels will be larger, and therefore, more effective in image correction. The number of SRAFs is closely related to the prevalence of the seed pixels, and the number of seed pixels can be increased by reducing the number of directions a local maxima must satisfy. Siemens EDA recommends that you measure the effectiveness of SRAF with the default 3dir keyword before modifying it. Use the verification script to help determine if more or less SRAFs are needed. Both the seedpixels and candidates debug layers are useful for analysis of SRAFs that are placed.

Once the spatial frequency and density of SRAFs seems reasonably effective, optimize the values of width and squarewidth to maximize positive image benefit while maintaining a low level of SRAF printing. The largest process window advantage comes from leaving some limited SRAF printing at the insertion stage and relying on sraf_print_avoidance in OPC to resolve this printing. A more conservative choice would be to keep the SRAF size smaller, but it is still possible for OPC edge movement to combine with these smaller SRAFs to induce printing, so sraf_print_avoidance is not optional for a production solution.

In combination with some experimental runs yielding varying SRAF width, this is the appropriate time in the recipe development to experiment with the mode. As a rough rule, collectno45, which bases SRAF length on the gradient peaks rather than target size, has proven effective on both island layers (generally contacts) as well as line layers (generally metal). However, as the relative shape of layers continue to change as additional interconnect solutions are adopted, there is no way to state that either mode will work better without experimentation.

In more classical line layer topologies, the square SRAFs at line ends may be unnecessary if line end pullback requirements are not very strict. The corner SRAFs help significantly with formation of 2D images, which is why cornerpref square is default. For many illuminators, the square SRAFs are so important that the squareonly option is required so that square SRAFs are placed at a higher priority than edge-oriented SRAFs.

Specific case solutions

The following solutions are provided as additional guidance for specific layer and illuminator scenarios.

Line layers

As mentioned, square SRAFs may be unnecessary, depending on the image requirement specifications. In addition, the restriction of a single width for each localmaxgradient can diminish the achievable process window. The solution is to not specify maxoffset (maximum distance from targets that SRAFs are placed) and instead have additional localmaxgradient templates, which have (generally) increasing SRAF widths (and square widths if desired). The benefit of the larger SRAFs at the larger distance to targets is more image influence, while the separation from OPC movement reduces the chance of SRAF printing (still controlled in OPC by sraf_print_avoidance).

Layers of mixed size targets in close proximity

Several layers include a mix of smaller and larger shapes in close proximity to each other. An example would be the smaller signal lines and larger power lines in a standard cell metal layer. For these environments, the gradient typically provides a grid of seed pixels that can be transformed into SRAFs which are parallel to either shape. If small shapes are marked as members of the prioritylayer, the SRAF output provides better support of small images at a modest expense of large shape imaging. Because the tolerance of the larger shapes to image imperfections is typically larger, this allows a general method of directing SRAFs to focus the benefit on the smaller target shapes.

Inadequate process window after optimization

In the majority of layouts and processes, simulation with a single optical condition (best focus or defocus) provides adequate SRAF insertion. However, at times the competing forces on proper image placement may require that the gradient consider more than one condition. Using Calibre OPCverify conditions for process windows, identify the conditions that lead to image failure. If these (or slightly more off-nominal) conditions are provided to the gradient calculations, the placement of SRAFs will shift and usually resolve the insufficient solution. At times, the weight of these pw_conditions may need to be in the 10-100 range, so that they more fully dominate the gradient calculation. Of course, if a single condition is needed to achieve the process window that condition can be provided as the nominal model (defined by the mbsraf optical command).

Negative SRAF insertion

In large target shapes, which are near tight spaces, imaging can be improved by inserting negative SRAFs, which are SRAFs that remove material inside the target (typically generating slots) in the vicinity of a tight space, increasing the amount of light that can form the outer edges. To enable negative SRAF generation with Calibre mbSRAF, several steps are required.

  1. Define a localmaxgradient template with a unique prior value. In the associated priority definition, specify the keyword negative along with any other priority level controls desired (for example minfeaturespace).

  2. Define styles specific to negative SRAF such as maxoffsetnegative, cornerprefnegative, and intersectionnegative, which refine negative SRAFs you generate.

  3. Introduce negative SRAFs into all shapes large enough for MRC-legal insertion. While it is mathematically valid that inserting negative SRAFs on isolated shapes improves many process variation artifacts, including PVbands, many customers desire to only have negative SRAFs inserted near tight spaces where they have the most significant imaging benefit. To control SRAF placement in these spaces, use a tag (from taglayer or using a tag script) to define the space edges in the localmaxgradient template used for negative SRAF. If the space where this is needed is unknown, it is probably better to choose a much larger space than suspected, and to adjust it tighter as the process matures. Again, the critical imaging metrics should be the guide for the definition of spaces that have restricted access.

Miscellaneous Recipe Development Aids

Using the available Calibre mbSRAF keywords and constraints properly speed recipe development.

  1. Start by defining the MRC constraints to match those used by OPC.

  2. Use the mode keyword to define required SRAF length controlled by target length or length controlled by gradient.

  3. Optimize width and squarewidth values based on process window printing.

Lithomodel Format Usage for Calibre mbSRAF Execution

Siemens EDA recommends using litho model format in the development and execution of Calibre mbSRAF because:

  • The litho model format provides a more organized repository for model data.

  • The litho model format is becoming more widely used for many Siemens EDA litho modeling tools.

The litho model format has several operational constituents:

image_options

Specifies the image_options block name.

litho_model

Specifies the directory containing the Lithomodel data.

Lithomodel file

References optical and resist models contained in the directory specified by litho_model.

In this Lithomodel file, a DDM model is specified:

version 1
mask 0 {
	optical optical_0
	optical optical_-40 focus -40nm
	optical optical_+40 focus 40nm
	background 1 0
	mask_layer 0 TRANS -.9785 -.0085 DDM omog.dark.quad.ddm
	mask_layer 1 TRANS -.9785 -.0085 DDM omog.dark.quad.ddm
	dose 1
}
resist resist.mod
image_threshold 0.246

This Lithomodel file specifies a non-DDM model:

version 1
mask 0 {
	optical opt
	background atten 0.06
	mask_layer 0 TRANS clear CATEGORY visible
}
resist ctr.mod

This example presents a complete LITHO FILE statement, implementing the litho model format, for generating SRAF with Calibre mbSRAF:

LITHO FILE mbsraf_setup [
	modelpath ./models
	layer M1
	layer M1_sraf
	layer M1_OPC
	tilemicrons 25
	progress_meter on

	image_options LMO {
		lithomodel MX1
		layer M1 visible mask_layer 1
		layer M1_sraf visible mask_layer 2
		layer M1_OPC visible mask_layer 406
	}
	sraf_options opts {
		algorithm 1
		sblayer M1_sraf
		correctionlayer M1_OPC
		image_options LMO
		minfeaturespace 0.045 minoffset 0.025 maxoffset 0.220
		minwidth 0.015 minlength 0.030
		minsquarelength 0.022 minsquarearea 0.000529
		gradientshift 0
		localmaxgradient prior 1 width 0.02 squarewidth 0.03
	}
	setlayer mb_sraf = mbsraf M1 IMAGE_OPTIONS LMO OPTIONS opts
	setlayer seedpixels = mbsraf debug mb_sraf seedpixels 1
]
mb_sraf = RET mbsraf M1 M1_sraf M1_OPC FILE mbsraf_setup MAP mb_sraf

For detailed information regarding the Lithomodel file and litho model format, see the Calibre nmOPC User’s and Reference Manual.

correctionlayer Usage for Calibre mbSRAF

The correctionlayer keyword is used during simulation to define the initial image that forms the basis of the gradient calculation.

Siemens EDA recommends using the correctionlayer argument for better performing SRAF, especially with bright field masks. The correctionlayer for mbsraf can be a sized-up target layer or an OPC layer (a layer that has been resized by several iterations of nmOPC). The image generated using the correctionlayer should be within plus or minus 20 percent of the expected nominal contour image.

Priorities for Calibre mbSRAF Tagging

Priorities are vital for the segregation of SRAFs generated by different templates. Priorities also apply to tag different groups, and ensure that minfeaturespace and minoffset enforcement for these various tag groups do not conflict. Additionally, localmaxgradient templates with different tags and the same priority generate Calibre warning messages.

Example

This code sample:

localmaxgradient tag mytag1 width 0.018 squarewidth 0.025 prior 1
localmaxgradient tag mytag2 width 0.018 squarewidth 0.025 prior 1

produces this warning in the transcript:

/-- WARN: nmSraf: Priority 1, templates 0 and 1 do not have the same tags. Check tags or move one template to a different priority.

Setting different priorities for each template with differing tags parses correctly:

localmaxgradient tag mytag1 width 0.018 squarewidth 0.025 prior 1
localmaxgradient tag mytag2 width 0.018 squarewidth 0.025 prior 2
