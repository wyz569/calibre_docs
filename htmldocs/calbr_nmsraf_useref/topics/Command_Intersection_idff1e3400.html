intersection

Style Keywords

Enables valid SRAFs to be merged at various junctions. Used by RET SBAR and RET MBSRAF.

Usage

intersection {45 | C | J | L | N | P | T}
[maxjogwidth value]
[minjogwidth value]
[minstickout value]
[minwidthfree value]
[minwidthoverlap value]
[widthc value]

Arguments

45

Enables valid, generated SRAFs to be merged at 45-degree junctions. Only 45-degree SRAFs of the same widths are merged. A 45-degree merge requires axis-aligned SRAFs; the convex corner is the common point, and the intersection of the two SRAFs defines the concave corner.

The intersection 45 keyword only supports applications for the rectedge extension45 keyword. The intersection 45 keyword is incompatible with cleanupversion 0 and requires specification of cleanupversion 1.

Note:

Siemens EDA does not recommend using intersection 45 where combined with intersection J and lineendmerge. This may allow 45-degree SRAFs that are not aligned to merge with a jog defined by minjogwidth. Although Siemens EDA does not recommend this combination, if intersection 45 is used with lineendmerge, all SRAFs to be merged must be the same width and must be specified with cleanupversion 1.

C

Enables valid, generated SRAFs to be merged at 90-degrees with an intermediate 45-degree junction. The C option can be used with either positive or negative SRAFs. The C option may be specified with the widthc option, providing a target width for the C-segment. Where the C and L options are specified together, the C option is given priority. If application of the C option fails, an attempt is made to apply the L option. Application of the C option fails if the orthogonal SRAFs to be joined are too close together, resulting in corners of the 45‑degree SRAF extending past the orthogonal SRAF edges. Intersection C is incompatible with cleanupversion 0.

J

Enables valid, generated SRAFs to be merged at jog point intersections. The J option can be used with either positive or negative SRAFs. When using intersection J with lineendmerge, jogs are merged only when SRAF widths match. This is the default behavior with cleanupversion 1.

Note:

Siemens EDA recommends using intersection J with the lineendmerge keyword set to a value greater than minoffset. This provides more continuous SRAF results. If no lineendmerge parameter is set, its value is automatically set. This value is reported to the transcript during processing. During Calibre nmOPC and SRAF Print Avoidance (SPA) after SRAF generation, fragments near concave corners cannot be aligned and eliminated. SPA cannot resolve printing at or near jogs because of the inability to align fragments near concave corners.

L

Enables valid, generated SRAFs to be merged at 90-degree junctions, irrespective of orientation angle. The L option can be used with either positive or negative SRAFs. Where the C and L options are specified together, the C option is given priority. If application of the C option fails, an attempt is made to apply the L option. Intersection L is incompatible with cleanupversion 0.

N

Disables SRAF merging. The N option can be used with either positive or negative SRAF generation. The N option is default.

P

Enables valid, generated SRAFs to be merged at orthogonal, non-45-degree junctions (forming four-way intersections or multiple T-junctions). The P option can be used with either positive or negative SRAFs. Intersection P is incompatible with cleanupversion 0.

T

Enables valid, generated SRAFs to be merged where lineend and long edges of SRAFs meet (forming T-junctions). The T option can be used with either positive or negative SRAFs. Intersection T is incompatible with cleanupversion 0.

maxjogwidth value

An optional keyword and argument specified as a positive, floating-point number in microns. Sets the maximum overlap of a jog. It can be set globally or by priority. The maxjogwidth parameter does not apply when smaller than the overlap between line-ends. The default for maxjogwidth is the maximum of the width parameters for all templates at the applied priority.

minjogwidth value

An optional keyword and argument specified as a positive, floating-point number in microns. This keyword overrides the default jog width (equal to the template-based width keyword). The default for minjogwidth is minwidth (if specified) or the minimum width parameter of any template of any priority. This default ensures that no MRC violations can occur when SRAFs of different priorities are jogged.

minstickout value

An optional keyword and argument that defines the minimum length a leg can extend from an intersection formed by intersection L, T, or P. The minstickout keyword controls both existing overlapping SRAFs and subsequently generated intersections. Existing overlapping SRAFs not meeting the minimum value specified by minstickout are cleaned. The default for minstickout is minwidth (if specified) or the minimum width parameter of any template of any priority. The minstickout value must be specified as a positive, floating-point number in microns.

minwidthfree value

An optional keyword and argument specified as a positive, floating-point number in microns. Specifies the minimum non-overlapping portions of jog intersections. See Figure 3 for more information.

minwidthoverlap value

An optional keyword and argument specified as a positive, floating-point number in microns. Specifies the minimum overlap of line widths forming jog intersections. See Figure 3 for more information.

widthc value

An optional keyword and argument, applied only to the C option, that specifies the width of C-segment SRAFs joining two orthogonal SRAFs. The larger orthogonal segment width must be less than or equal to 2 times the smaller orthogonal width for the C-segment to be correctly placed. By default, the width of the inserted segment is equal to the larger orthogonal SRAF segment. The widthc parameter can be used to specify a different width within a valid range of two orthogonal SRAF widths:

smaller_width * sqrt(2) <= widthc <= larger_width / sqrt(2)

If the specified widthc value is greater than the range, the maximum value is used; if the specified value is less than the valid range, the minimum value is used. See Figure 10 for more information. The parameter must be specified as a positive, floating-point number in microns.

Description

An optional keyword that enables valid, generated SRAFs to be joined at various specified intersections. The intersections allowed are represented by characters approximating their final shape: 45 (for diagonal SRAFs), C (for 45s joined with L SRAFs), J (for jogs), L, T, and P (for plus, an orthogonal cross at any orientation). The intersection keyword can be used with either positive or negative SRAFs. Optionally, the intersectionnegative keyword can be used globally to control negative SRAF intersections.

Formation of Intersection Candidates

The search range to locate candidates for the formation of intersections is defined by the parameter lineendmerge. If lineendmerge is not specified, the maximum of one of the minoffset parameter lineend, or the intersection parameters minjogwidth, or maxjogwidth, plus an additional 15% of the maximum value, defines the search distance. All SRAFs within the search distance are considered for the formation of a legal intersection by constructing one or two extensions of the existing SRAFs. For intersection J, three attempts are made: one symmetric attempt by extending both SRAFs, and two asymmetric attempts by extending either one of the SRAFs. Attempts to construct intersections between SRAFs fail if the resulting SRAF candidate conflicts with any existing SRAFs within the search distance.

When using intersection J with lineendmerge, jogs are merged only when SRAF widths match. This is the default behavior with cleanupversion 1.

Note:

Regarding intersection usage:

  • Siemens EDA recommends using intersection J with the lineendmerge keyword set to a value greater than minoffset. This provides more continuous SRAF results. If no lineendmerge parameter is set, its value is automatically set for you. This value is reported to the transcript during processing. During Calibre nmOPC and SRAF Print Avoidance (SPA) after SRAF generation, fragments near concave corners cannot be aligned and eliminated. SPA cannot resolve printing at or near jogs because of the inability to align fragments near concave corners.
  • The SRAF merging order for multi-option intersection keywords is always: J, L, T, and then P. The specified order of the intersection keyword option letters has no effect on SRAF merging. In all other cases where one option is specified, only merging consistent with that option can be performed.
