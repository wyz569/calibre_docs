DRC RVE Check Text Override File (CTO File) Format

Input for: Calibre RVE for DRC.

The DRC RVE check text override file (CTO file) allows you to specify DRC RVE rule check comments for a DRC results database and load the settings into a Calibre RVE for DRC session. DRC RVE rule check comments allow you to specify highlight colors, display layers, document links, and other settings related to highlighting results. The CTO file also defines categories for filtering by category.

See “DRC Rule Check Comments for Calibre RVE” for the complete list of rule check comments allowed in a CTO file. The CTO file is loaded directly into Calibre RVE, as explained in the section “Specifying and Loading the DRC RVE Check Text Override File”. Unlike the case when DRC RVE rule check comments are specified in the rule file, there is no need to edit the rule file and run Calibre nmDRC.

DRC RVE rule check comments found in a DRC RVE check text override file take precedence over settings from the rule file, as described in the section “DRC RVE Rule Check Comment Precedence”.

See “Creating a Check Text Override (CTO) File from the Rule File” for information on automatically creating a CTO file.

For support of the CTO file outside of Calibre RVE for DRC, see these topics: “Specifying Highlight Colors in DRC HTML Reporting” and DRC Rule Check Comments for Calibre RealTime Custom in the Calibre RealTime Custom User’s Manual.

Format

  • ASCII text file. Blank lines are ignored, and white space at the beginning of a line is ignored.

  • Comments — Lines beginning with the # character are ignored.

  • Contents

    # rule check comment block
checkname [checkname ...]
[@ check_text_comment]
RVE_rule_check_comment
...

# category definitions
category_definition
...

    Multiple rule check comment blocks and category definitions may be specified.

Parameters

checkname

A required parameter giving the name of one or more rule checks that the following DRC RVE rule check comments apply to.

The * wildcard character may be used in checkname. Rule check comments accumulate if more than one rule check comment section matches a specific rule check due to the use of wildcard characters in the checkname. If there is a conflict between accumulated rule check comments, then the latest rule check comment section in the Check Text Override file takes precedence; see the “Examples” section. Rule check sections without a wildcard in the check name always have the highest precedence.

If using a CTO file with Calibre PERC LDL CD and P2P results, the check name depends on the type of experiment. See the topic relevant for your experiment in “Viewing LDL CD Results in Calibre RVE” or “Viewing LDL P2P Results in Calibre RVE” in the Calibre PERC User’s Manual.

@ check_text_comment

Text appearing after the @ character and on the same line is processed as a check text comment for the checkname rule check; check text is optional. Check text comments are displayed as follows:

  • Calibre RVE for DRC — in the check text pane (View > Check Text Pane)

  • DRC HTML Report — in the Check Text column if ReportType=DRCResults

Check text present in the results database as a result of check text comments in the rule file is merged with check text comments in the CTO file.

RVE_rule_check_comment

A DRC RVE rule check comment. More than one rule check comment may be specified. See “DRC Rule Check Comments for Calibre RVE” for a list of rule check comments.

The RDB rule check comment cannot be placed in the Check Text Override file.

category_definition

An optional entry specifying a category definition to enable filtering by category. The category_definition has the following form:

RVE CATEGORY: category_name
category_name cat_branch [rule_check ... ] [cat_branch ...]
...

  • RVE CATEGORY: — Literal text. There must be a space after the colon.

  • category_name — The category name. If the name includes white space enclose the name in double quotes.

  • cat_branch — The name of a category grouping, or branch. The category branch element must specify at least one rule check or category branch. Specifying a category branch within a branch element creates a hierarchical structure. The specified category branches must be defined within the same category definition.

  • rule_check — The name of a rule check. Separate multiple rule checks with a space. A rule check may be listed in more than one category branch. Use wildcards to specify multiple rule checks.

See Example 4: Category Definition and Example 5: Category Definition from Rule File Group Statements for an example category definitions. See “Filtering DRC Results by Category” for how to use categories.

Examples

Example 1: Basic Check Text Override File

This example uses an environment variable DOC_DIR to specify the directory name of the documents being referenced in the RVE Link rule check comments. A check text comment beginning with the @ character is used to describe the rule checks.

# DRC RVE check text override file for XYZ

min_spacing_metal1
@ Minimum metal1 spacing = 1.0
RVE Show Layers: metal1 
RVE Highlight Color: blue
RVE Link: $DOC_DIR/wide_space.html#Procedure TEXT doc:wide metal rules
min_ext_diff_poly 
@ Minimum enclosure of poly by oxide = 1.25
RVE Show Layers: poly diff 
RVE Highlight Color: green
RVE Link: $DOC_DIR/min_ext.html TEXT doc:min enclosure rules

Example 2: Precedence of Rule Check Comments With Wildcards in the Check Names

The following example demonstrates the precedence of rule check comments when the use of wildcards in the rule check name causes a rule check comment to apply to more than one rule check. For this example, assume rule checks with the following names: min_spac_metal1, min_spac_metal2, poly_enc_cont, and metal1_enc_cont.



Final settings:

     min_spac_metal1:

RVE HIGHLIGHT COLOR: blue
RVE Priority: 1

     min_spac_metal2:

RVE Priority: 1
RVE HIGHLIGHT COLOR: yellow

     poly_enc_cont:

RVE HIGHLIGHT COLOR: green

     metal1_enc_cont:

RVE Priority: 1
RVE HIGHLIGHT COLOR: green

Example 3: Check Text Override File with Check Text and Wildcards in the Check Names

This example demonstrates the use of multiple check names and wildcard characters in the check name. Wildcards are used to match all spacing checks (*spac*) and assign a highlight color of yellow, then to match all enclosure checks (*enc*) and assign a highlight color of green. The RVE Show Layers rule check comment is used in each specific check to specify the layers that are displayed when highlighting.

# check text override file
# all spacing checks highlighted in yellow
*spac*
@ spacing comment from CTO file
RVE HIGHLIGHT COLOR: yellow

# all enclosure checks highlighted in green
*enc*
@ enclosure comment from CTO file
RVE HIGHLIGHT COLOR: green
# two enclosure checks use the same RVE Show Layers comment
poly_enclose_cont metal1_enclose_cont
RVE Show Layers: contact metal1 poly

min_spacing_metal1
@ min_spacing_metal1 comment from CTO file
RVE Show Layers: metal1

min_spacing_metal2
RVE Show Layers: metal2

Example 4: Category Definition

This example creates a category named “ViaMetalChecks.” Hierarchy is included in the category definition with the definition of the METAL_ALL and VIA_ALL branches, which specify other branch names. A wildcard character (?) is used to target all checks that begin with “MT1.”.

RVE CATEGORY: ViaMetalChecks
ViaMetalChecks METAL1 MT1.?
ViaMetalChecks VIA1 V12.1 V12.2 V12.3
ViaMetalChecks METAL2 MT2.1 MT2.2 MT2.3
ViaMetalChecks VIA2 V23.1 V23.2 V23.3
ViaMetalChecks METAL_ALL METAL1 METAL2
ViaMetalChecks VIA_ALL VIA1 VIA2

Example 5: Category Definition from Rule File Group Statements

You can easily create a category definition from the Group statements in your rule file. Name the category “Group” and copy each Group statement into the definition to define the category branches.

RVE CATEGORY: GROUP
GROUP CONTACT_CHECKS CO1.2 CO1.3 CO1.4 CO1.5
GROUP POLY_CHECKS PO1.1 PO1.2 PO1.3 PO1.4 PO1.5 PO1.6
GROUP METAL1_CHECKS MT1.1 MT1.2 MT1.3
GROUP METAL2_CHECKS MT2.1 MT2.2 MT2.3
GROUP VIA1_CHECKS V12.1 V12.2 V12.3
GROUP VIA2_CHECKS V23.1 V23.2 V23.3
GROUP METAL_CHECKS METAL1_CHECKS METAL2_CHECKS
GROUP VIA_CHECKS CONTACT_CHECKS VIA1_CHECKS VIA2_CHECKS
