Parameters

Note:

All parameter values for masksim use mask-level dimensions, and should be set accordingly.

make_ddm_edge_library

This required keyword triggers masksim to recognize that this input file contains a simplified DDM library creation specification.

wavelength val_in_um

This required keyword specifies the simulation wavelength in microns.

gridsize val_in_um | {dx dy dz} | {[dx dy] baseunitcells [num_cells_per_baseunit]}

This required keyword specifies the size of the simulation grid in microns. The grid size is specified using one of the following:

  • val_in_um for a cubic grid.

  • {dx dy dz} for a non-cubic grid.

  • {[dx dy] baseunitcells [num_cells_per_baseunit]} for EUV sources.

When a single val_in_um is provided, gridsize has the following guidelines:

  • The gridsize should be less than wavelength/15.

  • Typically, values between 0.001 and 0.005 give the best accuracy, but masksim run time increases as gridsize decreases.

  • In EUV, the multilayer period should be an integer multiple of the gridsize for best accuracy.

When three arguments (dx, dy, dz) are specified, gridsize uses a non-cubic grid. It has the following guidelines:

  • The dx and dy values must be equal (for example, gridsize 0.001 0.001 0.008).

  • The relative relationship between dx and dz should not be larger than 8, in order to prevent numerical calculation issues.

For EUV sources only, specifying the baseunitcells argument ([dx dy] baseunitcells [num_cells_per_baseunit]) creates non-cubic grid cells based on the thickness of the defined bilayer or multilayer. It requires that the “euv_multilayer numbaseunits” or “euv_multilayer numbilayers” parameter must be specified.

  • The optional dx and dy arguments set the cell X and Y dimensions. They are set to 1 nm by default.

  • The num_cells_per_baseunit argument sets the cell height for the euv_multilayer dz as the total thickness of the bilayers or multilayer in the euv_multilayer parameter / num_cells_per_baseunit. The default is 10; any value specified must be an integer value of 10 or greater.

Suggested val_in_um values are:

  • 0.002 for DUV.

  • 1/10 of the bilayer thickness for EUV multilayer.

However, there is an additional check performed for cells per wavelength (cpw):

            cpw = wavelength/(max_n × gridsize)

where max_n is the maximum real part of the refractive index of the mask film stack.

On rare occasions when cpw is smaller than 18, the gridsize is adjusted to:

  • DUV: wavelength/(18 × max_n)

  • EUV: bilayer_thickness/ceiling(18 × bilayer_thickness × max_n/wavelength)

edgelength val_in_um

The length in microns of the edge to be stored in the library. Uses mask dimensions (4X). The value is rounded during generation to make it an integer multiple of twice the grid size.

(DUV default = 8.0; EUV default = 2.0)

mincycle int

Minimum number of time periods that masksim will compute.

(Default = floor[edgelength/(2 × wavelength)])

maxcycle int

Maximum number of time periods that masksim will compute.

(Default = floor[edgelength/(2 × wavelength)])

cycles int

Total number of time periods that masksim will compute. This effectively sets the maxcycle and mincycle to the same value int.

(default = floor[edgelength/(2 × wavelength)])

substrate n [phasedepth phase_in_deg] [underetch val_in_um] [slope val_in_deg] [rounding radius_in_um]

A required parameter that specifies information about the substrate. The refractive index (real number) is the only required parameter, and can be for glass or other materials. The phasedepth, underetch, slope, and rounding are shown in Figure 1 and defined as follows:

Figure 1. Example Substrate Description

Sub-keywords to substrate:

  • phasedepth — The phase (in degrees) of the associated etch into the substrate material created for Phase Shifting Masks (PSMs). A 180 degree phase well would be specified as phasedepth 180.

    Default = 0.0

  • underetch — Describes how far the substrate material is etched back (in microns) away from the given film edge.

    Default = 0.0

  • rounding — Describes the radius (in microns) of a circle that inscribes the rounded corner.

    Default = 0.0.

  • slope — The angle (in degrees) of the slope associated with the substrate walls due to phase etching. A 90 degree value represents a perfectly vertical sidewall.

    Default = 90.

film t n k [“comments”] [local_slope_in_degrees] [sidewall sw_w sw_n sw_k]

A required parameter defining the film(s) in the mask stack. Each film is given its own line, and four parameters must be defined for each film: the thickness in microns, the real and imaginary part of the refractive index, and a comment. The optional comments must be contained within quotation marks (“ ”).

The optional local_slope_in_degrees argument follows the comment. If specified, it overwrites the universal film slope defined in the stackslope argument, and sets the slope for this film only; all other films that do not also have a local_slope_in_degrees argument are given a default slope of 90 degrees (a perfectly vertical sidewall). In addition, if the global slope is invalid, the stackrounding only affects the last film, no matter how large the rounding radius is.

The optional sidewall argument can be specified to represent the sidewall coating at the film boundary. The sidewall is specified with a width of sw_w in microns in the X and Y directions, and is assumed to have a film thickness equal to the film. The sidewall is also specified with its own independent refractive indexes sw_n and sw_k. The sidewall coating is given the same stackslope and rounding properties as the related film.

Note that the first film is assumed to be the one closest to the substrate. The sign of k (imaginary part of refractive index) is not relevant, with the understanding that the simulation code models lossy media.

You define the shape of the entire film stack using the keywords as shown in Figure 2.

Figure 2. Example Film Description (DUV)

stackslope val_in_deg

The slope of the mask edge at the film stack. A value of 90 is a perfectly vertical sidewall. However, note that if the local_slope_in_degrees argument is specified with any film parameter, this argument is ignored.

(Default = 90; recommended best practice value is 86)

stackrounding val_in_um

Defines corner rounding at the top (farthest from substrate) of the film stack.

(Default = 0.)

objectplane val_in_um

objectplane imageshift [mode {through_slit | field_maximum | horizontal_edge}] [cd feature_size] [background dark | clear] [threshold value] [center_slit_phi phi_value]

The location(s) of the plane to which the scattered electric field is propagated.

When a single value is specified, val_in_um is considered to be measured with respect to the end of the last film (farthest film from the substrate), with positive values being in the direction moving away from the substrate and towards the wafer (downwards). For universal libraries with DDM version 4 or later, use the default 0um and interpolateddmsectors specifies the objectplane value. (Default = 0um.)

If more than one object file is specified, output filenames are generated with modified filenames starting with “bright_outputfile” or “dark_outputfile”, appending the objectplane value in %5f format and ending in “.ddm,” such as “dark_outputfile0.05323.ddm”.

Note:

Objectplane imageshift should only be used with EUV lithography due to the non-telecentric illumination of EUV systems.

Use the imageshift argument to calculate an object plane with the best minimized shift based on the specified mode. When run, masksim outputs the DDM library for that object plane.

  • mode {through_slit | field_maximum | horizontal_edge}

    • through_slit — Minimizes the slope of the shift for the edges of the approximate image of an isolated vertical line across the center slit. This is the default mode, except for universal libraries.

    • horizontal_edge — Minimizes the average shift of the edges of the approximate image of an isolated horizontal line at the center slit.

    • field_maximum — Minimizes the shift of the maximum location of the DDM edge signal simulation for a horizontal feature of the center slit chief ray field, filtered to three orders. This was the method used for releases previous to version 2017.4.

  • cd feature_size

    Controls the feature size to be used for the through_slit and horizontal_edge modes, in microns. The default value is 0.2.

  • background dark | clear

    Controls the mask background tone for approximate image simulations. The default is dark.

  • threshold value

    Controls the threshold that defines the image edge for the through_slit and horizontal_edge modes. This value assumes an image with clearfield = 1.0. The default is 0.3.

  • center_slit_phi phi_value

    Computes the imageshift for the specified central phi_value, which must be set to a value of either 90 or -90 degrees, relative to the +x axis. If a center_slit_phi argument is also specified for the chief_ray parameter, the objectplane imageshift setting for center_slit_phi is used in place of the chief_ray center_slit_phi setting. When generating a ready library at non-center slit locations, specify center_slit_phi and a phi_value.

euv_multilayer {numbilayer options} | {numbaseunits options}

For EUV use only. Creates one or more EUV multilayers. Specifying “numbilayer” creates bilayers. Specifying “numbaseunits” creates sets of multilayers.

  • euv_multilayer numbilayer num_layers bottom thickness n k top thickness n k

    Creates an EUV multilayer with a number of bilayers specified with the numbilayer sub-keyword.

    The bottom keyword gives the thickness in um and refractive index (n and k) of the bottom half of each bilayer, whereas the top keyword gives the thickness and refractive index of the top half.

    It is not necessary that the thickness values be integer multiples of the grid size. If they are not, masksim performs weighted averages of the permittivity of the top and bottom materials, depending on the amount of each in each Yee cell.

    Figure 3. Diagram Showing Bilayers and Capping Layers in EUV-DDM

    In EUV, the film stack is above the capping layers, if any, and the multilayer, which in turn is above the substrate. You may have only one euv_multilayer command in your DDM input file.

  • euv_multilayer numbaseunits num_baseunits {film film_index thickness_in_um n k} [film...]

    For EUV use only. Creates multilayers that can be made up of repeating units of three or more different materials.

    • Each repeating unit is a base unit, and the number of base units is specified in num.

    • The film keyword set defines a base unit film, starting with the bottom (the film nearest the substrate). Each film keyword requires a film index, which must be specified in order from 1 to the number of films, then the thickness in microns (n) and refractive index (k).

      Note:

      Only one euv_multilayer command of either syntax is permitted in the DDM library input file.

capping_layer thickness n k

For EUV use only. Specifies the addition of a capping layer with the given thickness and refractive index between the multilayer and the film stack. You may have multiple capping layers. Successive capping layers will be placed above earlier ones, and are considered farther from the multilayer and closer to the film stack.

substrate_capping_layer thickness n k

Specifies the addition of a capping layer with the given thickness and refractive index between the substrate and a second layer. For EUV masks, the substrate capping layers are between the substrate and the multilayer (see Figure 3). For DUV masks, the substrate capping layers are between and films (see Figure 2). You may have multiple substrate capping layers. The first substrate capping layer is placed adjacent to the substrate; successive substrate capping layers are placed farther from the substrate and closer to the multilayer or films.

illumination path_to_optical_file | {universal [dsigma dsigma] [dphi dphi] [na NA] [chief_phi_range {phi_range | nxe3400 | nxe3300 | nxe3100}]}

Required. Adds an illumination condition from source sectors to be simulated.

  • path_to_optical_file specifies the location of an optical file that must have HHA active (the “hha 1” setting) and a source sectorization scheme.

    Note:

    Previous versions of the DDM library input format file required manually setting theta and phi. This has been replaced by only specifying the optical file, and the sector information is automatically calculated by Calibre WORKbench. Older files which use the old illumination argument format must be converted to the new syntax starting with the 2013.3 release.

    When the illumination path_to_optical_file version of the command is used, then the fullddm keyword passed through to the DDM library is determined by the ddm_model keyword in the optical file referenced. If the optical model file uses “ddm_model full,” then the created DDM library is generated with “fullddm yes”. If the optical file has “ddm_model approx,” then the DDM library is generated with “fullddm no”. If the masksim input file has more than one illumination line pointing to optical models that have different ddm_model keywords, an error message is generated.

  • If the universal keyword is supplied as the first argument to illumination, a “universal” library is created. A universal library samples the entire source pupil and is intended to be used to create “ready” libraries for a specific source by interpolation.

    A universal library has had its EM simulations completed offline, and the actual scattered electric field is given in the file. It can only be used in three ways:

    • As input to the interpolateddmsectors function, which generates a “ready” DDM library

    • As a crosstalk_ddm_shortcut_file or corner_ddm_shortcut_file to generate a “universal” DDM library

    • As input to Calibre® pxSMO™

    The density of the sampling can be controlled by the dsigma, dphi, na, and chief_phi_range arguments.

    • dsigma sets the approximate step size in the radial direction. It has units of sigma. The default value for dsigma is 0.1.

    • dphi sets the approximate stepsize at sigma=1 in the azimuthal direction. It has units of degrees. The default value for dphi is 4.0.

      dphi and dsigma are approximate, because the software will round them to the nearest value which evenly samples the source.

    • na is the wafer NA of the lithography system. It is needed to calculate the angles of plane waves in the DDM library. The default value for na is 1.35.

      These default values were chosen to produce approximately 500 physical sectors in the source.

    • chief_phi_range specifies the range of chief_ray phi values for which signals are generated. This parameter allows a single universal DDM library to contain signals for a full EUV slit. It accepts one of the following parameters:

      • phi_range — Specifies a range in degrees to include. Half of the specified range is used on either side of 0 degrees, so a phi_range of 60 runs the range from -30 to 30. The default is 0.

      • nxe3100, nxe3300, and nxe3400 — Specifies that the range generated is set to the values appropriate for the full range of the specified ASML tool.

compute_diagonals {0 | 1}

If the value 0 is specified, the signals for the 45-degree edges will not be computed. Turning off computation of the 45-degree edges will cut the run time roughly in half. (Default = 1.)

domain {spatial | fourier}

This optional parameter specifies whether the output electric field is reported either in the spatial or Fourier domain. (Default = spatial.) Fourier is not currently supported.

{bright_outputfile filename | dark_outputfile filename | {bright_outputfile filename dark_outputfile filename}

Both bright_outputfile and dark_outputfile can be specified to generate two libraries, but at least one must be specified.

Specifies the output filename to contain the DDM library information for a bright field mask or dark field mask, respectively.

reduction_factor val

Specifies the reduction factor to be used for DDM. This value is only passed into the output edge library for use by DDM later, and is not used by Calibre MaskSim. However, this value affects the gridsize parameter.

chief_ray nxe3400_throughslit | {theta theta_value phi {phi_value | {‘[’phi_list‘]’ center_slit_phi phi_list_value}}}

Specifies the off-axis (theta) and azimuthal (phi) angle in degrees of the direction that is to be used by default for simulation. The following rules apply for theta, phi, and center_slit_phi values:

  • theta cannot be negative, and it must be less than 45 degrees.

  • If theta equals zero, phi must also equal zero.

  • If theta is greater than zero, the absolute value of phi must be between 0 and 180 degrees.

  • center_slit_phi can only be 90 or -90.

Note:

The chief ray parameter is required in EUV, because the non-telecentric system requires a non-zero chief ray. Chief ray no longer has a default setting for EUV. (The default for DUV is theta=0 phi=0.)

Previous to 2016.3, the default for center slit EUV was theta=6 phi=90 for ASML ADT and NXE3100, and theta=6 phi=-90 for ASML NXE3300.

Multiple angles can be specified for EUV center slit purposes by supplying a phi_list that lists each angle of incidence for the source. The multiple angle syntax uses the following MATLAB-style1 conventions:

  • The entire list must be enclosed in brackets (“[ ]”). A single value not enclosed in brackets reverts to the single phi_value behavior above.

  • List values must be in increasing order, and separated by commas.

  • List values are relative to the angle of incidence, and are listed from left to right, corresponding to the EUV slit aperture.

  • List values must either be single integers or a start:step:end range (for example, [‑1:2:10] generates the values -1,1,3,5,7,9).

  • List entries can have mixed range and single values (for example, [1:2:10,15, 19,20:3:30]).

  • For ASML NXE3400 EUV scanners, the “chief_ray nxe3400_throughslit” option is recommended. The “chief_ray nxe3400_throughslit” option is the exact equivalent of the following setting:

    chief_ray theta 6 phi [-110 -108 -105 -102 -99 -96 -93 -90 -87
 ‑84 -81 -78 -75- 72 -70] center_slit_phi -90

    For other EUV scanners, the “center_slit_phi phi_list_value” argument is required. It specifies one of the listed angles as the main phi (center slit) value. Starting with the 2016.3 release, phi_list_value must be 90 or -90.

    In addition, if the “objectplane imageshift” parameter is set, the objectplane computed with the main phi will be reused for all other phis. If “objectplane imageshift center_slit_phi” is set, it overrides the center_slit_phi value set for chief_ray.

When you generate a DDM library using the EUV list syntax, Calibre WORKbench creates multiple DDM libraries. Calibre WORKbench names the libraries by appending “_phi_phi_value.ddm” to the original name, and each phi library contains the same center_slit_phi argument to indicate which file is the center slit library.

ddm_verbosity snapping

Specifies an optional argument that adds the line “verbosity snapping” to the output DDM file. This flag instructs Calibre WORKbench to report in an INFO line when an HHA simulation chooses a DDM signal of a different angle or polarization than actually requested.

Cross Talk Input Parameters

crosstalk_model {none | bright_gap | dark_gap | all}

An optional keyword that controls the generation of a cross talk model.

  • none — Specifies that no cross talk model is created. This is the default for EUV models.

  • bright_gap — Specifies that cross talk signals are generated only for opposing edges when the space between them corresponds to the higher mask transmission (the maximum absolute value between the transmission of the layer and the background).

  • dark_gap — Specifies that cross talk signals are generated only for opposing edges when the space between them corresponds to the smaller mask transmission (the minimum absolute value between the transmission of the layer and the background).

  • all — Specifies that cross talk signals are generated for both types of opposing edges (separate cross talk signals for opposing edges surrounding bright_gap space or dark_gap space). This is the default for DUV models.

crosstalk_orientation {all | horiz | vert | horizontal | vertical}

Controls whether to compute crosstalk values for horizontal or vertical gaps. The default value of “all” computes both signals.

crosstalk_range [bright_gap ‘[’ r1low[:r1step:r1high] [r2low[:r2step:r2high]] … ‘]’] [dark_gap ‘[’ t1low[:t1step:t1high] [t2low[:t2step:t2high]] … ‘]’]

An optional keyword for crosstalk modeling that sets the lower and upper bounds of distances between opposing edges for which cross talk signals will be generated. (See also the sections “Older Format for crosstalk_range” and “"auto" Behavior for crosstalk_range”.)

Values are specified in mask dimensions.

Arguments use one of the following formats to define a value or a group of values, separated by spaces:

  • low — An individual value can be used for low if :step:high is not also specified.

  • low:step:high — A range of values with a step spacing amount to increment from the low to the high value, with each element separated by colons (:). The step element must exactly divide between the low and high values.

Additionally, the entire range must be enclosed in brackets ([ ]) as part of the syntax.

For example, the following code defines the values for bright_gap as 0.04 to 0.40 incremented in steps of 0.01 (37 elements) followed by values of 0.45 to 0.5 in steps of 0.05 (2 elements), and dark_gap is defined as the values 0.07, 0.09, then 0.1 to 0.3 in steps of 0.1 (3 elements).

crosstalk_range bright_gap [0.04:0.01:0.400 0.45:0.05:0.500] dark_gap \
 [0.07 0.09 0.1:0.1:0.3]

Older Format for crosstalk_range

Previous to the 2013.3 release, the syntax for the arguments was:

[bright_gap r1_um r2_um] [dark_gap r3_um r4_um]

where values were limited such that 0< r1_um < r2_um and 0 < r3_um < r4_um. Also, the difference between r2_um and r1_um must be an exact multiple of the crosstalk stepsize, as must the difference between r4_um and r3_um. That is, the endpoints of the range must be hit exactly. Also, neither r2_um nor r4_um may exceed the domain size. We recommend that you set the range to be slightly smaller than the smallest anticipated mask feature (SRAFs included) up to 480nm (4x).

Both the old and new syntaxes are supported, but cannot be mixed within a single crosstalk_range statement.

At least one range (for bright_gap or dark_gap) must be specified. This line is copied as-is to the output ddm model file.

“auto” Behavior for crosstalk_range

If crosstalk_range is not specified or “crosstalk_range auto” is specified, a line is added to the output ddm model file showing the lowest and highest values for bright_gap and dark_gap, in the older format.

The “auto” mode directs masksim to automatically choose crosstalk distances, with no further input from the user. In particular, crosstalk_stepsize is not allowed and returns an error. The program performs the following actions:

- Allocates a fixed number (currently 60) of crosstalk distances to simulate.

- Distributes the distances from a low value of 20 nm (mask scale; that is, 5 nm wafer scale for DUV) to a high value of 40% of the domain size. It distributes them as evenly as possible on a logarithmic scale between these two values, subject to the constraint that every gap size must be a multiple of twice the Yee-cell size. The total number of distances simulated may be less than the number attempted because of this constraint.

crosstalk_stepsize val

An optional keyword (required for cross talk models) specifying the pixel size in microns of the cross talk signal, or equivalently the distance in microns between consecutive points of the cross talk signal. This parameter must be specified in mask dimensions (dividing by the reduction factor converts this to wafer dimensions).

The value of this parameter controls the number of simulations executed when computing the cross talk model. For this reason, it has a significant effect on the runtime. Recommended values of crosstalk_stepsize are approximately 5-10nm in 193nm wavelength.

Note:

This option is only used if the argument specified for crosstalk_range is the “start end” range format. It is ignored if the argument is the “low:step:high” range format.

This option cannot be used with the “crosstalk_range auto” setting and returns an error.

crosstalk_domainsize val

An optional keyword that specifies the size of the mask domain used for masksim simulations in order to generate the cross talk signals. The size is specified in micrometers. Note that this parameter must be specified in mask dimensions (dividing by the reduction factor converts this to wafer dimensions). The default value is 2um.

crosstalk_ddm_shortcut_file existing_ddmfile

An optional keyword that imports DDM signals from an existing library and then has masksim just generate the crosstalk signals. The existing_ddmfile must be a bright-background DDM file.

CAUTION:

Use this feature carefully. Only minimal checking is done to verify the DDM signals in the existing_ddmfile are consistent with the other settings in the input file.

Corner Model Input Parameters

corner_model {none | bright_corner | dark_corner | all}

An optional keyword set that controls the generation of corner signals.

  • none — Do not compute corner signals. This is the default.

  • bright_corner — Compute corner signals for bright features.

  • dark_corner — Compute corner signals for dark features.

  • all — Compute corner signals for all types of corners (bright and dark).

The tone in bright_corner or dark_corner refers to the tone of the region between the two edges as shown in Figure 4.

Figure 4. Bright Corner and Dark Corner

corner_type {sharp | chopped | rounded}

An optional keyword set that defines the shape of the corners being modeled.

  • sharp — Creates a library with only sharp 90-degree corners. This is the default.

  • chopped — Creates a library which may contain both sharp 90-degree and chopped corners using corner_range to specify the corner chop size.

  • rounded — Creates a library which may contain both sharp 90-degree and rounded corners using corner_range to specify the corner rounding size.

corner_range {[bright_corner ‘[‘ r1low[:r1step:r1high]  [r2low[:r2step:r2high]]... ‘]’]   [dark_corner ‘[‘ t1low[:t1step:t1high]  [t2low[:t2step:t2high]]... ‘]’]} | auto

A keyword set required with “corner_type chopped” and “corner_type rounded” but may not be specified with “corner_type sharp”. Values represent the corner chop or corner rounding size specified in mask dimensions. A corner chop or corner rounding size of 0 is equivalent to a sharp 90-degree corner.

At least one of auto, bright_corner, or dark_corner must be specified. You can also specify both bright_corner and dark_corner. The keyword auto cannot be combined with the others.

Arguments use one of the following formats to define a value or a group of values, separated by spaces:

  • low — An individual value can be used for low if :step:high is not also specified.

  • low:step:high — A range of values with a step spacing amount to increment from the low to the high value, with each element separated by colons (:). The step element must exactly divide between the low and high values.

Additionally, the entire range must be enclosed in brackets ([ ]) as part of the syntax.

For example, the following code defines the values for bright_corner as 0.04 to 0.40 incremented in steps of 0.01 (37 elements) followed by values of 0.45 to 0.5 in steps of 0.05 (2 elements), and dark_corner is defined as the values 0.07, 0.09, then 0.1 to 0.3 in steps of 0.1 (3 elements).

corner_range bright_corner [0.04:0.01:0.400 0.45:0.05:0.500] \
dark_corner [0.07 0.09 0.1:0.1:0.3]

The keyword auto is approximately equivalent to 0:gridsize:0.08. This range accurately and efficiently samples the range of corner chop and corner rounding values applicable to deep submicron photomasks.

corner_ddm_shortcut_file existing_ddmfile
CAUTION:

Use this feature carefully. Only minimal checking is done to verify the DDM signals in the existing_ddmfile are consistent with the other settings in the input file.

An optional keyword that imports DDM signals from an existing library and then has masksim generate the corner signals only. The existing_ddmfile must be a bright-background DDM file.

corner_domainsize val

An optional keyword that specifies the size of the simulation domain used for rigorous corner simulation. The domain is square with side lengths of val microns. For DUV, the default val is 2.0. For EUV, the default val is 0.512 for DDM version 5 or later and 0.6 otherwise.

corner_gridsize {auto | disable | val_in_um | {dx dy dz} | {[dx dy] baseunitcells [num_cells_per_baseunit]}}

An optional keyword that specifies the cell size used in the FDTD simulations for corner DDM signal calculation. The cell size is specified using one of the following:

  • auto sets the corner dx and dy values to the maximum of dx from gridsize and:

    • For DUV, 5 nm.

    • For EUV, 2 nm for DDM version < 5 and 4 nm otherwise.

    The corner dz value is set by wavelength and DDM version:

    • For DUV, the maximum of dz from gridsize and 5 nm.

    • For EUV, the value of dz from gridsize for DDM version < 5 and the dz-equivalent of 5 baseunitcells otherwise.

  • disable sets the corner dx, dy, and dz values to the gridsize values.

  • val_in_um sets the lengths for a cubic cell. The corner cell length values must be greater than or equal to the values for equivalent gridsize lengths.

  • {dx dy dz} sets the lengths for a non-cubic cell. The corner cell length values must be greater than or equal to the values for equivalent gridsize lengths. The dx and dy values must be equal.

  • {[dx dy] baseunitcells [num_cells_per_baseunit]} creates non‑cubic corner cells only for EUV sources based on the thickness of the defined bilayer or multilayer. For more information, see the gridsize description of baseunitcells.

The limits for corner_gridsize are the same as gridsize.
