Model-Layer File Format

CM1 and VEB Model Formats

Defines a simpler setup file (containing just a model path and layer definition).

A model-layer file is used in place of a Calibre OPCpro setup file for CM1 and VEB calibration. It requires fewer parameters than a Calibre OPCpro setup file.

Input for: modelflow_v2 -mlfile and -mlstring arguments

Format

A model-layer file is separated into the following sections:

  • Models — A required section defining model paths and inline models.

  • Layers — An optional section defining layer and background parameters. If the Models section does not include the mask_model keyword, this section is required.

Each line of the model-layer file must begin with a keyword from the Parameters section, be part of an inline file (keywords beginning with “inline”), or begin with a comment character (#). No more than one parameter per line is allowed.

Models Section

Use this section to include all models (optical, flare, DDM, resist, etch, mask, and so on) used by the modelflow_v2 script. The syntax, keywords, and arguments include those accepted in both sparse and dense setup files, and can be mixed.

Options in this section are required if the modelflow_v2 stage that uses them is active.

[modelpath path]
[optical_model mask_number {foldername | ‘{’optical_model_string‘}’}]…
[opticalmodel {foldername | inline}]
[secondOpticalmodel {foldername | inline}]
[thirdOpticalmodel {foldername | inline}]
[topo_model filename | ‘{’ inline_model_string ‘}’]
[flare_model {filename | inline}[png pngfile]]
[mask_model [masknum dose] {filename | ‘{’mask_model_string‘}’}]…
[shadow_bias_model filename | ‘{’ inline_model_string ‘}’]
[ddm_model ddm_id filename |‘{’ inline_model_string ‘}’]…
[ddm_groupM name { filename | ‘{’ inline_group_string ‘}’]
[resistpolyfile {filename | inline}]
[vetchbiasfile {filename | inline}]
[inline_optical1 ‘{’ optical_model_string ‘}’]
[inline_optical2 ‘{’ optical_model_string ‘}’ ]
[inline_optical3 ‘{’ optical_model_string ‘}’ ]
[inline_flare ‘{’ flare_model_string ‘}’ ]
[inline_resist ‘{’ resist_model_string ‘}’ ]
[inline_etch ‘{’ etch_model_string ‘}’ ]

Layers Sections

The layers section describes the background and layers. It is required if you do not use the mask_model keyword and must follow the model input section.

The layers can be described in one of three formats: mask model format, “conventional” format, and “underlying” format. There is only one format for background.

  • Use mask model format when the layers are contained in a mask model. (See “Mask Model Format”.)

  • Use conventional format when you are working from older setup files.

  • Use underlying format for implant layers with topographical modeling.

[background {bgtype [factor]} … ]
#
# Mask model format
[layer layerNum mask maskNum INPUT mlNum]
#
# Conventional format
[layer num {trans [att]}|{real imag} [bias x y] [maskNum dose] [ddm name]
[map_layer_to_maskedge layernum maskedge_number]]
#
# Underlying format (topo modeling)
[underlying {active | poly | finfet} layernum ]

# Etch underlying format (veb underlying)
[layer layernum etch_underlying]

Parameters

Parameters (Models Section)

modelpath path

Defines the directory paths to search for models; separate multiple directories by colons (:). Required if any model is not inline.

optical_model mask_number {foldername | ‘{’optical_model_string‘}’}

Specifies the name of an optical model directory using a multiple mask format, where mask_number can be 0 to 2. This keyword can be used in place of the opticalmodel, secondOpticalmodel, and thirdOpticalmodel keywords.

If an inline optical model is specified using an optical_model_string, it must be enclosed in braces ({}).

opticalmodel {foldername | inline}

This option is only maintained for backwards compatibility; use optical_model 0 instead. Specifies the name of the optical model directory. If inline is specified instead of a filename, it indicates that the model is defined using the inline_optical1 keyword.

secondOpticalmodel {foldername | inline}

This option is only maintained for backwards compatibility; use optical_model 1 instead. Specifies the name of a second optical model directory. This keyword is designed for double exposure flows. If inline is specified instead of a filename, it indicates that the model is defined using the inline_optical2 keyword.

thirdOpticalmodel {foldername | inline}

This option is only maintained for backwards compatibility; use optical_model 2 instead. Specifies the name of a third optical model directory. This keyword is designed for implant modeling flows. If inline is specified instead of a filename, it indicates that the model is defined using the inline_optical3 keyword.

topo_model filename | ‘{’inline_model_string‘}’

Specifies the name of a topographical model or an inline string defining the model, which must be enclosed in braces ({ }). Contact your Siemens representative for information on topographical models.

flare_model {filename | inline} [png pngfile }]

Specifies a flare model file for use with the optical model, which can be inline. Can only be used if modelflow_v2 “dense” is specified in stage1 or stage2. If the calibration would otherwise not require stage1 and stage2, specify “stage2 dense try” explicitly to load the flare model.

If long-range flare data is included, use a .png file to specify a file representing the long-range convolution results. You can generate the file using RET FLARE_CONVOLVE, described in the Calibre nmOPC User’s and Reference Manual.

shadow_bias_model filename | ‘{’inline_model_string‘}’

Specifies the name of a shadow bias model or an inline string that must be enclosed in braces ({}). Shadow bias models are used in combination with optical models for calculating intensities of dense images.

Shadow bias models are not compatible with sparse aerial image simulations; they can be used only with modelflow_v2 stage1 dense or stage2 dense options specified.

mask_model [masknum dose] {filename | ‘{’mask_model_string‘}’}

Specifies the name of a mask model or an inline string which must be enclosed in braces ({}). There can be as many mask_model statements as the number of optical models.

Can only be used if modelflow_v2 “dense” is specified in stage1 or stage2. If the calibration would otherwise not require stage1 and stage2, specify “stage2 dense try” explicitly to load the mask model.

See the section “Mask Model Format” for more information.

ddm_model ddm_id filename

Optionally specifies a DDM model file. Cannot be used with mask_model; if mask_model is specified, supply DDM models via the mask model-format layer statement. Multiple ddm_model statements can be specified.

  • ddm_id — An alias for the specified filename.

  • filename — The DDM model file. The file must be created by masksim. It can be either a bright or dark output file.

Can only be used if modelflow_v2 “dense” is specified in stage1 or stage2. If the calibration would otherwise not require stage1 and stage2, specify “stage2 dense try” explicitly to load the DDM model.

ddm_groupM name {filename | ‘{’ inline_group_string ‘}’}

Specifies a group of dense DDM models, where M is the index number of the dense DDM group in the range of 0 to 4. The name you supply is later used to identify the group to link to a layer. DDM groups use the DDM Group Format.

resistpolyfile {filename | inline}

Specifies the name of the resist model file. Required if modelflow_v2 stage2 is active. If inline is specified instead of a filename, it indicates that the model is defined using the inline_resist parameter.

vetchbiasfile {filename | inline}

Specifies the name of the etch model file. Required if modelflow_v2 stage3 is active. If inline is specified instead of a filename, it indicates that the model is defined using the inline_etch parameter.

inline_optical1 ‘{’ optical_model_string ‘}’

Specifies an optical model definition string, which must be enclosed by braces.

inline_optical2 ‘{’ optical_model_string ‘}’

Specifies the second optical model definition string, which must be enclosed by braces.

inline_optical3 ‘{’ optical_model_string ‘}’

Specifies the third optical model definition string (used only for implant modeling), which must be enclosed by braces.

Tip

Optical models are described in the section “Optical Parameters File Format”.

inline_flare ‘{’ flare_model_string ‘}’

Specifies a flare model string, which must be enclosed by braces.

inline_resist ‘{’ resist_model_string ‘}’

Specifies a CM1 resist model definition string (described in the section “CM1 Model File Format”), which must be enclosed by braces.

inline_etch ‘{’ etch_model_string ‘}’

Specifies an etch model definition string (described in the section “Variable Etch Bias (VEB) Model File Format”), which must be enclosed by braces.

Parameters (Layers Section)

background bgtype [factor] [bgtype [factor]]…

Specifies the background transmission for the masks. Multiple backgrounds can be specified in a single background parameter. Cannot be used when the Models section includes mask_model.

  • bgtype — Required. Must be either clear, dark, or attenuated.

  • factor — Optional. Specifies the factor for attenuated backgrounds.

Mask Model Layer Format

layer layerNum mask maskNum INPUT mlNum

Specifies layer parameters. This format can be used only if a mask model is present.

  • layer layernum — Specifies the layer number in the layout file. It must not exceed 555.

  • mask masknum — Specifies the mask number or exposure index for the mask model that controls the layer. Use 0 if the mask model does not specify a number.

  • INPUT mask_model_layer_num — Specifies the layer number from the mask model to associate with the layer declaration.

Conventional Layer Format

layer num {trans [att] | real imag} [bias x y] [maskNum dose] [ddm name]

Specifies the layer number within the layout. An optional datatype can be specified as part of the layer number after a period.

  • num — The layer number. An optional datatype can be specified as part of the layer number using a period.

  • trans [att] | real imag — The transmission. This is a required argument. The transmission can be expressed as either a keyword and optional attenuation factor, or as real and imaginary components. This determines the optical transmission value of the layer.

    Accepted values for trans are attenuated, phase60, phase90, phase 120, phase180, or phase 270. The optional att argument is only valid when trans is attenuated.

  • bias x y — An optional keyword specifying the bias for vertical edges (x) and horizontal edges (y) in microns for the specified layer. This biasing computation is done while the simulator generates the aerial image.

  • maskNum dose — An optional pair of values that specify the number for the mask that corresponds to the layer statement and the energy applied to the mask. It is required when using the conventional layer format with multiple exposures.

    The default for maskNum is 0. Legal values are 0, 1, 2, or 3. Legal values for dose are between 0 and 1.

  • ddm name — An optional keyword that maps the layer to the specified DDM model. It can be either a DDM group name or a simple DDM model.

map_layer_to_maskedge layernum maskedge_number

Optional. Used for version 1 DDM only.

  • layerNum — Specifies a layer number from the layer list to apply the DDM library calculations to.

  • maskedge_num — Specifies a mask edge identifier number.

DDM modeling is described in the appendix “3D Mask Modeling With DDM and HHA”.

Underlying Layer Format (Topo Modeling)

underlying {active | poly | finfet} layernum

Optional. Used only for topographical modeling configurations that have multiple densities. This keyword is used to designate a layer in the layout that is associated with the underlying layer type (active, poly, or finfet) for the process.

Etch Underlying Layer Format (VEB Etch Modeling)

layer layernum etch_underlying

Optional. The etch_underlying layer specification is used by VEB models that have a specific underlying layer to associate with the designated layernum. This option is used with the modelflow_v2 etch_underlying argument and the Calibre OPCverify veb_simulate command. It has no relationship to the underlying keyword above, used for topo modeling.

Examples

Example Model-Layer Input File 

# model info
modelpath models
opticalmodel optical
ddm_model D1 ddm1.mod
ddm_model DDM2 ddm2.mod
flare_model flare_2.mod 209 210 211 215
vetchbiasfile etch.mod
resistpolyfile inline
inline_resist {
version 9
modelType CM1
DEFINE M0 k=0 n=1 +b=0.000000e+00 s=0.000000e+00 p=0
ANCHOR 0
COEFMAX 1.200000e-04
THRESHOLD 1.386654e-01
MCOEF 0 1.000000e+00
}
# layer info
background clear dark
layer 1 dark 0 1 ddm D1
layer 3 clear 1 1.1 ddm DDM2
layer 4 phase180 1 1.1
layer 5 dark 0 1 ddm D1

Example Mask Layer Declaration

This example shows a simple mask model with one mask and the related layer statements.

Given a mask model defined as follows:

version 2
background clear
mask_layer 0 TRANS atten 0.06 CX 0.0125 CC 0.017 BIAS 0 DDM hha_bright.ddm
mask_layer 1 TRANS atten 0.06 CX 0.0125 CC 0.017 BIAS 0 DDM hha_bright.ddm

The layer statements to assign mask_layer 0 to the layout layer 2 and mask_layer 1 to the layout layer 4 in the model layer file would look like the following code:

layer 2 mask 0 INPUT 0
layer 4 mask 0 INPUT 1

Example Usage in modelflow_v2

The following examples show how to use model-layer files with the modelflow_v2 command.

Example 1

In this example, the model layer file is specified directly on the command line as a string. Notice the braces enclosing the -mlstring model keyword block. Each model file keyword is on its own separate line (without a continuation character).

modelflow_v2 -m flat.oas -gf sample.gg \
	-mlstring {
	modelpath models
	opticalmodel simple
	resistpolyfile cm1_ctr.mod
	background clear
	layer 4 dark
	} \
output log out_2.log gfout sample_sim2.gg

Example 2

This example demonstrates the use of a Tcl variable to set the model-layer file string:

set str “modelpath models \nopticalmodel simple\nresistpolyfile cm1_ctr.mod\nbackground clear\nlayer 4 dark”
modelflow_v2 -m flat.oas -gf sample.gg -mlstring $str \
output log out_2.log gfout sample_sim2.gg

Example 3

This example demonstrates the use of -mlstring in command-line mode (inside a makefile) with Tcl code to generate the multiple-line string during execution. 

${CALIBRE_HOME}/bin/calibrewb -a modelflow_v2 \
	-m flat.oas \
	-gf sample.gg \
	-mlstring [ \
	join { \
		{ modelpath models } \
		{ opticalmodel simple } \
		{ resistpolyfile cm1_ctr.mod } \
		{ layer 4 dark } \
	} “\n” ] \
	output log out_2.log \
	gfout sample_sim2.gg

Example 4

This example shows the use of the model-layer file using inline optical resist models, and the accompanying modelflow_v2 command using mask models and DDM.

set mlstr {
    layer 0 mask 0 INPUT 0
    modelpath .
    opticalmodel inline
    mask_model mask.mod
    resistpolyfile inline
    vetchbiasfile inline
    inline_optical1 {

 	version 6
 	engine TCCcalc
 	opticalsystem 0.193 0.750
 	defocuslevels 1 0.000 0.020
 	magnification 4.
 	approxorder 10
 	hoodpix 0.96
 	kerngrid 0.01
 	illumtype COMPOSITE
	ANNULAR ERF weight 0.528 sigma 0.864 sigma_in 0.540 \
		dsigma 0.247 dsigma_in 0.446
 	imagediffusion 0.020000
	ddm_model approx
	forcePower2LUT 0
	vectormodelflag 1
    }

    inline_resist {
	version 9
	modelType CM1
	ANCHOR 0
	COEFMAX 1.2E-4
	THRESHOLD 0.3011
	DEFINE M0 k=0 n=1 +b=0 s=0
	MCOEF 0 1
    }
    inline_etch {
	version 1
	SKERNEL D1 type=gauss s = 300 u=-20
	SKERNEL D2 type=visible s = 300 u=20
	btermCount 3
	BTERM -20
	BTERM 20 D1^1
	BTERM 20 D2^1
    }
}
modelflow_v2 -l $l_obj -gf gauges.gg -mlstring $mlstr \
    -turbo output log out.log skipddminlog \
    stage1 dense threshold auto search front explore 7 \
       na 0.72 0.78 1 \
       mask_model_layer0_cc 0 0.02 3 \
       mask_model_layer0_cx = {$mask_model_layer0_cc} 0 0.02 \
    stage2 dense search full \
       s0 0 25 33 \
    stage3 search full \
       es1 50 150 33
