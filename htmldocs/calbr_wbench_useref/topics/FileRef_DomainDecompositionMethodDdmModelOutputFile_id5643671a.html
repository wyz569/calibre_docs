Domain Decomposition Method (DDM) Model Output File

Optical Model File Formats

Domain decomposition method (DDM) model output files primarily contain the near electromagnetic field edge diffraction information that is needed in order to apply the modeling effects of light scattering for three-dimensional mask topography. Each DDM model is associated with one or more layers in the setup file.

FileRefTopic-classic-Formats

For instructions on the use of DDM model output files, refer to “3D Mask Modeling With DDM and HHA”.

The output file is a text file with keywords on individual lines. It is automatically generated from a DDM Library Input File Format file processed by the Calibre MaskSim tool and may be in one or two files (depending on whether a brightfile and/or darkfile output was specified in the input file).

Note:

If you are not using Calibre MaskSim, you must create the header file yourself at the top of your data file.

Note the following general rules:

  • Empty lines are allowed anywhere in the file.

  • Keywords precede most numerical settings.

  • Any line that starts with a pound sign (#) is a comment.

FileRefTopic-classic-Arguments

modeltype {ready | universal}

This is a required setting that needs to exist in the beginning of the file. Both “ready” and “universal” signify that the EM simulations have been completed offline, but “universal” libraries have many signals covering the entire source pupil. Universal libraries can only be used as input to the interpolateddmsectors function, which generates a ready library, or as a crosstalk_ddm_shortcutfile or corner_ddm_shortcut_file, which generates a universal library.

version num

This is a required keyword. Starting in 2020.3, all new DDM model output files must use version 7 (the default). Version 7 improves accuracy by improving the orders calculation for signal calculation.

object_plane val_in_um

An optional keyword valid only for DDM version 4 and later libraries that specifies the mask-side focus object plane for simulation. When a single value is specified, val_in_um is considered to be measured with respect to the end of the last film (farthest film from the substrate; this is the bottom film), with positive values being in the direction moving away from the substrate and towards the wafer (downwards). (Default is 0um).

domain [spatial | fourier]

Specifies whether the edge diffraction fields are given in the spatial or fourier domain. Currently only the spatial type is supported. (Default is spatial.)

reduction_factor val

An optional keyword that specifies the reduction factor from mask to wafer dimensions; the electromagnetic simulation of the mask edge must take place at the mask dimensions, and the results are converted to wafer dimensions. For 4X projection systems, specify a value of 4.

efieldtype [complex]

Specifies that the ddm signals (electric fields) are complex values (previous versions allowed for “real”). (Default is complex.)

edge_location val

The location of the mask edge from the start of the signal in microns. Note that the first point of the signal is assumed to be at x=0um. Also, note that the edge location must be specified in mask dimensions. This is a required keyword.

signal_limit_bg val

This parameter limits the effective length of the ddm signal on the background side of the edge to the specified value in microns. Note that this parameter must be specified in mask dimensions. This is a required keyword.

signal_limit_lr val

This parameter limits the effective length of the ddm signal on the layer side of the edge to the specified value in microns. Note that this parameter must be specified in mask dimensions. This is a required keyword.

small_edge_limit val

Edges in the design that are smaller than this value in microns do not receive the additional ddm signal around them — effectively they are ignored from DDM. This setting may be used to achieve some speedup in the image simulation at the expense of reduced accuracy by ignoring small edges. Note that only the additional ddm signal is not added to the mask transmission for small edges, whereas their normal contribution to the mask transmission is accounted for as usual. (Default is 0um (all edges receive the ddm signal).)

This option does not control crosstalk modeling. To ignored small edges in crosstalk, use small_edge_limit_crosstalk, documented in “Cross Talk Model Output Format”.

fullddm [ yes | no]
Note:

For releases previous to 2017.2, the fullddm setting needed to match the ddm_model setting used to create the optical model. Starting with the 2017.2 release, the fullddm setting in the DDM library is ignored, and is set automatically based on the ddm_model setting in the optical model.

This is an optional setting that can trigger a special mode of DDM simulation.

If the setting of this parameter is “yes”, it means that the regular full DDM simulation is performed. The full DDM simulation requires two response signals for linear polarizations of the electric field parallel and perpendicular to the mask edge.

If the setting of this parameter is “no,” the intensity computation is done in one step, whereby a single signal is used for all edges regardless of incoming electric field polarization.

reverse_signal [ no | yes ]

The convention used in DDM models is that the left side of the signal must correspond to the layer side of the layout (mask) and the right side of the signal must correspond to the background side of the layout. This flag is useful for using the same DDM library for both mask polarities (darkfield and lightfield). In this manner a DDM library generated for a lightfield binary mask could be used on a darkfield binary mask by simply adding the reverse_signal keyword. By setting this flag to “yes” the program is instructed to do the appropriate signal flipping internally before applying the ddm signals to the respective layer (eliminating the need for you to reverse the signal by other means). (Default is no.)

Note:

The reverse_signal keyword cannot be used with HHA optical models; the DDM library must be generated with the correct tone in these cases.

numpts num

The number of simulation points. This is a required keyword.

stepsize val

The pixel size in microns of the ddm signal or equivalently the distance in microns between consecutive points. Note that this value must be specified in mask dimensions. This is a required keyword.

chief_ray theta theta_val phi{ phi_val | {‘[’phi_list ‘]’ center_slit_phi phi_list_value }}

Specifies the off-axis and azimuthal angle (in degrees) of the direction that is to be used by default for simulation. The phi value can alternatively be a list that represents the angles of incidence for the source, with one list element designated as the center slit with the center_slit_phi argument.

If a phi_list is supplied, it must adhere to these restrictions:

  • The entire list must be enclosed in brackets (“[ ]”). A single value not enclosed in brackets reverts to the single phi_value behavior above.

  • List values must be in increasing order, and separated by commas.

  • List values are relative to the angle of incidence, and are listed from left to right, corresponding to the EUV slit aperture.

  • List values must either be single integers or a start:step:end range (for example, [‑1:2:10] generates the values -1,1,3,5,7,9).

  • List entries can have mixed range and single values (for example, [1:2:10,15, 19,20:3:30]).

  • The “center_slit_phi phi_list_value”argument must also be added to specify one of the listed angles as the main phi (center slit) value.

verbosity snapping

Specifies that during HHA runs, if the program cannot find a DDM signal at the desired angle and polarization (within tolerance), Calibre WORKbench reports this fact in an INFO message, along with the angle and polarization of the signal that it actually used. If this keyword is not specified, then the program will still choose another signal, but without informing the user.

normalize_clearfield real imaginary

This optional keyword is only used for modeltype “ready”. It specifies the value to which the bright electric field signal will be normalized to a specified value.

First of all, Calibre will search to see which DDM signal has the closest theta and phi values to the chief ray, as specified by the chief_ray keyword.

For the closest theta and phi represented, if there is an UNPOLARIZED signal or if both TE and TM signals exist, then the UNPOLARIZED signal is used (usually meaning the average of TE and TM). If on the other hand there is only TE or only TM, then the TE or TM signal for that theta and phi is used.

The value to be normalized is then the layer_trans or background_trans value for the signal described above, depending on which is brighter (larger complex magnitude). That value is scaled to equal the value specified in normalize_clearfield, and all other layer_trans, background_trans, and electric field values in the DDM file are scaled correspondingly. This scaling is done during the run and is not saved to the DDM file.

This is an optional keyword. A value pair of 1 0 is recommended; for example, if the background is clear and the DDM model states that the background transmission is 1.3 0, then the normalization factor is 1/1.3. Note that the values in the DDM file override values in the setup file.

chief_ray_bright_trans real imaginary

An optional keyword that modifies the normalize_clearfield keyword. If chief_ray_bright_trans is specified, normalize_clearfield normalizes the chief ray rather than the angle closest to the chief ray. That is, chief_ray_bright_trans uses the transmission of the chief ray, even if no signals are at the chief ray angle. This makes it easier to compare DDM models.

layer_trans real imag theta offaxis_angle_in_degrees polarization {TE | TM}

A required keyword that specifies the transmission value of the layer side of the edge, as above, but for the illumination specified. This keyword may be specified as many times as desired, for different illumination conditions. In general there should be one layer_trans line for each distinct theta and polarization represented in the file. Both real and imaginary values must be specified; real-only DDM signals are not supported.

background_trans real imag theta offaxis_angle_in_degrees polarization {TE | TM}

A required keyword specifying the transmission value of the background side of the edge, as above, but for the illumination specified. This keyword may be specified as many times as desired, for different illumination conditions. In general there should be one background_trans line for each distinct theta and polarization represented in the file. Both real and imaginary values must be specified; real-only DDM signals are not supported.

Data Blocks

A DDM model output file contains one or more data block sets using the general format:

keyword { 
datavalues
…
}

Each data value set must be on its own line. At least one data block is required as part of the output file, and it must follow the general format for that type as shown in Table 1.

Table 1. DDM Output File Data Block Requirements

Keywords Required

Data Values Required

One or more of efield_left,efield_up, efield_down, efield_right, efield_nw, efield_ne, efield_se, or efield_sw

real imag only
efield_left theta offaxis_angle_in_degrees phi azimuthal_angle_in_degrees polarization {TE | TM | UNPOLARIZED}

Signifies the start of the scattered electric field, for a vertical edge on the left side of the layer, under the specified illumination condition. Here theta is the angle in degrees that the incident illumination is from vertical, whereas phi is the angle in degrees measured counterclockwise from the x-axis. If theta is zero, then phi must also be zero, and in this case, by convention:

  • TE means that the electric field is polarized along the y-axis, when phi = 0

  • TM means that the electric field is polarized along the x-axis, when phi = 0

  • Normally, the polarization will be either TE or TM. UNPOLARIZED signals are computed automatically by the program.

    Note:

    There are also similar keywords efield_up, efield_right, efield_down, efield_nw, efield_ne, efield_se, and efield_sw, with identical syntax, corresponding to the other seven possible horizontal, vertical, and 45-degree edges.

  • In HHA runs, the simulation engine will use the signal present in the file whose angle is closest to the actual illumination angle being simulated (distance is taken to be Euclidean distance in k-space). All needed edges and polarizations must have signals corresponding to the theta and phi of the nearest signal represented in the file; otherwise, a runtime error will be generated.

    In non-HHA runs, identical remarks apply, except that the signal chosen is the one closest to the angle given by the chief_ray keyword, documented as part of the “Domain Decomposition Method (DDM) Model Output File”.

    If the 45-degree edges are not given, the program will infer their signals by averaging those from the adjacent horizontal and vertical edges. Therefore, if this approximation is considered acceptable, it is allowable to run masksim with “compute_diagonals 0” to produce the DDM library, even if there are 45-degree edges that one desires to simulate.

    The values of the electric field along the edge follow in the next lines.

real imag

A required parameter specifying pairs of (real, imag) numbers for the complex scattered electric field along the edge. One pair is given per line. The number of pairs must equal the number that was specified with numpts. The first pair must follow immediately after the opening brace (“{“) of the respective efield_par, efield_per or efield line and the last pair must be followed by a line with just the closing brace (“}”). This is a required argument, and in a typical case these lines will comprise the majority of the file.

The closing brace (“}”) signifies the end of the scattered electric field values along the edge.

Example 2

If you add the following line to the DDM model of Example 1:

reverse_signal yes

The effect of this action is a one to one exchange of the layer and background sides of the ddm signal. Since by convention the left side of the signal must correspond to the layer side and the right side to the background, the previous example is appropriate for simulating a clear field mask (clear background). In order to re-use the same DDM model, but apply it to a dark field mask (dark background), the signal must be reversed (either explicitly or by using the reverse_signal keyword).
