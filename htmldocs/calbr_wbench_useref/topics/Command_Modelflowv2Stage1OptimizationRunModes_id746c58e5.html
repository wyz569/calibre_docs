Arguments

spatialtcc

Optionally replaces the input TCC engine with a spatialTCC engine. It cannot be set if the nominal model or optimization parameters include any of the following non-zero Gaussian diffusion length keywords:

  • kerneldiffusion

  • imagediffusion

  • imagediffusions1

  • imagediffusions2

The spatialTCC engine is used during optical calibration only. The default engine is used for all other parts of the simulation.

For spatialTCC results to agree with the standard TCCcalc engine, the dimensionality of the TCCmatrix N must be greater than or equal to 20:

N = 1 + 2*int(hoodpix*na*(1+sigma)/λ)
thresholdstability value

Optionally sets a percentage threshold tolerance where all geometries crossed by gauges are supposed to remain resolved without printing extra features or holes. If set, it tells the tool to penalize, within the specified range of threshold values, any gauge that fails to resolve at all, or which resolves with printing extra features that were not intended for printing. A “penalized gauge” has its associated simulated CD set to zero so that the contribution of the penalized gauge to the total RMS error becomes proportional to the CDmeas squared.

This option is designed for gauge-based model calibration and has no impact on contour-based calibration, which involves a small number of dummy gauges.

This option only penalizes the gauges that do not print precisely at the threshold. (Recommended value if thresholdtolerance is not used: 4 percent)

thresholdstability supports any search method, and directly verifies the model stability.

This option cannot be specified if thresholdtolerance is specified.

thresholdtolerance value

Optionally sets the percentage for threshold tolerance, a nonnegative quantity for the range of threshold values over which equality-constrained gauges are supposed to be resolved correctly. If set, modelflow_v2 computes simulated CDs not only at a specific threshold (th) but also at two more values, (th - delta) and (th + delta), where delta = 0.01 * value * th. The gauge is considered to be resolved correctly if it prints at all of the three threshold values, and if the simulated CD remains a linear function of the threshold within the specified tolerance. If the latter is not the case, the gauge is penalized by setting the associated simulated CD to zero, such that the contribution of this gauge to the total RMS error becomes proportional to CDmeas squared.

This option is a more robust alternative to thresholdstability. Both are designed to filter out unstable models during calibration, but the implementation of these options is different.

This option cannot be set if thresholdstability is specified. The recommended value is 1.5.

dense

Optionally uses a dense optical routine to generate aerial image grids for stage 2 calibration. The dense engine uses a square coarse (at the Nyquist rate) aerial image grid over the area of interest and interpolates the grid intensity at the specified control points. It operates at a linear speed, and can be faster than the default sparse engine for large numbers of gauges (> 1000).

Note:

In CM1 Center only, selecting stage1 (Optics “O” or “T” buttons) automatically outputs “stage1 dense” to the output, which makes dense the default mode for CM1 Center.

fastfocus [value1 value2 value3]
Note:

This option requires a super gauge data file to use.

Optionally speeds up optical calibration for multiple process conditions with four or more different defocus values. When set, modelflow_v2 uses a three-point parabolic approximation (instead of the full process condition set) in defocus for computing AI grids of off-focus gauge replicas. This approximation is active during calibration only; final error statistics and simulated CDs are computed without it.

value1, value2, and value3 are optional sub-arguments that explicitly set defocus values corresponding to a gauge sets matching those defocus values to be used for the parabolic approximation. The default defocus values used are the minimal and maximal defocus value in the super gauge data file, and an intermediate value between them.

fasttopo value

Specifies what type of topo model signal calibration is to be used. Fasttopo 2 is the default setting starting with the 2016.2 release. Accepted values are the following:

  • 0 — Selects slow calibration, where both aerial image cutlines and topo signals are generated at each iteration.

  • 1 — Selects fast calibration, where aerial image cutlines are generated in advance and only topo signals are generated at each iteration.

  • 2 — Selects internal calibration of the topo signal’s linear and quadratic terms using a gradient descent method. The following topo signal linear terms are calibrated in this mode if they are specified in the input topo model. Their equivalent topomf forms are ignored.

    cterm

    fterm

    fwterm

    gterm

    iterm

    sterm

    stermim

    tterm

    ttermim

    t2term

    t4term

    vterm

    so2term

    to2term

    t2o2term

    t4o2term

    The following topo signal quadratic terms are calibrated in this mode if they are specified in the input topo model:

    cquadratic_mult

    squadratic_mult

    squadratic_mult_im

    Because the preceding optimization linear and quadratic options use implicit mode, the following explicit mode options cannot be specified when modelflow_v2 is in fasttopo 2 mode:

    topo_cterm_N

    topo_fterm_N

    topo_fwterm_N

    topo_gterm_N

    topo_iterm_N

    topo_sterm_N

    topo_stermim_N

    topo_tterm_N

    topo_ttermim_N

    topo_t2term_N

    topo_t4term_N

    topo_vterm_N

    topo_cquadratic_mult

    topo_squadratic_mult

    topo_squadratic_mult_im

Optimization results obtained with this flag set to 0 or 1 or 2 can differ from each other. The fasttopo option can be set only if topo model signals are to be optimized.

topoiterations value

Optionally specifies the max number of iterations in the loop for calibrating topo parameters internally. Must be a positive integer. This option can be set only if dense is specified and fasttopo 0 or 1 is not specified. The default is 2000.

search {full | gradient | newton | front | zoom | frontzoom}

Sets the optimization algorithm (required if the search interval for at least one independent parameter is specified).

  • full means full factorial search over all possible combinations of parameter values. The latter are determined by begin, end, and num_levels, where begin and end are the lower and upper limit of the parameter interval, while num_levels is the number of parameter values to be used in full factorial search. These values are generated on a regular grid using the formula:

    value = begin + (i - 1)*step, i = 1, ... , num_levels
step  = (end – begin)/(num_levels – 1), if num_levels > 1

    or

    value = (end – begin)/2,  if  num_levels  = 1

    The full search can use as many CPUs as are available if multi-threading (-turbo) or distributed processing (-remotefile and -remotestring) options are specified.

  • gradient is a straightforward implementation of the method of local variations. During the search, the optimization moves each parameter forward and backward by the associated step size and then chooses the parameter value corresponding to the minimal CD DIFF RMS. If explore is explicitly set to 0, the input model is used as a starting point of the newton search. Otherwise, exploration runs are performed to generate the starting point. The number of exploration runs is determined by explore (or its default value if explore is not specified). If explore is greater than or equal to the product of level numbers (num_levels), the full factorial exploration search is performed. Otherwise, a partial factorial search is performed. The epsilon value does not affect the gradient search.

    Although the full and gradient search are both performed with fixed values of the parameter steps (see above), they can converge to different results. The gradient search can be by orders of magnitude faster than the full search, if the total number of parameter combinations to be explored is large.

    The gradient search can use a maximum of 2 CPUs if multi-threading (‑turbo) or distributed processing (‑remotefile and ‑remotestring) options are specified.

  • newton is for quasi-Newton optimization. Unlike the gradient and full search which are both performed on a regular grid, the quasi-Newton search is done with varying self-adaptive parameter steps that are automatically adjusted after each iteration. If explore is explicitly set to 0, the input model is used as a starting point of the newton search. Otherwise, exploration runs are performed to generate the starting point. The number of exploration runs is determined by explore (or its default value if explore is not specified). If explore is greater than or equal to the product of level numbers (num_levels), the full factorial exploration search is performed. Otherwise, a partial factorial search is performed. The optimization process is limited by simple constraints determined by the begin and end values of the parameter intervals.

    The newton method can use a maximum of 2Nvar CPUs if multi-threading (-turbo) or distributed processing (-remotefile and -remotestring) options are specified, where Nvar is the number of independent variables involved in calibration.

  • front is used for global optimization. Unlike the gradient and newton search, which are both local and therefore depend crucially on the starting point (initial guess for optimization variables), the front search is done on the whole parameter space, and does not require a good initial guess and/or large number of exploration runs. The optimization process is limited by the constraints determined by the begin and end values of the parameter intervals. The number of iterations to be performed is specified using the explore or runtime values. If explore and runtime are not set, the number of iterations is limited to 1000 by default. The number of levels (num_levels) and epsilon value have no effect on the front search.

    The front search method uses a maximum of 32 CPUs if multi-threading (-turbo) or 128 if distributed processing (-remotefile and -remotestring) options are set. However, the actual scaling is effectively limited to between 4 and 20 CPUs with the current implementation; Siemens EDA therefore recommends a setting of 20 CPUs for stage 1 and 4 CPUs for stage 2 if you are using the front or mtfront search method even if you have more CPUs available.

  • zoom is a local optimization routine. Unlike gradient, the zoom search is performed on a self-adaptive grid, and it does not involve numerical differentiation as is the case for the newton search. The search is limited by simple constraints determined by the begin and end values of the parameter intervals. It stops when a certain convergence criterion controlled by epsilon is met. If explore is explicitly set to 0, the input model is used as a starting point of the zoom search. Otherwise, exploration runs are performed to generate the starting point. The number of exploration runs is determined by explore (or its default value if explore is not specified). If explore is greater than or equal to the product of level numbers (num_levels), the full factorial exploration search is performed. Otherwise a partial factorial exploration search is performed. The MT scalability of zoom is that the zoom search can use no greater than Nvar CPUs where Nvar is the number of independent variables involved in calibration.

  • frontzoom is a combination of a global (front) and local (zoom) optimization. The former is used to explore the search space and find a good initial guess (starting point) for the latter. The number of front iterations is determined by the explore or runtime options. If neither totaliteration nor totalruntime is set, the zoom search stops when a certain convergence criterion controlled by epsilon is met. The smaller the epsilon value the greater the number of zoom iterations required for convergence.

    If totaliteration is specified, the zoom search stops if the convergence criterion is met, or if it performs totaliteration - explore iterations, whichever comes first. In this case, the total number of frontzoom iterations done is less than or equal to the value specified by totaliteration.

    If totalruntime is specified, the zoom search stops if the convergence criterion is met or if the zoom runtime exceeds the difference totalruntime - runtime, whichever comes first. In this case, the total runtime of the frontzoom search is less than or equal to the value specified by totalruntime.

    The number of levels that must be specified for each independent optimization parameter (num_levels)have no effect on the frontzoom search. The MT scalability of frontzoom is determined by that of the front and zoom searches.

Note:

For a given level of accuracy, which is determined by steps for the gradient and full search and by epsilon for the newton search, the newton search is the fastest of these three methods.

Newton optimization will fail if an invalid combination of optical parameters occurs, such as when def_start is out of the resist stack or sigma_in is larger than sigma_out. You should be careful when setting limits for optimization variables. Make sure that any point inside the cube defined by the begin and end values constitutes a correct model.

explore integer

Determines the maximal number of iterations to be performed in the front search (Default: 1000) or the gradient and newton search exploration stage. (Defaults:128 in 2006.4, 20000 since 2007.1.) This option cannot be set if runtime is specified.

This stage precedes the gradient and newton optimization processes. To skip exploration simulations and start newton search with the nominal parameter settings, set explore to 0. The parameter values to be used in the exploration stage are generated in the same way as in full search. The maximal number of possible parameter combinations (exploration runs) Nmax is given by the product of the num_levels values specified. If this number is smaller than the explore number, only Nmax exploration runs will be performed. Otherwise, a subset of the explore size will be randomly chosen of the total Nmax combinations to be used in the exploration stage.

There is a limit of 80000 (8e+04) for the front search (4e+04 in versions previous to 2007.3) and 10000000 (1e+7) for the gradient and newton searches.

totaliteration integer

Determines the total number of iterations to be performed in case zoom or frontzoom is specified. (Default = run the zoom search until the convergence criterion is met) This option can be set only if the zoom or frontzoom search is specified. The totaliteration value must be greater than the explore value, and it cannot be set if runtime or totalruntime is present.

runtime hours

Specifies the runtime in hours after which optimization is to be stopped.

This option cannot be set if explore is specified, and is only used for the front search. If runtime is set, the front optimization is stopped after the specified amount of time or 1000000 iterations, whichever comes first. (Default = stop the front search after the maximal number of iterations specified by the explore value)

totalruntime value

Specifies the total runtime in hours after which frontzoom optimization is stopped. (Default = run the frontzoom search until the convergence criterion is met.) This option can only be set if runtime is present and the frontzoom search is specified. The totalruntime value must be greater than the runtime value, and it is not compatible with the explore and totaliteration options.

Using the Independent, Constrained, and Dependent Optimization Parameters

There are three types of optimization parameters – independent, constrained, and dependent. These parameters are abbreviated in this reference page as optim_set.

Independent

Variations of an independent parameter are determined by three numbers (begin, end, and num_levels) that specify the search interval [begin,end] and the number of parameter values (num_levels) to be used in a full factorial exploration run, such as:

sigma_out 0.6 0.8 5

If “auto” is used in place of begin or end, modelflow_v2 selects the value automatically using best practice recommendations. For example:

beamfocus_ex2 auto auto 10

“auto” can be specified for the following parameters:

  • beamfocus and beamfocus_exn

  • def_start and def_start_exn

  • hbias_dj and hbias_sj

  • sn

  • shrink_radiusj

  • swbias_d and swbias_s

  • vshrink_s

Constrained

Variations of a constrained parameter are determined by an inequality symbol, Tcl expression and three numbers (begin, end, and step) that specify the search interval using the following syntax:

		param inequality_symbol tcl_exp begin end num_levels

Accepted inequality types are < (less than), <= (less than or equal to), > (greater than), or >= (greater than or equal to).

begin and end specify the start and end values of the search interval.

Starting with the 2009.4 release, adding a mod prefix to an equality (mod <, mod <=, mod >, or mod >=) causes modelflow_v2 to treat it as a double inequality (a parameter that can be both positive and negative, such as def_start and beamfocus). The specified inequality constraint will be applied to the module of the optimization parameter rather than the parameter.

tcl_exp must be an independent parameter, enclosed in curly braces.

Constrained variable optimization is only available starting with the 2009.2 release and only works with the front search.

Dependent

Variations of a dependent parameter are determined by a Tcl expression and two numbers tcl_expression begin end; the two numbers specify the search interval, while the Tcl expression specifies the relationship between this parameter and other independent parameters involved in optimization. The Tcl expression must be enclosed in curly brackets, and include only independent parameters.

Description

Definitions

  • CTR — Constant Threshold Resist Model of type CM1.

  • Nominal optical, resist, and etch models are the initial models specified in the setup file.

  • optim_set — Defined as one of the optimization expressions as described in Using the Independent, Constrained, and Dependent Optimization Parameters of this topic.
