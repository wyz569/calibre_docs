VT5 Resist Model Commands List

This command creates an optimized VT5 model and an optimized optical model (or maskwriter model) using a fixed model optimization procedure that begins with a pair of “nominal” models.

Usage

modelflow
   {-ss spreadsheet | -sss superspreadsheet}
   [-f setup] {-m gds | -l layout}
   [-biasfirst]
   [-foptions parallel] [-ftolerance normtol]
   [-force][-force_symmetry]
   [-vss verification_spreadsheet]
   [-csf csffile -semweight weight]
   [{-cli clifile | -clistring cstring} [-contourweight weight]]
   [-turbo | -remotefile rfile | -remotestring rstring]

Arguments

-ss spreadsheet | -sss superspreadsheet

Required argument supplying the path to the spreadsheet (‑ss) or super spreadsheet (‑sss) file. Calibre WORKbench uses super spreadsheets for calibration over a process window. See the procedure “Calibrating Over a Process Window Using Super Spreadsheets” for more information.

-f setup

Optional argument supplying the path to a Calibre OPCpro setup file. (Default = use already loaded setup).

If ‑f is specified, modelflow uses the input setup file data, and then deletes the information (not the file) along with the input and optimized models after the modelflow run completes. This is the normal behavior previous to 2011.2

If ‑f is not specified, modelflow attempts to use the global (previously loaded using openlib or in the Calibre WORKbench GUI) setup. In this case, after the run is done, modelflow keeps the optimized models and deletes the input models if the run is successful, or deletes the failed optimized models if the modelflow run fails.

Setup files must obey the following rules:

  • gridsize must match the layout database unit.

  • siteinfo must be of SAMPLE type.

  • The minimum length of the site, which is (num - 1) * spacing, must be greater than 0.64 um. Since num cannot be greater than 256, the smallest spacing allowed is

    0.64 / (256 - 1) = 0.0025 um.

    However, enabling the environment variable CWB_SITEINFO_NOCHECK disables the check if the number of points in site is sufficient (lifts the restriction on the minimum site length) and thus allows ones to use spacing less than 0.0025 um.

  • The layer info should include an OPC layer where EPE measurements are performed. This OPC layer must be a target layer in the sense that all its geometries should look the way you want them to be printed.

  • The modelflow command does not support OPC layers that involve geometries with negative assist features (geometries with a thin gap or hole inside) because in this case the tool substantially overestimates EPEs (underestimates simulated CDs). A layer involving geometries with negative assist features should be specified as external, in which case the AI simulation will be performed using only this layer while all other layers including OPC will be ignored.

    Another way to resolve this problem is to introduce a visible layer that includes negative assist features only.

-m gds | -l layout

One of either ‑m or ‑l are required even if ‑f is not supplied.

This argument and value pair supplies either the path to the GDSII test pattern file or the Tcl handle of the layout containing the test pattern data.

-biasfirst

An optional keyword instructing the software to calculate the constant offset (epeBias) that delivers the best threshold polynomial fit.

The -biasfirst option does not work with the -sss, -cli, and -csf options.

–foptions parallel

An optional argument that affects the front algorithm. It improves the scalability of the search by adding more front candidates. This argument can be set only if ‑turbo or ‑remotefile or ‑remotestring is specified. It is intended to make the front search 100% scalable after 5xN iterations where N is the number of CPUs and remotes available. In order to achieve this, the number of jobs to be done simultaneously is forced to be a multiple of N.

–ftolerance normtol

An optional keyword that specifies the norm tolerance. The norm tolerance is a nonnegative number, 0 <= normtol <= 1, which controls the accuracy of the front optimizer. (convergence criterion). Setting this parameter to a value greater than FLT_EPSILON (which is the default) prevents the optimizer from digging too deeply into the lowest local minimum and thus forces it to better explore other local minima.

-force

An optional keyword that specifies that sites that are not centered in the middle of an edge are still considered valid. By default, this option is unset, and modelflow exits with an error if it encounters sites that are not valid, such as not being centered in the middle of an edge.

-force_symmetry

An optional keyword that forces kernel symmetry of the final optical model. This goal is reached by increasing the number of kernels used. This option does nothing if no optical parameters are selected for optimization in stage1.

-vss verification_spreadsheet

An optional argument that verifies the optimized model after Stage 3a completes, using the supplied verification_spreadsheet.

-csf csffile

An optional argument specifying a calibration session file (.csf). This argument does not need to be specified if a .csf file was previously loaded using the scalfit or scalcontour commands. If you are using a super spreadsheet, it must contain a spreadsheet containing best process conditions, since SEM contours are assumed to be taken at dose = 1, defocus = 0.

-semweight weight

An optional argument specifying the weight in the objective function for the SEM contours. The CD measurements from the standard spreadsheet file are given a weight of 1‑weight. modelflow optimizes the total RMS T:



where w is ‑semweight, F is the SEM fitness function, described in the scalfit command, and S is the EPE RMS of the CD measurements from the sample file. When w=0 or if csffile is absent, the SEM fitness is ignored, and modelflow calibrates the model from CD measurements.

Default is 0.25.

-cli clifile

An optional argument specifying a Contour Layout Information file (.cli), described in “Importing Contour Calibration Files”.

-clistring cstring

An optional argument and alternative to the ‑cli argument. Allows for the specification of contour layers in a CLI string to be used to calculate error RMS. This argument cannot be used if ‑biasfirst is specified. The default is no contour layer information being used.

-contourweight weight

An optional argument specifying the weight in the objective function for the SEM contours. The CD measurements from the standard spreadsheet file are given a weight of 1‑weight, similar to using ‑semweight.

-turbo

An optional argument used for multi-threaded calibration. Parallel execution on multiple CPUs is used to create optical models for stage1 calibration regardless of the search method chosen. The number of threads used is dependent on the CPUs available on your machine and the MT scalability of the search method chosen with a maximum of 32 by default.

You can specify and limit the number of threads used by adding the ‑threads option to your Calibre WORKbench invocation string. For example:

calibrewb -threads 2 modelflow.tcl

The threads used will be limited to two when running the modelflow command in the modelflow.tcl file, even if the machine has 4 CPUs.

The use of ‑turbo does not require additional licenses. This option cannot be used if ‑remotefile or ‑remotestring is specified.

-remotefile rfile | -remotestring rstring

Optional arguments that input a remote configuration file (rfile) or string equivalent (rstring) in order to implement distributed processing during optimization of the optical model in stage1 calibration. The maximum number of CPUs allowed is 128.

The use of these options does not require additional licenses. Distributed processing is off by default.

Description

Tip

Nominal models are the initial models, specified by the setup file. In this case, the models are the optical model and the VT5 model used as a starting point for creating the optimized models.

The nominal models provide default values for parameters that are not optimized. modelflow evaluates variations to the nominal models, as specified by the command line. It optimizes parameters in distinct optimization stages that are performed sequentially.
