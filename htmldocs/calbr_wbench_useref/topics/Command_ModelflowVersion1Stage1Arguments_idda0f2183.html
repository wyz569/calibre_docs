Arguments

Stage 1 Arguments (Maskwriter)

In maskwriter mode, Calibre WORKbench performs optimization on a maskwriter model instead of the optical model. This mode uses a special optical model with customized kernels to model mask proximity, using the supplied Gaussian lengths to tune a triple Gaussian function.

Calibre WORKbench tunes the maskwriter model by changing the following parameter:

maskwriter_model g1 optim_set g2 optim_set g3 optim_set

Sets maskwriter model optimization based on a triple-Gaussian convolution. The three specified Gaussian lengths (sigma) in microns are fitted to mask proximity data using a full factorial search by default. As of version 2009.2, all constrained optimization parameters are available for maskwriter model fitting. modelflow calibrates the model to CD data from the spreadsheet file (-ss option).

A search interval (optim_set, described in Dependent, Constrained, and Independent Optical Model Variables) must be specified for all three Gaussian lengths.

This optimization is off by default if it is not specified.

CAUTION:

The maskwriter model assumes a fixed aerial image threshold of 0.5. If your maskwriter modeling data assumes a different aerial image threshold, you may end up with false errors in the results.

Note:

Stage2 must be turned off to use maskwriter mode.

Stage 1 Arguments (Optical Model)

stage1 [off]

Stage 1 tunes the optical model using the empirical data. In this stage, the optical model is varied by changing various parameters. For each variation of the optical model, an optimized CTR (resist) model is used to determine the EPE RMS fitness of the model. Stage 1 is off by default, if it is not specified.

filter filter

An optional spreadsheet filter for this stage only. Default = use entire spreadsheet file.

exclude_2D_geometry

An optional argument that generates an optical simulation, then filters out all the points that require 2D optical simulation (such as line ends and T-junctions) from the spreadsheet. The spreadsheet points that are kept for stage 1 optimization are reported in the log file.

optics_model_1D

An optional argument that excludes all spreadsheet points that require 2D optical simulation and then performs all the optimization steps of Stage 1 using only 1D optical models. Because 1D optical models are very fast to generate, using this argument enables rapid optimization of the optical parameters based on the measurements from a subset of the original spreadsheet points (only the 1D data are kept, such as lines, iso lines, and spaces). The spreadsheet points that are kept for Stage 1 optimization are reported in the log file.

epsilon value

This parameter is used to control the convergence of the zoom and frontzoom searches. The smaller the epsilon value the greater the number of zoom iterations required for convergence (Default=0.001).

search {full | mtfull | gradient | mtgradient | newton | mtnewton | front | mtfront}

Sets the optimization algorithm.

  • full — causes a full factorial search over all combinations of variables.

  • mtfull — is a multi-threaded version of the full search algorithm. It uses up to 32 CPUs.

  • gradient — uses a simple conjugate gradient descent method.

  • mtgradient — is a multi-threaded version of the gradient algorithm. It uses up to 32 CPUs and no more than 2 CPUs during the gradient search.

  • newton — uses a quasi-Newton optimization method, in which the second derivatives are evaluated in terms of the sequence of gradients from successive iterations using an update formula for the Hessian matrix.

  • mtnewton — A multi-threaded version of the newton mode; multi-threading is invoked during exploration runs and gradient calculations. It uses up to 32 CPUs.

  • front — uses a global search optimization method that is independent of a starting point. It performs search iterations based on the explore or runtime options, and uses up to 32 CPUs in -turbo mode.

  • mtfront — A multi-threaded version of the front mode.

  • frontzoom is a combination of a global (front) and local (zoom) optimization. The former is used to explore the search space and find a good initial guess (starting point) for the latter.

    • The number of front iterations is determined by the explore and runtime options.

    • The zoom search stops when the certain convergence criterion controlled by epsilon is met. The smaller the epsilon value the greater the number of zoom iterations required for convergence. The number of levels that must be specified for each independent optimization parameter (num_levels) has no effect on the frontzoom search. The MT scalability of frontzoom is determined by that of the front and zoom search.

  • mtfrontzoom is a multi-threaded version of frontzoom.

  • zoom is a local optimization routine. Unlike gradient, the zoom search is performed on a self adaptive grid and it does not involve numerical differentiation as is the case for the newton search. The search is limited by simple constraints determined by the begin and end values of the parameter intervals. It stops when a certain convergence criterion controlled by epsilon is met. If explore is explicitly set to 0, the input model is used as a starting point of the zoom search. Otherwise, exploration runs are performed to generate the starting point. The number of exploration runs is determined by explore or its default value if explore is not specified. The MT scalability of the zoom search can use no greater than Nvar CPUs where Nvar is the number of independent variables involved in calibration.

  • mtzoom is a multi-threaded version of zoom.

Two things to keep in mind:

  • The full search and gradient search can converge to different solutions.

  • The gradient search can be orders of magnitude faster, especially for large search spaces.

You configure your optimization process using begin and end search intervals, which must also be within the parameter boundaries as shown in “Global Recommendations for Optical Models,” Table 5.

The step value is actually used as the criterion to end optimization – iterations stop if the differences between successive values of the optimization variables are less than the step value. Accordingly, the smaller the step value, the more accurate the result and the longer the execution time.

EPEs obtained using a newton search are usually lower than that achieved using the gradient method.

You can constrain the number of CPUs used by the mtfull and mtnewton options by setting the following environment variable: 

setenv CWB_MT_THREAD_COUNT value

to a value below the maximum CPU limit (32). The multi-threading options do not consume additional licenses.

Default = full.

explore runcount
Tip

This option cannot be specified with the runtime option.

This option sets the maximum number of exploration stage runs performed for search methods. The exploration stage is skipped (and the newton and gradient simulations start from the nominal optical model settings) if explore is explicitly set to 0.

For the newton and gradient search, the exploration stage is designed based on a 3-level full factorial search: each optimization variable is set on a grid of three values corresponding to 25%, 50%, and 75% between begin and end of the search interval (step is ignored), and a full factorial search on all possible combinations is performed.

In this approach, the number of potential exploration runs (N) grows exponentially with the number of variables (M) involved, as N = 3M. Specifying the explore option allows you to avoid long run times. If runcount < N, a partial factorial search is used. In such a case, Calibre WORKbench randomly selects from the N potential full factorial runs a subset of explorations.

The default value for explore is 128. This means that for cases involving up to 4 variables (81 experiments) the full factorial exploration design is used, and a partial factorial design otherwise.

Tip

Using explore with gradient gives different results than using gradient without explore. To use the standard gradient search, set explore to 0.

runtime time
Tip

This option cannot be specified with the explore option, or any search method other than front.

An optional parameter that specifies the runtime limit in hours to use for explorations with the front search mode only. The optimization run stops when the time specified has expired, or when 100,000 iterations have been performed, whichever occurs first.

Dependent, Constrained, and Independent Optical Model Variables

The variable optim_set is used with multiple optical model parameters. It represents one of three methods of specifying the optimization search range associated with an optical model parameter: independent, constrained, and dependent.

  • independent

    Variations of an independent variable are determined by three (or two in the case of num_planes) numbers (begin, end, and step) that specify the search interval and the step size.

    For example: 

    sigma_out 0.6 0.8 0.05

  • constrained

    Variations of a constrained variable are determined by an inequality symbol, Tcl expression and three numbers (inequality, Tcl_exp, begin, end, and step) that specify the search interval:

    param inequality_symbol tcl_exp begin end step

    For example:

    stage1 search front explore 501 \
	sigma_out1 35 100 5 \
	sigma_out2 >= {$sigma_out1} 35 100 5

    means that during Stage1 optimization only optical models that satisfy the constraint sigma_out2 >= sigma_out1 will be considered as possible solutions.

    Constrained variable optimization is only available starting with the 2009.2 release and only works with the front search.

  • dependent

    Variations of a dependent variable are determined by a Tcl expression and two numbers (Tcl_exp, begin, and end); the two numbers specify the search interval, while the Tcl expression specifies the relationship between this variable and the other independent variables involved in optimization.

    For example: 

    sigma_in = {$sigma_out - 0.2} 0.4 0.6

    The Tcl expression must be enclosed in braces ({}) and can only include independent variables. No dependent variables or parameters that are not involved in optimization are allowed in this expression. If the value of a dependent variable obtained using the associated Tcl expression is outside the search interval, this variable is set to the nearest boundary value (either begin or end).
