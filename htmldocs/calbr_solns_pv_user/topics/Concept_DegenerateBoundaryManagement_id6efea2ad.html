Degenerate Boundary and Polygon Management

A degenerate boundary (GDS) or polygon (OASIS®) is a record in a layout database that represents a malformed shape.

A degenerate polygon is indicated by the following warnings in the run transcript or DRC Summary Report:

WARNING: Degenerate (vertex count = <n>) polygon at location (x,y) in cell <cell> on layer <n> datatype <n>.

WARNING: zero-area boundary at location (x,y) in cell <cell> on layer <n> datatype <n>.

By default, such polygons are discarded by Calibre on layout database read-in. These warnings should not be ignored, and the problems should be corrected.

The cause of these warnings is bad data was generated in the layout, often in this sequence:

  1. Polygon data was generated at a greater precision than is used for database output or than is used for the rule file Precision in a subsequent Calibre run.

  2. Polygons have vertex coordinates that are on grid for the greater precision, but are not on grid at the lesser precision.

  3. Polygon vertices are snapped to a new grid when the precision is reduced. As a result, they can become degenerate, and Calibre cannot process them in subsequent runs.

Avoid Degenerate Polygons on Output

You can avoid creating degenerate polygons in Calibre by checking these things:

  • Ensure the rule file Precision value is less than or equal to the DRC Results Database Precision value. They are the same by default, but you should check the rule file for a DRC Results Database Precision statement.

    PRECISION 1000
DRC RESULTS DATABASE PRECISION 10000 // OK.

    The output database precision should generally never be less than the input database precision, as this causes magnification and grid snapping issues.

    PRECISION 10000
DRC RESULTS DATABASE PRECISION 1000 // BAD.

  • Ensure that if the DRC Magnify Results statement is used that it meets one of these criteria:

    • The product of the Layout Magnify value (1 by default) and the DRC Magnify Results value is exactly 1 or gives all 9s in the decimal representation, such as 0.999999.

    • If the magnification of the output data is less than the input data, ensure that the DRC Results Database Precision value is increased by a corresponding factor. For example, if the magnification is decreased by a factor of 10 on output, then the precision should be increased by a factor of 10 to avoid grid snapping.

    See “Reducing Layout Magnification” for a related discussion.

  • Ensure that any DRC Check Map MAG and PREC keyword values are chosen using the criteria mentioned previously in this section. MAG corresponds to DRC Magnify Results and PREC corresponds to DRC Results Database Precision.

Additionally, you should use rule checks that output geometry that is subject to grid snapping in a follow-on Calibre run. Here is an example:

PRECISION 10000
LAYOUT MAGNIFY 10
LAYOUT PRECISION 1000
DRC RESULTS DATABASE output.gds GDS

// do this for all output layers
short_edge {@ layer edge could be snapped for PRECISION < 10000
  LENGTH layer < 0.001
}

offgrid_10 {@ layer vertex could be snapped for PRECISION = 1000
  OFFGRID layer 10
}

Detect Degenerate Polygons on Input

You control how degenerate polygons are handled on input using the Layout Input Exception Severity POLYGON_AREA_ZERO and POLYGON_DEGENERATE exceptions.

The default for POLYGON_AREA_ZERO is 1, which means to warn and ignore (drop) the polygon. You can set this to 2 if you want the condition to be a fatal error. The default for POLYGON_DEGENERATE is 2, which is a fatal error.

You can use Layout Input Exception RDB to store degenerate polygons when the exception settings are 0 or 1.
