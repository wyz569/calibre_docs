Layout Database Versus Calibre Database

At runtime, the layout database is converted to a Calibre database in memory. Calibre loads the layout database and keeps the number of grid points between two vertices the same. However, the distance between grid points may or may not be the same as the layout. If the distances are different, then scaling of the data occurs in the Calibre database. Additionally, magnification record values in the layout, as well as magnification factors specified in the rule file, are applied to the Calibre database.

The preceding considerations all apply as an aggregate magnification of the layout data within the Calibre database. Magnification changes the number of grid points between two points in the layout database.

Therefore, these things should always be borne in mind when magnification occurs:

  • Pay attention to what the output grid unit is in comparison to the input grid unit.

  • When decreasing layout magnification, consider increasing the rule file precision and the output database precision. If you do this, grid snapping of cell placements and layers can occur. To mitigate this, you can use Size … UNDEROVER with a 1 measurement dbu value on the affected layers to fill in possible gaps due to grid snapping. Be aware that Size can cause longer runtimes due to layer merging across the hierarchy. Another method is to adjust the measurement constraints for affected layers by 2 dbu measurement values.

  • Any output layout data should be validated after magnification before combining with other layout data.

  • All rule file elements that manage magnification should be validated before combining with other rule processes.
