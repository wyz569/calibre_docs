Hcell Recommendations for Digital Place and Route Flows

If you are using a digital place and route (P&R) flow, the following guidelines pertain to selection of hcells.

Hcells are explicitly declared in the rule file using the Hcell specification statement or in an hcell list referenced by the -hcell command line option. Hcells may be implicitly declared for LVS circuit comparison with the -automatch or -genhcells command line options, but it is frequently preferable to explicitly declare hcells.

  • All of the P&R library cells should be declared as hcells.

  • Cells containing devices and placed more than once, embedded memory cells, and large macro blocks all generally make good hcells. For large memory blocks, include one or two matching levels of hierarchy within the core array of the memory, if possible.

  • All hcells should have their ports texted. This makes for easier debugging of LVS results. (Hcell pins are not treated as logically equivalent.)

  • When multiple cell libraries are used, there is the possibility that cell names are duplicated. P&R streamout can cause cells to be renamed. If this occurs, the hcell list must be updated to include the renamed cells.

See “Hcells and Circuit Comparison” and “Hcells and Circuit Extraction” in the Calibre Verification User’s Manual to learn more about how hcells are used, and discrepancies that can result from an non-optimized hcell list.

The LVS Hcell Report specification statement can be useful in optimizing an hcell list.

Note:

If you do not have a good hcell selection methodology, then you should consider using the ‑genhcells command line option. If you get a CORRECT comparison using that option, then use the hcell list that this option generates in subsequent runs by specifying the list with the ‑hcell option, and omit ‑genhcells.
