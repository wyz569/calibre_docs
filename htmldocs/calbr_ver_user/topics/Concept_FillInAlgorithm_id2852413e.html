Fill-In Algorithm

The fill-in algorithm tries to find a unique match to a DEVice statement by duplicating existing pins to fill in for missing pins. More precisely, a provisional match is made between the initial pin set and any DEVice definition for which certain conditions are true.

Provisional pin matching occurs under these conditions:

  • The pins and layers of the initial set can be matched with a subset of the pins and layers in the DEVice statement.

  • The initial pin set contains at least one pin on each pin layer of the DEVice statement.

  • The initial pin set contains exactly one pin on each pin layer of the DEVice statement for which there are any missing pins.

Device recognition classifies an instance as a device and assigns the missing pins on each layer to the same net as the initial pin on that layer if exactly one DEVice statement generates a provisional match.

The instance is classified as ill-formed if more than one DEVice statement generates a provisional match.

This is an example of where the fill-in algorithm would be successful. Assume you have this five-pin device defined in the rule file:

DEV BPL base coll(C) base(B) emitt(E) sub(S) emitt(E2)

A four-pin device instance, with one pin each on layers coll, emitt, base, and sub, is classified as a BPL device because it satisfies the three conditions for fill-in (this assumes there are no other device statements that could provide an exact match). The pins E and E2 in the DEVice statement are considered shorted together for this instance because they are from the same layer.
