Layer of Origin

Because polygon reference data associates each edge with a specific polygon, you must pay close attention to the layer of origin, that is, where the data comes from. For example, it might appear that the following layer definitions produce the same data in the layer X:
Table 1. Layer of Origin Example
 
X = A COINCIDENT EDGE B  
X = B COINCIDENT EDGE A 
 
// Edges on layer A that are coincident 
// with edges on layer B. 
// Edges on layer B that are coincident 
// with edges on layer A.

However, the polygon reference data stored with the results is different depending on the layer of origin. While both operations appear to generate the same geometric data, one contains edges selected off layer A, and the other contains edges selected off layer B. In Figure 1, inside and outside are different depending on the layer of origin.

Figure 1. Reference Data Dependence on Layer of Origin

Dimension checks, which measure distances in a specific direction relative to the edge, provide a good example of just how important the layer of origin is. For example, the INTERNAL operation in the following sequence A, has an entirely different effect than the INTERNAL operation in the sequence B, which follows it:

X = metal coincident edge poly 
internal X metal < 3 
 
// Edges on layer metal that are 
// coincident with edges on layer poly. 
// Find  internally facing edges closer
// than 3 microns. 
X = poly coincident edge metal 
internal X metal < 3 
// Edges on layer poly that are
// coincident with edges on layer metal. 
// Find  internally facing edges closer
// than 3 microns.

In sequence A, layer X contains geometric data carrying polygon reference data from layer metal, while in sequence B, this data carries polygon reference data from layer poly. Figure 2 shows how this results in the internal check, which looks inside from edge X, looking in opposite directions for the lower coincident edge.
